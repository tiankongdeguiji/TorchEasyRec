LLM- and SID-based Generative Recommendation in TorchEasyRec

Design RFC: SID tokenization, SFT and RL pipelines for OneRec-family generative recommenders, and how tzrec leverages PyTorch, Megatron-LM, Transformers, Accelerate, and verl for distributed training.

Date2026-06-10
Branchrfc_llm4rec (tzrec master @ 7886e4c)

Algorithms coveredOneRec V1/V2, OneRec-Think, OneReason, OpenOneRec, ALGR/FORGE, LC-Rec, MiniOneRec, GRID, PLUM, PIT, Latte, RPG, STATIC

1. Executive summary

This RFC proposes extending TorchEasyRec (tzrec) into a production framework for LLM- and SID-based generative recommendation: predicting the next item directly, as a short sequence of Semantic-ID (SID) tokens emitted autoregressively by a transformer language model, instead of scoring candidates through a retrieval/ranking cascade. The design covers the full pipeline — SID tokenization, supervised fine-tuning (SFT) and continued pre-training, reinforcement learning (RL), and constrained-decoding inference/serving — and specifies precisely how tzrec leverages PyTorch 2.11, Megatron-core 0.18, HuggingFace Transformers, Accelerate, and verl for efficient multi-GPU and distributed training.

The proposal is grounded in a systematic study of the OneRec family (V1, V2, OneRec-Think, OneReason [onerec-v1][onerec-v2][onerec-think][onereason]), the open reproductions OpenOneRec and MiniOneRec [openonerec][minionerec], industrial SID-construction work (ALGR/FORGE at Alibaba, PLUM at YouTube [forge][plum]), the academic line (LC-Rec, GRID, Latte, PIT, RPG [lcrec][grid][latte][pit][rpg]), and source-level reviews of the five infrastructure stacks plus NVIDIA's recsys-examples two-world integration [repo-recsysexamples]. It supersedes the internal v1 spec under experiments/llm4rec/: the architecture and canonical names are retained, while a small number of decisions are revised with new evidence (each marked Revision in place; see §4).

1.1 The shape of the design

Two-world training. Recommendation models have two kinds of parameters with incompatible scaling behavior. Sparse embedding tables (item IDs, user features, SID codebooks — up to 250M rows) stay in the TorchRec world tzrec already owns: model-parallel sharding via DistributedModelParallel, fused in-backward optimizers, dynamicemb hash tables. The dense LLM trunk (0.1B–8B+ parameters) joins a new dense world: FSDP2 fully_shard by default, Megatron-core tensor/expert parallelism at scale. Both worlds share one torch.distributed WORLD, one DeviceMesh, two optimizer trees, and one unified DCP checkpoint. The bridge between them is activation-level: TorchRec produces context embeddings that enter the LLM as inputs_embeds. This is the same integration shape NVIDIA's recsys-examples proves out for HSTU at scale, adapted to tzrec's existing seams (§11).

Four subsystems, all config-driven. (1) A SID tokenizer subsystem (tzrec/sid/) builds on the ResidualQuantizer/BaseSidModel base classes already on master: RQ-KMeans and RQ-VAE quantizers, pluggable collision resolution, training-free quality metrics, and a many-to-many item↔SID catalog (§5). (2) An SFT/pretraining pipeline runs the staged curriculum every successful system converges on — itemic-text alignment → co-pretraining → multi-task SFT → rejection-sampling fine-tuning — inside tzrec's existing train loop via a new pipeline class (§8). (3) An RL subsystem targets verl 0.8 with zero framework patches for Qwen3-class backbones, plus a tzrec-native rollout loop for architectures vLLM cannot serve, and GBPO-style learning from logged real feedback with no rollouts at all (§9). (4) A decoding/serving stack pairs the tzrec GR serving runtime (a BackboneAdapter fork of recsys-examples sid-gr-inference) with the AOTI scorer path — an AOTInductor-exported single-step scorer under a host-side beam driver — both decoding through a STATIC-style CSR tensorized trie (§10).

1.2 What is new since the v1 spec

  • FSDP2 becomes the default dense backend (D8); Megatron-core is the scale-out path for MoE/EP and ~8B+. OpenOneRec trains all stages on raw fully_shard; PyTorch 2.11's DCP, DeviceMesh, and DDP-composability APIs make the TorchRec bridge native (§11).
  • Constrained decoding is resolved, not open: YouTube's STATIC flattens the SID trie into CSR tensors — one batched gather per decode step, ~1.2–3.1 GB at 3×8192 over 250M items [static] (D12). Unconstrained beam with post-grounding is a supported config, with GRID/PLUM evidence that it is often sufficient.
  • Single-step scorer export is de-risked empirically: a Qwen3 decode step with a static KV cache was validated on this machine (torch 2.11, transformers 4.53) — the torch.export module matches eager bit-exactly, host-side beam reordering included; the AOTInductor compile is fp32-exact, with bf16 logits within ≈5e-3–8e-3 fusion-reorder noise — with two sharp edges (input contiguity, GLIBCXX) documented in §10 (D13).
  • The vocabulary-extension math is corrected (total-vocab padding to 176,384, not block padding), and a latent OpenOneRec bug is fixed by contract: HF mean_resizing leaves the first 267 itemic rows uninitialized (§7).
  • Collision resolution defaults to RandomCollisionResolver: FORGE's Random-5 strictly dominates every KNN setting on both AL-GR and Amazon [forge], so the default flips from KNN-σ; KNN-σ remains an option where last-level semantics matter (§5).
  • Stage-1 embedding freezing is backend-specific (D7): the DTensor restore-after-step trick is proven on FSDP2 but broken under Megatron's DistributedOptimizer fp32 masters; the correct Megatron mechanism is gradient zeroing after finalize_model_grads plus a weight-decay override (§8).
  • Item↔SID is many-to-many from day one (D4): FORGE collision buckets, PIT's dynamic top-B index, and Latte's latent trees all break one-SID-per-item; the catalog, decoder, and metrics are specified accordingly (§5, §10).
  • New algorithm coverage: OneRec-Think/OneReason reasoning recipes (two-stage rollouts, stage-wise clipping, RFT, on-policy distillation), PLUM's co-pretraining and evaluation lessons, PIT/Latte/RPG extensions — folded into D10, D15, D16.
  • Serving implementation is analyzed to a decision (2026-06-10): a GR service decomposes into six layers with an engine-invariant core that stays tzrec code under every topology; wide beams turn on sharing KV bandwidth, not storage — beams-in-query measured 135× over per-sequence attention in a kernel-level A10 microbenchmark; recsys-examples' sid-gr-inference (PR #414) is adopted as a fork behind the BackboneAdapter contract — the tzrec GR serving runtime, serving all softmax-attention GR backbones (Qwen3/Qwen2.5 token-only now; OneRec-V1/V2 via planned enc_dec/lazy_decoder adapters), with HSTU the sole exception on tzrec's own stack [repo-recsysexamples]; and W=512-synchronous serving on Qwen3-class backbones is demoted to a demand-gated contingency (§10).

1.4 Non-goals (v1)

  • No raw-pixel/multimodal encoders inside tzrec: the multimodal item embedding is an upstream artifact consumed as a parquet/ODPS column (§5.1).
  • No general-purpose LLM pretraining from scratch; every recipe starts from an open checkpoint (Qwen3 family).
  • No replacement of TorchRec for sparse sharding; no DeepSpeed; no pipeline parallelism in v1 (PP=1).
  • Reward models (P-Score multi-tower) are trained with tzrec's existing multi-task stack and consumed by RL as a scoring service; their training is not redesigned here.
  • No synchronous LLM reasoning on the request path: CoT models serve via nearline/Think-Ahead caching patterns (§10.7).

2. The generative-recommendation landscape

This section surveys the systems this RFC is built on: the Kuaishou OneRec family (the only line with multi-generation production evidence at 400M-DAU scale), its open reproductions, the industrial SID-construction studies from Alibaba and Google, and the academic line that supplies the alignment recipes, expressiveness analysis, and decoding accelerations. The goal is not completeness — the companion survey [survey-itemid] and its living bibliography [awesome-list] index well over a hundred papers — but to extract, with exact numbers, the design constraints that the rest of this document turns into decisions D1–D16. Readers who want only the conclusions can skip to the comparison table and rollup in §2.6.

2.1 From cascaded scoring to end-to-end generation

Industrial recommendation today is a cascade — retrieval, pre-ranking, ranking, re-ranking — each stage a separate model with separate objectives. OneRec V1 quantifies what this costs at Kuaishou: more than 50% of serving resources go to communication and storage rather than computation, and the legacy ranking model achieves only 4.6% training MFU / 11.2% inference MFU on flagship GPUs, versus roughly 40% for LLM workloads on comparable hardware, under production constraints of peak QPS > 400k and latency < 500 ms [onerec-v1]. The cascade also fragments objectives (hundreds of targets patched in at different stages) and structurally blocks the scaling-law and RLHF-style techniques that drive modern LLMs. The generative alternative replaces the cascade with one autoregressive model that generates the next item’s identifier directly from user context, turning recommendation into a sequence-modeling problem that inherits the entire LLM toolchain: pretraining corpora, SFT, RL, beam search, and inference engines.

The enabling representation is the semantic ID (SID): a short tuple of discrete codes per item, produced by quantizing a content (or content-plus-collaborative) embedding. TIGER established the canonical recipe — RQ-VAE over content embeddings yielding hierarchical code tuples, decoded autoregressively by a T5-style encoder-decoder [tiger] — and essentially every system below is TIGER plus deltas. SIDs give a generative model three properties atomic item IDs cannot: a vocabulary that is orders of magnitude smaller than the catalog (24,576 tokens can address 8192³ ≈ 5.5×10¹¹ combinations), prefix structure that lets coarse semantics transfer to cold-start items, and a token interface that an off-the-shelf LLM can absorb through vocabulary expansion.

The survey [survey-itemid] organizes this space into exactly three pillars — creation of item IDs (tokenizer training, collision handling, augmentation), alignment of IDs with the backbone (instruction-tuning task suites, joint/co-trained tokenizers), and generation of next-item IDs (validity-aware decoding: direct generation, closed-set generation, post-grounding, constrained decoding; plus acceleration: speculative, pruned, and parallel decoding) [awesome-list]. This taxonomy maps one-to-one onto the subsystems a framework must provide, and onto this RFC’s section structure: creation is §5, alignment is §7/§8, generation is §10. Within the taxonomy, the systems studied here sit as follows: FORGE, PLUM’s SID-v2, and PIT are creation-pillar work; LC-Rec, OpenOneRec’s stage-1 alignment, and OneRec-Think’s itemic alignment are alignment-pillar work; STATIC, RPG, Latte, AtSpeed, and LLaDA-Rec are generation-pillar work; and the OneRec reports are full-stack systems that touch all three.

2.2 The OneRec family: four generations of production evidence

OneRec V1 — the existence proof, and where the win actually came from

OneRec V1 [onerec-v1] is an encoder-decoder trained from scratch: a four-pathway feature encoder (user-static; last-20 short-term; 256 positive-feedback interactions; a lifelong pathway compressing up to 100,000 raw behaviors into 128 QFormer query tokens — 405 encoder positions total) feeding a decoder with causal self-attention, cross-attention, and SwiGLU MoE FFNs balanced with DeepSeek-V3’s auxiliary-loss-free routing [dsv3]. Items are tokenized by RQ-KMeans, 3 levels × 8192 codes, over aligned multimodal embeddings: miniCPM-V-8B encodes caption/tags/ASR/OCR plus cover and frames into 1280 vectors of dimension 512, a 4-query QFormer compresses them, and the representation is trained with item-pair InfoNCE (Swing-similarity positives) plus a caption loss. Against RQ-VAE on the same data, RQ-KMeans achieves 25% lower reconstruction loss and 100% codebook utilization at every level. Training is streaming next-token prediction over point-wise targets (one item = 3 tokens, ~54B tokens/day), then post-training runs RSFT (drop the bottom 50% of sessions by play duration) jointly with RL: ECPO, a GRPO variant that early-clips the old-policy denominator (π′_old = max(sg(π_θ)/(1+ε+δ), π_old), δ=0.1, no KL term), with rewards from a P-Score reward model — a SIM-backbone multi-tower multi-task ranking model with a fused tower trained on all objectives, i.e. stock tzrec territory (§9, D14).

Three V1 findings are load-bearing for this design. First, the economics: 23.7% training / 28.8% inference MFU (5.2×/2.6× over the cascade), OPEX at 10.6% of the legacy pipeline, serving ~25% of total feed QPS. Second, RM selection carries the online win: the pure generative arm only reached parity with the full cascade (App Stay Time +0.01%/+0.06%, interaction metrics negative), while reranking the generated list with the P-Score model delivered +0.54%/+1.24% App Stay Time with all interaction metrics positive. A generative stack should plan for the reward-model reranking stage from day one. Third, the RL pathologies: enabling ECPO drove SID legality below 50% via the squeezing effect (R-LEGALITY), and the fix — a format-reward pass on K=5 of 128 rollouts where legal rollouts get A=1 and illegal rollouts are discarded, never penalized — only works with random selection of the K; picking the top-5 by probability teaches legality only inside the top subset and overall legality decays. Beam search beats top-k/top-p sampling for SID rollouts, rollout group size saturates around 4× the serving beam width, and production pairs beam-512 RL rollouts with a served beam of 128 (§10). The earlier OneRec paper [onerec-v0] contributed the session-wise list-generation formulation and DPO-style Iterative Preference Alignment, both since abandoned for point-wise generation plus ECPO — a useful reminder that the family’s recipes churn.

OneRec V2 — compute reallocation and RL without rollouts

V2 [onerec-v2] starts from a compute audit: in V1 at context length 512, 97.66% of FLOPs go to encoding the user context and only 2.34% to the target decoding that actually carries loss. The Lazy Decoder-Only architecture deletes the encoder entirely: the context is embedded once by a Context Processor, RMSNorm’ed, and consumed directly as cross-attention K/V with no learned W_k/W_v projections at all; all transformer compute runs on the 3-token target. This cuts total compute ~94% at equal loss (1B model: 18.9 vs 634.8 GFLOPs/sample against a naive decoder-only) and unlocks dense scaling 0.1B→8B with a clean Chinchilla-form fit (E=3.13, A=3660, α=0.489). Training data uses new-impression-only NTP: samples stay impression-organized, but loss applies exclusively to the newest impressed item — exactly tzrec’s per-impression sample shape (§6).

V2’s post-training is the opposite pole from V1’s: because OneRec now serves 25% of traffic, logged real feedback on self-generated recommendations is abundant, so V2 drops reward-model rollouts entirely. Duration-aware reward shaping log-buckets video duration, ranks the target’s playing time as a per-user per-bucket percentile, and assigns advantage +1 to the batch top-quartile, −1 to explicit dislikes, and filters everything else — deliberately without advantage normalization. The optimizer is GBPO (Gradient-Bounded Policy Optimization): no ratio clipping, instead an advantage-sign-dependent denominator (max(π_old, sg(π_θ)) for positives, max(π_old, 1−sg(π_θ)) for negatives) that bounds RL gradients by BCE-loss gradients, giving full sample utilization with no rollout engine and no critic. Online, V2 beat V1 by +0.467%/+0.741% App Stay Time with every interaction metric positive. The deployed model is 1B parameters, context ≈3000, beam 512 at 36 ms on L20 GPUs with 62% inference MFU — the serving-feasibility anchor this RFC cites instead of any speculative-decoding scheme, achievable because the lazy decoder shares context KV by construction; it is not a Qwen3-class latency target (§10). One warning travels with V2: in the caching-disabled full-traffic arm, cold-start video views fell 44.7% and cluster density rose 11.7% — an ecosystem regression the authors flag as open, and a reason this RFC treats exploration/diversity rewards as first-class (D14).

OneRec-Think — making an LLM bilingual in items

OneRec-Think [onerec-think] is the family’s first move onto a pretrained text LLM (Qwen 1.7B public / Qwen-8B industrial) with explicit chain-of-thought. Its three stages name the recipe this RFC adopts for the Qwen3 path: (i) itemic alignment — vocabulary expansion (industrial: 3×8192 = 24,576 itemic tokens plus boundary tokens), a token warm-up phase training only the new embeddings at LR 5e-4 with the backbone frozen, then full multi-task integration at 3e-4 over a four-task mixture (interleaved persona grounding 24.3%, sequential preference 65.7%, itemic dense captioning 4.9%, general LM 5.0%); (ii) reasoning activation — bootstrap rationales by pruning the history to the top-k=10 target-relevant items, letting the aligned model itself write the rationale, then SFT on (full noisy history → rationale → target SID); (iii) reasoning enhancement — GRPO (group 16, verl) with a Rollout-Beam reward: for each sampled CoT, run a beam-32 search over the SID head and credit the best per-digit match across the beam — reward computation is generation-in-the-loop, a requirement that shapes the rollout architecture in §9. Industrially it trains daily on 80 GPUs (~20B tokens/day) and serves through Think-Ahead: offline reasoning caches each user’s top 2-of-3-level SID prefixes; a cheap online OneRec model finalizes the last level under a prefix constraint — the first of the nearline serving patterns in §10. A one-week 1.29%-traffic A/B yielded +0.159% App Stay Time.

OneReason — when thinking finally pays

OneReason [onereason] confronts the embarrassing result that in both OneRec-Think and OpenOneRec, thinking mode does not beat non-thinking mode — SFT-only CoT actively hurts (teacher-forced ΔLL of the ground truth given the model’s own CoT is negative in all four domains). Its answer is scale plus structure: a 578B-token, four-granularity pretraining corpus (token-level compositional prefix semantics 2.5%, item captioning/grounding/QA 31.8%, relational interleaved chains 3.2%, user-granularity multi-domain sequences 34.1%, plus ~28% general text/multimodal) trained in three stages (110B tokens updating only the extended-vocabulary embedding and LM-head rows, 449B full-parameter, 19B long-context with 48K sample packing); an R0–R3 capability hierarchy (perception → derivation → evolution → recommendation) organizing both SFT data and the benchmark; and specialize-then-unify RL — four domain-specific GRPO teachers, unified either by RFT (rejection-sampled trajectories, the only method that guarantees thinking > non-thinking at all K) or by MOPD, multi-teacher on-policy distillation with single-point reverse-KL advantages. The GRPO recipe contributes three reusable mechanisms: two-stage rollout (N CoT traces × K SID completions each, cutting step time from ~350s to ~150–200s while densifying reward), stage-wise clipping (looser clip range on CoT tokens than on the short SID segment, without which the SID distribution entropy collapses — R-ENTROPY), and response-level down-weighting of zero-reward rollouts. Deployment is fast–slow thinking: a nearline path runs offline inference with hierarchical decoding (beam at the top SID level, greedy below) into a Redis candidate pool, with the realtime OneRec as fallback, plus representation-level distillation of a “Thinking Token” into the online model. Combined online effect in a local-services ad scenario: +10.33% impressions / +8.23% revenue, one of the largest uplifts reported in that scenario, at ~600 GPUs/day serving 400M users with ROI > 5. OneReason is the first reported system where thinking mode consistently beats non-thinking on recommendation benchmarks — and its cost profile is exactly why this RFC keeps CoT serving nearline-only (§10).

2.3 Open reproductions: the recipes we can actually inspect

OpenOneRec [openonerec] is Kuaishou’s open release of the LLM-based line, and the single most load-bearing artifact for this RFC because its full stack is inspectable [repo-openonerec]. It tokenizes with RQ-KMeans 3×8192 over Qwen3-Embedding-8B item embeddings (plain faiss k-means per residual level, no balancing), expands the Qwen3 vocabulary by 24,578 tokens (<s_a_*>/<s_b_*>/<s_c_*> plus <|sid_begin|>/<|sid_end|>), and trains in stages: stage-1 itemic-text alignment updating only the new embedding rows (implemented as a DTensor row save/restore around optimizer.step() under FSDP2 — the mechanism behind D7, §8), stage-2 full co-pretraining at 62.34% general / 37.66% recommendation tokens (33B tokens open, 130B industrial), multi-task SFT over 8 RecIF task families, then on-policy distillation against the original Qwen3 as teacher (reverse-KL token advantages clamped to [−30, +5], itemic tokens masked from the teacher) to repair general-capability loss (R-FORGET: AIME falls 0.79→0.51 after SFT, distillation recovers 0.71), and finally Rec-RL: GRPO with a two-stage rollout — sample a CoT to </think>, then a vLLM beam-32 expansion of exactly 3 SID tokens, with the 32 beams forming the GRPO group. Both RL phases run on verl forks (the Rec-RL stage on FSDP2 workers, the distillation stage on FSDP) — the proof point for D8/D9 (§11, §9). It also contributes RecIF-Bench (8 tasks, 4 capability layers, ~120M interactions/200K users), validated recommendation scaling laws (N_opt ∝ C^0.44, D_opt ∝ C^0.56 — data-hungrier than Chinchilla), and the cross-domain transfer result that text-augmented itemic tokens (pretrained 3-level SID + appended keywords) cut Amazon collision rates from >30% to 0.47% and beat per-domain baselines by +26.8% average Recall@10.

MiniOneRec [minionerec] is a small-scale, fully-HF-stack reproduction (Qwen2.5 ≤ 7B, TRL + DeepSpeed ZeRO-2, no vLLM) whose lasting contribution is a trick: in its forked GRPO trainer, the rollout group is one trie-constrained beam-search call (num_beams = G = 16, sampled), so every group member is a valid, unique catalog item for free, and beam rank doubles as the ranking signal for an NDCG-shaped, rank-aware reward [repo-minionerec]. Its trie is an HF LogitsProcessor keyed on prompt-suffix token paths — fragile (the repo’s own issue log shows constraint failures with Instruct checkpoints), which is part of why this RFC standardizes on a tensorized CSR trie instead (D12, §10). In this design MiniOneRec survives as a config preset over the LCRec model class plus GRPO, never as a class of its own.

semantic-ids-llm [repo-semanticids] is a single-GPU, single-category (Amazon Video Games) prototype that nevertheless documents practitioner details the papers omit: an RQ-VAE with k-means codebook init, the rotation trick in place of straight-through estimation, and aggressive dead-code resets every 2 steps; a post-hoc uniqueness digit appended per collision group (89.08% unique 3-tuples → 100% with a 4th positional code); a two-stage SFT (embedding-only warm-up at LR 1e-3, then full fine-tune at 2e-5); and graceful-degradation grounding that resolves a hallucinated SID by prefix fallback 3→2→1 levels. Its codebook-health metrics (per-level usage, unique-ID proportion, residual norms) read as a checklist for the tokenizer dashboards in §5 and §13.

2.4 Industrial SID construction: what actually moves the metric

FORGE / AL-GR [forge] is not a model but a benchmark and ablation matrix over the SID construction pipeline, validated on a 250M-item Taobao corpus and deployed at full scale (+0.35% transaction count in a 7-day A/B, served asynchronously via RTP-LLM writing 1200 candidates to a cache table within 200 ms). Its central inversion of conventional wisdom: collision post-processing is first-order; the quantizer choice is second-order. At identical post-processing, RQ-KMeans ≈ OPQ ≈ RQ-VAE; but switching the last-level assignment from capacity-25 KNN buckets to Random-5 — round-robin assignment of the third code in insertion order, discarding last-level semantics entirely — gains +29.4% HR@20 (and +101% on Amazon), strictly beating KNN-5’s +15.8%. Leaf fairness beats leaf semantics, which is the evidence behind D2’s pluggable collision-resolver stage (§5), including the iterative ODPS-SQL variant FORGE itself uses at 250M items [repo-alsid]. FORGE’s second contribution is training-free tokenizer quality metrics: the Gini coefficient of the SID-assignment histogram and an Embedding HitRate (i2i retrieval over the pre-quantization embeddings) both correlate with downstream generative-retrieval HitRate, while reconstruction loss and feature-fidelity metrics do not correlate at all — the anti-metric finding behind D3’s rule that RQ-VAE checkpoints must never be model-selected by reconstruction loss. Its geometry sweep confirms 3×8192 as the sweet spot for 10⁸-scale corpora (3×512 is catastrophically under-capacity at −19% HR@20). The released AL-GR dataset (131M users, 251M items, 14B interactions, 512-d multimodal embeddings, base and final SIDs) is the only public industrial-scale corpus for this pipeline and anchors the evaluation plan in §13.

PLUM [plum] is Google/YouTube’s production existence proof for exactly the pipeline shape this RFC proposes: SID-tokenize → vocab-expand a pretrained LLM (Gemini-1.5 MoE, 900M activated / ~4.2B total in production) → continued pre-training → task SFT → beam-search retrieval, launched on YouTube surfaces at billions-of-users scale (Panel CTR +0.76% LFV / +4.96% Shorts; effective vocabulary 2.6×/13.2× the embedding-model baseline). Its SID-v2 tokenizer is an RQ-VAE with three refinements: multi-resolution codebooks (cardinality halving per level from 2048), progressive masking so every SID prefix is trained as a usable coarse representation, and an in-batch co-occurrence InfoNCE term with positives from items co-occurring in user sequences — the largest single contributor (uniqueness 91.8%→96.7%, downstream Recall@10 12.6%→14.4%), and operationally superior to fusing drift-prone CF embeddings into the quantizer input. Both refinement axes feed D1’s tokenizer option set (§5). PLUM’s 2×2 ablation is the cleanest published stage-prioritization evidence: random-init + CPT reaches Recall@10 0.27 versus 0.23 for Gemini-init without CPT — continued pre-training matters more than the choice of base LLM, which is why D6 makes CO_PRETRAIN a first-class SFT stage (§8). Alignment to user experience enters purely through reward-weighted SFT example sampling (sample ∝ handcrafted engagement reward, then unit weights) — no RL anywhere. And its decoding is unconstrained beam search with <5% hallucination tolerated, evidence (jointly with GRID below) that the unconstrained-plus-post-grounding configuration in D12 is launch-viable. Its iso-FLOPS scaling study adds three operational lessons: scale data and model size together, larger models generalize better to next-day distributions, and eval Recall keeps improving after eval loss saturates — never early-stop on loss.

2.5 The academic line and the decoding-acceleration frontier

LC-Rec [lcrec] defined the vocabulary-extension playbook for decoder-only LLMs: RQ-VAE 4×256 SIDs (with a Sinkhorn-based uniform last-level assignment for collision groups) added to LLaMA-7B, then a suite of six alignment task families — sequential prediction, bidirectional SID↔text mutual prediction, asymmetric prediction, intention-based item search, and preference inference. Its decisive ablation: applying the alignment suite lifts every indexing scheme by a large margin — even random multi-level indices with alignment beat all baselines — so alignment-task design matters more than tokenizer sophistication [repo-lcrec]. A table cut from the camera-ready measured MMLU dropping 35.2→25.8 after recommendation fine-tuning — the earliest documented instance of R-FORGET, later answered by OpenOneRec’s distillation stage. GRID [grid], Snap’s open framework and benchmark [repo-grid], supplies the controlled small-scale ablations: RK-Means ≥ residual VQ > RQ-VAE despite RQ-VAE getting 5× the training steps (validating D1’s RQ-KMeans default); (3,256) is the best codebook geometry at academic scale and more levels actively hurt; sliding-window data augmentation is the single biggest lever (Beauty R@10 0.0597 vs 0.0447 without — a dataset-builder requirement for §6); TIGER-style hashed user tokens are useless; unconstrained beam matches or beats constrained; and encoder-decoder beats decoder-only decisively only at this small scale — a finding that must not be extrapolated against the LLM-backbone paths, which compensate with scale and language pretraining.

Two papers attack the one-item-one-SID assumption. Latte [latte] proves an expressiveness limit of AR SID generation: items close in the decoding tree have generation probabilities correlated across users (Pearson ρ ≈ 1.0 for sibling leaves), bounding rank reversals and forcing similarity transitivity through the tree’s ultrametric. The fix is minimal: prepend one of M′=8 latent tokens, uniformly sampled per training target, and aggregate item scores over latent paths at beam time — +3.45% average NDCG@10 with identical SIDs, tokenizer-agnostic [repo-latte]. PIT [pit] makes the tokenizer itself dynamic: an autoregressive item-to-token model co-evolves with the recommender in Kuaishou’s streaming-training production stack, with the recommender acting as critic to select pseudo-label SIDs from per-item beam candidates, and a one-to-many beam index (top-B momentum-weighted SIDs per item) as the serving mapping — ablation-wise the single most important component (removing it costs −14.9% R@10). Online it beat the OneRec-V2 production baseline by +0.217%/+0.402% App Stay Time over 24 days. Together with FORGE’s capacity-σ collision buckets, Latte and PIT are why D4 makes the item↔SID catalog many-to-many from day one (§5): the index, decoder, and metrics layers must never hard-code a bijection. PIT’s full co-evolution loop, requiring in-training-loop beam search and streaming index sync, is deferred to a phase-2 streaming extension (D16).

On the generation pillar, RPG [rpg] abandons hierarchical codes entirely: unordered OPQ product-quantization codes (up to m=64 tokens/item), trained with multi-token prediction (conditionally independent per-digit cross-entropies) and decoded with one model forward followed by a similarity-graph propagation — +12.6% NDCG@10 over the strongest baselines and ~15× decode speedup, constant in catalog size [rpg-repo]. Its sharpest cautionary number: running ordinary AR beam search over unordered codes scores exactly 0.0000 — decode strategy and code structure are inseparable, which is the direct motivation for D5’s rule that the decoder, not the model, owns the number of scorer invocations (§7). LLaDA-Rec [llada-rec] independently confirms the parallel-code direction with discrete diffusion over multi-head VQ codes (beating RPG by 10–20% Recall@10), but with no code, no latency accounting, and unspecified validity enforcement — watch-list only. STATIC [static] resolves the constrained-decoding performance question for the AR path: flattening the legality trie into a static CSR sparse matrix turns per-step constraint application into vectorized gathers at 0.033 ms/step — 948× faster than a CPU trie (TPU v6e figures at V=2048/L=8; see §10 for the GPU caveat) — in the first production strictly-constrained generative retrieval system at YouTube [static-repo]; this is the design D12 adopts and de-risks (§10). AtSpeed [atspeed] shows speculative decoding can be made beam-aware (N-to-K verification, 1.8–2.5× at K≤20) and composes with trie constraints, but our arithmetic puts its ceiling at ~1.5× for 3-level SIDs — not worth v1 complexity. Finally, DeepSeek-V3’s MTP module [dsv3] — a single-depth sequential extra head, λ-weighted auxiliary loss, discarded at inference with consistent quality gains — is the only multi-token-prediction piece cheap enough to ship as an optional training flag (D15, §7).

2.6 Synthesis: the design space at a glance

The table below compresses the thirteen systems into the seven axes a framework must parameterize. Two readings matter more than any cell. Read column-wise: every axis has at least three materially different production-grade answers — tokenizers span frozen RQ-KMeans to co-evolving AR tokenizers; backbones span from-scratch encoder-decoders to 8B text LLMs; RL spans “none” through rollout-free GBPO to two-stage CoT×beam GRPO — so a framework that hard-codes any single recipe will be obsolete within one paper cycle. Read row-wise: the systems with online wins (V1/V2/Think/OneReason, FORGE, PLUM, PIT) disagree about almost everything except the artifact boundaries: an item-embedding table in, an item↔SID catalog out of tokenization, and a decode loop that is a thin strategy over a per-step scorer.

What a framework must therefore support

Distilling the survey into requirements, each anchored to the decision that answers it and the section that owns it:

  • A tokenizer menu, not a tokenizer — RQ-KMeans default with an RQ-VAE option carrying the practitioner knobs (k-means init, dead-code reset, rotation trick, Sinkhorn, co-occurrence InfoNCE), both as trainable tzrec models: OneRec/OpenOneRec and GRID argue for RQ-KMeans; FORGE and PLUM productionize RQ-VAE; GRID and FORGE agree the choice is second-order (D1, §5).
  • Collision handling as a pluggable post-processing stage, decoupled from the quantizer, with KNN-σ, random round-robin, and uniqueness-digit policies plus an ODPS-SQL path for 10⁸-item catalogs — FORGE’s first-order finding (D2, §5).
  • Training-free quality gates — Gini, codebook utilization, unique-SID ratio, Embedding HitRate — and a hard rule against reconstruction-loss model selection (D3, §5).
  • A many-to-many item↔SID catalog with multi-path scoring and dedup, since FORGE buckets, PIT’s top-B index, and Latte’s latent trees all break the bijection (D4, §5); Latte ships as a cheap option and PIT co-evolution as a phase-2 streaming extension (D16, §5).
  • Multiple backbones behind level-agnostic abstractions — encoder-decoder (V1), lazy decoder (V2), pretrained Qwen3 (Think/OneReason/OpenOneRec/PLUM), and an export-clean in-house HSTU path — with the decoder owning the number of scorer invocations, because RPG’s 0.0000-on-AR-beam result proves decode strategy cannot be an afterthought (D5, §7; D15, §7).
  • A staged SFT curriculum with co-pretraining as a first-class stage (PLUM’s 2×2; OneReason’s 578B tokens), response-masked CE, and 32K sample packing (D6, §8), plus backend-correct itemic-row freezing for the warm-up stage (OpenOneRec, OneRec-Think) (D7, §8).
  • FSDP2-default dense parallelism with a Megatron scale-out path, following OpenOneRec’s proven stack and the OneRec family’s MoE trajectory (D8, §11).
  • RL as a plugin surface on verl: GRPO default with ECPO and GBPO as policy-loss/advantage plugins, on-policy distillation for capability retention, and a per-backbone rollout decision matrix — including the no-rollout GBPO regime and MiniOneRec-style in-process beam groups (D9–D11, §9); rewards composable from RM scores, duration-aware shaping, legality credit (discard, never penalize), and diversity (D14, §9).
  • Constrained decoding as data, not code — a STATIC-style CSR tensorized trie — with unconstrained-plus-post-grounding as a supported configuration per GRID and PLUM (D12, §10); serving as a host-side beam driver over an exported single-step scorer, with Think-Ahead/fast–slow nearline patterns for CoT models (D13, §10).

Note — what the survey does not settle

Three questions the literature leaves open shape this RFC’s hedges. First, thinking mode: only OneReason makes CoT pay, and only after a 578B-token pretrain plus domain-specialized RL and RFT — this RFC therefore treats reasoning as an optional path with non-thinking serving as the default. Second, ecosystem effects: OneRec-V2’s −44.7% cold-start views under full traffic is the strongest published warning that generative rankers need explicit diversity/exploration machinery. Third, parallel and diffusion decoding (RPG, LLaDA-Rec) have no published validation on LLM backbones with interleaved text — they are future work whose interface requirements D5 deliberately preserves.

3. TorchEasyRec today: baseline and gap analysis

This section inventories the tzrec baseline at master commit 7886e4c from the perspective of the generative-recommendation requirements distilled in §2: what the orchestration, data, and model layers already provide, which seams the design in §4 deliberately exploits, and which baseline assumptions break and must be engineered around. Every claim below is grounded in a source location at this commit. The headline finding is that tzrec already ships most of the sparse-world machinery (DMP sharding, dynamicemb, jagged data plumbing, a host-side beam-search precedent, and the PR #538 SID base classes), while the dense LLM world — HF/Megatron backbones, FSDP2, packed-sequence training, RL, and autoregressive serving — is entirely absent and is exactly what this RFC adds.

3.1 Orchestration: one loop, one process group, one DMP wrap

All CLIs (tzrec/train_eval.py, eval.py, export.py, predict.py) are thin argparse wrappers over tzrec/main.py (1488 lines), which owns the entire lifecycle. The training entry train_and_evaluate (main.py:543-788) executes a fixed order of operations:

load EasyRecConfig (text proto)            main.py:568-580
  -> init_process_group: ONE global WORLD PG, device=cuda:{LOCAL_RANK}   dist_util.py:57-75
  -> create_features (BaseFeature registry)                              main.py:94-112
  -> create_dataloader: IterableDataset yields full Batch objects        dataset.py:751-840
  -> _create_model: oneof-name -> BaseModel.create_class -> TrainWrapper main.py:127-159

  -> planner.collective_plan(model, sharders, dist.GroupMember.WORLD)    main.py:682

Three structural facts here shape the whole design. First, the process-group topology is flat: init_process_group creates only the global WORLD with backend nccl and never builds TP/PP/EP sub-groups; the planner receives dist.GroupMember.WORLD directly (main.py:682). Any dense-parallel backend must layer its meshes on top of this WORLD after init, which is precisely the contract D8 (§11) adopts. Second, sparse optimization is fused into backward: BaseModel.sparse_parameters() (models/model.py:153-192) BFS-collects every EmbeddingBagCollectionInterface/EmbeddingCollectionInterface parameter and apply_optimizer_in_backward attaches the fused optimizer before planning; per-table trainable=false features are frozen by attaching SGD with lr=0.0 while keeping the fused kernel (main.py:666-668) — an existing precedent for parameter-group-level freezing relevant to D7. Third, one DistributedModelParallel wraps the entire TrainWrapper, and with init_data_parallel=True every non-sharded parameter is DDP-replicated by DMP's default DataParallelWrapper (dist_util.py:164-195) [torchrec]. tzrec's DMP wrapper does forward a custom data_parallel_wrapper argument (dist_util.py:172), which is the hook the FSDP2 integration uses (§11).

The optimizer tree is already a three-way composite, which is the single most important reuse point for a second (dense LLM) optimizer world:

TZRecOptimizer (grad_scaler + gradient_accumulation)        optim/optimizer.py:26-68
 +- [GradientClippingOptimizer]                             main.py:723-741
     +- CombinedOptimizer([
          model.fused_optimizer,   # sparse, in-backward; .step() is a no-op
          dense_optimizer,         # KeyedOptimizerWrapper(remaining params)
          *part_optimizers,        # one per regex_pattern over DMP FQNs

Parameters are split by in_backward_optimizer_filter(model.named_parameters()) and then by re.fullmatch per dense_optimizer.part_optimizers[i].regex_pattern (optimizer_builder.py:219-237); each node gets its own BaseLR schedule, and base_lr==0 freezes a group outright (optim/lr_scheduler.py:34-41). Anything that speaks the KeyedOptimizer protocol can be appended to the CombinedOptimizer list. Two caveats matter for the LLM loop: gradient accumulation only gates the dense side (in-backward sparse optimizers step on every backward, tzrec-core notes §4), and the entire fwd+bwd+step sequence is owned by pipeline.progress with no hook between backward and step — the origin of R-GRAD (§15).

Checkpointing

Checkpoints are torch DCP directories model.ckpt-{step}/{model, optimizer, plan, dataloader_state.json, dynamicemb?} written by the free functions save_model (checkpoint_util.py:873-913) and save_dataloader_state (checkpoint_util.py:922); saves are synchronous, with only pruning offloaded to a daemon thread that keeps the newest keep_checkpoint_max plus the best checkpoint by eval metric (checkpoint_util.py:267-588). Save triggers include step/epoch intervals and Kafka event-time quorum reconciled across ranks over a side gloo group (checkpoint_util.py:392-456). Restore goes through PartialLoadPlanner (checkpoint_util.py:50-148), which (a) skips keys missing from checkpoint metadata with a warning — partial restore for fine-tuning works today, (b) remaps names via the fine_tune_ckpt_param_map file ({new}\t{old} lines), and (c) is DTensor-aware because DCP load itself is. dynamicemb already demonstrates the “foreign parameter world” pattern: a side-by-side custom dump (DynamicEmbDump) keyed by module discovery, living next to model/ and optimizer/. This is exactly how the dense LLM checkpoint root attaches in §11.

Export and predict paths

Export (main.py:891-1007, utils/export_util.py) runs on rank 0 only and forces WORLD_SIZE=1 (export_util.py:159-168), restoring DCP weights into an unsharded module. The default artifact is FX symbolic_trace followed by torch.jit.script; under ENABLE_TRT/ENABLE_AOT, split_model (export_util.py:1068-1206) FX-splits the graph at marker boundaries into a sparse GM and a dense GM, compiling the dense part with torch_tensorrt or torch.export + aoti_compile_and_package with a dynamic torch.export.Dim("batch"). The RTP export path hard-rejects DTensor state (raise ValueError("DTensors are not considered yet."), export_util.py:484) — relevant because FSDP2 produces DTensor parameters; the AOTI path has no such rejection and is the route D13 builds on (§10). Notably, the predict driver already supports variable-cardinality “GR-style” outputs: when TARGET_REPEAT_INTERLEAVE_KEY (constant.py:40) appears in predictions, flattened per-candidate outputs are regrouped into pa.ListArray rows via fbgemm.asynchronous_complete_cumsum offsets (main.py:1017-1037) — beam outputs of K SIDs per user ride this convention unchanged.

3.2 Data and feature layer

The data layer is an Arrow-centric streaming pipeline: readers yield Dict<str, pa.Array> micro-batches, the dataset (an IterableDataset) applies sampling/masking and feature generation, and DataParser.to_batch assembles a fully formed Batch per iteration — the DataLoader runs with batch_size=None, collate_fn=identity (dataset.py:827-833). The Batch dataclass (datasets/utils.py:298-512) implements TorchRec's Pipelineable protocol and carries dense_features (KeyedTensor), sparse_features (KeyedJaggedTensor), sequence_dense_features, labels, jagged_labels: Dict<str, JaggedTensor>, sample_weights, additional_infos: Dict<str, Tensor> (a free-form tensor side-channel), plus serving/streaming metadata (tile_size, dummy, checkpoint_info, data_timestamp). One discipline applies to any extension: a new payload must be threaded through to(device), pin_memory(), record_stream(), and to_dict() or it silently drops on the H2D copy (utils.py:344-512).

The jagged_labels slot is the established precedent for variable-length, per-token supervision: DataParser.parse routes list<int>/list<float> label columns into values+lengths pairs (data_parser.py:231-250, 458-467), and both HSTUMatch._get_label (models/hstu.py:392-394) and DlrmHSTU._get_label (models/dlrm_hstu.py:246-258) consume them. This is the carrier for batch.jagged_labels["response_sid_ids"] and the separate batch.jagged_labels["loss_mask"] defined in §6; no Batch schema change is needed. What does not exist is any per-token mask semantics — masks today are sample-level feature dropout or implicit in jagged lengths (tzrec-data notes §8) — hence the loss-mask label convention rather than an overload of .weights().

Feature generation has four modes (data.proto:47-60): FG_NONE (inputs already encoded), FG_NORMAL/FG_DAG (pyfg C++ feature generation), and FG_BUCKETIZE. Two properties matter for the LLM route. First, FG_NONE accepts native Arrow list columns: _parse_fg_encoded_sequence_sparse_feature_impl takes list<int> directly (feature.py:262-272), ODPS ARRAY<BIGINT> maps onto it (odps_dataset.py:65), and jagged label columns likewise — so pre-tokenized prompt ids and precomputed SID columns flow end-to-end with zero new transport code. Second, TokenizeFeature (tzrec/features/tokenize_feature.py) already runs BPE/sentencepiece tokenization inside pyfg against a HF tokenizer.json or spiece.model vocab, and with tokens_as_sequence=true (feature.proto:757) emits the token stream as a jagged sequence feature (tokenize_feature.py:185-217). However, it offers no chat template, no BOS/EOS/special-token control, and no verified byte-level parity with the HF Qwen3 tokenizer — which is why §6 standardizes on offline pre-tokenized ARRAY<BIGINT> columns for v1 and treats any fg-side SID join as future work (R-FGJOIN).

The reader fleet is production-hardened and reused wholesale: OdpsReader (MaxCompute Storage-API Arrow sessions created on rank 0 and broadcast, LZ4 transport, row-offset checkpoint resume, odps_dataset.py:364-642), ParquetReader with global row-range rebalancing, CsvReader, and KafkaReader (each message a serialized Arrow IPC RecordBatch; per-row offsets and event timestamps injected and popped into Batch.checkpoint_info/data_timestamp, driving the event-time checkpoint quorum, kafka_dataset.py:128-485). Two adjacent primitives seed generative needs: cost-based batching (sample_cost_field + batch_cost_size, dataset.py:567-585) is a greedy token-budget batcher when cost = token count, and the GraphLearn sampler cluster (sampler.py) provides distributed “fetch attributes for sampled ids and append rows” — reusable for SID-level contrastive negatives and eval candidate expansion, though string-attr-centric and CPU-side. Known limitations the design must respect: one reader schema per dataloader (no ratio-based mixing of heterogeneous corpora, odps_dataset.py:424-438), batch-granular rather than row-granular shuffling, and the sequence-feature default value "0" colliding with SID code 0 unless the vocab layout reserves it (feature.py:549-561; the flat itemic-offset encoding in §6 addresses this).

3.3 Model and module layer

Models auto-register by class name via the metaclass in load_class.py:117-145; _create_model resolves config_util.which_msg(model_config, "model") — the protobuf message class name of the set oneof model arm — to BaseModel.create_class(name) (main.py:127-159, config_util.py:73-75). Adding a model is therefore one proto message plus one identically named class in tzrec/models/*.py (the import is non-recursive, load_class.py:53-99), with config_to_kwargs bridging proto fields to module kwargs. The ModelConfig oneof currently occupies tag blocks 100 (ranking), 200 (multitask, including DlrmHSTU=205), 300 (matching, including HSTUMatch=304), 400 (TDM), and 500 (RocketLaunching); the 600 block is free and is where the five generative arms and the 610 SID-tokenizer arms land (§12). The BaseModel contract is narrow — predict(batch), init_loss()/loss(), init_metric()/update_metric()/compute_metric() — and the eval loop only ever calls predict once per batch and feeds (losses, predictions, batch) to metrics (main.py:162-226).

EmbeddingGroup (modules/embedding.py:139-519) routes feature groups to pooled EmbeddingBagCollections or sequence EmbeddingCollections and emits a uniform Dict<str, Tensor>: DEEP/WIDE groups as dense [B, sum_dims], SEQUENCE/JAGGED_SEQUENCE groups as "g.query"/"g.sequence"/"g.sequence_length", with JAGGED_SEQUENCE keeping jagged values [sum(len_i), D] (embedding.py:1370-1375). A SID-token history is representable today as a JAGGED_SEQUENCE group over an EC table — bringing TorchRec sharding, ZCH, and dynamicemb along for free — and these grouped outputs are the payload that context-embedding injection (D8's bridge, R-INJECT, §7) feeds into the LLM.

The HSTU stack is the in-house generative backbone precedent. Its jagged convention — (values [total, D], seq_offsets [B+1], max_seq_len, num_targets [B]) — pervades tzrec/ops; hstu_mha (ops/hstu_attention.py:43-196) is the full causal kernel and delta_hstu_mha (:199-267) the incremental/cached variant. STULayer already implements a two-phase KV cache: prefill splits just-computed k/v at kv_caching_lengths into per-layer caches (modules/gr/stu.py:128-175), and cached_forward concatenates cache plus delta and runs delta_hstu_mha (stu.py:564-641). Constraints to design around: the delta size must be uniform per row (L % B == 0, hstu_attention.py:221) — compatible with stepwise SID decode where the host replicates beam rows — the cache is plain Python instance state (not traceable, reinforcing the host-side-loop rule), and CUTLASS falls back to Triton on the delta path. DlrmHSTU contributes two conventions reused verbatim: predictions[TARGET_REPEAT_INTERLEAVE_KEY] = num_targets for jagged per-candidate outputs (dlrm_hstu.py:192-244, key at constant.py:40), and enable_global_average_loss, which all-reduces the average batch size to weight variable per-rank token counts (dlrm_hstu.py:64-69, 280-284) — the same normalization a globally averaged token CE needs in §8.

TDM retrieval (tzrec/tools/tdm/retrieval.py:121-414) is the codebase's proof that autoregressive decode belongs on the host: a scripted, stateless scorer is driven by a Python loop that expands candidates per tree level, re-materializes features in a worker process, runs torch.topk beam pruning at width 2*recall_num, and keeps all beam state in pyarrow on the host. A SID beam decoder is structurally identical — children become the trie-constrained next codes instead of GraphLearn tree children, and the exported scorer gains explicit KV-cache I/O instead of full re-encoding (§10, D13). The TDM export also set the two-artifact precedent (embedding/ + model/, main.py:981-998) for splitting a prefix encoder from a per-step scorer.

SID baseline from PR #538 — and what is deliberately not there yet

Commit 7dc1c18 landed the tokenizer-training substrate this RFC builds on. ResidualQuantizer (modules/sid/residual_quantizer.py) implements the residual chain with precise gradient semantics — quantized vectors are subtracted detached so the residual chain carries no gradient, while the aggregate sum keeps codebook gradients flowing (_residual_pass, rq.py:119-153) — plus get_codes/decode_codes as the offline SID-assignment API and abstract _quantize_layer/_lookup_code primitives for backends. BaseSidModel (models/sid_model.py:54-74) consumes a precomputed item-embedding dense feature (embedding_feature_name/input_dim/normalize_residuals/codebook config fields), declares the internal-loss convention (losses computed in predict and passed through predictions; init_loss() is a no-op), and registers mse plus unique_sid_ratio (metrics/unique_ratio.py — explicitly a batch-level diversity proxy, not global codebook coverage). What is not in the repo: no concrete quantizer backend (the docstring names the planned k-means and VQ backends; neither exists), no SID proto messages in the oneof model (the 600/610 blocks are empty), no collision resolution, no item↔SID catalog or trie, no quality gates beyond UniqueRatio, and no decoder or generative model of any kind. §5 fills the tokenizer half (D1–D4); §7 the model half (D5).

3.4 Gap analysis

Table 3.1 maps the requirement clusters of §2 onto existing assets, the concrete gap, and the owning section of this RFC. The pattern is consistent: transport, sharding, checkpointing, and the host-side decode precedent exist; everything that touches a HF transformer, a packed 32K sequence, an RL rollout engine, or an autoregressive serving loop is new.

The four integration seams the design exploits

Four seams, zero forks

The architecture in §4 deliberately routes every two-world integration through four existing seams rather than forking the loop:

  • Pipeline factory. create_train_pipeline (dist_util.py:335-376) is the single pipeline-selection point, called at exactly three sites (main.py:176, 430; predict uses PredictPipelineSparseDist directly). The loop relies only on pipeline.progress(iterator) -> (losses, predictions, batch) plus StopIteration semantics, and a no-sparse fallback to TrainPipelineBase already exists — a custom GR pipeline that orders sparse dist, dense backward, finalize_model_grads, and the two optimizer steps slots in here (§11).
  • Optimizer tree. part_optimizers regex groups (optimizer_builder.py:219-237) and the open-ended CombinedOptimizer list (main.py:693-755) give the LLM its own AdamW + schedule without touching sparse Adagrad; per-node BaseLR registration covers warmup-cosine (§8, §12).
  • Checkpoint roots. save_model (checkpoint_util.py:873) is the single write site; the dynamicemb side-dump and PartialLoadPlanner's skip-missing/remap behaviors make an additional dense-world subtree (and warm-start from converted HF weights via fine_tune_ckpt_param_map) additive rather than invasive (§11).
  • Batch side-channels. jagged_labels carries response_sid_ids and loss_mask; additional_infos carries packing metadata; TARGET_REPEAT_INTERLEAVE_KEY carries variable-K outputs through loss, metrics, and the predict writer — all without a Batch schema change (§6).

Assumptions that break

Baseline assumptions the design must engineer around

  • Single DMP wrap vs. an FSDP2 subtree. DMP wraps the whole TrainWrapper and DDP-replicates every non-sharded parameter; naively placing a fully_sharded LLM inside would have DMP's DataParallelWrapper attempt to DDP-wrap DTensor modules. The custom data_parallel_wrapper hook (dist_util.py:172) and DDP-ignoring the FSDP FQNs are the mitigation (D8, R-DMP; §11). Side effects extend to tooling: _log_train special-cases ShardedTensor but not DTensor histograms (main.py:229-314), and RTP export rejects DTensor outright (export_util.py:484).
  • The planner ignores dense memory. The TorchRec planner budgets HBM from device total_memory minus HeuristicalStorageReservation(0.15); a multi-billion-parameter dense model plus activations and optimizer state is invisible to it, so plans that fit today OOM under the LLM (R-PLANNER). Mitigations — raising STORAGE_RESERVE_PERCENT, forcing row-wise constraints — are specified in §11.
  • pipeline.progress owns fwd+bwd+step; accumulation is asymmetric. There is no hook between backward and step, and in-backward sparse optimizers step every backward while TZRecOptimizer accumulates the dense side — the two-world step-sequencing problem (R-GRAD) is resolved by the GR pipeline in §11.
  • FX/JIT cannot trace a HF transformer. The default export (symbolic_trace + jit.script) will not swallow dynamic control flow and cache objects; the AOTI single-step-scorer route (D13) with a host-side decode loop is mandatory, never tracing the autoregressive loop (§10; R-EXPORT, de-risked).
  • Eval-after-save is too expensive for generation. The loop runs a full eval after every checkpoint save (main.py:490-497), and the eval contract assumes one cheap predict per batch. Constrained beam decode over the catalog per save is unaffordable; generative eval cadence and decode-driver wiring are decoupled in §13.
  • One flat process group. init_process_group creates only WORLD and the planner assumes it; Megatron's initialize_model_parallel and FSDP2 meshes must layer on top after init without re-creating groups (D8; §11).

In sum, the baseline supplies a stable chassis — registry-driven models, Arrow-jagged transport, fused sparse optimization, DCP checkpointing, and a proven host-side beam pattern — and the gaps concentrate precisely where the two-world design of §4 introduces its new packages. The remainder of this RFC fills those gaps without modifying the contracts cataloged above.

4.2 Two-world training: why the split is shaped this way

Sparse stays TorchRec. The sparse world's defining workloads have no dense-world equivalent: a 250M-row item table only fits via dynamicemb HKV hash tables with ROW_WISE sharding (dynamicemb_util.py:216), per-token SID and id lookups need EmbeddingCollection semantics with all-to-all exchange, bounded-HBM exact-id mapping needs ZCH, and the fused in-backward optimizer is what makes embedding training throughput-viable. tzrec already owns all of this (§3) [torchrec]. A strategic simplification keeps the burden tractable: after SID assignment, day-to-day LLM training is dominated by dense compute — the giant item tables are stressed mainly offline during tokenization and in HSTU/encoder-style context paths, while the itemic vocabulary itself is a small 24,576-row block living inside the LLM's embedding matrix.

Dense leaves the DMP/DDP world. Today DMP wraps the entire TrainWrapper and DDP-replicates every non-sharded parameter (dist_util.py:164-195). For a 1.7B–8B trunk that means a full parameter copy, full gradients, and full AdamW state per rank — DDP shards nothing. FSDP2's fully_shard shards parameters, gradients, and optimizer state as DTensors over a DeviceMesh derived from the same WORLD process group, composes with TorchRec's DTensor-aware DCP checkpointing, and is exactly how OpenOneRec trains every stage of its 1.7B/8B models [openonerec][pytorch]. Megatron-core 0.18 remains the scale-out backend where FSDP2 runs out of road: expert parallelism for the OneRec MoE arms, tensor-parallel vocab embedding and vocab_parallel_cross_entropy for the 176K-row vocabulary at 8B+, and alignment with verl's megatron engine [megatron][verl].

Revision pointer (D8). This inverts the prior spec's decision 3(f) (Megatron-core default, FSDP2 fallback); the canonical rationale lives in the revision callout in §11.1, and §11 owns the full decision.

The bridge and its risks. The two worlds meet at exactly one activation boundary: TorchRec produces all-to-all'd context embeddings (user towers, HSTU-encoded histories, SID-history embeddings), and the dense backbone consumes them as inputs_embeds (HF path) or decoder_input (mcore path), overwriting designated feature slots without disturbing rope position ids — R-INJECT. Optimizer trees stay disjoint, partitioned by in_backward_optimizer_filter, but step inside one CombinedOptimizer; the ordering of Megatron's finalize_model_grads versus the fused in-backward sparse step is R-GRAD. DMP must shard the sparse subtree without DDP-wrapping the FSDP/TP-managed dense parameters — R-DMP — and the TorchRec planner must be told about dense memory it cannot see — R-PLANNER. All four are tracked with proof obligations in §15; the concrete mechanisms live in §11.

4.4 Model classes and how the literature maps onto them

Five generative model classes and two SID tokenizer-training classes enter the oneof model registry; the proto message name equals the Python class name, so registration needs no main.py edits (§3). MiniOneRec is deliberately not a class: it is a config preset over LCRec plus GRPO [minionerec].

The newly surveyed literature maps onto this taxonomy without new classes. TIGER and GRID configure the encoder-decoder shape at small scale [tiger][grid]; PLUM's lessons (co-pretraining dominates base-model choice, reward-weighted SFT sampling) land as stage and sampling knobs in D6/D14 [plum]; OneRec-Think and OneReason contribute RL recipes (two-stage rollouts, RFT, MOPD, stage-wise clipping) and the Think-Ahead serving pattern to D10/D13 [onerec-think][onereason]; Latte's latent tokens and PIT's tokenizer–recommender co-evolution are scoped extensions under D16 [latte][pit]; RPG-style parallel decode and LLaDA-Rec diffusion are future work that D5's level-agnostic interfaces must not preclude (D15) [rpg][llada-rec].

4.5 Pipeline lifecycle and artifact hand-offs

Every stage is a separate launch with a file-level artifact contract; no stage holds another's in-process state. Checkpoints chain through the existing --fine_tune_checkpoint mechanism, and decode-time assets (catalog, trie, extended tokenizer) ship through --additional_export_config into model_acc.json alongside the exported model, following the TDM precedent of separately serialized artifacts (§3).

Two properties of this chain are deliberate. First, the SID asset table is the single integration seam between the tokenizer subsystem and everything downstream — training joins, vocabulary id arithmetic, trie construction, and HR@K expansion all read the same table, so a re-tokenization (cadence risk R-RETOK) is a table swap plus dependent-artifact rebuild, not a code change. Second, every numbered stage is restartable from its inputs: the pipeline has no hidden coupling that would force a full rerun when, say, only the RL stage changes.

5. SID tokenizer subsystem (tzrec/sid)

This section specifies the subsystem that turns a catalog of item embeddings into the Semantic-ID space everything downstream consumes: quantizers (D1), collision resolvers (D2), training-free quality gates (D3), and the assignment/asset/catalog layer publishing the item↔SID mapping as a many-to-many index (D4). The package is tzrec/sid/ plus two tokenizer-training model classes (SidRqkmeans=610, SidRqvae=611) extending the BaseSidModel/ResidualQuantizer substrate already on master from PR #538 (§3). One empirical theme governs effort allocation throughout: the quantizer algorithm is a second-order choice, while collision post-processing, the embedding's collaborative signal, and the evaluation protocol are first-order [forge] [grid].

5.1 Item embedding sources

tzrec does not run image or video encoders. The tokenizer's input is an upstream artifact: a parquet/ODPS table with an item-id column and a fixed-width fp32 vector column (parquet list<float32>, ODPS ARRAY<FLOAT> or a base64-packed string per the AL-GR release format [repo-alsid]). This is exactly the surface BaseSidModel already consumes — a RawFeature with value_dim=input_dim read via _extract_feature (tzrec/models/sid_model.py:76-89) — so the embedding contract requires no new data-layer code. The asset must carry an embedding_version tag, because every downstream artifact (codebooks, SID asset table, trie, vocabulary alignment) is keyed to it (§5.5).

Two evidence points steer how teams should produce this artifact. First, collaborative signal dominates content: FORGE's modality ablation (Table 6) shows an i2i-co-occurrence-only embedding reaching HR@20 4.63% versus 3.01% text-only and 3.06% image-only, with the full fusion at 4.89% — content embeddings mainly add tail coverage at large K (HR@1000 29.01% vs 23.15% for i2i-only) [forge]. Second, not all alignment positives help: aligning the encoder with query–item pairs or same-category positives hurt retrieval by 4.6–15% relative HR, while i2i co-occurrence positives and side-info text concatenation helped (FORGE Table 5). PLUM independently found the co-occurrence InfoNCE term to be the single largest tokenizer-side win — removing it dropped SID uniqueness from 96.7% to 91.8% and downstream Recall@10 from 14.4% to 12.6% (PLUM Table 4) [plum]. PLUM also makes the operational case for injecting behavior as a training objective on content-only inputs rather than fusing CF embeddings into the quantizer input (the LETTER-style alternative [letter]): CF embeddings drift with popularity and would couple the re-tokenization cadence to CF retraining (§5.5, R-RETOK). The RQ-VAE trainer therefore exposes an optional i2i co-occurrence InfoNCE auxiliary loss (§5.2), and the recommended upstream recipe is an i2i-aligned multimodal encoder — with the caveat that a plain i2i-only embedding is already a strong, cheap baseline.

Encoder size, by contrast, is not worth GPU budget: GRID's sweep from Flan-T5-Large to XXL (14× parameters) moved Amazon-Beauty Recall@10 only from 0.0639 to 0.0646 [grid]. Marginal effort belongs in co-occurrence pair mining (the same pairs double as Embedding-HitRate ground truth in §5.4), not in a larger encoder.

5.2 Quantizers

Decision D1 — RQ-KMeans default, RQ-VAE option; both packaged as tzrec models

The default tokenizer is ResidualKMeansQuantizer (3×8192, faiss residual k-means over a frozen embedding, training-free), matching the OneRec lineage and the GRID benchmark verdict that residual k-means meets or beats RQ-VAE at a fraction of the complexity [onerec-v1] [openonerec] [grid]. ResidualVectorQuantizer (gradient-trained RQ-VAE) is a supported option with the knobs the strongest implementations converge on: k-means codebook init, commitment loss, dead-code reset, optional rotation-trick estimator, no EMA (every strong implementation rejected it), optional per-level Sinkhorn (last level in practice), and an optional co-occurrence InfoNCE auxiliary loss; progressive prefix masking is specified but deliberately cut from the v1 proto surface (Table 5.2). Both are exposed as tzrec models — SidRqkmeans (proto tag 610) and SidRqvae (611) over BaseSidModel — so tokenizer training reuses the stock dataloader, distributed, metric, and checkpoint machinery instead of growing a parallel trainer.

The class plan extends the abstract ResidualQuantizer from PR #538 (tzrec/modules/sid/residual_quantizer.py), whose contract is already final: subclasses populate one layer per level and implement _quantize_layer/_lookup_code; the shared _residual_pass (rq.py:119-153) fixes the gradient semantics (residual chain detached, aggregate sum carries codebook gradients) and get_codes/decode_codes (rq.py:155-168, 197-213) are the offline assignment/reconstruction API. Per-level codebook sizes are already first-class (normalize_n_embed, rq.py:21-37), which is deliberate: PLUM's multi-resolution geometry (halving codebook size per level) improved both uniqueness and recall [plum], so the config must not hard-code "L levels × fixed K" even though the shipped default is uniform 3×8192. The 3×8192 default itself is FORGE's empirically supported sweet spot for 1e8-scale corpora: under-capacity geometries are catastrophic (3×512 cost 19–21% HR@20 on a 251M-item corpus), 2-level variants trade HR@20 for HR@1000, and 4×4096 is accuracy-neutral at strictly higher decode cost (FORGE Tables 8/10) [forge]. For small academic corpora the GRID result (3×256 optimal at TIGER scale [grid]) is reachable through the same config fields.

The RQ-VAE knob set is the union of what the strongest open implementations found necessary to avoid codebook collapse, each having rediscovered a subset independently: al_sid ships gradient codebooks with dead-code restart and EMA disabled (configs/rqvae_i2v.yml) [repo-alsid]; semantic-ids-llm implemented EMA, rejected it, and relies on k-means init + rotation trick + frequent dead-code resets [repo-semanticids]; GRID needed embedding whitening (RQ-VAE) and residual normalization (RK-Means/R-VQ) [grid]. LC-Rec contributes the per-level Sinkhorn epsilon list — balanced optimal-transport assignment on the last level only (ε=0.003, 50 iterations), a training regularizer while inference stays greedy argmin (index/models/vq.py, get_indices(use_sk=False)) [repo-lcrec]. The co-occurrence InfoNCE head follows the al_sid construction (three symmetric terms with learnable temperatures, positives from an i2i pair table, embeddings all-gathered with gradient across ranks); pairs arrive as a second input table on the same item-id space. Progressive prefix masking (train against a randomly truncated prefix of levels so every SID prefix is a usable coarse representation) is cheap and directly benefits prefix-decodable beam search [plum]; it is deliberately left out of the minimal v1 proto, and adding progressive_mask_prob later is a backward-compatible field extension.

Tokenizer-as-model: SidRqkmeans / SidRqvae

BaseSidModel already defines the packaging: the proto message carries embedding_feature_name, input_dim, normalize_residuals, and repeated codebook; init_loss() is a no-op because SID models compute losses internally and return them through predictions (sid_model.py:54-74); init_metric() registers mse and unique_sid_ratio (tzrec/metrics/unique_ratio.py). SidRqvae.predict runs the encoder + _residual_pass + decoder and emits {"reconstruction": ..., "codes": ..., "loss_recon": ..., "loss_commit": ..., "loss_infonce": ...}; loss() pulls the loss tensors out of predictions, so the stock TrainWrapper sums them unchanged. Both classes add the new gate metrics of §5.4 (per-level utilization and entropy as torchmetrics; Gini and Embedding HitRate run offline over the assignment table, not in-loop). Following LC-Rec, the eval-time checkpoint selector for SidRqvae is collision rate / unique-SID ratio, never reconstruction loss (§5.4, D3) [repo-lcrec].

Distributed training for large corpora

The tokenizer model is tiny — three 8192×64 codebooks plus a small MLP is a few million parameters — so the distributed problem is data volume, not model size. SidRqvae trains as ordinary data-parallel: no sharded embedding module exists, so tzrec's pipeline factory selects the plain TrainPipelineBase path and DDP-replicates the model; the only cross-rank coupling is the InfoNCE all-gather, which al_sid already demonstrates at 2-node scale [repo-alsid]. Reference batch/optimizer settings come straight from the published configs (al_sid: batch 2048, AdamW, cosine; LC-Rec: batch 1024, AdamW 1e-3, k-means init [repo-lcrec]).

SidRqkmeans is gradient-free, so "training" through the stock loop means reusing its plumbing, not its optimizer: the tzrec.gr.train_tokenizer entry (a thin wrapper over tzrec/main_gr.py, §12) iterates the standard dataloader to fill a per-rank reservoir of embedding rows up to a fixed sample budget (faiss's default per-centroid training-point cap; deliberately not a v1 proto field), gathers the reservoir to rank 0, runs faiss residual k-means level by level (the OpenOneRec train_kmeans loop: fit on residuals, assign, subtract, recurse [repo-openonerec]), and broadcasts the centroids into the quantizer's non-trainable ParameterList; a standard eval pass then computes the in-loop metrics, and the checkpoint saves through the normal DCP path. A subsample is sufficient and intended — k-means quality saturates far below full-corpus scale (faiss itself caps training points per centroid by default), and OpenOneRec fits its production tokenizer in one in-memory CPU pass — whereas the full-corpus work is the encode in assign_sid, chunked and embarrassingly parallel (§5.5). For a streaming/trainer-integrated k-means (not needed in v1), GRID's trick of expressing mini-batch k-means as a weighted-MSE loss under SGD — letting DDP gradient averaging perform the cross-GPU reduction — is the documented fallback [repo-grid].

5.3 Collision resolution

Decision D2 — collision resolution is pluggable post-processing, decoupled from the quantizer

Collision handling lives in tzrec/sid/collision.py as a separate pipeline stage operating on the raw assignment table, never inside the quantizer. Resolvers: RandomCollisionResolver (round-robin last-level reassignment, default for GR targets), KnnCollisionResolver (capacity-σ walk over ranked last-level candidates), UniquenessDigitResolver (TIGER-style appended digit), and SinkhornGroupResolver (LC-Rec collision-group re-encoding). For 1e8-scale catalogs the resolver compiles to FORGE's iterative ODPS SQL pattern rather than running in-process.

FORGE's central finding is that this stage, not the quantizer, is where retrieval quality is won: with everything else fixed, no collision handling costs 18% relative HR@20 versus a σ=25 baseline, tightening to σ=5 gains 15.8%, and the non-semantic Random-5 policy gains 29.4% HR@20 / 4.8% HR@1000 — a strictly monotone ordering from "no handling" through KNN to Random (FORGE Table 7; on sparse Amazon data the Random-5 gain exceeds 100%) [forge]. The reading: leaf fairness beats leaf semantics — the third residual level carries little information anyway (LC-Rec's level-wise case study agrees [lcrec]), so equalizing the SID-occupancy distribution matters more than assigning the semantically nearest leaf. This is also why the FORGE "RQ-VAE beats RQ-Kmeans" headline does not move D1: the comparison is confounded (only RQ-VAE received the Final post-processing), and at equal post-processing the encoders are within noise of each other (FORGE Table 9).

Revision vs prior spec — default resolver flips from KNN-σ to Random

The prior spec (3a) made KnnCollisionResolver (σ=5) the default. Given the FORGE Table 7 evidence above — Random-5 strictly dominates every KNN setting on both AL-GR and Amazon — v1 defaults to RandomCollisionResolver with the σ=5 capacity convention. KNN-σ remains the recommended choice when last-level code embeddings are reused as semantic features elsewhere (Random deliberately destroys level-3 semantics), and is required anyway as the fallback for quantizers that emit ranked candidates.

The four resolvers, concretely:

  • RandomCollisionResolver (FORGE Algorithm 2): keep the first L−1 semantic levels; assign the last level by a round-robin counter keyed on the (c_1, …, c_{L-1}) prefix — IndexMap[key] = (IndexMap[key]+1) mod n_L — dispersing items maximally. With n_L = 8192 and the σ=5 convention, a bucket only overflows past σ once a prefix holds >40,960 items. The counter is insertion-order dependent, so determinism requires (a) a pinned snapshot ordering — the resolver sorts by item_id before assignment — and (b) persisted IndexMap state for incremental arrivals: the counter table is part of the published asset so that tomorrow's new items continue the rotation instead of restarting it (§5.5).
  • KnnCollisionResolver (FORGE Algorithm 1): requires the quantizer to emit a ranked last-level candidate list (al_sid's get_sorted_index — the argsort of all last-level distances; production exports the top 200 [repo-alsid]); walk candidates nearest-first and take the first bucket with fewer than σ items. ResidualKMeansQuantizer.get_codes therefore gains an optional return_topk_last_level argument; this ranked emission is the one quantizer-side hook the resolver layer needs.
  • UniquenessDigitResolver (TIGER / GRID / semantic-ids-llm): append one extra digit numbering items within each collision group, guaranteeing strict uniqueness [tiger] [repo-semanticids]. Useful when downstream requires a bijection, but it lengthens decode by one step, the digit is purely positional (observed group sizes up to 50 against a reserved range of 256 [repo-semanticids]), and GRID found it near-indistinguishable from simply tolerating collisions (Recall@10 0.0597 vs 0.0591) [grid]. tzrec validates group sizes against the digit range, fails loud on overflow, and uses 0-based positions uniformly (GRID's released tags are inconsistently 1-based).
  • SinkhornGroupResolver (LC-Rec): re-encode only the colliding groups as single batches with last-level Sinkhorn assignment enabled, repeating up to 20 rounds; transport balancing splits the ties while keeping the last level semantic [repo-lcrec]. Only applicable to SidRqvae (needs the trained codebook distances); residual collisions among exact-duplicate items are tolerated, as in the original.

Scaling to 250M items: the iterative ODPS SQL pattern

An in-memory resolver is fine to ~1e7 items; beyond that, tzrec ships the resolver logic as a templated MaxCompute SQL job, directly modeled on the AL-GR production pipeline that processed 251M items (SID_generation/ID_collision.sql [repo-alsid]): (1) a first pass keeps up to σ random items per SID via ROW_NUMBER() OVER (PARTITION BY sid ORDER BY rand()); (2) overflow items join against their exploded top-200 ranked candidate codes (TRANS_ARRAY, priority 1..200); (3) an iterated LEFT ANTI JOIN loop removes already-assigned items and full buckets, assigns by priority with capacity truncation, and merges, until empty. The Random variant replaces the ranked candidate generator with a random/rotating code. The output schema — (item_id, origin_codes, final_codes, bucket_pos) — is the asset-table schema of §5.5 verbatim, so the SQL job and the in-process resolver are interchangeable producers.

5.4 Training-free quality gates

Decision D3 — tokenizers are selected by training-free gates, never by reconstruction loss

A sid_eval step (part of assign_sid, §5.5) computes four gates from the assignment table and the embedding table alone: Gini coefficient over SID occupancy, codebook utilization per level and for full SIDs, global unique-SID ratio, and Embedding HitRate against held-out co-occurrence pairs. These correlate with downstream GR HitRate; reconstruction fidelity and consistency metrics do not, and must never drive model selection [forge].

The motivation is cost: validating a tokenizer by training the GR model is an 80-hour, 32-GPU experiment per configuration at FORGE's scale [forge], while every gate below is a SQL aggregation plus a small ANN job. FORGE validated the correlation empirically (its Figure 3: lower Gini and higher Embedding HitRate track higher downstream HR across the whole noco→Random-5 family), and PLUM's harness independently arrived at the same two-axis protocol — uniqueness plus a cheap downstream recall probe [plum].

Gini coefficient. With SID occupancy counts a_(1) ≤ … ≤ a_(N) and Lorenz curve L(i) = C(i)/C(N), G = (2/N) · Σ_i (i/N − L(i)); lower is fairer. One convention gotcha must be pinned: the FORGE paper defines N over all possible SID slots (8192^3, including zero-count slots) while the released script uses used SIDs only — the two produce very different absolute values [forge] [repo-alsid]. tzrec adopts the cleaner decomposition: Gini over used SIDs (fairness among occupied buckets) plus the separate utilization metric below (how much of the space is occupied at all). The implementation is the al_sid histogram closed form — one ODPS GROUP BY builds the count→frequency histogram; arithmetic-series rank sums then compute G in O(#distinct counts) memory — so it runs on a 250M-item table without moving the data.

Codebook utilization — the fraction of codes used per level and the fraction of realized full SIDs — plus per-level coverage and entropy diagnostics logged during training, following GRID's _compute_output_stats (distinct-ids/n_clusters per level, categorical entropy of the per-level code distribution, batch unique-SID fraction, residual-norm ratios) [repo-grid]. These are the early-warning signals for dead codes and level collapse, registered as torchmetrics on both SidRqkmeans and SidRqvae.

Unique-SID ratio. The in-batch UniqueRatio metric already exists (tzrec/metrics/unique_ratio.py) and is documented as a batch-level diversity proxy; sid_eval adds the global ratio (distinct full SIDs / corpus size before resolution; exactly 1.0 after a uniqueness-guaranteeing resolver) computed over the full assignment table. PLUM supplies the essential caveat: uniqueness alone is an insufficient selection metric — its multi-embedding ablation produced higher uniqueness (96.9% vs 96.7%) with lower downstream recall (12.8% vs 14.4%) [plum]. Gates are evaluated jointly, never as a single scalar.

Embedding HitRate. FORGE shows the metric predicts downstream HR but never publishes a precise protocol [forge], so tzrec defines one: build a faiss inner-product index over the item vectors, query each item in a held-out set, retrieve top-K neighbors (excluding self), and score hits against held-out i2i co-occurrence pairs (the same pair table used for the InfoNCE loss, split before training); report HR@K for K ∈ {20, 100, 500} with per-query macro averaging. Two variants run under one flag: embedding-gate mode indexes the raw input embeddings (validating the upstream artifact before any quantization, FORGE's published usage), and quantized mode indexes decode_codes reconstructions (validating that quantization preserved neighborhood structure, the reading in the al_sid release [repo-alsid]). The notes for the two FORGE sources disagree on which variant the paper measured, which is precisely why the tzrec protocol names its variant in the metric key (emb_hitrate@K/raw vs /quantized).

Anti-metric: reconstruction loss does not predict retrieval quality

FORGE tested feature fidelity, style consistency, and homogeneous consistency against downstream HR and found no correlation: Multiple-VQ wins two fidelity metrics yet is the worst retriever in the comparison (FORGE §4.3.5, Figure 4) [forge]. Selecting RQ-VAE checkpoints by validation reconstruction loss — the common practice the default HF/Lightning training loop encourages — silently optimizes the wrong objective. tzrec's SidRqvae therefore keys best-checkpoint tracking to collision rate / unique-SID ratio (LC-Rec's best_collision_model precedent [repo-lcrec]) and surfaces the gate metrics, with reconstruction MSE logged for debugging only.

5.5 SID assignment, asset table, and catalog index

Decision D4 — the item↔SID mapping is many-to-many from day one

The catalog layer (SidCatalogIndex) treats both directions as one-to-many: one SID resolves to up to σ items (collision buckets), and one item may carry multiple SID paths (PIT's top-B dynamic beam index [pit], Latte's latent-token trees [latte]). All consumers — HR@K evaluation, trie construction, generation grounding, RL legality — go through this index and its multi-path scoring/dedup API rather than assuming a bijection.

assign_sid: chunked batch inference

tzrec.gr.assign_sid loads a tokenizer checkpoint and runs get_codes over the full embedding table in chunks — the al_sid full-inventory pattern [repo-alsid], but through tzrec's standard predict pipeline: the ODPS/parquet reader streams the embedding column, the model emits codes [B, L] (plus ranked last-level candidates when a KNN resolver is configured), and the predict writer lands the raw assignment table. The collision resolver then runs as a second pass (in-process or the ODPS SQL job of §5.3), followed by the sid_eval gates. Encoding is embarrassingly parallel across workers, so 1e8-item catalogs are a throughput problem, not a design problem.

Asset table schema

Training data consumes this asset by an offline join: a batch job maps each interaction history's item ids through the asset table and materializes SID sequences as plain ARRAY<BIGINT> columns, which flow through the existing FG_NONE path as fg-encoded integer sequence features with zero new transport code (§6). Performing the join inside the feature-generation pipeline instead (fg-side SID lookup at training/serving time, which would let one asset update propagate without re-materializing datasets) is explicitly future work tracked as R-FGJOIN (§15).

SidCatalogIndex

SidCatalogIndex (tzrec/sid/catalog.py) is the single in-memory view of the asset table, loaded once per job. Its API: items_for_sid(codes) → List[item_id] (bucket members in bucket_pos order), sids_for_item(item_id) → List[path], expand_beams(beam_paths_in_order, k) → top-k item ids (walk completed beams in score order, expand each SID into its bucket, de-duplicate, truncate — the AL-GR calc_hr protocol, where 100 beams × σ=5 yields up to 500 candidates [repo-alsid]), and aggregate_item_scores(scored_paths, agg=sum|max) → Dict[item_id, score] for the multi-path case where several beams — latent-token trees, PIT index entries, or plain collision siblings — reach the same item. Consumers: HR@K evaluation (§13) must expand SID→items in beam order rather than assume one item per SID; the GR serving runtime (tzrec-gr-serving) runs the same σ-bucket beam-order expansion in its grounding layer, tensorized at ~100 LOC (§10.6); the CSR trie of §9/§10 (D12) is built from the set of valid code paths; unconstrained-decode grounding maps generated SIDs back to catalog items, with the prefix-fallback trick (exact path, else longest valid prefix 3→2→1) as a graceful-degradation option [repo-semanticids]; and RL legality rewards query the same index (§9, D14). Multi-path scoring is forced anyway: collision buckets alone (σ>1) make the mapping non-bijective, and PIT's ablation shows its multi-view index is the single largest contributor (removing it: −14.9% Recall@10) [pit].

Re-tokenization cadence for streaming corpora

New items arrive continuously in production; the SID space must follow without invalidating a trained LLM. The v1 policy is frozen-codebook incremental assignment: between major versions, codebooks are pinned; new items get codes via the frozen get_codes plus the collision resolver running with persisted state (the Random resolver's IndexMap counters and bucket occupancies are part of the asset, so incremental assignment is deterministic and capacity-correct). Full re-tokenization — retraining codebooks and regenerating every SID — redefines token semantics and forces an LLM re-alignment pass (at minimum the CO_PRETRAIN stage of §8); it is scheduled with major model refreshes, not on a timer. The cadence question is tracked as R-RETOK (§15), bounded by three evidence points. First, under the Random resolver the leaf codes of late-arriving items are semantically meaningless by construction — acceptable for GR targets, but leaf-code semantics then degrade monotonically between re-tokenizations, so no consumer may reuse level-3 code embeddings as features under this policy [forge]. Second, PLUM's two choices — behavioral signal as an objective over static content-only inputs, and progressive masking so every prefix stays a usable coarse code — are exactly what decouples tokenizer freshness from CF drift; tzrec inherits the co-occurrence objective as a v1 RQ-VAE knob and earmarks progressive masking as its follow-up field (§5.2) [plum]. Third, genuinely streaming tokenizer maintenance (incremental codebook merges without a global re-fit) exists in the literature but is unproven in any system we studied; noted as future work (§14) [awesome-list].

Decision D16 — Latte latent tokens as a cheap option; PIT tokenizer co-evolution as a phase-2 streaming extension

Two published extensions of the SID layer are adopted at different speeds. Latte latent tokens ship in v1 as a cheap option, off by default (n_latent_tokens=0): the asset and catalog layers need nothing new, and the model-layer slice (extra vocabulary rows, restricted beam step 0, multi-path score aggregation) is specified in §7.4. PIT tokenizer–recommender co-evolution is deferred to phase 2 streaming training — it requires train-time generation and a recommender-as-critic path that only the RL machinery of §9 provides — while v1 commits to the seams it needs: weighted multi-SID sid_paths (D4), a versioned asset table, and a reusable SID-emission head (D5). The two hooks below elaborate.

Hook: PIT tokenizer–recommender co-evolution (D16, phase 2)

PIT replaces the frozen tokenizer with a small autoregressive item-to-token model that co-evolves with the recommender in streaming training: the tokenizer proposes B candidate SIDs per item by beam search, the recommender-as-critic picks the most predictable one as a pseudo-label, both update jointly, and a momentum-weighted top-B beam index replaces the static asset table — deployed online against the OneRec-V2 baseline with positive A/B results [pit]. tzrec deliberately scopes this to phase 2, streaming-only: it requires train-time generation inside the training loop (eager mode, incompatible with graph capture) and a critic-scoring path that only the RL machinery of §9 provides, and its offline form degenerates anyway (PIT freezes the tokenizer on academic datasets). What v1 does commit to are the seams PIT needs later: SidCatalogIndex already supports weighted multi-SID entries per item (Table 5.3's sid_paths), the asset table is versioned for streaming sync, and the decode-strategy abstraction of §7 (D5) keeps the SID-emission head reusable as a standalone item-to-token model.

Hook: Latte latent tokens (D16, cheap option)

Latte addresses a structural expressiveness limit of single-tree SID decoding — items sharing a prefix have strongly coupled probabilities across users, provably suppressing user-specific rank reversals between tree-close items — by prepending one of M′ latent tokens (M′ ∈ {4,8}) to each training target and aggregating item scores over latent paths at inference, lifting NDCG@10 by ~3.5% on average with identical SIDs [latte]. The integration cost in tzrec is deliberately near-zero: training needs only a uniformly sampled latent-token id prepended to the target SID (one extra CE position, a dataset-builder flag in §6); the vocabulary needs M′ extra rows, which fit in the existing alignment headroom after the SID specials without changing the padded vocab total (§7); decoding restricts beam step 0 to the latent vocabulary and then proceeds normally (trie root fan-out in §10); and score aggregation across latent paths reuses SidCatalogIndex.aggregate_item_scores — the same component collision buckets and PIT already require (sum vs max aggregation is empirically a wash [latte]). The one real cost is beam budget: with M′ trees the effective per-tree beam shrinks, so eval keeps ≥50 beams and final retrieval ~500, per the reference configuration [repo-latte]. n_latent_tokens=0 (the default) disables the feature entirely.

6. Data and feature pipeline for generative recommendation

This section defines how raw interaction logs, item catalogs, and general-text corpora become training rows for the five generative model families, and what new code lands in tzrec/datasets/. The design stance follows directly from the baseline inventory in §3.2: every sample format the LLM pipeline needs is expressible as a column schema over the existing readers plus label routing, so the data layer ships no new reader plumbing and no Batch schema change — the new code is three focused modules (tzrec/datasets/gr/instruction.py, packing.py, mixture.py), one thin feature module (tzrec/features/semantic_id_feature.py, §6.1), plus offline build jobs. Everything between source table and model stays Arrow and jagged; the canonical label contract is batch.jagged_labels["response_sid_ids"] for target SID token ids and a separate batch.jagged_labels["loss_mask"] for per-token supervision masks.

6.1 Sample schemas

Five row families cover the full lifecycle. Each is a flat columnar table (ODPS or Parquet; Kafka for streaming) produced by an offline builder job, with column types chosen so the existing FG_NONE parse path consumes them without conversion. The first two families and the GBPO family flow through tzrec readers into Batch; the RL-prompt and distillation families are materialized as plain Parquet because their consumer is verl's file-based dataset (data.custom_cls / RLHFDataset), not the tzrec dataloader [verl] (see §9, R-ENV).

Two schema details deserve elaboration. For pretrain/CPT rows, the loss_mask column implements OneReason's data-composition rule: noisy, template-synthesized business context (serialized profiles, raw behavior dumps) is routed into loss-masked context positions, while loss-bearing targets are reserved for higher-quality text, at roughly 1:1 QA-style versus free-form targets [onereason]. This is strictly more general than OpenOneRec's segments handling (loss on everything except the terminator) and costs nothing: the mask is just a second integer list column. For GBPO rows, the data job precomputes everything that requires per-user state — the log-bucket index F(d) = floor(log_b(d + eps)) and the within-bucket percentile rank of watch time over the user's history — while the batch-relative top-quartile thresholding and the advantage assignment {+1, −1, filtered} remain train-time reward logic owned by DurationAwareReward (D14, §9) [onerec-v2]. Teams that shape advantages fully upstream can populate the advantage column directly; the reward reads its configured columns unconditionally and raises KeyError on absence rather than degrading silently.

Naming and dtype conventions

All token-id and SID columns are native list columns end to end: ODPS ARRAY<BIGINT> maps onto Arrow list<int64> in the storage-API reader (odps_dataset.py:65), data.proto already declares the matching FieldTypes (data.proto:20-29), and the FG_NONE parse path accepts list<int> directly (_parse_fg_encoded_sequence_sparse_feature_impl, feature.py:262-272). Nested SID histories use ARRAY<ARRAY<BIGINT>>. Delimiter-encoded strings are prohibited for new GR columns: the FG_NONE defaults (fg_encoded_multival_sep = \x03, GraphLearn chr(29)) are invisible control characters that corrupt silently when tables round-trip through text tooling. Label routing is the existing DataParser behavior: list<int>/list<float> columns named in label_fields become jagged_labels values+lengths pairs, scalars become labels tensors (data_parser.py:231-250, 458-467).

Label contract — response_sid_ids plus a separate loss_mask

The target SID token ids ride in batch.jagged_labels["response_sid_ids"] and the per-token mask in batch.jagged_labels["loss_mask"] — two independent jagged labels, never an overload of JaggedTensor.weights() (which the weighted-id-feature path already claims). For the packed decoder path the CE loss in §8 derives shifted labels from input_ids and loss_mask alone; response_sid_ids is still carried on every supervised row because the non-text backbones (OneRecV2's lazy decoder consumes exactly the 3 target codes) and the teacher-forced hit-rate metrics (§13) consume it directly. This reuses the in-tree jagged-label precedent (DlrmHSTU._get_label, dlrm_hstu.py:246-258) with zero transport changes.

Feature classes: SemanticIdFeature and the transport-only contract

Two thin feature classes land in tzrec/features/semantic_id_feature.py for SID columns consumed as model inputs by the non-text backbones of §7 (the in-house HSTU and encoder-decoder paths embed SID history themselves rather than receiving it inside tokenized text). SemanticIdFeature(IdFeature) reads a precomputed per-item code list with fixed value_dim = num_levels and fused table geometry num_embeddings = num_levels × codebook_size, using cumulative per-level offsets (id = level × 8192 + code, no text-vocab base — the layout proven in NVIDIA's SID-GR example [repo-recsysexamples]); SequenceSemanticIdFeature is its grouped sequence variant, flowing through the existing sparse_features KJT plus sequence_mulval_lengths transport with no Batch change. Both own their _fg_json (the in-tree MatchFeature precedent for features that build their own feature-generator config); in v1 that json declares only the standard id-feature keys because codes are joined offline at build time (§6.2) — a parse-time pyfg asset-map join is the deferred optimization tracked as R-FGJOIN (§15).

Pre-tokenized prompt columns (input_ids in Table 6.1; prompt_token_ids in the §12.9 example) need the opposite treatment: they must reach the model as raw jagged ids, and no TorchRec embedding table may exist for them, because the dense backbone owns the 176,384-row token embedding (§7). This is the transport-only contract, enforced on the model side rather than the parser side. DataParser parses the column like any sequence id feature into Batch.sparse_features (values/lengths, ids untouched), but each generative model family declares which of its feature groups it consumes as raw token ids (for OpenOneRec, the prompt group), and BaseGenerativeModel constructs its EmbeddingGroup only over the remaining groups: no EmbeddingConfig is generated for transport-only features, no EmbeddingCollection table is built, and the planner never sees a spurious 176,384-row shardable table (which would otherwise inflate the sharding plan and HBM budget, R-PLANNER, §15). The schema-required embedding_dim/num_buckets on such a feature are inert metadata, set only to satisfy the IdFeature schema.

6.2 Tokenization route: pre-tokenized offline, FG_NONE pass-through

Tokenization for the LLM path runs offline, in the data-build job, with the HF tokenizer asset — the vocab-expanded Qwen3 tokenizer produced by the expansion tool in §7 — and training tables carry integer ids. At train time the ids pass through FG_NONE untouched. This is a deliberate rejection of the in-tree alternative: TokenizeFeature (tzrec/features/tokenize_feature.py) does run BPE/sentencepiece inside pyfg C++ against a HF tokenizer.json vocab, and tokens_as_sequence=true emits a jagged token sequence, but it is insufficient for chat-LLM training today on three counts. First, there is no chat-template engine: Qwen3's template is a Jinja program over a messages structure [transformers], and pyfg offers only a text normalizer. Second, there is no special-token control — no BOS/EOS policy, no <|im_start|> role scaffolding, no added-token registration — while correctness depends on exact specials placement. Third, byte-level parity between pyfg's BPE and the HF Qwen3 tokenizer (byte-level pretokenizer plus added-token semantics) is unverified; any divergence silently shifts every token id. A parse-time C++ tokenize-or-join path remains a future optimization (R-FGJOIN, §15).

Chat-template handling is therefore a builder responsibility. The builder applies apply_chat_template per row, derives loss_mask from message roles at build time (no train-time pattern matching of <|im_start|>assistant token spans, which is what OpenOneRec's loader does at runtime), and implements the two Qwen3-specific conventions OpenOneRec validated: the think-mode suffix rule (add_think_pattern: an assistant turn containing a non-empty <think> block appends /think to its user turn; otherwise /no_think is appended and an empty <think>\n</think>\n is prepended to the response), and the terminator convention that the appended end-of-document token is <|endoftext|> (id 151643, the pad token) rather than <|im_end|>, loss-masked [repo-openonerec].

Itemic tokens never pass through string tokenization in the hot path. The builder joins item_id → SID from the catalog asset of §5 (the SidCatalogIndex; for many-to-many catalogs the primary path renders histories and the full path set populates ground-truth columns, D4) and emits ids arithmetically: id = 151669 + level*8192 + code, with the full itemic block at itemic_id_range = [151669, 176245) and the boundary specials <|sid_begin|> = 176245, <|sid_end|> = 176246 (§7 owns the vocab math). The surface form <|sid_begin|><s_a_i><s_b_j><s_c_k><|sid_end|> exists in the messages column for provenance and for verl prompts, where rendered strings are required. Because added tokens map 1:1 to single ids, arithmetic emission and string tokenization must agree exactly; the builder asserts a round-trip (render → tokenize → ids) over a sample of the catalog at startup and fails loudly on any mismatch — a sub-word split of an itemic token is the classic silent-zero failure mode observed downstream in the MiniOneRec lineage [repo-minionerec]. One tokenizer asset is shared by builder, trainer, rollout, and serving.

6.3 InstructionTaskBuilder

InstructionTaskBuilder (tzrec/datasets/gr/instruction.py) is a registry of task generators that renders interaction logs and catalog text into family-2 rows. It consolidates three proven template suites plus two reasoning-data recipes:

For OneRec-Think-style rationale bootstrapping the builder includes the context-pruning synthesis job: retrieve the top-k (k=10) target-relevant history items by item-embedding cosine, prompt the stage-1-aligned checkpoint itself to write a rationale over the pruned context, then emit the SFT sample conditioned on the full noisy history with loss on both rationale and target SIDs [onerec-think]. Both reasoning recipes are generation-heavy offline jobs, not dataloader transforms; they write ordinary family-2 rows.

Three sampling rules are builder policy rather than template content. First, the one-sampled-template-per-datum rule: each datum draws exactly one paraphrase variant per epoch (LC-Rec's train_prompt_sample_num 1,...,1 with a random prompt id per __getitem__), which prevents overfitting to a single phrasing and is what makes the per-paraphrase mean/min/max eval protocol of §13 meaningful [repo-lcrec]. Second, sliding-window augmentation: each user sequence expands into contiguous sub-sequence→next-item pairs (with a per-user sample cap), which GRID's ablation identifies as the single biggest data-side lever — Beauty Recall@10 0.0597 with versus 0.0447 without, Toys 0.0548 versus 0.0442 [grid]. The builder implements it as enumerate-then-subsample (GRID's collate enumerates all contiguous sub-sequences of length ≥2 and samples a budget per batch). Third, the reward-weighted sampling knob (D14, §9): following PLUM, rows are sampled into the training set with probability proportional to a handcrafted engagement reward and then weighted equally, rather than loss-weighted — PLUM's production-proven, near-zero-cost alignment mechanism [plum]. The sample_weight column carries the raw reward; the sampling pass materializes the resampled table so training-time readers stay unchanged.

6.4 Co-pretrain mixtures: explicit ratio control

Every successful recipe in the survey treats the general-text/recommendation mixture as a load-bearing hyperparameter, yet none of the open implementations gives it a first-class control surface. OpenOneRec literally pd.concats the general and recommendation tables and inherits whatever sample-count ratio results — 62.34% general at token level in the open variant, shifting to 53/47 for Pro — with re-sharding into 1000-row Parquet files as the only mixing mechanism [openonerec][repo-openonerec]. OneReason budgets ≈28% general (26.85% text + 1.59% multimodal) and, within the recommendation share, deliberate four-granularity weights (token 2.50%, item 31.84%, relational 3.18%, user 34.05%) [onereason]. OneRec-Think balances four alignment tasks at 65.73/24.30/4.94/5.03 [onerec-think]. PLUM uses an example-level 50/50 split of behavior versus item↔text grounding for the domain corpus and leaves the general ratio undisclosed [plum]. The spread (5%–62% general) and PLUM's evidence that CPT is the highest-leverage stage (D6) mean tzrec must make ratios reproducible config, not an artifact of table sizes.

Design rule (section-local, no D-number) — mixture ratios are explicit config via a weighted multi-source reader

A new MixtureReader (tzrec/datasets/gr/mixture.py) wraps N inner readers, each declared as (input_path, reader options, weight). Today one dataloader binds one reader with one schema (comma-separated input_path shares a single schema, odps_dataset.py:424-438), so heterogeneous-corpus interleaving is impossible without offline union tables. MixtureReader samples the next micro-batch's source from the weight vector (seeded per worker for reproducibility), requires only that sources agree on the GR columns of Table 6.1 family 1, and reports realized per-source token counts so drift between configured and realized token-level ratios is visible in metrics. Checkpoint state composes naturally: dataloader state is already a per-source-key dict ({source_key: row}), so the union over inner readers needs no new resume machinery. Per-source weights live in the stage config (§12), giving each D6 stage its own mixture without re-materializing data.

Because weights are enforced at the sampler, mixture changes never require rebuilding corpora — the practical failure mode of the concat approach, where reweighting means re-running the join and re-sharding tens of billions of tokens. The loss-masked context-span convention of §6.1 composes with mixing: per-source loss_mask policies (segments-style full-loss, chat-style response-only, OneReason-style masked context) are fixed at build time, so the train loop applies one uniform masked-CE regardless of source.

6.5 SequencePacker: 32K packing with cu_seqlens

SequencePacker (tzrec/datasets/gr/packing.py) is an in-worker transform that coalesces parsed, variable-length documents into fixed-budget packed sequences for the dense LLM stages. It implements the OpenOneRec recipe: greedy sequential packing into a pack_max_length budget (32768 default; OneReason packs 48K with per-sample caps that widen 4K→32K by stage, so both knobs are config [onereason]), boundary-sample truncation to fill the tail, padding to an 8-multiple, and a trailing pad segment appended to cu_seqlens [repo-openonerec]. Per packed batch it emits flat input_ids, cu_seqlens (int32 prefix sums — one torch.ops.fbgemm.asynchronous_complete_cumsum over lengths, the same op the HSTU path uses; total tokens must stay below 2^31), position_ids reset to zero at every sample boundary so RoPE is per-document, the packed loss_mask, the itemic_id_mask (id ∈ [151669, 176245), driving the itemic-versus-text loss split of §8), and sample_idx for per-source loss metrics. The existing cost-based batching primitive (sample_cost_field + batch_cost_size, dataset.py:567-585) is the upstream token-budget seed — with cost = token count it already does greedy budget slicing — and the packer adds what it lacks: attention-isolation metadata and intra-pack layout.

Interaction with Batch is deliberately conservative. The packed tensors ride batch.additional_infos (Dict[str, Tensor]), which is already threaded through all four transport methods; any future dedicated field must be added to to(device), pin_memory(), record_stream(), and to_dict() together (utils.py:344-512) or it silently drops on the overlapped H2D copy — the single most common way to corrupt a Pipelineable extension. The packed path stays jagged end to end: one packed sequence per GPU per step (OpenOneRec's batch_size=1 StatefulDataLoader convention, i.e. a 32K-token global batch per rank [repo-openonerec]), and never a to_padded_dense materialization — [B, 32768, d] dense activations are both an OOM and a contract violation.

Packed batches are only valid under a varlen attention backend (guard owned by §8, D6)

cu_seqlens only isolates documents if the attention kernel honors it; SDPA and eager attention silently cross-attend across packed documents. The canonical guard lives with the trainer — §8 fails loud at startup when packing is enabled without an FA/TE varlen backend — and the packer merely re-asserts that invariant at runtime (defense in depth, not a second policy). The packer-owned obligation is the boundary-label invariant, inherited from OpenOneRec: the shifted label at a document's last position points into the next document, but that position's terminator is loss-masked by construction, so no cross-document leakage enters the loss — preserving this invariant is a packer unit-test obligation, not an accident.

6.6 Streaming and RL data

GBPO real-feedback streaming. OneRec-V2's logged-feedback RL needs no rollout engine (D10/D11): the trainer consumes the exposure stream directly, so its data path is the tzrec-native one. The KafkaReader is reused unchanged — each message a serialized Arrow IPC RecordBatch, per-row offsets resumed via consumer-group on_assign, and per-row event timestamps popped into Batch.data_timestamp (kafka_dataset.py:128-485, dataset.py:319-344), which already drives event-time checkpointing on master (PR #541). The upstream join (exposure log × realized feedback → family-4 rows) is a streaming ETL outside tzrec; the contract it must honor is that serving logs per-token generation probabilities at exposure time (§10) so gen_prob_logged can populate pi_old; serving emits the post-mask (renormalized) per-token probabilities of the constrained distribution actually decoded from (§10.6 — sid-gr-inference's stock response carries pre-mask logprobs, so the fork changes this). Rows without the column fall back to ratio = 1, the regime GBPO is explicitly designed to keep stable [onerec-v2]. For streaming corpora generally, the re-tokenization cadence when the SID catalog rotates is tracked as R-RETOK (§15): family-1/2 tables embed itemic ids, so a tokenizer or catalog change invalidates built data, and the builder stamps every table with the tokenizer-asset and catalog-version ids to make staleness detectable.

RL prompt and distillation Parquet. Builders write family-3/5 rows through the existing writers (OdpsWriter or Parquet); a thin snapshot job materializes Parquet files for verl, since verl's dataset stack is file-based and runs in its own environment (R-ENV). Ground truth is carried twice deliberately: reward_model.ground_truth holds rendered SID strings for the regex-style rule rewards (OpenOneRec's compute_score parses <s_a_(\d+)><s_b_(\d+)><s_c_(\d+)>), while extra_info holds structured code arrays and item ids for catalog-aware rewards and the SID→item expansion of D4. Multi-target ground-truth sets are first-class from day one — OneReason's R3 targets average up to ~14 items per user in the video domain [onereason].

Group expansion for GRPO. The dataset emits exactly one row per prompt; group expansion is the rollout's job. The load-bearing ordering, validated by OpenOneRec's trainer, is that the group uid is assigned to the prompt before beam expansion, so all K beams (or N×K trace×completion rollouts) of a prompt share one GRPO group, with the batch repeated to align afterwards [repo-openonerec]. Pre-expanding rows in the data job would break both grouping and verl's dynamic batching; where an offline dataset already carries G logged responses per prompt, the row-expansion precedent is the TDM/sampler machinery (expand_tdm_sample, utils.py:515-585), not the builder.

Negative-sampler reuse. The generative CE objective needs no explicit negatives — the softmax over the itemic block is the implicit all-negative contrast, as in LC-Rec and MiniOneRec [lcrec][minionerec] — so the GraphLearn cluster is not on the GR hot path. It remains useful at the edges: fetching item attributes by id for eval candidate expansion, mining hard negatives for label_pred-style discrimination prompts, and supplying co-occurrence pairs for the tokenizer's optional InfoNCE auxiliary loss (D1, §5) where an ODPS SQL job is not already in place. Its row-expansion model (sample ids, fetch attrs, append rows) carries over unchanged; note it is string-attr-centric and CPU-side, so it should not be put inside the packed-path token budget.

7. Model layer: backbones, vocabulary, and context injection

This section specifies the dense half of the two-world architecture: the BaseGenerativeModel contract that plugs generative recommenders into tzrec's existing registry/loss/metric machinery, the four backbone modules behind decision D5, the corrected itemic-vocabulary contract that every other section consumes, the prediction-head and decode-strategy abstractions that keep the design level-count- and decode-strategy-agnostic, and the context-injection bridge from TorchRec sparse embeddings into the LLM (R-INJECT). Everything here is a module-layer specification; training-loop mechanics live in §8 and §11, and decode/serving in §10.

7.1 BaseGenerativeModel(BaseModel) contract

Generative models are ordinary tzrec models. A new abstract base, BaseGenerativeModel(BaseModel) in tzrec/models/generative/, follows the same registry pattern as BaseSidModel (tzrec/models/sid_model.py, PR #538): the metaclass from tzrec/utils/load_class.py:117-145 auto-registers every subclass under its class name, and tzrec/main.py:_create_model resolves the class from the protobuf oneof model arm via config_util.which_msg (config_util.py:73-75). The concrete classes and their proto tags are fixed by this RFC: OpenOneRec=600, OneRecV1=601, OneRecV2=602, ALGR=603, LCRec=604 (message name == class name; see §12 for the proto surface). MiniOneRec is a config preset over LCRec + GRPO, never a class. One registration caveat from the baseline study: auto_import scans tzrec/models/*.py non-recursively (load_class.py:53-99), so classes under tzrec/models/generative/ must be re-exported from a module directly under tzrec/models/ (or auto_import gains the subpackage — a one-line change owned by §12).

The base contract mirrors BaseModel's three surfaces with generative-specific semantics:

  • predict(batch) -> Dict[str, Tensor]. In training/eval mode this is the teacher-forced scoring pass: assemble inputs (token ids and/or injected context embeddings, §7.5), run the backbone, emit per-level logits through SidPredictionHead (§7.4), and compute the token-level cross-entropy inside predict, passing loss tensors through the predictions dict. This adopts the BaseSidModel convention ("SID models compute their losses internally and pass them through predictions", sid_model.py:91-97 init_loss docstring) rather than the RankModel scaffolding, which assumes per-row scalar labels and cannot express jagged per-token CE.
  • loss(predictions, batch) selects the loss tensors from predict's output and applies global token normalization. Per-rank token counts vary under packing; the established pattern is DlrmHSTU.enable_global_average_loss (dlrm_hstu.py:64-69, 280-284): all-reduce the global valid-token count and weight the local sum accordingly, which is also exactly the HF Trainer num_items_in_batch fix replicated in §8.
  • Labels. Target SID token ids arrive as batch.jagged_labels["response_sid_ids"] and the response mask as a separate batch.jagged_labels["loss_mask"]. The mask is never smuggled through JaggedTensor.weights() — weights remain reserved for sample weighting. Both are jagged over response tokens; the model never calls to_padded_dense on the packed path.
  • Generated outputs. In offline predict/eval mode, generation runs through the host-side BaseSidDecoder driver in eager mode (the decode loop is never traced into a graph; §10). The model emits one prediction row per generated candidate — jagged over the top-K beam results — and sets predictions[TARGET_REPEAT_INTERLEAVE_KEY] (tzrec/constant.py:40) to the per-user candidate count, exactly the DlrmHSTU variable-output convention. PredictWrapper passes the key through unconditionally (model.py:332-335) and the predict writer already expands per-user reserved columns by it, so beam outputs need zero writer changes.
  • init_metric registers generative-retrieval metrics (HitRate@K / Recall@K over decoded item ids after SID→item expansion; §13 owns definitions). Because metrics consume item ids, swapping decoders (§7.4) is metric-transparent.

7.2 Backbones (D5)

Decision D5 — four backbones behind one interface; decode strategy owned by the decoder, not the model

v1 ships four backbone modules under tzrec/modules/llm/, all implementing a common GRBackbone forward contract (inputs: token ids and/or context embeddings plus packing metadata; output: final hidden states): Qwen3GRBackbone (HF weights, FSDP2 default or Megatron-core scale-out), LazyDecoderGRBackbone (OneRec-V2), EncoderDecoderGRBackbone (OneRec-V1), and InhouseHstuGRBackbone (export-clean, reuses STUStack). Two abstractions keep the model layer decode-agnostic: SidPredictionHead emits per-level logits (never hard-coding 3 levels), and BaseSidDecoder owns the number of scorer invocations (3 AR steps, 1 RPG-style parallel pass, or T diffusion steps). No interface in this section may assume AR-only decoding or num_decode_steps == num_sid_levels (see D15/D16 in §7.4).

Qwen3GRBackbone

The Qwen3 architecture is GQA with a signature twist: per-head-dim RMSNorm on q and k (q_norm = Qwen3RMSNorm(head_dim), applied over the last axis before RoPE; modeling_qwen3.py:222-291 [transformers]). head_dim is an explicit config field (128 for every released size), not derived from hidden_size/num_heads. SwiGLU MLPs, RMSNorm eps 1e-6, rope theta 1e6, full attention on all layers in released checkpoints. Embedding tying is size-dependent and load-bearing for §7.3: Qwen3-0.6B/1.7B/4B ship tie_word_embeddings: true; 8B/14B/32B ship false. The HF vocab is 151,936 rows of which only 151,669 are used tokens. One memory lever worth wiring from day one: Qwen3ForCausalLM.forward(logits_to_keep=...) slices hidden states before the lm_head (modeling_qwen3.py:467), which matters greatly at a 176K-row vocabulary; the training-side equivalent is the chunked-lm_head CE owned by §8.

Two construction routes, selected by the parallelism config (§11):

  • FSDP2 route (default). from_pretrained loads the vocab-expanded checkpoint directly (meta-init + sharded safetensors), then fully_shard is applied per Qwen3DecoderLayer and root — the layering OpenOneRec uses in production (pretrain/onerec_llm/training/distributed.py [repo-openonerec]). tzrec pins the transformers 4.53 line (OpenOneRec's pin, and the version the AOTI export proof in §10 was validated on); the v5 line renames torch_dtype→dtype and nests rope_theta under rope_parameters, so version assumptions must not leak across the pin.
  • mcore route. Qwen3-dense maps onto a stock TransformerConfig with no custom modules — verified field-by-field in the research notes: num_query_groups = num_key_value_heads, kv_channels = head_dim (decoupled from hidden size, exactly Qwen3's convention), qk_layernorm=True (the layer spec's q_layernorm/k_layernorm = TENorm(kv_channels) reproduces per-head q/k RMSNorm), gated_linear_unit=True + SiLU, normalization="RMSNorm", add_bias_linear=False; GPTModel(position_embedding_type="rope", rotary_base=1e6, share_embeddings_and_output_weights=tie_word_embeddings) [megatron]. The same mapping is what verl's converter applies (qk_layernorm=True if "Qwen3" in architectures, verl/models/mcore/config_converter.py:167-184 [verl]). Weight conversion goes through mbridge's AutoBridge.from_config(hf_config, dtype) [mbridge], which handles the q/k/v→fused linear_qkv interleave and gate+up→linear_fc1 concat, and — critically for §7.3 — builds the model at exactly the HF vocab size with no extra padding.

LazyDecoderGRBackbone (OneRec-V2)

OneRec-V2's lazy decoder removes the encoder entirely: in V1, context encoding consumed 97.66% of FLOPs at context length 512 while the loss-bearing target decode was 2.34% [onerec-v2]. The lazy design embeds the heterogeneous context (user profile + short/long behavior pathways) once in a Context Processor, projects every context position to a single dimension d_context = S_kv · L_kv · G_kv · d_head, partitions the feature dimension into S_kv·L_kv chunks of G_kv·d_head each, and uses RMSNorm'ed chunks directly as cross-attention K/V: k_l = RMSNorm(C_{l·S_kv}), with v_l = k_l when S_kv=1. There are no learned W_k/W_v matrices over the context anywhere in the decoder — the only per-layer learned pieces on the context path are RMSNorm gains. Layer l selects its KV slice via l_kv = floor(l · L_kv / N_layer); since the paper's ablations show the aggressive settings are loss-neutral (loss flat at 3.27 across (L_kv, S_kv); MQA G_kv=1 shrinks context KV 13×), tzrec defaults to L_kv=1, S_kv=1, G_kv=1. All transformer compute (pre-norm blocks ordered cross-attn → causal self-attn → FFN/MoE) runs only on the 3-token target [BOS, s1, s2]; output is 3 position-specific RMSNorm+Linear heads over 8192-way softmaxes with loss = mean of the 3 CEs (V1 used the sum — loss values are not comparable across versions). The result: 18.89 GFLOPs/sample at 1B, independent of context length, versus 634.8 for a naive decoder-only at near-identical loss (the naive baseline was trained only at 0.1B, where the comparison is 3.54 vs 3.57). Scaling anchors for config validation: 1B = d_model 1792, 18 layers, 14 heads, SID embedding dim 90. The Context Processor maps onto tzrec feature groups via per-pathway projections; data-side assembly, including request-grouped "common context compression", is owned by §6.

EncoderDecoderGRBackbone (OneRec-V1)

The V1 architecture is a pre-norm encoder over a four-pathway context — user static (1 token), short-term (20 interactions), positive-feedback (256), and lifelong (up to 100K items compressed TWIN-V2-style, then QFormer'ed to 128 query tokens) — concatenated to a 405-token encoder input with learned positions, each pathway ending in a Dense(LeakyReLU(Dense)) projection to d_model [onerec-v1]. The decoder interleaves causal self-attention, cross-attention to the encoder output, and an MoE FFN (SwiGLU experts, top-k routing, DeepSeek-V3 aux-loss-free balancing — worth −0.2 loss in their ablation); NTP loss is the sum of the 3 token CEs. Reference family points: 0.935B = 8 layers, d_model 1024, 24 experts top-2 in the decoder only; 2.633B = 24 layers, 24/4 experts in both encoder and decoder. Construction routes: mcore T5Model (which takes separate encoder/decoder configs and layer specs, t5_model.py:136-159 [megatron]; PP=1 keeps the legacy encoder/decoder pipeline-placement scheme out of play) or an in-house pre-norm module set when mcore is not in the build. The pathway feature engineering is §6 territory; rollouts for this backbone use the tzrec-native in-process beam loop since no serving engine speaks this architecture (D11, §9).

InhouseHstuGRBackbone

The fourth backbone needs no new dependencies: STUStack (tzrec/modules/gr/stu.py) is already a causal, target-aware, jagged-native transformer with per-layer KV caching and an incremental-decode path (cached_forward over delta_hstu_mha, hstu_attention.py:199-267) whose uniform per-row delta-size constraint matches stepwise SID decode exactly (each beam row appends one token per step). SID history enters as a JAGGED_SEQUENCE feature group; per-level codebooks are fused into one embedding table with cumulative per-level id offsets, the layout proven in NVIDIA's SID-GR example (sid_sequence_dataset.py:95-141 [repo-recsysexamples]), with a learnable BOS embedding parameter. With a fused vocab of 24,576 the head is small enough that no vocab parallelism is ever needed. This backbone is the export-clean lane: DlrmHSTU already exports through tzrec's AOTI path, and the same machinery carries the single-step decode scorer of §10 (D13). Its limits are inherited and documented: the delta path falls back from CUTLASS to Triton, and beam forks require host-side re-indexing of the per-layer k_cache/v_cache plus offsets (a small helper this RFC adds; §10).

7.3 Itemic vocabulary: the corrected contract

Revision — vocab padding applies to the total, not the itemic block (supersedes prior spec §4.7.4)

The prior internal spec padded the itemic block to a TP-divisible size (24,578 → 24,832). That formula is wrong and is retired: padding must apply to the total vocabulary, matching OpenOneRec's tool (expand_qwen3_vocab.py:169: align(151669 + 24578, 256) = 176384 [repo-openonerec]) and required for exact HF↔mcore shape parity (below). Padding the block instead of the total would shift every special-token id, break the id = 151669 + level·8192 + code arithmetic, and desynchronize tzrec checkpoints from the OpenOneRec-compatible layout. All sections of this RFC use the numbers in the table below.

Token ordering is load-bearing. Itemic tokens are added in strict layer-major order — all <s_a_0>..<s_a_8191>, then <s_b_*>, then <s_c_*>, then the two specials last — so that id = 151669 + level·8192 + code holds; OpenOneRec hard-codes this arithmetic in at least four places across training, RL metrics, and distillation configs [repo-openonerec], and tzrec's RL legality and loss-split code relies on it too. A rendered SID is the string <|sid_begin|><s_a_{c0}><s_b_{c1}><s_c_{c2}><|sid_end|>. The tokens are added as ordinary added tokens, not special tokens, so they survive skip_special_tokens=True decoding. The vocab-expansion tool — python -m tzrec.gr.extend_vocab, the stage-3 lifecycle entry point of §4 (offline single-process, deliberately not one of the §12 CLI verbs) — asserts padded_vocab % 256 == 0; the training driver asserts vocab % tp == 0 at startup — fail loud, no silent re-padding.

New-row initialization and the 267-row gotcha

Initialization uses HF resize_token_embeddings(new_num_tokens=176384, mean_resizing=True). The exact mechanics (modeling_utils.py:3071-3094 [transformers]): cast the old embedding to fp32, compute the row mean μ and covariance Σ = (W−μ)ᵀ(W−μ)/N, then sample new rows from MultivariateNormal(μ, 1e-9·Σ) — i.e. the mean plus tiny correlated noise (the Hewitt vocab-expansion trick, which keeps pre/post-resize next-token KL small) — falling back to plain mean-fill if Σ is not positive-definite. The lm_head receives the same treatment, and tie_weights() re-ties afterwards so for tied checkpoints the input-embedding init is what survives. Because the algorithm needs the full unsharded [V, H] matrix in fp32, the resize runs offline, in the HF representation, before any bridge conversion or sharding — OpenOneRec loads the model with torch_dtype=torch.float32 explicitly for this reason, and both mbridge and Megatron-Bridge then copy row values verbatim into the TP-sharded model.

Gotcha (R-VOCAB-INIT) — 267 itemic tokens land on stale dummy rows

resize_token_embeddings(mean_resizing=True) initializes only rows ≥ old_embedding_rows = 151936. The first 267 itemic tokens occupy ids [151669, 151936) — Qwen's shipped dummy rows — and therefore keep Qwen's untrained values after the resize. OpenOneRec shipped this bug and silently trained through it [repo-openonerec]. tzrec's expansion tool must explicitly re-initialize rows [len(tokenizer_before_add), old_embedding_rows) from the same N(μ, 1e-9·Σ) distribution, in both the input embedding and (for untied models) the lm_head, and the tool's self-check must verify that no row in the itemic range is bit-identical to the pre-resize dummy rows.

Two initialization options layer on top. First, codebook-vector init: after the mean-resize, itemic rows may be overwritten with embeddings derived from the RQ codebook centroids (a plain in-place write of get_input_embeddings().weight.data[151669:176245]). Since centroids live in the item-embedding space (e.g. 4096-d Qwen3-Embedding-8B vectors for the OpenOneRec-compatible tokenizer) and generally do not match the LLM hidden size, this requires a projection or truncation step; it ships as an off-by-default option — the OpenOneRec evidence is that plain mean init trains fine through stage 1. Second, per-domain begin tokens: reasoning/multi-domain recipes use domain-specific begin markers (e.g. separate begin tokens per vertical) in the OneReason style [onereason]; tzrec supports an optional list of extra special tokens appended after <|sid_end|>. The 137 alignment-dummy rows absorb small additions without changing the padded size; if the total ever exceeds 176,384, the tool re-aligns to the next multiple of 256 and the new padded size propagates everywhere (it is computed, never hard-coded).

Tied-weight handling per model size. For tied checkpoints (0.6B–4B) there is one logical [V, H] matrix: the resize path updates the input embedding and re-ties; under mcore with PP=1 the output layer is built with skip_weight_param_allocation and shares the embedding Parameter, so row-level operations (init, freezing) apply exactly once. For untied checkpoints (8B+), embed_tokens and lm_head are independent [V, H] matrices with identical row geometry: both receive the 24,448 resize-initialized rows covering the 24,578 added tokens, both need the 267-row re-init for the tokens landing on pre-existing dummy rows, and both must be covered by the stage-1 freezing recipe (OpenOneRec's masker matches both by name).

TP sharding compatibility

Megatron's VocabParallelEmbedding splits rows via VocabUtility.vocab_range_from_global_vocab_size, whose divide() hard-asserts padded_vocab % tp == 0 (tensor_parallel/utils.py:113-121 [megatron]). 176,384 = 2^8 · 689 satisfies this for every realistic TP degree, and — verified across both bridges — mbridge and Megatron-Bridge build GPTModel(vocab_size=176384) with no additional padding, so Megatron-side and HF-side shapes match exactly [mbridge] [megatron-bridge]. Do not route any tzrec flow through megatron.training's tokenizer-padding convention (ceil(v/(128·tp))·128·tp), which would yield 176,640 at tp=4 and 177,152 at tp=8 and desynchronize checkpoints. For stage-1 row freezing (D7, mechanics in §8), the per-rank frozen-row count follows from the module attributes vocab_start_index / num_embeddings_per_partition: n_frozen_local = clamp(151669 − vocab_start_index, 0, rows_per_rank).

7.4 Prediction heads and decode-strategy abstractions

SidPredictionHead is the single module boundary between backbone hidden states and per-level token distributions: it takes hidden states plus the level count m and emits per-level logits — conceptually [B, m, vocab_per_level], concretely a list of per-level logit tensors so unequal per-level codebook sizes stay legal. Loss code consumes the list, never the literal 3. Three implementations ship in v1:

  • Shared-lm_head slice (Qwen3 path): the head is the stock 176K-row lm_head; "per-level logits" are views of the itemic block [151669 + l·8192, 151669 + (l+1)·8192). Training CE for this path runs over the full vocabulary (the response tokens are itemic, but the model must keep competing them against text tokens), so this implementation is a thin indexing adapter used by decoding and metrics rather than by the SFT loss.
  • Position-specific heads (OneRec-V2): m independent RMSNorm+Linear heads over per-level codebooks, teacher-forced at the m target positions [onerec-v2].
  • Per-hierarchy heads over a fused vocab (in-house path): either one shared linear over ΣC_l with per-level slicing or a ModuleList of per-level linears — both layouts proven in the GRID-tokenized SID-GR example [grid] [repo-recsysexamples].

BaseSidDecoder owns generation: generate(scorer, n_return) -> item ids, where the decoder — not the model — decides how many scorer invocations occur. ConstrainedBeamDecoder (3 AR steps over the CSR trie, D12) is the v1 default; an unconstrained variant with post-grounding is a supported config (§10). The abstraction exists because the literature has diverged from one-token-per-step AR: RPG predicts all m unordered PQ digits from one forward and walks a precomputed item-similarity graph (+12.6% NDCG@10, ≈15× decode speedup, constant in catalog size [rpg]); LLaDA-Rec denoises SIDs in T ≤ m diffusion steps [llada-rec]. A future GraphPropagationDecoder (1 invocation) and a diffusion decoder (T invocations) are peers behind the same interface; because metrics consume item ids (§7.1), decoder swaps are metric-transparent. Decode-strategy-specific artifacts (trie, similarity graph) are optional members of the export bundle (§10). At serving time these two abstractions map directly onto the runtime's BackboneAdapter contract (§10.6) — encode_context, the step-aware decode_step, and per-level head/vocab-layout metadata — keeping the training-side and serving-side abstractions aligned.

Decision D15 — MTP auxiliary loss as a v1.5 flag; parallel/speculative/diffusion decode as future work the interfaces must not preclude

DeepSeek-V3-style MTP — D sequential modules, each a shared embedding/output-head plus one transformer block and a d×2d projection, with CE weighted λ/D (λ = 0.3 early, 0.1 late) — is a pure training regularizer that is discarded at inference and showed consistent benchmark gains at identical inference cost in DeepSeek's ablations [dsv3]. On the mcore route this is config-only (TransformerConfig.mtp_num_layers / mtp_loss_scaling_factor + get_gpt_mtp_block_spec [megatron]), so tzrec exposes it as an optional v1.5 flag. MTP-based speculative beam decoding is explicitly out of v1 scope: with only 3 SID levels the ceiling is ~1.5× (3 sequential calls → 2) and AtSpeed's own N-to-K acceptance degrades past K=20 beams [atspeed] — at L=3 the structurally better answer to decode cost is RPG-style parallel decode, which D5's interfaces already accommodate (§14 for roadmap placement).

Latte latent tokens (D16, model-layer slice). Latte breaks the AR decoding tree's probability coupling by prepending one of M′ latent tokens (uniformly sampled per example) to the target SID only — never to the history — and aggregating item scores over latent paths at inference (sum or max; near-identical) [latte]. Model-layer cost is M′ extra vocabulary rows (M′ ∈ {4, 8}; fits the spare-row headroom on the Qwen3 path, or M′ extra fused-vocab rows in-house), beam step-0 restricted to the latent vocabulary, and item-level score aggregation across beams — the same multi-path item scoring that SidCatalogIndex already provides for collision buckets (D4, §5). The data-side hook (sampled virtual token column) is one line in §6; the +3.45% avg NDCG@10 evidence makes this a cheap optional extension rather than a default.

7.5 Context injection (R-INJECT)

The two-world bridge moves TorchRec-computed context embeddings (user profile, behavior-sequence side features) into the dense LLM as inputs. The injection surface differs per backbone, which is why R-INJECT concentrates on the Qwen3 path: LazyDecoderGRBackbone and EncoderDecoderGRBackbone consume context embeddings natively (the context K/V and encoder pathways are the architecture), and InhouseHstuGRBackbone reads jagged grouped features through the existing EmbeddingGroup contract. For Qwen3 the mechanisms are:

  • HF route — inputs_embeds. Qwen3Model.forward enforces a strict XOR between input_ids and inputs_embeds (modeling_qwen3.py:388-389 [transformers]), and — verified in the notes — RoPE consumes position_ids only, never token ids, so an embeds-only forward is fully supported. Two generation caveats are inherited: generate() uses inputs_embeds only on the first iteration before switching to generated ids, and the attention mask cannot be inferred from embeds, so it is always passed explicitly.
  • mcore route — decoder_input. GPTModel.forward(decoder_input=...) bypasses the embedding lookup and accepts pre-computed hidden inputs in mcore's internal [s, b, h] layout (gpt_model.py:479-507 [megatron]); partial injection computes the token embeddings first, scatters context rows, then passes the result.

Two placement strategies are supported, selected per feature group in config (§12):

  • Itemic-slot overwrite (default for text+SID SFT). The prompt template reserves placeholder token positions (e.g. a context slot adjacent to each history SID, or fixed user-profile slots); the model computes embed_tokens(input_ids) and index_copy_s projected context vectors into those rows. Token count is unchanged, so packing math, cu_seqlens, loss masks, and the chat template all survive untouched — the property that makes this composable with the 32K packed path of §8.
  • Prefix prepend (soft prompt). N_ctx projected vectors are prepended to the sequence; positions shift by N_ctx and the injected positions are excluded from loss via the mask. This is the natural form for prompt-light rec-only models and is the degenerate case of V1/V2's "context first, target last" layout.

Bridge conversions are mechanical but specified once: sparse lookups emit fp32 (tzrec's embedding output dtype; recsys-examples likewise pins FP32 qcomms), cast to bf16 at the bridge; jagged [sum_len, D] values are gathered directly into flat packed positions — never via to_padded_dense on the packed path; and the projection MLP adapting embedding dims to the LLM hidden size belongs to the dense model, so its parameters live in the dense optimizer tree (§11). When TP is enabled (mcore lane), the DMP↔TP impedance mismatch is solved at the activation level, exactly as recsys-examples does: each TP rank performs the sparse lookup on its own micro-batch, then applies GradScale × tp (compensating TorchRec's division of model-parallel embedding grads by tp×dp when only dp is wanted) followed by an autograd-aware jagged allgather over the TP group, with labels allgathered via dmp_batch_to_tp (commons/distributed/dmp_to_tp.py:15-68 [repo-recsysexamples]).

R-INJECT is an explicit proof-obligation task, not an assumed-working hook

The first milestone of the model-layer work item is a test battery, run before any full training: (1) gradient reach — a probe embedding table injected via inputs_embeds must show its fused in-backward optimizer firing (parameter delta after one step), proving autograd crosses the bridge; (2) packing equivalence — injected forward over a packed batch is bit-comparable to the same samples unpacked; (3) generate parity — first-step inputs_embeds + subsequent token-id steps reproduce teacher-forced logits on the same prefix; (4) TP equivalence (mcore lane only) — GradScale + allgather matches the tp=1 reference. Sequencing of the two optimizer trees around this bridge is R-GRAD/R-DMP territory, owned by §11; the consolidated register is in §15.

7.6 MoE

MoE matters here for two reasons: OneRec-V1/V2 both report MoE as the parameter-scaling lever (V1: 24-expert top-2 decoder MoE at 0.935B; V2: 53 routed + 1 shared experts, top-3, replacing dense FFNs after the first 2 blocks — its 4B-total/0.5B-active MoE beats a 2B dense model at 0.5B-dense compute [onerec-v1] [onerec-v2]), and both adopt DeepSeek-V3's aux-loss-free load balancing [dsv3]. On the mcore route this entire design space is configuration, not code: TransformerConfig exposes num_moe_experts, moe_router_topk, moe_ffn_hidden_size, moe_shared_expert_intermediate_size, moe_grouped_gemm, moe_token_dispatcher_type (allgather/alltoall/flex), moe_router_load_balancing_type ∈ {aux_loss, seq_aux_loss, global_aux_loss, sinkhorn, none}, and the aux-loss-free pair moe_router_enable_expert_bias + moe_router_bias_update_rate=1e-3 (transformer_config.py:604-822 [megatron]). tzrec's MoE presets default to the OneRec recipe (aux-loss-free balancing, alltoall dispatcher, grouped GEMM); one verl-sourced nuance is recorded for the RL stage: verl sets moe_router_load_balancing_type="none" during RL because the aux loss hurts policy optimization [verl]. Hard constraints to validate at config time: EP>1 requires num_moe_experts set, and TP>1 combined with EP>1 requires sequence_parallel=True.

On the HF side, Qwen3-MoE exists and is complete (Qwen3MoeForCausalLM: identical Qwen3 attention, top-k softmax router with optional renorm, 3D expert parameter layout, Switch-style aux loss via output_router_logits [transformers]), so the FSDP2 lane can fine-tune MoE checkpoints — but without expert parallelism or grouped-GEMM dispatch its efficiency ceiling is low. The decision boundary is therefore the one already drawn in D8: MoE and EP are the Megatron lane's reason to exist; expert-parallel sizing, placement, and its interaction with the TorchRec planner are owned by §11.

8. SFT and pretraining pipeline

This section specifies everything between "the backbone has an expanded itemic vocabulary" (§7) and "the policy is ready for RL" (§9): the staged curriculum (D6), the loss and masking engineering required by a 176,384-row vocabulary at 32K packed context, the backend-specific stage-1 row-freezing mechanics (D7), the integration of the dense training step into tzrec's existing loop, and checkpoint/resume semantics. The stages are not separate trainers: they are one launch (tzrec.gr.sft → tzrec/main_gr.py, §12) differing only in data mix, trainable-parameter set, learning rate, and loss mask, chained through checkpoints. The reference recipe is OpenOneRec's, which trains all pretrain/SFT stages through a single FSDP2 entry point differing only in flags [openonerec] [repo-openonerec].

8.1 Stage curriculum

Decision D6 — one staged curriculum: ITEMIC_TEXT_ALIGN → CO_PRETRAIN → MULTI_TASK_SFT → RSFT

SFT is a four-stage curriculum selected by a single stage enum, with CO_PRETRAIN as a first-class, non-optional stage (PLUM evidence below). All stages use response/loss-masked cross-entropy driven by batch.jagged_labels["loss_mask"], chunked lm_head CE (or vocab-parallel CE on the Megatron route) so the full-vocab logit tensor is never materialized, and sample packing to 32K via cu_seqlens varlen flash-attention. Stage chaining is by --fine_tune_checkpoint; no stage forks the trainer.

Stage 1 exists because freshly initialized itemic rows would otherwise inject a large representation shock into a full-parameter update; training only the new rows first lets them converge into the model's existing semantic space (OpenOneRec's ablation shows the alignment stage is most valuable for small models and for Label-Pred AUC at all sizes, e.g. 0.6488 vs 0.5807 at 0.6B [openonerec]; OneRec-Think uses the same freeze-then-integrate two-substage pattern at LR 5e-4/3e-4 [onerec-think]). Note that stage 1 keeps the full NTP loss over all tokens: a text-token target still produces useful gradient into itemic embedding rows that appear in the context, so the loss is not restricted to itemic targets — freezing, not masking, is what confines learning.

PLUM evidence — CPT outweighs base-LLM choice; reward enters via sampling

PLUM's 2×2 ablation at 900M-activated scale is the cleanest published evidence for making CO_PRETRAIN first-class: random init + CPT reaches Recall@10 = 0.27, beating Gemini init without CPT (0.23); LLM init adds only +0.01 on top of CPT (0.28). CPT also accelerates SFT convergence and amortizes across downstream tasks sharing one checkpoint [plum]. Budget consequence: CO_PRETRAIN receives the dominant token budget and is never skipped to save compute; swapping the base checkpoint is the lower-leverage knob. PLUM's second transferable lesson is reward-weighted SFT: sample training examples proportional to a handcrafted engagement reward, then weigh them equally — production-proven alignment at near-zero infra cost, surfaced here as a data-sampling knob on RSFT (reward composition itself is D14, §9).

Long-context variant (OneReason). The curriculum is a schedule, not a constant: OneReason's 578B-token pretrain runs three sub-stages — 110B tokens with only extended-vocab embeddings + LM head trainable (2e-4 → 1e-4), 449B full-parameter (1e-4 → 1e-5), then a 19B long-context anneal (1e-5 → 1e-6) — packed to 48K with a per-sample cap that grows from 4K to 32K in the final stage [onereason]. tzrec expresses this as chained CO_PRETRAIN runs with different pack_max_length/cap values; both are config fields, not constants (§12).

Anti-forgetting strategy

The in-stage defense is the general-domain mix: anti-forgetting is purely a data-distribution property of CO_PRETRAIN and MULTI_TASK_SFT, with no special loss term. OpenOneRec holds 62.34% general tokens through both pretrain stages and ~65% general samples through SFT; PLUM discloses no general-text ratio at all, defensible only because its product surface is retrieval-only — the general ratio should scale with whether language-facing tasks (captioning, reasoning, chat) are in product scope [plum]. The threat is well measured: LC-Rec's own (cut-from-camera-ready) measurement shows MMLU dropping 35.2 → 25.8 after rec fine-tuning [lcrec]; OneReason's reproduction has LC-Rec-8B SFT-only collapsing to 9.73 MMLU-Pro while the curriculum-trained model stays ≈72, on par with base Qwen3-8B [onereason]; and OpenOneRec's stage-evolution tables show SFT alone dropping AIME'24 0.79 → 0.51 and MMLU-Pro 0.72 → 0.53 [openonerec]. The mix is the prevention; the post-hoc repair is on-policy distillation from the base teacher (§9.5), which recovers most of the loss (AIME back to 0.71) at the cost of a transient dip in rec metrics that RL then recovers. The early-warning signals live in this section's diagnostics (text_token_loss, §8.2) and the general-benchmark sanity suite (§13).

Packing contract

All stages consume pre-packed samples: one (1, L ≤ pack_max_length) token row per rank per step, with cu_seqlens marking document boundaries and per-sample position_ids resets — OpenOneRec's greedy packer with boundary truncation, where the shifted label at a sample boundary is harmless because each sample's terminator is loss-masked [repo-openonerec]. The packer itself (SequencePacker) and the parquet/ODPS sample schema are owned by §6; this section's contract is only that the packed row is never densified per document and never padded per document.

Packing requires a varlen flash-attention backend — SDPA/eager silently cross-attend

cu_seqlens are FlashAttention varlen arguments: OpenOneRec's Qwen3 fork forwards them as cu_seq_lens_q/k + max_length_q/k into FA2 (modeling_qwen3.py:543-550); an SDPA or eager attention implementation ignores these kwargs and attends across packed document boundaries without any error, corrupting training while losses still go down. On the Megatron route the equivalent constraint is structural: THD PackedSeqParams require the TransformerEngine layer spec — the local DotProductAttention asserts packed_seq_params is None (megatron/core/transformer/dot_product_attention.py:153) [megatron]. The GR trainer therefore fails loud at startup if packing is enabled and the resolved attention implementation is not an FA/TE varlen backend; falling back silently is forbidden.

8.2 Loss engineering

Label contract. Supervision arrives as two jagged label streams built by the data layer (§6): batch.jagged_labels["response_sid_ids"] (target token ids) and the separate batch.jagged_labels["loss_mask"] — never an overload of .weights(). The model derives one further mask itself, because it owns the static vocab geometry (§7): itemic_id_mask = (labels >= 151669) & (labels < 176245), the single itemic_id_range shared with distillation masking and RL legality. The objective is shifted causal-LM cross-entropy normalized by the global masked-token count:

L = ( Σ_t m_t · CE(logits_t, labels_t) ) / N_eff
        m_t = loss_mask_t ;  N_eff = Σ_t m_t   summed over the DP group (and the accumulation window)

The mask is built in the data layer and the model is a pure consumer. This explicitness is deliberate: the OpenOneRec release documents an only_assistant_loss config key that the studied training code never reads [repo-openonerec] — a silently non-binding knob is exactly the failure mode tzrec's fail-loud convention forbids, so tzrec has no model-side masking switches at all; what is in loss_mask is what trains.

Itemic-vs-text split (logged, never weighted). Every log step emits train/itemic_token_loss and train/text_token_loss by masking the per-token CE with itemic_id_mask and its complement (OpenOneRec computes the same split, train_qwen3.py:818-833). These are the primary stage-health signals: in stage 1, text_token_loss should stay flat (body frozen) while itemic_token_loss falls; a rising text_token_loss in CO_PRETRAIN is the forgetting alarm that the general mix ratio is too low. We deliberately do not add a per-class loss weight — OpenOneRec ships a single CE and only logs the split; stage masking already controls what learns, and an extra knob would violate the minimal-proto rule.

Chunked lm_head cross-entropy (FSDP2 / in-house route)

At vocab 176,384, the fp32 logits for one 32K packed row are 32768 × 176384 × 4 B ≈ 23 GB before backward — materializing them is not an option. The FSDP2 and in-house backbone routes therefore use a reimplementation of OpenOneRec's ChunkedLossComputer (pretrain/onerec_llm/losses/ce.py:72-199): the backbone is configured to return final hidden states in place of logits (skipping lm_head, modeling_qwen3.py:876-879); the loss computer then chunks the sequence dimension by minibatch_size (16384 in OpenOneRec's runs), runs lm_head + fp32 CE per chunk, accumulates the lm_head weight gradients and the hidden-state input gradients via manual torch.autograd.grad, and finally triggers the single trunk backward with hidden.backward(grad_input_total / N_eff); the returned loss scalar is detached (display/metric only). This lands as tzrec/loss/chunked_ce.py, reimplemented with an in-code citation rather than imported.

Vocab-parallel cross-entropy (Megatron route)

On the mcore backend the same memory problem is solved by never gathering the vocabulary: GPTModel is built with parallel_output=True so logits stay TP-sharded [s, b, V/tp], and passing labels to forward returns the per-token loss [b, s] via tensor_parallel.vocab_parallel_cross_entropy, whose backward is in-place on the saved softmax [megatron]. mcore's convention is that masking and reduction are the caller's job: the tzrec loss_func multiplies the per-token loss by loss_mask and returns (loss_sum, num_tokens, report); with calculate_per_token_loss=True, the schedule's finalize_model_grads broadcasts num_tokens, all-reduces it over the dp-cp group, and scales gradients by 1/num_tokens (finalize_model_grads.py:529-542) — exactly the global-N_eff semantics of the formula above, implemented in the framework rather than the model. One library sharp edge: the unfused vocab_parallel_cross_entropy reads the global mpu TP group, so the mcore route initializes parallel_state even at TP=1 (§11).

Gradient-accumulation normalization (the GA loss fix)

Dividing each micro-batch's mean loss by the accumulation step count is wrong whenever micro-batches carry different valid-token counts — which under packing with response masking is always. HF Trainer's fix counts labels.ne(-100).sum() across all micro-batches of the accumulation window up front (num_items_in_batch, trainer.py:2095-2140) and normalizes the summed CE by that total [transformers]. Any tzrec-owned loop replicates this: masked-token counts are accumulated across the window and all-reduced across DP before scaling. The per-step DP half is already part of the two-world step sequence (§8.4); accumulation extends the same counter across micro-steps. On the mcore route the num_tokens machinery above covers it natively.

8.3 Stage-1 itemic-row freezing

Decision D7 — stage-1 freezing is backend-specific

Freezing the original vocab rows [0, 151669) while training [151669, 176384) uses a different mechanism per dense backend. FSDP2: OpenOneRec's EmbeddingGradientMasker pattern — save the frozen rows of the local DTensor shard once, restore them after every optimizer.step() — or equivalently zero the gradient rows before step. Megatron: the restore trick is broken under DistributedOptimizer fp32 master shards; the correct mechanism is zeroing param.main_grad rows after finalize_model_grads and before optimizer.step(), plus a ParamGroupOverride(wd_mult=0.0) on embedding/output params so decoupled weight decay cannot move zero-gradient rows. Whole-tensor body freezing uses requires_grad=False set before the DDP wrap on both backends.

Revision vs prior spec — restore-after-step is not portable

The prior spec (3(d)/05) pinned a single ItemicEmbeddingFreezer with OpenOneRec's restore-after-step strategy for all backends, calling it "robust to AdamW weight decay/momentum". Source-verified mcore 0.18 data flow shows the opposite for the Megatron backend: restore-after-step silently un-freezes the rows (mechanism below). D7 supersedes that design with per-backend mechanisms; the FSDP2 half keeps the OpenOneRec behavior.

FSDP2 mechanics

OpenOneRec stage 1 sets requires_grad=False on everything except embed_tokens/lm_head (train_qwen3.py:535-541), then applies row-level freezing with EmbeddingGradientMasker (pretrain/onerec_llm/training/gradients.py:76-173): at init it computes this rank's DTensor dim-0 shard overlap with [0, 151669) and clones those rows; after every optimizer.step() it copies them back (restore_frozen_params, called at train_qwen3.py:1224-1230). The restore is exact in their setup for one specific reason: the optimizer is plain AdamW directly on the FSDP2 DTensor parameters with no separate fp32 master copy — the parameter is the single source of truth, so overwriting it wins. The known warts are benign there but worth fixing in tzrec: frozen rows' gradients still inflate the global clip norm, and Adam moments accumulate garbage state for rows that get overwritten anyway. tzrec's Fsdp2Parallelizer (§11) therefore supports both modes — restore-after-step (proven) and pre-step gradient row-zeroing (clip-norm-clean) — and adds one guard: if a future configuration introduces a separate fp32 master copy on this path (e.g. an accelerate-style upcast where the optimizer state diverges from the live param), restore-after-step inherits exactly the Megatron failure below, so the freezer asserts the optimizer steps the parameter it restores.

Megatron mechanics: why restore-after-step drifts, and the correct recipe

Under mcore DDP + DistributedOptimizer (ZeRO-1), the parameter the model computes with is not what the optimizer updates. The per-step data flow (mcore 0.18, verified): backward hooks accumulate into param.main_grad, a param-shaped view into the flat grad buffer; finalize_model_grads runs the DP reduce-scatter and the tied-embedding allreduce; _copy_model_grads_to_main_grads slices the reduced buffer into fp32 master-shard gradients; Adam steps the fp32 master shards; _copy_main_params_to_model_params writes master values back into the bf16 param buffer; a per-bucket all-gather propagates them (optimizer/distrib_optimizer.py:2593-2703, 2828-2855) [megatron]. If frozen bf16 rows are restored after step(), the fp32 master rows have already moved (non-zero reduced gradients plus decoupled weight decay); the restore touches only the bf16 copy, and on the next step the master-to-model copy and all-gather overwrite the restored rows with the drifted master values. Net effect: the frozen rows evolve exactly as if never frozen, and the restore is a one-step cosmetic fix undone every iteration.

The correct mechanism acts on the gradient before the step, plus a weight-decay override:

# RowFreezer, Megatron backend (registered once after DDP + optimizer build)
n = clamp(151669 - emb.vocab_start_index, 0, emb.num_embeddings_per_partition)
targets = [(emb.weight, n)]            # + output_layer.weight rows for untied / PP>1 tied

# per train step, between forward_backward_func and optimizer.step():
for p, n in targets:
    p.main_grad[:n].zero_()            # 2-D view into the flat grad buffer

# optimizer build: kill decoupled weight decay on the embedding/output group
overrides[ParamKey(attr="is_embedding_or_output_parameter")] = ParamGroupOverride(wd_mult=0.0)

The zeroing runs after finalize_model_grads (zeroing identical rows on every rank commutes with the linear DP reduce and the embd-group allreduce, so post-reduce zeroing is exact) and before optimizer.step(), which also keeps frozen rows out of the gradient clip norm — an improvement over OpenOneRec. With zero gradient, Adam's exp_avg/exp_avg_sq stay zero and the update is exactly zero; with wd_mult=0 the decoupled decay term (p -= lr·wd·p, which AdamW applies even at zero gradient) is also zero; the master-to-model copy then writes back bit-identical rows. No divergence channel remains. The residual trade-off: wd_mult=0 also removes weight decay from the trainable itemic rows (a deviation from OpenOneRec, which effectively decayed them). If decay on trainable rows is required, the alternative is a periodic fp32 master-row restore through model_param.main_param + _get_model_param_range_map, keeping the grad zeroing for clip hygiene; the default accepts wd_mult=0 — the rows are freshly mean-initialized and embedding weight decay is of dubious value. DDP backward post-hooks and post-step param.data restores are explicitly non-viable hook points (they race the bucketed reduce-scatter under overlap_grad_reduce and the deferred all-gather under overlap_param_gather). Stage 1 on this backend pins plain bf16 + Adam + DistributedOptimizer; exotic modes (use_precision_aware_optimizer, Muon, FP8 param gather) are out of scope until re-verified.

Whole-tensor body freezing, tying, and TP geometry

Freezing the transformer body is a different, simpler problem: requires_grad=False set on whole tensors before the DDP wrap. mcore DDP then excludes those params from grad buffers entirely and _get_param_groups skips them, so they consume no fp32 masters and no Adam state — a large stage-1 memory saving (distributed_data_parallel.py:115-117, optimizer/__init__.py:330-331). The hook point is mbridge's post_model_creation_callbacks (precedent: freeze_moe_router, mbridge/utils/post_creation_callbacks.py:20-34) [mbridge]. Flipping requires_grad after the wrap is forbidden: the param stays in buffers and the optimizer with permanently-zero grads, and weight decay still moves it. Per-row freezing can not be expressed this way — requires_grad is per-tensor — which is why the embedding needs the grad-masking machinery above. Per-rank-asymmetric requires_grad (e.g. TP rank 0 fully frozen) is also rejected: param-group keys are all-gathered across ranks to align distributed-checkpoint layouts, and asymmetry breaks that.

Weight tying determines the target set. PP=1 tied (Qwen3 0.6B/1.7B/4B): the output layer borrows the embedding weight — one Parameter, both roles' weight-grads accumulate into one main_grad, one zeroing site freezes both. PP>1 tied: the last stage holds a duplicate copy whose grads are all-reduced over the embd group inside finalize_model_grads (after the DP reduce), so the generic rule "every rank zeroes its local frozen rows after finalize" covers both copies, and the wd_mult override matches both (the is_embedding_or_output_parameter attribute is set on both). Untied (Qwen3-8B+): freeze rows [0, 151669) in both embed_tokens and lm_head/output_layer — matching OpenOneRec, whose stage 1 trains the itemic rows of both. The mcore output layer shares the embedding's row geometry but lacks the vocab_start_index attribute; compute its range via VocabUtility.vocab_range_from_global_vocab_size. On the FSDP2 side, tied weights must land in the same fully_shard group (list form), per §11.

8.4 Trainer integration

Pipeline seam. The GR trainer plugs in at tzrec's single pipeline-selection point: create_train_pipeline (dist_util.py:335-376), called at exactly the train and eval sites (main.py:176, 430). A new GRTrainPipeline is returned for generative models; the surrounding loop is untouched because it relies only on pipeline.progress(iterator) -> (losses, predictions, batch) plus StopIteration semantics (§3). The new pipeline exists because tzrec's stock pipelines expose no hook between backward and step — pipeline.progress owns fwd+bwd+step — while the two-world step needs finalize_model_grads and the row freezer in exactly that gap. Owning the whole sequence in one class is the resolution of R-GRAD (register: §15; backend wiring: §11), and it follows the step sequence NVIDIA recsys-examples has in production for TorchRec-sparse + mcore-dense training (JaggedMegatronTrainPipelineSparseDist.progress, commons/pipeline/train_pipeline.py:735-859) [repo-recsysexamples]:

GRTrainPipeline.progress(iterator)            # Megatron backend
  1  model.module.zero_grad_buffer(); optimizer.zero_grad()
  2  global_tokens = batch.num_loss_tokens(); all_reduce(global_tokens)   # DP group
  3  losses, predictions = model(batch)       # sparse lookup -> inputs_embeds -> backbone
  4  (losses.sum() * dp_size / global_tokens).backward()
        # sparse fused optimizer steps IN backward; dense grads -> main_grad buffers
  5  finalize_model_grads([model.module])     # DP reduce-scatter + TP/embd allreduces
  6  row_freezer.zero_frozen_grads()          # ITEMIC_TEXT_ALIGN only (8.3)
  7  dense_optimizer.step()                   # DistributedOptimizer
  8  lr_schedulers.step()

The loss scaling in line 4 is the token-normalization identity: mcore DDP averages DP-summed grads by dp_size, so pre-multiplying by dp_size and dividing by the global token count yields the exact global per-token mean — and the in-backward sparse optimizer, which sees raw backward gradients without DDP division, receives a consistent scale through TorchRec's own world-size division (ranking_gr.py:118-120). On the FSDP2 backend the shape is simpler: there is no separate finalize step (reduce-scatter happens in FSDP2's backward hooks), the freezer's restore or grad-mask brackets TZRecOptimizer.step(), and the dense AdamW joins tzrec's existing CombinedOptimizer tree directly (§11).

Gradient accumulation gates dense only. In-backward sparse optimizers step on every backward, so accumulation semantics apply only to the dense world (set_requires_gradient_sync(False) on FSDP2; no_sync buckets on mcore DDP). The default for joint two-world phases is accumulation = 1; pure-dense stages (no TorchRec table in the loss path, the common case for ITEMIC_TEXT_ALIGN through MULTI_TASK_SFT without context injection) may set GAS > 1, with the num_items_in_batch normalization of §8.2 applied across the window.

Eval interleaving cost control. tzrec runs eval after every checkpoint save (main.py:490-497). For generative models a full eval is a beam decode plus catalog expansion per batch — recsys-examples runs generate() per eval batch and it dominates eval wall-clock [repo-recsysexamples]. The GR trainer therefore splits eval into two tiers: the per-save tier computes only teacher-forced metrics (perplexity, the itemic/text split losses, hierarchical prefix accuracy — cheap, single forward), while full constrained-beam HR@K runs on its own generation_eval cadence over a bounded user sample, decoupled from checkpoint cadence (metric definitions in §13).

LR scheduling. Per-stage schedules reuse tzrec's existing machinery unchanged: the dense LLM params form their own optimizer node via the part_optimizers regex seam (optimizer_builder.py:219-237), each node carries its own BaseLR schedule (registry pattern; CosineAnnealingLR with warmup exists, a WSD variant is one subclass + proto message), and base_lr == 0 freezes a whole group (lr_scheduler.py:34-41). The stage-1 split "embedding + head at 1e-3, body frozen" maps onto a regex group over embed_tokens/lm_head FQNs plus requires_grad=False on the body; the row-level boundary inside the embedding is the freezer's job, never a param group's.

8.5 Checkpointing

One DCP root for both worlds (FSDP2 default). DCP's planner treats plain Tensor, ShardedTensor, and DTensor uniformly in one state dict (planner_helpers.py:380-432, verified in torch 2.11 [pytorch]), so the default backend keeps tzrec's existing layout: model.ckpt-{step}/model holds TorchRec ShardedTensor tables and FSDP2 DTensor params in one dict, and optimizer gains the dense AdamW state via get_optimizer_state_dict (FQN-keyed, reshardable). The planner mechanics and HF-format bridges are §11's material; this section pins the trainer-facing semantics.

HF safetensors at the boundaries. The first load of a run chain is the vocab-expanded HF checkpoint produced by the §7 tool, read via DCP's HuggingFaceStorageReader (each rank reads only its slice ranges). The SFT → RL boundary and any external serving handoff need the reverse: HuggingFaceStorageWriter(save_distributed=True, enable_consolidation=True) emits consolidated safetensors plus index — the artifact verl and vLLM consume (§9, §10). OpenOneRec needed an offline DCP→HF converter between every stage because its stages are separate processes [repo-openonerec]; tzrec stays in DCP across stages and converts to HF only at these two boundaries. For serving specifically, each backbone family ships a DCP→safetensors serving manifest export — config.json + weights + tokenizer/SID manifest — consumed by the tzrec-gr-serving runtime; this is the designed train-to-serve checkpoint contract that prevents a sid_gr-style train/serve rift (§10.6).

Megatron second root. When the mcore backend is active, the dense world checkpoints through megatron.core.dist_checkpointing.save(model.sharded_state_dict()) (the torch_dist backend) into a sibling subdir, e.g. model.ckpt-{step}/dense_mcore/, next to model/, optimizer/, and dynamicemb/ — the same side-by-side foreign-world pattern dynamicemb already established (§3). mcore dist-checkpoints are TP/PP/DP-reshardable [megatron], so stage 1 trained at TP=8 can resume stage 2 at TP=4. We explicitly do not copy recsys-examples' per-rank torch.save for the dense part, which locks the checkpoint to one world size [repo-recsysexamples]. Checkpoint pruning (keep_checkpoint_max, async daemon) works unchanged because it deletes whole step directories.

Resume semantics. Three pieces. (a) Dataloader state: tzrec's dataloader_state.json merges per-source row offsets by max across ranks — correct for offset-resumable readers, but a packed iterable dataset over sharded parquet consumes files non-monotonically, so the GR dataset adds a per-(file, epoch) consumption bitmask under a new reserved key (OpenOneRec's finish_dict bitmask is the proven exact-resume design for this shape [repo-openonerec]; merge-by-max is the only existing merge op, hence the new key). (b) Optimizer lazy init: the existing peek-step dance — run one dummy pipeline.progress on a peeked batch to materialize optimizer state, restore, then re-chain the batch (main.py:441-454) — is retained for the FSDP2 AdamW (lazy, like all torch.optim); the Megatron DistributedOptimizer builds its master shards at construction, so the peek is a harmless no-op there. (c) Stage transitions: stage N starts from stage N−1 via --fine_tune_checkpoint; model weights load through PartialLoadPlanner (skip-missing with warning, name remap via the fine_tune_ckpt_param_map {new}\t{old} file, checkpoint_util.py:50-148), and optimizer state is dropped across stage boundaries (--ignore_restore_optimizer) because the trainable set and schedule change. The freezer's cached rows are never checkpointed — they are recomputed from loaded weights at stage start, with a fail-loud assert if the configured freeze boundary disagrees with the checkpoint's vocab metadata. One operational default borrowed from OpenOneRec: emit early checkpoints at small step counts (e.g. steps 20 and 200) so stage-misconfiguration (wrong freeze boundary, wrong mask) is caught before a long run burns its budget.

9. Reinforcement learning pipeline

RL is the stage where the generative recommender stops imitating the logged exposure distribution and starts optimizing a reward: next-token prediction can only fit what the legacy cascade chose to expose, so it inherits that cascade's ceiling [onerec-v1]. This section owns four decisions: the algorithm family (D10, §9.1), the verl integration contract (D9, §9.2), the composable reward library (D14, §9.3), and the per-backbone rollout decision matrix (D11, §9.4); it closes with on-policy distillation and reasoning RL (§9.5) and operational guidance (§9.6). The constrained-decoding artifacts the rollout depends on (CSR trie, SidConstrainedLogitsProcessor) are specified in §10 (D12); the dense training engines verl reuses are covered in §11 (D8); the RlConfig proto surface is in §12.

9.1 Why RL, and the algorithm family (D10)

The empirical case for RL in this domain is consistent across the OneRec line. OneRec-V1 measured that RL concentrates quality at the top of the generated list — the gain over no-RL is largest at small serving K (Pass@32 App Stay Time +0.49% vs +0.09% at Pass@512) — and that beam-search rollouts beat top-k/top-p sampling for SID generation, with group size G ≈ 4× the serving K as the saturation rule of thumb [onerec-v1]. OpenOneRec showed the staging effect: on-policy distillation dips recommendation metrics while restoring general capability, and the Rec-RL GRPO stage then recovers and surpasses the SFT checkpoint (video-rec Recall 0.0324 → 0.0304 → 0.0370) [openonerec]. At the same time, PLUM is the production proof that RL is an additive phase, not a launch prerequisite: SID + continued pretraining + reward-weighted SFT alone beat a tuned embedding-table retrieval stack [plum]. tzrec therefore treats RL as phase P3 of the roadmap (§14) and designs it so the SFT pipeline of §8 is independently shippable.

Decision D10 — GRPO is the default; ECPO and GBPO are registered plugins, never forks

The default algorithm is GRPO with the OpenOneRec-proven configuration (group-relative outcome advantage, KL-in-loss low_var_kl with coefficient 0.001, asymmetric clipping with clip_ratio_high=0.28) — this is stock verl, zero tzrec code [repo-openonerec][verl]. ECPO (OneRec-V1) and GBPO (OneRec-V2) ship in tzrec/rl/ as user-side registrations against verl's @register_adv_est / @register_policy_loss registries, which verl explicitly supports for out-of-tree estimators [verl] (verl/trainer/ppo/core_algos.py: ADV_ESTIMATOR_REGISTRY, POLICY_LOSS_REGISTRY). GBPO additionally has a verl-free implementation in the tzrec-native loop (§9.4) because it needs no rollout engine at all. DAPO-style asymmetric clipping and GSPO sequence-level ratios are configuration toggles on the GRPO path (clip_ratio_low/high, policy_loss.loss_mode=gspo), mirroring MiniOneRec's ablation surface [minionerec].

The three core objectives, in the notation of the source papers (sg = stop-gradient, G = group size, A_i = advantage of rollout i):

# GRPO (verl compute_grpo_outcome_advantage; norm_adv_by_std_in_grpo=True)
A_i = (r_i - mean(r_1..G)) / (std(r_1..G) + eps)
L   = PPO_clip(ratio, A_i; clip_low=0.2, clip_high=0.28) + kl_loss_coef * KL_low_var(pi_theta || pi_ref)

# ECPO — Early-Clipped GRPO (OneRec-V1 Eq. 32-34), delta = 0.1, no KL term
J_ECPO = E[ (1/G) * sum_i min(rho_i * A_i, clip(rho_i, 1-eps, 1+eps) * A_i) ]
rho_i  = pi_theta(o_i|u) / pi'_old(o_i|u)
pi'_old = max( sg(pi_theta(o_i|u)) / (1 + eps + delta),  pi_old(o_i|u) )

# GBPO — Gradient-Bounded Policy Optimization (OneRec-V2 Eq. 11-12), no clipping
J_GBPO = -E[ (1/G) * sum_i (pi_theta(o_i|u) / pi'_old(o_i|u)) * A_i ],   A_i in {+1, -1}
pi'_old = max(pi_old, sg(pi_theta))        if A_i >= 0
        = max(pi_old, 1 - sg(pi_theta))    if A_i <  0

The mechanics matter for stability on 3-token SID outputs. ECPO's early clip bounds the denominator rather than the ratio: for negative-advantage rollouts where pi_theta >> pi_old, plain GRPO lets the effective ratio explode; substituting pi'_old = max(sg(pi_theta)/(1+eps+delta), pi_old) caps it near 1+eps+delta, which OneRec-V1 identifies as the fix for the gradient explosion that otherwise collapses legality (§9.3) [onerec-v1]. GBPO goes further: no clipping at all, full sample utilization, and a sign-dependent denominator that reproduces BCE gradient magnitudes — positives bounded like 1/pi, negatives like 1/(1-pi), so negative gradients shrink exactly where the policy already assigns low probability. Its operational profile is unique: it trains on logged real feedback (top-quartile duration-shaped positives, dislikes as negatives, advantage in {+1, −1}, deliberately not normalized), needs no rollout engine, no critic, and no reward model — but it requires that serving logs the per-token generation probability of every exposed item; impressions from non-generative traffic fall back to pi_old := sg(pi_theta) (ratio ≡ 1), the regime GBPO is specifically designed to keep stable [onerec-v2]. This logging requirement is a serving contract tracked in §10.

MiniOneRec contributes the group-equals-beams construction: one constrained beam-search call with num_beams = num_return_sequences = G and do_sample=True produces the whole GRPO group, guaranteeing valid and mutually distinct items per group, and the beam rank doubles as a ranking signal for an NDCG-shaped reward (high-ranked wrong items penalized more, and only in groups where the target was findable) [repo-minionerec]. OneReason contributes the reasoning-era refinements that tzrec adopts as configuration rather than new algorithms: (i) two-stage rollout — sample N reasoning traces, then K parallel SID completions per trace, paying for N long generations while getting N×K group members (Kuaishou reports per-step time dropping from ~350 s to 150–200 s with quality gains); (ii) stage-wise clipping — a looser clip range on CoT tokens than on SID tokens, because a shared clip lets the 3-token SID distribution move too aggressively and triggers entropy collapse (R-ENTROPY); (iii) negative-rollout down-weighting — a response-level loss weight β<1 on non-hit rollouts (advantages untouched) so abundant zero-reward rollouts do not dominate gradients; and (iv) the specialize-then-unify schedule — per-domain RL teachers consolidated by rejection-sampling fine-tuning (RFT), the only method in their study that guarantees thinking mode beats non-thinking across all domains and K [onereason]. Items (i)–(iii) require token-span and group-structure metadata in the loss; verl's driver-assignable loss callable (set_loss_fn) is the sanctioned injection point [verl].

9.2 verl integration: the zero-patch plan (D9)

verl 0.8 (studied at commit 8232babc, version 0.8.0.dev) is a Ray single-controller harness: a TaskRunner builds worker groups and drives RayPPOTrainer.fit, while the legacy per-backend workers have been replaced by one unified ActorRolloutRefWorker/TrainingWorker pair parameterized by a model-engine registry (fsdp/fsdp2/megatron/torchtitan/veomni) [verl]. Rollout is exclusively async server mode: each replica boots a full vllm serve (or SGLang) on the trainer GPUs, and CPU-side agent loops drive token-in/token-out generation per sample through a load-balanced server client. Rewards run in a parallel reward-loop worker pool that can score streamingly as rollouts finish; reference and teacher models are fused into the same worker or split into dedicated resource pools. Weight resync after each actor update goes through a pluggable CheckpointEngine (colocated all-gather by default; NCCL/NIXL/Mooncake for disaggregated layouts; a Qwen3-30B-A3B full resync benchmarks at ~7 s on 4×8 H100). The Megatron engine path requires mbridge (assert engine_config.use_mbridge) and consumes HF checkpoints directly — including our vocab-resized Qwen3, since embedding sizes come from the HF config [mbridge]. A TransferQueue-based successor trainer (main_ppo_sync.py) coexists with the classic path and shares the same extension surfaces, including first-class multi-trajectory agent-loop outputs — relevant for group-equals-beams; tzrec builds against the classic path and keeps its custom pieces portable to both.

Decision D9 — RL runs on the verl 0.8 line with zero framework patches

Every tzrec RL requirement lands through documented verl extension surfaces; we fork nothing and patch nothing. The user-side inventory, all living under tzrec/rl/ and a config bundle:

  • Dataset: an offline tzrec job exports RL parquet in verl's schema (prompt chat messages with SID history rendered as itemic tokens, data_source, reward_model.ground_truth, extra_info); where on-the-fly construction is needed, one TzrecRLDataset class registered via data.custom_cls.{path,name}.
  • Rollout: a sid_agent AgentLoop registered via rollout.agent.agent_loop_config_path YAML ({name, _target_} entries; no fork) and selected by rollout.agent.default_agent_loop or a per-row agent_name column.
  • Constraints: the vLLM-v1 variant of SidConstrainedLogitsProcessor (§10, D12) injected engine-wide via ++actor_rollout_ref.rollout.engine_kwargs.vllm.logits_processors=[...] — verl assembles vLLM CLI args from this passthrough — with per-request arming via SamplingParams.extra_args, which verl forwards verbatim (vllm_async_server.py:502).
  • Rewards: reward.custom_reward_function.{path,name} plus a tzrec RewardManager loaded via reward.reward_manager.source=importlib for raw-token-id scoring (§9.3).
  • Engines: FSDP2 (model_engine=dp, strategy fsdp2) for ≤8B dense, Megatron (model_engine=megatron, mbridge mandatory) for MoE/EP — config-only, aligned with D8 (§11). Checkpoint-engine weight resync is used as-is.

Preconditions: vLLM is the pinned rollout backend for beam-faithful OneRec-parity rollouts (SGLang still has no beam search, though it is a viable alternative for sampling-only GRPO rollouts — §9.6, §10.5); the engine must run with logprobs_mode="processed_logprobs" (verl 0.8's default, verl/workers/config/rollout.py:203); and engine_kwargs.vllm.max_logprobs must be raised to ~2× the beam width. verl's OOV guard masks logits at len(tokenizer), so the padded dummy rows above 176247 (§7) can never be sampled — itemic tokens must be genuine tokenizer-added tokens.

Constrained beam search: the one true gap

verl 0.8 has no beam search anywhere in its rollout path, and vLLM's beam search is not an engine feature: in both the sync LLM.beam_search entrypoint and the OpenAI-server path (use_beam_search: true on completions/chat), it is a client-side emulation that, per token step, submits one fresh 1-token request per live beam with SamplingParams(logprobs=2*beam_width, max_tokens=1, temperature=0) and re-ranks candidates host-side [vllm]. Two consequences follow. First, the emulation never copies constraint fields into its per-step SamplingParams — so per-request constraints do not reach beam steps. Second, and more subtly, the logprobs the ranking consumes are by default computed from raw logits, before any logits processor runs:

logprobs_mode is the linchpin of constrained beam ranking

With vLLM's default logprobs_mode="raw_logprobs", an engine-level logits processor masks what gets sampled but is invisible to the logprobs that emulated beam search ranks by — the beam happily ranks illegal continuations. With logprobs_mode="processed_logprobs", beam steps are all-greedy, so the returned logprobs are the log-softmax over masked logits — exactly the renormalized-over-legal-set distribution constrained beam search needs. verl 0.8 defaults to processed_logprobs (rollout.py:203); plain vllm serve and offline LLM() do not, and must set it explicitly (§10, §13). The option landed in vLLM v0.10.0 (PR #21398) [vllm] and does not exist on the 0.8.5 line OpenOneRec pinned — their beam ranking was unconstrained by construction [repo-openonerec]. Companion knob: engine max_logprobs defaults to 20, failing validation for beam width >10; OpenOneRec sets 320 (width ≤160), and width 512 needs ≥1024.

Because each emulated beam step is a fresh request whose prompt ends with the partial SID, an engine-level trie processor can derive its state from the prompt tail (walk back to <|sid_begin|>) instead of requiring extra_args — one processor then constrains verl rollouts, offline LLM.beam_search, and OpenAI-server beam requests without forking any beam loop. With that in place, three designs were evaluated for verl-side generation (full analysis in the research notes; ranked):

The canonical recipe: OpenOneRec two-stage rollout

The reference rollout recipe for reasoning-capable Qwen3 backbones is OpenOneRec's two-stage generation, reimplemented as a sid_agent AgentLoop (their OneRecvLLMRollout subclassed the retired SPMD rollout and cannot be ported directly) [repo-openonerec]. Stage 1 samples one CoT with stop=["</think>"] (max 1024 tokens), filters any OOV ids ≥ tokenizer length, and splices the literal \n<|sid_begin|>; stage 2 runs beam width 32 for exactly 3 itemic tokens. All 32 beams are returned as separate trajectories that share one GRPO uid — the trainer assigns uids before beam expansion, so the group is the beam set (rollout.n=1). Run parameters worth copying verbatim: actor LR 2e-6, kl_loss_coef=0.001 (low_var_kl), clip_ratio_high=0.28, temperature 1.0, max_logprobs=320, prompt length 10240 / response length 2048. One structural caveat: group members share a single sampled CoT, so the advantage differentiates items, not reasoning — CoT diversity comes only across prompts and epochs. OneReason's N CoT × K SID two-stage rollout is the generalization that restores reasoning-level credit assignment and is the configuration tzrec exposes for reasoning RL (§9.5) [onereason].

9.3 Composable rewards (D14)

Decision D14 — rewards are composable tzrec-side functions, scored on raw token ids

All rewards live in tzrec/rl/reward.py as plain callables composed by weighted sum, surfaced to verl through reward.custom_reward_function and — for anything that needs exact SID matching or group structure — a tzrec RewardManager registered via reward.reward_manager.source=importlib whose run_single reads the raw responses token ids. The same callables are reused by the tzrec-native rollout loop (§9.4) and the eval harness (§13), so reward semantics never fork between training and evaluation.

verl's custom-function contract (as of 0.8.0.dev): compute_score(data_source, solution_str, ground_truth, extra_info, **kw) -> float | {"score": float, ...}, where dict extras land in reward_extra_info for logging and for GDPO's per-component keys; sync and async functions are both supported [verl]. The gotcha that forces the RewardManager path: solution_str is decoded with skip_special_tokens=True. Our itemic tokens are deliberately ordinary added tokens (§7) and survive that decode — OpenOneRec relies on exactly this and regex-parses <s_a_(\d+)><s_b_(\d+)><s_c_(\d+)> — but string-level matching is fragile against template drift and cannot see token positions; id-level matching against itemic_id_range = [151669, 176245) is exact and is the tzrec default. Token-level shaped rewards (e.g., per-level credit at specific positions) also require the manager, since the naive path writes one scalar at the last valid token.

The reward inventory, each item grounded in its source system:

  • RewardModelReward (P-Score). OneRec-V1's P-Score reward model is, structurally, a stock tzrec multi-task ranking model: SIM-style sequence backbone, one BCE tower per objective (ctr/lvtr/ltr/vtr/…), and a fused P-Score tower trained with weighted BCE against all labels [onerec-v1]. tzrec trains it with the existing multi-task stack (§3) and serves it externally; the reward callable decodes generated SIDs to item ids via SidCatalogIndex (§5, D4) and batch-scores over RPC. In verl this maps to either the RM resource pool (reward.reward_model.enable_resource_pool, RM served on rollout infrastructure) or an external endpoint called from the custom function, which receives reward_router_address for exactly this purpose. The investment is justified by V1's online result: pure generative was only at parity with the cascade; the reward model contributed most of the win (+0.54%/+1.24% App Stay Time with RM selection).
  • DurationAwareReward (OneRec-V2). Log-bucket the video duration (F(d) = floor(log_beta(d + eps))), rank the target's playing time as a percentile within the user's own history for that bucket, then assign +1 to batch top-quartile non-disliked samples, −1 to explicit dislikes, and filter the rest — deliberately without advantage normalization, which V2 found degrading [onerec-v2]. This is a data/label-pipeline job (per-user duration/playtime history, dislike labels) feeding an advantage column; it pairs with GBPO and with the hybrid arm (independent streams: rollouts → RM reward, logged exposures → feedback reward) that V2 found best-balanced.
  • SidLegalityReward / FormatReward. OneRec-V1's format reward is the canonical treatment of legality: under ECPO the legality ratio collapsed below 50% via the squeezing effect, and the fix is to take a random K of the G rollouts, assign A=+1 to legal ones, and discard illegal ones entirely — never penalize them, since negative updates squeeze probability mass further onto the argmax. Random-K selection climbs to ~95% legality and stays; top-K-by-probability rises then falls [onerec-v1]. Under constrained decoding (D12) the legality reward remains enabled as a diagnostic: MiniOneRec's CC counter precedent shows a non-zero illegal count means the constraint machinery silently regressed [repo-minionerec]. Legality-rate monitoring is mandatory in every RL run — R-LEGALITY. FormatReward additionally checks the surface form (a well-formed <|sid_begin|>…<|sid_end|> block of exactly num_levels itemic tokens; non-trivial <think>…</think> when reasoning is required).
  • DiversityReward (OneReason). A trace-level multiplicative factor R_div = max(0, m1 - 1)/(K - 1), where m1 is the number of distinct first-level SID digits among the K completions of one CoT — the first digit approximates the coarse category, so this rewards category-diverse generation and helps most at large K [onereason]. It needs the rollout grouping structure, not just (prompt, response) — RewardManager territory.
  • Hierarchical partial credit. OpenOneRec's partial_hit_reward grants 100 for a full triple match, 10 for the first two levels, 1 for the first level, averaged over predicted SIDs; in their released config it is a logged extra while the trained score is first-SID Pass@1 [repo-openonerec]. tzrec exposes it as an optional trained arm (MiniOneRec's HEPO 0.2/0.5/1.0 is the same idea) — useful against reward sparsity on large catalogs, with the caveat that prefix credit is only meaningful when the tokenizer hierarchy is semantically loaded (§5).
  • Multi-component normalization (GDPO). When mixing accuracy, diversity, and business components, verl's in-tree GDPO estimator (algorithm.adv_estimator=gdpo + algorithm.gdpo_reward_keys) normalizes each component's advantage separately — preferable to hand-tuning weights on a summed scalar [verl].
  • Reward-weighted SFT sampling (PLUM). Not an RL reward but part of the same design space: sample SFT examples proportionally to a handcrafted engagement reward, then weight them equally. Production-proven at YouTube at near-zero infrastructure cost [plum]; exposed as a sampling knob on the SFT data pipeline (§8) so teams get reward alignment before any RL is stood up.

9.4 Rollout decision matrix (D11)

Decision D11 — the rollout path is chosen per backbone, not globally

There is no single rollout engine that serves all five backbones of D5. tzrec routes each generative model class to the cheapest rollout path that is actually expressible: verl + vLLM for token-LLM backbones, an out-of-tree vLLM wrapper model only when per-request dense context embeddings must reach the engine, GBPO (no engine) for the lazy decoder, and a tzrec-native in-process loop for encoder-decoder and HSTU backbones. The matrix below is binding; RlConfig validation (§12) rejects unsupported combinations rather than degrading silently.

The tzrec-native in-process rollout loop

For backbones that cannot ride vLLM, the rollout runs inside the training process on the already-sharded model — a pattern with two existence proofs: recsys-examples' generate_beam_decode executes beam search on a DMP-sharded Megatron-style model with TorchRec sparse lookups in the loop (split context-KV / beam-KV caches, beam-ancestry index propagation) [repo-recsysexamples], and MiniOneRec runs a complete colocated GRPO loop (forked TRL trainer, constrained HF generate) on ≤7B models across 4–8×80 GB GPUs [minionerec]. The tzrec design, per training step modulo a rollout interval: (1) switch to eval mode and run generate_beam(batch, G) under no_grad on a user subsample (OneRec-V1 routed 1% of users; G ≈ 4× serving K); (2) take pi_old directly from the beam's per-token logprobs — the beam already computes them, no extra forward; (3) score rewards in-process (P-Score as a second eval-only DMP model on the same ranks, or RPC) plus the format/legality channel; (4) add the ECPO or GBPO loss to the streaming RSFT loss in the same backward — the fused sparse optimizers and the dense path are untouched, it is one extra loss term, exactly OneRec-V1's joint RSFT+RL recipe [onerec-v1]. A teacher-forced recompute of old logprobs is needed only when taking multiple gradient steps per rollout batch (mini-epochs); ECPO needs no reference model at all, and if a KL variant is wanted, only the dense parameters can be cheaply frozen-copied — duplicating the sparse world is not viable, one more reason to follow the no-KL design. At 3–4 decode steps even the naive re-forward generation is acceptable (context prefill dominates); the envelope is roughly ≤3B dense with short decodes, beyond which paged KV and continuous batching are missed badly enough to mandate the verl path. Longer term, this loop is expected to converge onto the tzrec GR serving runtime once the lazy_decoder and enc_dec adapter families land (§10.6), so rollout-in-validation and serving share one decode stack.

Sparse tables cannot ride weight resync

verl's checkpoint engines stream (name, full_tensor) pairs — name-agnostic transport, name semantics at the endpoints. A tzrec dense backbone could resync to a tzrec rollout server over this channel, but the sparse world cannot: a per-tensor stream has no representation for dynamicemb/TorchRec sharded hash tables with billions of rows. The production-proven answer is decoupling: push dense parameters to the rollout/inference side and serve embeddings from a parameter server or incremental dump — OneRec-V1 pushed dense params over a message queue every 1000 steps to its external beam-search service [onerec-v1], which is structurally verl's STANDALONE replica mode plus an NCCL/NIXL checkpoint engine. The SID-as-input representation (history rendered as itemic tokens rather than per-item sparse embeddings, OneRec-V1 §4.2.2) shrinks the "sparse" surface to an ordinary dense embedding matrix and makes the model fully resync-friendly — an additional argument, beyond the modeling ones in §6, for SID-everything inputs in RL-bound configurations.

9.5 On-policy distillation and reasoning RL

On-policy distillation is the capability-retention stage (R-FORGET): OpenOneRec's SFT checkpoint loses general capability hard (AIME'24 0.79 → 0.51, IFEval 0.86 → 0.62), distillation against the original Qwen3 recovers most of it (AIME 0.71, IFEval 0.77), and the subsequent Rec-RL roughly preserves it [openonerec]. The mechanism is reverse-KL-as-advantage run through the standard policy-gradient machinery: the student samples on-policy from 200K general-domain prompts, and each token's advantage is A_t = -(logp_student - logp_teacher), clamped to [-30, +5]; the teacher is the unmodified base Qwen3 of the same scale, loaded as the reference worker with an overridden model path. Sampling is deliberately hot (temperature 1.1, top_p 0.95, top_k 200) so that itemic-token leaks onto general prompts actually get sampled — and punished.

The vocab mismatch is the crux: the teacher has never seen ids ≥ 151669. The OpenOneRec recipe, adopted with one definitional change: (1) build a mask over response tokens whose id falls in itemic_id_range = [151669, 176245) — the same single range used by loss splitting and RL legality (§7) — plus, stated explicitly, the two SID specials <|sid_begin|>/<|sid_end|> (176245/176246), which the teacher equally does not know and which must therefore be covered by the teacher-side replacement and override of steps (3)–(4). One-time divergence note: OpenOneRec's released mask is the open-ended id ≥ 151669, which sweeps in its specials implicitly; tzrec deliberately reuses the shared itemic_id_range definition rather than introducing a second ad-hoc threshold; (2) truncate the trajectory after the first itemic token (response_mask zeroed beyond it; the itemic token itself stays in-loss so it absorbs the penalty); (3) before the teacher forward, replace masked input ids with <|im_end|> (id 151645) to avoid out-of-bounds embedding lookups; (4) override the teacher's logprob at masked positions with a strongly negative constant (−100 in the released config — workable only because of the −30 advantage clamp). Net effect: emitting an itemic token on a general-domain prompt earns the maximum negative advantage, pulling general behavior back to the teacher without unlearning the SID head [repo-openonerec]. verl 0.8 has on-policy distillation in-tree — a distillation config block with teacher resource pools, per-sample teacher routing, and a family of distillation losses (k3, reverse/forward KL, top-k variants) [verl] — so OpenOneRec's fork-era recipe maps onto configuration plus tzrec-side hooks: the trajectory truncation and teacher-query id swap land in the sid_agent loop (which owns response_mask and the teacher-logprob fetch), and the loss-side clamping lands in the driver-assigned loss callable. No verl patches, consistent with D9.

For reasoning models, tzrec adopts OneReason's pipeline shape rather than inventing one [onereason]. Single-domain GRPO teachers are trained with the two-stage N×K rollout, stage-wise clipping, and the diversity-factored reward (§9.1, §9.3); unification is then RFT — batch-sample trajectories from each domain teacher, keep only verified hits passing trace-quality filters, and consolidate by plain SFT. RFT is an offline pipeline (the §13 eval harness generates, the §8 SFT loop consumes); it requires no new RL machinery, and it is the only method in OneReason's study with a thinking-beats-non-thinking guarantee across domains and K. OneRec-Think's rollout-beam reward (beam search width 32 inside the reward to grant per-digit credit if any beam hits) is supported by the same server-side beam mechanism as design (b) of §9.2, with the reward worker calling the rollout replica's HTTP endpoint [onerec-think]. MOPD — multi-teacher on-policy distillation with single-point reverse-KL advantages, truncated importance weights, and information-gain trajectory filtering — is explicitly future work: verl's multi-teacher routing makes it reachable, but RFT covers the v1 need at far lower complexity.

9.6 Operational concerns

Environment isolation (R-ENV). The RL stage runs in its own conda environment or container image, as a separate process tree from tzrec training. This is not optional: tzrec pins grpcio-tools<1.63.0 and the protobuf 4.x API line, while ray/vllm/verl pull newer protobuf and grpcio. The handoff surfaces are files, not imports — RL datasets as parquet, policies as HF checkpoints, the trie as a versioned .npz artifact (§10) — so the two environments never share a Python process. The tzrec.gr.rl CLI is a thin launcher that materializes the verl Hydra config and submits the Ray job into that environment (§12).

Version pins (R-VLLM-PIN). verl 0.8.0.dev's declared pin vllm>=0.8.5,<=0.12.0 is stale — the code carries live branches for vLLM ≥0.13 and ≥0.17 — and the constrained-decoding parameter surface drifted across that span (guided_decoding → structured_outputs; logprobs_mode landed only in v0.10.0; per-request logits processors rejected by V1). tzrec pins an exact (verl commit, vLLM release) pair per image, records it in the §14 dependency table, and tags every API-level claim with its verl version, as the prior spec already mandated. vLLM remains the pinned backend for beam-faithful OneRec-parity rollouts: SGLang still ships no beam search of any kind (a repo-wide grep at the HEAD studied returns zero hits) [sglang]. For sampling-only GRPO rollouts, however, SGLang is a viable and often superior alternative — its per-request custom logit processor sees the live request each step and can walk the SID trie, ReasonerGrammarBackend natively keeps constraints dormant until </think>, and its verl weight-update surface is the richest of the three engines (§10.5).

Entropy collapse (R-ENTROPY). The 3-token SID segment is the collapse-prone span. Standing dashboards: per-token entropy split into itemic vs language tokens at the 151669 boundary (OpenOneRec added exactly this metric to its trainer [repo-openonerec]), Validity@K, per-level token entropy and codebook utilization (shared with §5 tokenizer health), legality rate (R-LEGALITY), and the degenerate-group ratio (fraction of GRPO groups whose rewards are all zero — OpenOneRec's grpo_all_zero_hit_reward_group_ratio). Mitigations in order of preference: stage-wise clipping (tighter on SID tokens), ECPO/GBPO denominator bounds, and reward densification (partial credit, N×K rollouts) before any KL-coefficient surgery.

Sequence-length sizing. rollout.prompt_length and rollout.response_length drive padded tensor shapes through the whole verl pipeline. Pure SID responses are 3 itemic tokens plus markers — size response length in the tens, not the LLM-default thousands, or pay the memory for nothing; generation dominates RL step time, so this is the cheapest large win. Two-stage CoT configurations need response length sized for the CoT (OpenOneRec: 1024-token stage-1 cap inside a 2048 response budget, 10240 prompt). max_num_batched_tokens and max_model_len follow the same logic.

Reward hacking. OneRec-V1 found on-policy rollouts beat a frozen behavior model (vtr +5.61%) but attributed the residual gap to reward hacking against the imperfect P-Score model, flagging it as open [onerec-v1]. tzrec's mitigations: hybrid reward streams (logged real feedback alongside the RM, per OneRec-V2's best-balanced arm [onerec-v2]), the legality/diversity monitors above, and periodic held-out evaluation against non-RM metrics (§13) so an RM-gamed policy is caught by metrics it cannot influence.

GRPO uid discipline. Group identity is the per-prompt uid assigned in the fit loop before any expansion: beams of one prompt must share it (group = beams), repeated prompts must not. Custom samplers or datasets that duplicate rows must preserve uid semantics, and the seqlen-balancing pass reorders rows freely — safe only because advantage computation is uid-keyed, a property any custom estimator registered under D10 must also maintain.

10. Decoding, inference, and serving

This section specifies how a trained generative recommender becomes a serving artifact: constrained decoding (D12), the single-step scorer export driven by a host-side beam loop (D13), the KV-cache ladder, the engine landscape, and the serving topologies. The structural rule from §4 holds throughout: the autoregressive decode loop is never traced into an exported graph — it lives in host code invoking an export-clean scorer, exactly as TDM retrieval already does (tzrec/tools/tdm/retrieval.py:121-414). Research moved both owned decisions: the single-step export path is empirically proven on this machine (R-EXPORT de-risked), and STATIC's tensorized trie [static] resolves the constrained-decoding cost concern (R-TRIE resolved). The engine study (§10.5) found RTP-LLM's OSS build ships the AL-GR/FORGE constrained-decoding production path and SGLang the best-equipped RL sampling-rollout surface — engine-native constraint paths consume the same SidTrie artifact, they do not replace the D12/D13 defaults. The serving-implementation analysis (2026-06-10) adds the six-layer decomposition with its engine-invariant core (§10.1) and adopts recsys-examples' sid-gr-inference as the seed of tzrec's own runtime; the extension plan (2026-07-03) specifies that runtime as the tzrec-gr-serving fork behind the BackboneAdapter contract, serving all softmax-attention GR backbones (§10.6).

10.1 Requirements

The decode subsystem must cover a wide operating range: published systems disagree by an order of magnitude on beam width, and SID geometry is a config (D5).

The six-layer decomposition and the engine-invariant core

A GR inference service is six layers. L1 prompt frontend — user features → prompt and/or context embeddings, shared code with the training-side task builder (train/serve skew is the classic rec killer); serving assembles token ids directly: static template segments pre-tokenized offline, itemic ids computed arithmetically, never string-tokenized. L2 tokenizer — hot-path only if L1 assembles strings. L3 model execution — prefill, decode, batching, KV cache, quantization, TP: the only layer where the engine question lives. L4 decode control — beam widths, constraint masking, logprobs, dedup/exposure filters. L5 grounding — SidCatalogIndex, response format, gen_prob_logged. L6 ops — EAS deployment, versioned artifact swap, A/B, monitoring, fallback. The engine-invariant core (L1, the L4 artifacts, L5, L6) is tzrec code under every topology in §10.7: the shared prompt frontend and token-id assembly; the trie builder and CSR runtime (~500 LOC owned from STATIC's 836-LOC, zero-server reference [static-repo]); SidCatalogIndex grounding and gen_prob_logged emission; versioned swap of (tokenizer, SID assignment, trie, catalog) as one unit; an eval harness on the same L1/L4/L5 code as serving. RTP-LLM's in-tree recommendation_parser.py proves even engine deployments need this python layer [rtp-llm].

The GR workload also has structure general engines cannot exploit: decode length is fixed and tiny (3–4 SID tokens; no EOS uncertainty, no streaming); requests are homogeneous, so gang/bucketed batching replaces continuous batching, preemption, and paged eviction; cross-user token-prefix reuse is ~0 — the reuse that matters is beams sharing one user's context within a request (§10.4) and user-keyed session reuse; and beam search is the primary decode mode (RTP-LLM's native rec-purpose beam path is the one engine exception). A general engine imports machinery this workload does not need while missing what it does: L1, L4 fidelity, L5, batched-beam attention.

Latency anchors. OneRec-V2 serves a 1B lazy decoder at context 3000, beam 512, in 36 ms on L20 at 62% inference MFU [onerec-v2] — read carefully: the lazy decoder shares context KV across beams by construction and cuts ~94% of decode compute, so 36 ms is not a target a Qwen3-class backbone can reach (§10.4). For decoder-only transformer backbones, recsys-examples measures KV-cached generate_beam_decode at 2.1× / 4.7× / 11–12× over cacheless re-encoding at history 256/512/1024, the decode loop itself ~15 ms and dominated by small-batch dense projections [repo-recsysexamples]: KV caching across SID levels is mandatory; below width ~512 the GEMMs set the floor. FORGE's nearline path writes 1200 candidates to a cache table within 200 ms — generative retrieval need not sit on the synchronous request path at all (§10.7) [forge]. Engine cost models at beam 512 over 3000-token contexts sit at ~0.8–1.2 s/request — 3–30× off a purpose-built path depending on L2 residency (§10.4): context length, not beam width, dominates.

Grounding through the many-to-many catalog (D4). Decoded SIDs are not items. Three mechanisms from §5 make the SID↔item map many-to-many, and the decode layer honors all of them: (a) collision buckets — up to σ items share one SID under FORGE-style post-processing, so each generated SID expands into its item list in beam order before truncating to top-K (FORGE's calc_hr.py protocol: 100 beams × σ=5 yields up to 500 candidates [repo-alsid]); (b) Latte latent tokens — beams reaching the same item through different latent roots aggregate scores (agg_sum/agg_max; empirically equivalent [latte]); (c) PIT-style dynamic indices keep top-B SIDs per item [pit]. SidCatalogIndex (§5) owns this: the decode driver hands it (sid_tuple, score) pairs in beam order and receives a deduplicated, score-aggregated item ranking — dedup is item-level, not SID-level.

Level-agnostic driver. Per D5, the host driver delegates the number of scorer invocations to BaseSidDecoder (3 AR steps default, 1 for RPG-style parallel decode, T for diffusion — §7); nothing here may hard-code three levels or AR-only assumptions. The default 3×8192 geometry (§5) coexists with LC-Rec's 4×256 and semantic-ids-llm's 3+1 dedup-digit scheme [repo-semanticids]; trie, head, and catalog index take the level structure from the tokenizer artifact.

10.2 Constrained decoding (D12)

Decision D12 — per-prefix legality via a STATIC-style CSR tensorized trie; unconstrained beam is a supported config

tzrec implements catalog-grounded constrained decoding as a packed-CSR tensorized trie following STATIC [static][static-repo]: the catalog's SID set is flattened offline into CSR tensors (start_mask, dense head tables, packed_csr, indptr, layer_max_branches), so one constrained decode step costs a single batched gather. The trie is the canonical tzrec artifact; the masking logic ships as SidConstrainedLogitsProcessor with one consumer surface per execution environment — an HF LogitsProcessorList drop-in, a vLLM v1 engine-level processor, SGLang custom-LP and grammar-backend bindings, an RTP-LLM prefix_dict rendering, and a torch-native path inside the host beam driver — so engine-native constraint paths consume the same artifact rather than replacing it. Unconstrained beam search with post-hoc catalog grounding is a supported configuration, not a degraded mode: GRID found free-form decoding matches or beats constrained [grid]; PLUM launched at YouTube unconstrained with <5% hallucination [plum]. This resolves R-TRIE.

Why flat per-depth masks are insufficient

The cheap constraint — masking step-l logits to the level-l itemic range [151669 + l·8192, 151669 + (l+1)·8192) — guarantees only well-formed SIDs: at 3×8192 the leaf space holds 8192³ ≈ 5.5×10¹¹ combinations against ~2.5×10⁸ cataloged items, so 99.95% of well-formed leaves are vacant and beam mass parked on them is wasted width. Catalog-grounded generation needs per-prefix legality: which children of the emitted prefix lead to at least one real item. The naive implementations fail at scale: HF's prefix_allowed_tokens_fn is a per-row Python callback (a CPU stall per beam per step — the original R-TRIE concern), and GRID's vectorized alternative brute-force compares each candidate prefix against the full (N, H) codebook tensor, O(C·b·H) per step, with an in-code TODO conceding a trie is needed [repo-grid].

The STATIC CSR design

STATIC ("Vectorizing the Trie", YouTube/Google) is deployed at 100M–1B-item scale; its repo ships JAX and PyTorch references [static-repo]. build_static_index(sorted_sids (N,L), vocab_size, dense_lookup_layers=d) (static_decoding/csr_utils.py:21-203) emits five read-only tensors: start_mask (V,) of valid level-0 tokens; dense (V,)×d mask/state tables for the first d "hot head" layers where nearly every prefix exists; a row-major packed_csr (E+V, 2) int32 of [token, next_state] pairs, each state's children contiguous and token-sorted (V padding rows for clamp-safe reads); indptr (S+2,) int32 row pointers; and layer_max_branches, the per-depth maximum fan-out K fixing the static gather width. The constrained step (decoding_pt.py:45-116) is one batched "burst read":

starts = indptr[states]; lens = indptr[states + 1] - starts        # (B*W,)
pairs  = packed_csr[(starts[:, None] + arange(K)).clamp(max=last)] # (B*W, K, 2) one gather
cand_tok, cand_next = pairs[..., 0], pairs[..., 1]
cand_lp = logprobs.gather(1, cand_tok.clamp(max=V-1).long())
cand_lp = where(arange(K) < lens[:, None], cand_lp, -inf)          # invalid slots masked

Crucially this operates on candidate log-probs and never materializes a (B·W, V) mask: beam-score fusion (cum[:, :, None] + cand_lp.view(B, W, K), flat topk(W)) is strictly cheaper than mask-then-topk whenever K « V. The repo's sparse_transition_torch (decoding_pt.py:146-304) is a complete torch beam harness over this kernel — step 0 via start_mask, dense tables for depths < d, CSR elsewhere — the direct template for tzrec's host driver (§10.3). STATIC's deployed numbers (+0.033 ms/step, 948× over a CPU trie) are TPU v6e measurements at V=2048/L=8 — not quotable as GPU expectations, though our V=8192/L=3 burst reads are smaller per step.

One adaptation: STATIC assumes trie tokens are the model vocabulary; tzrec's itemic tokens live at offsets inside a ~176K LLM vocab. The trie stores per-level local ids 0..8191 (token id minus the level base, itemic_id_range, §7); consumers add the offset back when masking LLM logits — dense tables at V=176K would be absurd, at V=8192 they are 64 MB.

NaN guard. A row masked to all -inf makes log_softmax emit NaN, silently corrupting beam scores. Every reachable trie state has ≥1 child by construction, so the guard is an invariant assertion, not a fallback: arm only on states reached through the trie, and decode fixed-length with ignore_eos so EOS candidates are never siphoned into vLLM's completed-set rescoring.

SidConstrainedLogitsProcessor consumer surfaces

The same artifact backs five consumers. (i) HF: a LogitsProcessorList-compatible class for generate()-based eval — the vectorized CSR gather, explicitly not a prefix_allowed_tokens_fn closure (a Python loop over rows). (ii) vLLM v1: the engine-level interface (apply, update_state, is_argmax_invariant() = False), registered via --logits-processors, armed per request through SamplingParams.extra_args (RL rollouts, §9) or by detecting an <|sid_begin|> tail in prompt_tok_ids — prompt-tail arming also covers vLLM's emulated beam search, where each beam step is a fresh request carrying the full prefix. Two flags are load-bearing — logprobs_mode=processed_logprobs (default raw_logprobs snapshots logprobs before sampler-stage processors, hiding the mask from beam ranking) and max_logprobs ≥ 2×beam_width; composability limits in §10.5. (iii) SGLang, two grades: a per-request dill-serialized CustomLogitProcessor (zero server code; lazy-loads the trie from a custom_params path), or the production route — a sidtrie grammar backend registered via the public register_grammar_backend hook, one file implementing fill_vocab_mask/accept_token over the same CSR, inheriting scheduler integration, spec-decode rollback, and reasoning-gated activation [sglang]. (iv) RTP-LLM consumes the artifact as configuration: the build job renders the trie to tree_decode_config prefix_dict JSON for the in-engine TreeLogitsProcessor — small/medium catalogs; 250M items needs the loader patch of §10.5 [rtp-llm]. (v) The host driver (§10.3) calls the gather directly on the scorer's logits, sliced to the level-local range before log_softmax.

Unconstrained mode and post-grounding

The trie is a config knob, not an architectural dependency. GRID's ablation found free-form beam search matches or beats constrained (Beauty R@10 0.0609 vs 0.0597) and is cheaper — the model learns valid SID structure from SFT alone [grid]; PLUM shipped YouTube production retrieval unconstrained with <5% invalid SIDs [plum]; OneRec likewise serves without a trie [onerec-v1]. Unconstrained output is grounded post-hoc by SidCatalogIndex with an exact-then-prefix fallback (full tuple, then 2-token and 1-token prefix buckets) — semantic-ids-llm's graceful degradation, exploiting the hierarchy so even a hallucinated leaf resolves to semantically nearby items [repo-semanticids]. Recommended defaults: constrained for RL rollouts (legality shapes reward, §9) and serving at 10⁸-item scale; unconstrained as the cheap baseline for eval ablations and small catalogs.

Trie artifact build job

The trie is built offline from the same item_id → SID assignment table that feeds SidCatalogIndex (§5): sort the deduplicated (N, L) code array, run build_static_index (vectorized NumPy — minutes at 250M rows), and write an .npz with the five tensors, per-level bases, and the tokenizer fingerprint. On request the builder also emits the RTP-LLM prefix_dict JSON rendering — a format that enumerates per-prefix candidate lists explicitly, so bounded to small/medium catalogs (§10.5). The artifact is versioned with, and invalidated by, the SID assignment snapshot — the re-tokenization cadence (R-RETOK, §15) governs trie refreshes, and serving swaps trie, catalog index, and cached prefix sets atomically as one unit. With Latte latent tokens, the builder prepends the latent level as depth 0 (M′ extra root branches); nothing else changes (D16).

10.3 Single-step scorer export (D13)

Decision D13 — host-side beam driver over an AOTI-exported single-step scorer with explicit KV-cache I/O

The serving artifact for the default Qwen3 backbone is a torch.export + AOTInductor package exposing one decode step with the KV cache as explicit inputs, plus a prefill graph; the beam loop (level masking, trie gather, top-k, cache reorder) runs host-side in the serving processor. This is empirically proven on this machine (torch 2.11.0+cu129, transformers 4.53.3, tiny Qwen3, bf16, sdpa): the step exports strictly and matches eager StaticCache decoding bit-exactly over 5 sequential steps; compiles and runs under AOTI (fp32-exact; bf16 within ≈5e-3–8e-3 fusion noise); survives host-side index_select beam reorder; and the explicit-I/O variant compiles with Dim("batch") so one artifact serves any users×beams product. R-EXPORT is thereby de-risked from design assumption to validated recipe with enumerated residuals. The legacy FX-trace + TorchScript path is a NO-GO for HF models.

Revision — serving-implementation analysis (2026-06-10)

The host-loop recipe is hereby scoped by its KV memory. Qwen3 GQA KV (8 kv-heads × head_dim 128, bf16) costs 4 KB per token per layer — 112 KB/token at 28 layers (0.6B/1.7B), 144 KB at 36 (4B/8B) — so an S=3000 context holds ~344 MB (1.7B) / ~442 MB (8B) of KV per sequence. The D13 design duplicates that per beam (repeat_interleave prefill, one StaticCache row per beam): W=32 costs ~11 GB — eval-friendly on an L20-48G — but W=512 costs ~176 GB: impossible on any single GPU tzrec targets. The naive host loop is therefore an eval/small-W recipe (W ≤ ~32–64 at 3K contexts), not a beam-512 production path; wide synchronous beams need the batched-beam decode module of §10.4 or the tzrec-gr-serving runtime (§10.6), whose context KV is shared across beams by construction. Two clarifications: the smoke ran no prefill (prefill-graph export remains residual obligation 3), and per-beam duplication is the D13 design, not something the smoke proved. None of this displaces the decision: the AOTI single-step scorer plus host beam driver remains the export-clean default, the only path for HSTU, and the path for encoder-decoder and lazy-decoder backbones until the corresponding gr_models families land (§10.6, P4/P5).

The proven recipe

The export rides transformers' own supported path. On the 4.53.x pin, TorchExportableModuleForDecoderOnlyLM (integrations/executorch.py, v4.53.3 L30-72) wraps the model with a StaticCache whose per-layer [batch, num_kv_heads, max_cache_len, head_dim] tensors are registered as non-persistent buffers, and swaps attention to sdpa_without_vmap (4.53's default mask construction uses torch.vmap, which does not trace under torch.export). forward(input_ids, cache_position) derives positions, RoPE, and the causal mask from cache_position inside the graph; the cache's in-place index_copy_ mutation is preserved through export and AOTI — PyTorch 2.11's AOTI suite includes a literal KV-cache buffer-mutation test (test/inductor/test_aot_inductor.py:2835) plus bf16 SDPA coverage [pytorch][transformers]. Vocab extension to 176,384 rows (§7) is invisible to the recipe — it only resizes embed_tokens and the untied lm_head.

The HF recipe fixes batch×beam at export time (cache buffers are module state). The preferred tzrec variant therefore moves the cache to explicit I/O, which was also smoke-tested end to end including dynamic batch:

def forward_step(
    input_ids:      torch.LongTensor,   # [N, 1]   N = B_users x W_beams  (Dim("batch"))
    cache_position: torch.LongTensor,   # [1]      absolute position = prefix_len + step
    *kv_caches:     torch.Tensor,       # 2L x [N, H_kv, S_max, D] bf16, mutated in place
) -> torch.Tensor:                      # logits [N, 1, vocab ~ 176384]

Explicit I/O keeps all state host-owned (no hidden tensors in the artifact), matches tzrec's "dense module + explicit inputs" export style, and enables one artifact across beam widths. The buffers variant remains a proven fallback: AOTICompiledModel.load_constants(..., user_managed=True) adopts host tensors by storage pointer, so the host can still reorder cache rows between calls. Prefill needs no separate cache input — a prefill graph (dynamic sequence Dim) or host-side eager prefill writes positions [0, prefix_len) of the same tensors, once per user then repeat_interleave(W, dim=0), or directly at N with repeated prompts.

Host beam loop and measured parity

Per SID step the host masks logits to the legal level range (plus the trie gather of §10.2), takes log_softmax, accumulates beam scores, runs a flat topk(W) over [B, W·K], and reorders every cache tensor with kv.copy_(kv.index_select(0, beam_idx)). For 3-token SIDs only steps 2 and 3 reorder — step 1 is pure expansion — so total reorder traffic is two copies of [B·W, H_kv, S_max, D] per layer per request: microscopic at W ≤ 32 — and at 512 the copies would be affordable, but the duplicated cache itself is not (revision callout above). The smoke matrix (tzrec120 env, A10; R-EXPORT follow-up study, 2026-06-10) adds to the callout's results: strict export at batch 8; max_abs_diff = 0 over 5 steps; bit-exact next-step parity after host reorder; AOTI with user_managed host-owned KV; explicit-I/O in-place kv mutation correct under both ep.module() and AOTI; one Dim("batch", 1, 512) artifact serving B ∈ {4, 8, 24}.

Gotcha 1 — AOTI silently corrupts on non-contiguous inputs

Feeding the AOTI artifact a strided slice such as generated[:, -1:] (shape (B,1), stride (S,1)) produced garbage logits with no error — on CPU and CUDA, even with zero transformer layers. AOTI reads the raw data pointer assuming example-input strides; ep.module() handles the same input fine, so the bug hides from eager-parity testing — and host beam loops naturally produce exactly such slices. The tzrec step-runner must .contiguous() every input and assert strides before each call.

Gotcha 2 — the AOTI .so requires GLIBCXX_3.4.30

dlopen of the compiled package failed on a system libstdc++ lacking GLIBCXX_3.4.30; fixed via LD_LIBRARY_PATH=$CONDA_PREFIX/lib. PAI serving images must carry a matching toolchain runtime on the dlopen path — this is a deployment-image checklist item, not a code change.

Residual proof obligations (tracked in §14): (1) a real-scale artifact — Qwen3-0.6B/1.7B with the 176,384-row vocab — measuring compile time, .pt2 size, load time, and per-step latency on PAI GPUs; (2) per-row cache_position [N] for mixed prefix lengths in one batch (only scalar positions were tested; left-padding within a user group sidesteps it); (3) prefill-graph export with a dynamic sequence dim (documented HF recipe, not exercised here); (4) re-running the matrix before any transformers-v5 pin move — real drift: key_cache lists become cache.layers[i].keys, a new cumulative_length_{i} buffer needs constant handling and per-generation reset, and v5's Cache.reorder_cache rebinds attributes via index_select, breaking exported-buffer aliasing (reorder stays copy_-based). Cache-internal access goes through a thin version shim from day one.

Fit with tzrec's existing export machinery

The AOTI step package is a sibling of what tzrec already produces: tzrec/acc/aot_utils.py:export_model_aot runs torch.export.export with a dynamic Dim("batch") (L177) and aoti_compile_and_package into aoti/aoti_model.pt2 (L226-235), with load_model_aot on the predict side — the scorer reuses the same compile config and loading path. The default FX route (export_model_normal, export_util.py:148: symbolic_trace + torch.jit.script) does not survive HF's config-dependent control flow and cache objects. The RTP export path is likewise out of scope for the LLM trunk — embedding-table-oriented, hard-rejecting DTensor parameters (export_util.py:484) — though its batch_size==1 convention ("when batch_size=1, we assume gr model", export_util.py:1010-1011) already anticipates GR-shaped serving.

10.4 KV-cache strategies

Four cache designs form a ladder from "ship now" to "optimize later"; the host-driver contract (scorer in, logits out, cache reorderable by row) is identical across them, so they swap without touching decode logic. The organizing question for wide beams: does a design share KV storage or KV bandwidth across one user's W beams? The geometry (revision, §10.3): 4 KB/token/layer bf16, so an S=3000 context is ~344 MB (1.7B) / ~442 MB (8B) per sequence.

Sharing storage vs sharing bandwidth

Storage sharing is what engines do, and it does not touch the decode wall. RTP-LLM forks beam KV copy-on-write and vLLM's prefix cache dedups blocks — memory is shared well; attention still runs per sequence, every beam reading the full context KV each step. At S=3000, W=512 that is 176 GB (28 layers) / 227 GB (36 layers) of KV reads per user per step — 204–262 ms/step at L20's 864 GB/s DRAM as the upper bound. L20's 96 MB L2 can soften this (one user's ~12 MB/layer working set fits when few users are in flight), so a per-sequence engine beam sits 3–30× off a purpose-built path depending on L2 residency — the DRAM-bound branch matching RTP-LLM's own ~0.8–1.2 s/request cost model (§10.5). vLLM's emulated beam pays a further, different tax: per-step request re-issue and tail prefill put it at roughly 10–50× a native fork-KV beam step [vllm].

Bandwidth sharing is the mechanism that removes the wall: beams in the query dimension. Batching the W beams as query rows against the user's shared context KV — q [B, H, W, D] over kv [B, H, S, D] — reads context KV ~once per user per layer per step (~0.34 GB/user/step at 28 layers: sub-ms). This is measured, not modeled: on A10 at S=3000, W=512, one layer, beams-in-query ran 0.105 ms vs 14.2 ms for per-sequence attention over duplicated KV — 135× at the kernel level. It is also why OneRec-V2 serves beam 512 in 36 ms [onerec-v2] — with the mandatory caveat that V2 is a 1B lazy decoder: context KV shared across beams by construction, ~94% of compute cut. A Qwen3-class backbone with batched-beam attention removes the KV wall but does not approach 36 ms — the GEMM/weight-read floor plus full prefill remains.

(a) HF StaticCache with host index_select reorder — the default. Per-layer [N, H_kv, S_max, D] tensors, reordered by index_select at beam pruning. Proven bit-exact above; trivial host logic; two reorder copies total for 3-token SIDs, far below the dense-projection floor recsys-examples measured. The binding constraint is capacity, not traffic: per-beam duplication caps it at W ≤ ~32–64 at 3K contexts on an L20-48G (revision, §10.3) — the eval and narrow-beam default, not a wide-beam path.

(b) The batched-beam decode module — bandwidth sharing on a Qwen3-class backbone (demand-gated). Decode steps only; prefill and weights untouched; parity-testable against the naive path at small W. W=512-synchronous on a Qwen3-class backbone has no production precedent — OneRec-V1 served at W=128 [onerec-v1], V2's 512 rides the lazy decoder, OpenOneRec uses 32 [openonerec], FORGE's 1200 is nearline [forge] — so this module is a demand-gated contingency, never a committed milestone (§10.7). Three implementations of the mechanism:

  • (c1) recsys-examples gr_decode_atten — the kernel-proven implementation. Exactly this split: K1 runs context attention with beams as the query tile (plus split-KV) over the shared prefill slab, K2 gathers per-beam step-major generated KV by ancestry indices without reordering, K3 merges via LSE combine [repo-recsysexamples]. GQA-generic (head_q % head_kv == 0; Qwen3's 16/8 in-envelope), bf16/fp16, W tested to 1024, L20 (SM89) dispatches the SM80 kernel class; ragged contexts need the 3-kernel backend. Sober scoping: it is not a wheel (CuTe-DSL JIT, cutlass-dsl 4.4/4.5 pins); it is softmax-only (the LSE combine), hence inapplicable to HSTU's silu attention — recsys-examples' own HSTU stack never uses it; the flat beam-KV layout needs uniform per-step widths; its nonstandard attention signature forfeits the plain-HF export recipe; and their own Nsight profile shows the entire decode-attention bucket at 1.593 ms of a 23.3 ms decode stage — non-attention kernels dominate. The kernel is one implementation of the mechanism, not the magic: K1 is plain flash attention at Sq=W (what the two-phase flex path reproduces), and K2's ancestry-gather avoids a reorder worth ≤4 MB/layer at 3-token SIDs — microseconds via torch.gather.
  • (c2) Two-phase flex_attention + exact LSE combine — the torch-native default. Phase 1: flex_attention of the W beam queries over the shared context KV, requesting the log-sum-exp via return_aux (return_lse is deprecated); phase 2: per-beam attention over the ≤3 generated tokens (tiny); merge by the standard flash-decoding LSE combine, which is exact (verified to 6e-16 in fp64), not approximate. An export + AOTI smoke of flex_attention-with-LSE passes at toy scale on torch 2.11 [pytorch]; the Qwen3-scale artifact (real head counts, 3K context, W=512) remains a D13-style proof obligation (§14).
  • (c3) Stride-0 expanded-KV SDPA. Expanding context K/V across beams as a zero-stride view works on all SDPA backends and stays a view under export+AOTI (no materialization) — but it shares storage, not bandwidth: beams-in-batch still reads KV per batch row and measured ~100× slower than beams-in-query. A memory-saving fallback only.

(c) delta_hstu_mha jagged cache — the in-house backbone's native path. For InhouseHstuGRBackbone (D5), tzrec already ships the prefill/decode cache machinery: STULayer holds per-layer k_cache/v_cache with jagged offsets, prefill populates them via kv_caching_lengths, and cached_forward runs delta_hstu_mha over cache+delta (tzrec/ops/hstu_attention.py:199-267, tzrec/modules/gr/stu.py:564-641). Constraints: delta size uniform per row (beam steps append one token per row — fits), CUTLASS falls back to Triton on the delta path, and the cache lives as Python module attributes — fine because the loop is host-side; beam forks reorder by re-indexing k_cache/v_cache and offsets, mirroring (a).

(d) recsys_kvcache_manager for cross-request session reuse. For high-QPS deployments where a user's history prefix recurs across requests, NVIDIA's standalone paged GPU KV cache with host/FlexKV tiered offload keys cache identity by user_id — histories never share prefixes across users, so engine-style token-prefix matching buys nothing while user-keyed reuse skips re-prefilling thousands of history tokens. Measured on the HSTU inference example: 1.3–2.6× end-to-end with CUDA graphs [repo-recsysexamples]; the paged attention entry it feeds belongs to the same fbgemm_gpu_hstu wheel lineage tzrec already ships. This slots under topology (a) of §10.7 as a phase-2 optimization for the HSTU and lazy-decoder backbones.

Three OSS engines and one purpose-built runtime were studied at source level: vLLM at HEAD fe1d923af (v0.22.1+ dev), SGLang at b40f365732 (v0.5.12.post1), RTP-LLM at OSS v0.2.0 b0c09a344, and recsys-examples at 2550a9c4 (v26.05+; re-verified unchanged at 5dc46a217d, 2026-06-30 — zero drift in either serving tree) [vllm][sglang][rtp-llm][repo-recsysexamples]. vLLM — eval harness, emulated beams, and a rescoring channel

The seven claims established at v0.12 hold at HEAD [vllm], with refinements. BeamSearchParams still carries no constraint fields and no vllm#21546-class fix ever landed; logprobs_mode landed in v0.10.0; grammar bitmasks apply before the sampler, so structured-output masks are visible even in default raw_logprobs — only sampler-stage processors need processed_logprobs. Two hard conflicts: custom logits processors are rejected under speculative decoding (structured outputs do compose), and Model Runner V2, now default for Qwen3 dense, silently falls back to the V1 runner under custom LPs or prompt embeds. tzrec uses vLLM for offline eval at widths 10–50, OpenOneRec-parity checks, and SID-shortlist rescoring: the generative scoring API with SamplingParams.logprob_token_ids (≤128 ids) scores pre-retrieved candidates teacher-forced — one prefill per user, no decode loop. Constrained beams always run through tzrec's custom client loop (~200 lines) carrying per-step allowed_token_ids or extra_args, which also unlocks FORGE-style variable widths; prompt embeds now hash into prefix-cache keys but remain blocked in the beam API. Operational keys: --max-logprobs ≥ 2W, --max-num-seqs ≥ 512, --logprobs-mode processed_logprobs, --logits-processors. R-VLLM-PIN is re-anchored at v0.22.1+ (§15).

SGLang — the RL-rollout engine, not a beam engine

SGLang still has no beam search of any kind — a repo-wide grep at HEAD returns zero hits, not even a vestigial field [sglang] — so it stays off the beam-faithful eval and production paths. For GRPO sampling rollouts (§9), however, it is arguably the best-equipped engine. The per-request custom logit processor walks the full 250M-item trie with zero server code; the registered sidtrie grammar backend is the cleaner server-side route (both in §10.2); per-level itemic masks alternatively fit a ~0.4 MB EBNF — provided the itemic tokens are registered as non-special added tokens, or xgrammar cannot emit them. ReasonerGrammarBackend keeps any grammar dormant inside <think>…</think> and arms it after, with a per-request thinking_budget — CoT-then-constrained-SID in one request, the exact two-stage rollout shape. The radix cache at page_size=1 lets N CoT samples pay the ~3K-token prefill once; returned logprobs are post-constraint and renormalized (the semantics GBPO logging wants); the verl weight-update surface (update_weights_from_*, memory release/resume, --rl-on-policy-target fsdp bit-exact inference) is the richest of the three. Caveats: input_embeds forces --disable-radix-cache server-wide; the enc-dec whitelist is three audio/vision models; emulated beam-512 monopolizes the scheduler — a diagnostic, not a topology.

RTP-LLM — engine-native beams plus an OSS generative-rec constraint stack

The deep study refutes this RFC's earlier claim that the OSS build has no logits-processor or constrained-decoding hook [rtp-llm]. OSS RTP-LLM ships a C++ processor framework inside Sampler::forward, before beam top-k: TreeLogitsProcessor does catalog-trie constrained decoding configured purely by --tree_decode_config (a JSON prefix_dict of per-prefix candidate lists), per-beam DFA state forked and reordered on every beam transition; RecommendationLogitsProcessor (landed 2026-04-29) is purpose-built for SID rec — combo_token_size treats every k consecutive tokens as one item, banned_combo_token_ids/auto_parse_banned_combo filter already-exposed items parsed from the prompt, items dedup across beam steps, a Qwen3 think-prelude is skipped via DFA. This is the AL-GR/FORGE production path, in OSS [forge]. The native beam path is stronger than assumed: variable_num_beams gives per-step widths (step 1 always width 1; kernels padded to width 4096); beam KV forks block-level copy-on-write, so a 3000-token prompt is held once; masks apply before log_softmax, so cum_log_probs renormalize over the allowed set. Limits, stated plainly: speculative decoding does not compose with beam requests; the beam path is deterministic (per-beam temperature ignored — RL rollout diversity must come from the CoT sampling stage); and three patch surfaces gate large-scale adoption — (1) generateVocabMask builds a dense CPU [B, V] mask and H2D-copies it every step (~90 MB/step at beam 512; ~100–200 LoC sparse-write fix), (2) loadPrefixDict (JSON plus a string-keyed hash map) does not scale to 250M items — compact binary format plus a reload RPC, (3) single-request CoT-then-beam is inexpressible because the width schedule indexes absolute output position — patch numBeams to index from the think-end DFA position, or run two requests over the 64-token-block prefix cache (REUSE_CACHE=1, 3-tier). Cost model: ~0.8–1.2 s/request at beam 512 over 3000-token contexts, bf16 on L20 — the per-sequence KV wall of §10.4; fp8 KV halves it; FORGE's 200 ms envelope corresponds to ~500-token contexts. The trie build job (§10.2) also renders the SidTrie as prefix_dict JSON for small/medium catalogs.

recsys-examples sid-gr-inference — the purpose-built SID-GR runtime, adopted as seed

PR #414 added examples/sid-gr-inference: a standalone, pip-installable, pure-python SID-GR serving framework whose sole dependency is huggingface-hub [repo-recsysexamples]. Anatomy in one line: a minimal Qwen3 nn.Module over stock HF safetensors (tzrec's 176,384-row vocab flows through as config), dense ContextKV/BeamKV lease pools — exactly the beams-share-context split the bandwidth analysis of §10.4 demands — a continuous-batching scheduler grouping decode ticks by (step, W, next_W, ctx_len), decode/prefill CUDA graphs, an SGLang-compatible /generate server, TokenTrie + JSONL catalog with /catalog hot reload/rollback, and dynamic beam-width policies (fixed / scheduled / score-margin — the FORGE shape). Benchmarked against SGLang's beam-search PR branch (Qwen3-1.7B, ctx 1000/5000, W=256, 3 tokens): 1.90–2.27× offline, 1.85× online, top1 exact match 1.00. Caveats: a self-declared single-node, single-GPU alpha, validated on H100/bf16 only; the trie step-mask is a batch-1 per-beam python walk; the 176K-vocab topk is unoptimized; no CI; no torch.export/AOTI story — the maturity gap is tracked as R-SIDGR (registered in §15). Adoption is no longer a posture but a plan: §10.6 specifies the tzrec-gr-serving fork, its verified extension seams, and the per-backbone workstreams.

10.6 The tzrec GR serving runtime (BackboneAdapter over sid-gr-inference)

Scope and packaging — tzrec-gr-serving, a fork of sid-gr-inference behind the BackboneAdapter contract

The tzrec GR serving runtime is a fork of sid-gr-inference, shipped as a separate deliverable — tzrec-gr-serving, consuming tzrec export artifacts — and serving all softmax-attention GR backbones: Qwen3/Qwen2.5 token-only, the lazy decoder, and encoder-decoder. This retires the earlier "HF-Qwen3 family" scoping: the seam audit below shows the scheduler, KV pools, CUDA-graph runners, and server are model-agnostic, so the fork keeps ~13K lines of runtime while the engine-invariant core of §10.1 stays tzrec code in front of and behind it. HSTU is the sole exception — silu attention falls outside seam S3 — and stays on tzrec's own stack (topology (a), §10.7; delta_hstu_mha cache, §10.4). tzrec/acc/gr/ keeps the library pieces both paths consume: the AOTI scorer, the trie builder, SidCatalogIndex, and artifact versioning. examples/sid_gr is a kernel+concept donor only, never a checkpoint source (below).

Why the base holds: the verified extension seams

The fork bet rests on source-verified seams, not optimism. The model contract is duck-typed — the engine holds model: Any, and exactly one Qwen3 import exists outside gr_models (an optional JIT beam-KV write kernel): a new backbone is a new gr_models family plus a bring-up script, zero engine edits. "Prefill" is a model method, not a scheduler concept — the scheduler flips a stage flag and the executor calls the model's prefill — so a model that fills ContextKV by projection, with no causal prefill at all, is legal. allocate_context_kv is model-owned, making per-family pool schemas a model-side declaration. step is passed into the decode step, so position-specific heads are legal model-side. Beam widening is mechanically supported (next_W ≤ W·V; the scheduler already groups by (step, W, next_W, ctx_len); config defaults cap width 128 / 3 steps; the kernel is tested to W=1024). CUDA-graph capture wraps the whole decode step model-agnostically. RoPE is model-side and KV is stored post-rope, so non-RoPE architectures simply skip it. The CuTe decode kernel is optional per layer — SDPA fallback costs ≤1.6 ms per their own Nsight profile — so the kernel bounds performance, never feasibility. And decode control is beam-faithful and extensible: trie masks renormalize beam scores post-mask, and /generate ignores unknown JSON fields.

Five seams are genuinely fixed and bound the extension work:

  • S1 — token-ids-only ingress. /generate accepts input_ids only, and ctx_len := input_ids.shape[-1] drives pool leases and batch grouping. The one runtime extension both OneRecs force.
  • S2 — causal-prefill assumption in the stock prefill backends and prefill CUDA graph; bypassable model-side, since the model owns its prefill method.
  • S3 — softmax-only single-merged-softmax decode kernel, no attention-bias argument: cannot express enc-dec two-softmax sublayers, T5 relative bias, or HSTU silu. Kept optional; SDPA/flex fallback (§10.4 c2) for the enc-dec families.
  • S4 — batch-1 CPU python trie — replaced by tzrec's tensorized CSR gather (§10.2) regardless (R-SIDGR).
  • S5 — kernel W validation ceiling 1024: FORGE's 1200 needs a validation run (or a torch-topk path).

Per-target suitability

Two rows deserve emphasis. OneRec-V2 is the best architectural fit: its ContextProcessor emits per-layer K/V directly — it is the ContextKV abstraction minus the prefill — and with L_kv=1, K=V, and MQA the entire per-request context state is one [B, 3000, G_kv, d_head] slab, ~0.77 MB/request at S=3000 [onerec-v2]. It goes first in the extension sequence: the best fit and the production W=512 architecture. And the sid_gr row is a cautionary tale, not a work item: examples/sid_gr and examples/sid-gr-inference are two disjoint stacks in one repo — the table row's KJT/NoPE/per-rank training side against an HF-dirs-only loader with hardcoded Qwen3 anatomy (mandatory qk-norm, RoPE) and input_ids-only /generate — with no converter anywhere in the repo. The train-to-serve checkpoint contract must be designed, not assumed; WS4's export tool is the structural answer.

Workstreams WS0–WS6 and sequencing

  • WS0 — fork governance. Fork under tzrec test conventions; pin the upstream commit; send generic fixes upstream.
  • WS1 — the existing R-SIDGR items. CSR trie gather (S4), context-length bucketing, L20 validation, EAS packaging, the prompt frontend (token-id assembly shared with training).
  • WS2 — formalize BackboneAdapter, an ABC over the duck protocol, kept upstream-compatible: encode_context (causal prefill | encoder + per-decoder-layer cross-KV precompute | ContextProcessor projection), step-aware decode_step, allocate_context_kv, and head/vocab-layout metadata — flat 176,384 vs level-local 8192, the local layout matching the CSR trie's per-level local ids (§10.2).
  • WS3 — S1 embedding ingress. First: request-side fp16 context tensors (~0.8–1 MB/request), with admission/memory/grouping keyed on embedding count and session ContextKV reuse keyed user_id+feature-version instead of token hash. Phase 2: in-process tzrec EmbeddingGroup/fg for co-located tables.
  • WS4 — model zoo + checkpoint contract. gr_models families: qwen2_5 (QKV bias=True, no qk-norm; ~150–300 LOC), lazy_decoder (slab ContextKV, 3 position-specific heads; ~600–1000 LOC), enc_dec (bespoke SDPA attention, two softmaxes per layer, optional T5 rel-bias, MoE FFN; ~800–1200 LOC). Per family, a tzrec DCP→safetensors manifest export tool emits config.json + weights + tokenizer/SID manifest — the train-to-serve contract that prevents a sid_gr-style rift by construction.
  • WS5 — decode control. Level-offset mask processor (~50 LOC — a static range mask, not the trie); tensorized σ-bucket SID→item expansion (~100 LOC) in SidCatalogIndex grounding; beam caps raised plus the S5 W=1200 validation; post-mask per-token gen_prob emission (below).
  • WS6 — perf/ops. Per-family CUDA-graph validation (including SDPA-fallback graphs), the L20 matrix per family, session ContextKV reuse.

Sequencing. P2 delivers WS0–WS2 plus the FORGE-dec qwen2_5 adapter as the cheap contract proof (~0.5–1 wk for the adapter itself): a second family is what proves BackboneAdapter is an abstraction, not a Qwen3 rename. P4/P5 deliver WS3 plus lazy_decoder first (OneRec-V2: best fit and the production W=512 architecture; ~2–3 wk including ingress), then enc_dec (OneRec-V1; ~3–5 wk); T5 parity is optional.

gen_prob logging. The runtime's response per-token logprobs are currently pre-mask (true unconstrained probabilities), while beam scores renormalize post-mask. gen_prob_logged (§6) must reflect the masked distribution actually decoded from (the logging contract of §10.7; GBPO, §9), so WS5 adds a post-mask per-token variant — a small change, since the renormalized scores already exist at selection time.

10.7 Serving topologies

(a) The tzrec GR serving runtime — the engine-free default, now a three-rung ladder. The foundation is unchanged: load the AOTI step package (and prefill graph), the trie .npz, and the SidCatalogIndex snapshot; run the beam loop, trie gather, and grounding in the serving process; per-beam outputs ride the TARGET_REPEAT_INTERLEAVE_KEY flatten/regroup path into ListArray columns. It remains the only topology requiring no engine patches and the only one serving all backbones, including encoder-decoder and HSTU (which no OSS LLM engine speaks). M-a: a python EAS custom container running tzrec-gr-serving (§10.6), covering every softmax-attention backbone as its gr_models families land — gang/bucketed batching over the fixed 3–4-token decode, with honest ops caveats: the hot path holds python between kernels (one straggler stall inflates a gang batch's p99); per-replica assets are multi-GB (~1–1.5 GB CSR trie plus ~2–4 GB grounding map at 250M items, in tension with fractional-GPU pooling); EAS python containers lack the warmup/rolling-update/health conventions of the C++ processors. M-b: a C++ TorchRecProcessor extension — the AOTI runner plus a ~200-line libtorch beam loop — demand-gated on measured M-a QPS shortfall. M-c: the batched-beam decode module (§10.4) — demand-gated on a demonstrated W=512-synchronous product need, which no Qwen3-class production system has yet exhibited.

(b) RTP-LLM — engine-native beams with an OSS constraint stack: the FORGE production shape. Per §10.5, everything FORGE's serving contract needs is config-only: variable_num_beams=[300,600,1200] reproduces the dynamic schedule, tree_decode_config grounds beams in the catalog trie, and RecommendationLogitsProcessor supplies the exposure filter and cross-step item dedup [rtp-llm][forge][repo-alsid]. At small/medium catalog scale this is the lowest-code path to production-width beams in the nearline/short-context envelope — FORGE's actual deployment shape: tzrec exports the SidTrie as prefix_dict JSON, the engine returns per-beam cum_log_probs renormalized over the allowed set, and SidCatalogIndex grounds the result. At 250M items the two named patches gate adoption, and 3000-token synchronous contexts pay the per-sequence KV-read wall (§10.4; fp8 KV or shorter serialized contexts buy back the 200 ms envelope). Recommended where Taobao-side infra alignment matters or production beam widths rule out emulation; the engine is PAI-ecosystem native (iGraph, vipserver, kmonitor in-tree).

(c) vLLM serving — eval harness and emulated beams for token-only Qwen3 models. vLLM's beam search remains a pure-Python emulation at the API layer: one 1-token request per live beam per step — 2W+1 requests per 3-token decode — with prefix caching bounding per-beam recompute to a ≤16-token tail; block-aligning the prompt so SID tokens start a fresh block cuts step cost ~4× [vllm]. Constrained beams run through tzrec's custom loop plus the engine-level SidConstrainedLogitsProcessor with prompt-tail arming and processed_logprobs (§10.2). At width 512 on L20: ~0.2–1.4 s/step for Qwen3-8B (~1.5–3.5 s/request) — fine for offline eval and OpenOneRec-parity at W ≤ 50, marginal for synchronous production at 8B; 0.6–1.7B models drop to ~30–100 ms/step and are production-plausible. The cautions of §10.5 apply (V2-runner fallback; the spec-decode/custom-LP conflict; R-VLLM-PIN API drift, §15). The OpenAI-endpoint + custom-LP + processed_logprobs combination is source-verified but not yet GPU-smoke-tested — it gets a validation run before this topology is declared supported (§14).

SGLang's role — RL rollouts and offline CoT, never beams. With no beam path SGLang cannot serve beam-faithful retrieval, and emulation monopolizes its scheduler (§10.5). Its place is the training side — the recommended GRPO rollout engine (§9) — and the natural harness for offline CoT-generation jobs feeding distillation and nearline reasoning corpora.

(d) Nearline patterns — how reasoning-class models actually ship. No published system serves LLM reasoning synchronously; three deployed patterns cover the spectrum. FORGE async-trigger cache: generation fires on a user event (item-detail click), the RTP-LLM-served path writes 1200 candidates to a cache table within 200 ms (the cache write is application-side), and the existing ranking stage consumes the cache — generative retrieval as a feeder, off the latency path [forge]; the engine-side half is visible in OSS: RecommendationLogitsProcessor bans already-exposed items at generation time [rtp-llm]. OneRec-Think Think-Ahead: an offline reasoning job beam-searches the first 2 SID levels of T reasoning paths per user into a per-user prefix cache; online, a small realtime OneRec decodes the final level constrained to the cached prefixes [onerec-think] — "constrain to cached prefix set" becomes an alternative trie root, a small CSR generalization. OneReason fast–slow thinking: nearline hierarchical decoding (beam at the top level, greedy below) feeds a Redis candidate pool with realtime-model fallback, while "Thinking Token" representation distillation (supervising the realtime decoder's BOS hidden state) carries reasoning value into the synchronous path [onereason]. Think-Ahead prefix caches reuse the trie machinery; the distillation bridge is a §14 roadmap item.

First production for token-only Qwen3: the nearline envelope first; the substrate settled by a bake-off. The first production exposure goes behind the FORGE-style async cache table regardless of substrate — it removes the latency-wall problem class and is production-proven at Taobao scale [forge]. Underneath, the substrate is a genuine tie: (b) RTP-LLM — ops-hardened, PAI-native, per-sequence beams, two named patches at 250M catalogs — versus (a)/M-a running tzrec-gr-serving (§10.6) — beam/trie fidelity, dynamic widths, torch-native ownership, template parity by construction, upstream alpha maturity (R-SIDGR). An L20 bake-off settles it cheaply; both expose SGLang-compatible APIs. Capacity, not latency, is the nearline sizing question: 100M candidate refreshes/day at ~0.2 s GPU each ≈ 230 sustained L20s — active-user tiering and trigger policy, not engine choice, are the sizing levers. One bound holds on every path: 8B synchronous serving is prefill-bound (~0.4–0.8 s at 3K contexts on one L20) and needs TP — a flag for the engines, a new sharded-serving subsystem for the native runtime. In-house backbones are unaffected: they stay tzrec-native on (a), borrowing at most recsys_kvcache_manager and the paged fbgemm_gpu_hstu attention arguments — the same wheel lineage tzrec already ships (§10.4).

Exposure-time generation-probability logging. Every topology also owns the producing side of a training contract: GBPO (§9) populates pi_old from the family-4 column gen_prob_logged : ARRAY<DOUBLE> (§6) — the per-token generation probabilities of the exposed item's SID tokens, logged at exposure time. The values are computed in every topology and need only be carried through: the host driver (a) attaches per-token log_softmax scores from beam accumulation to each (sid_tuple, score) pair handed to SidCatalogIndex and emits them per grounded candidate (collision-bucket items expanded from one SID share that SID's token probabilities, D4) — under M-a via the post-mask emission of WS5 (§10.6); engines (b)/(c) return per-token logprobs natively — post-mask by construction in RTP-LLM and SGLang (masking precedes the softmax; no pre-constraint channel), via processed_logprobs in vLLM — so the logged value always reflects the masked distribution actually decoded from (§10.5); nearline patterns (d) persist them alongside candidates in the cache table for the downstream exposure×feedback join. The fallback is contractual: non-generative impressions carry no column and the trainer falls back to ratio = 1 — the regime GBPO is designed to keep stable — so logging is mandatory only on generative impressions, never a backfill of the full exposure log.

11. Distributed training: leveraging PyTorch, Megatron-LM, Transformers, Accelerate, and verl

This section is the framework-leverage contract for the whole RFC: for each of the five external frameworks (plus TorchRec, the rollout engines, and recsys-examples) it states exactly what tzrec depends on, what it deliberately does not adopt, and why — with every claim grounded in source inspection of the pinned checkouts (PyTorch release/2.11, megatron-core 0.18.0, transformers, accelerate 1.14.0.dev, verl 0.8.0.dev). It then specifies the two-world integration that joins the TorchRec sparse world and the dense LLM world on one torch.distributed WORLD (§4), owns decision D8 (dense parallelism backends), and closes with a parallelism selection guide anchored to published MFU numbers. 11.1 Verdict table

Revision vs prior spec §3(f) — FSDP2 becomes the default dense backend

The prior internal spec made Megatron-core the default dense-world backend with FSDP2 as fallback. This RFC inverts that: FSDP2 (fully_shard) is the v1 default and Megatron-core 0.18 is the scale-out backend for MoE/EP and ~8B+ models. Three findings drove the inversion: (a) the DMP+FSDP2 coexistence pattern is explicitly test-covered in PyTorch 2.11 (§11.2), resolving R-DMP with in-tree machinery; (b) OpenOneRec trained its full released model line on FSDP2 with plain AdamW over DTensor params [openonerec][repo-openonerec], so the recipe tzrec reproduces first is FSDP2-native; (c) megatron-core HEAD now pins requires-python >= 3.12 while the tzrec env is py3.11 — usable via a released wheel pin, but a dependency-surface asymmetry that argues against making it the default. Megatron remains fully specified (§11.3) and is the engine verl uses for large actors, so nothing is lost at the top end.

11.2 PyTorch 2.11 native stack: the foundation

fully_shard API surface

FSDP2's public API (torch/distributed/fsdp/_fully_shard/_fully_shard.py:90) is small and composes precisely the way the two-world design needs. fully_shard(module_or_list, *, mesh, reshard_after_forward, mp_policy, offload_policy, ignored_params) swaps each managed parameter for a dim-0-sharded DTensor in place — FQNs are unchanged because the class is swapped via MRO injection rather than wrapping. The properties tzrec relies on, all verified in source or tests:

  • List-form grouping for tied weights. Shared-parameter bookkeeping only spans a single fully_shard call (_fsdp_param_group.py:845-857), so tied embed_tokens/lm_head (Qwen3-0.6B/1.7B/4B ship tie_word_embeddings: true) must land in one call: fully_shard([embed, lm_head], ...) or be left to the root group. Getting this wrong silently double-shards one logical weight.
  • Lazy root detection lets us shard only the LLM subtree. The first FSDP module to run forward becomes the root (_fsdp_state.py:163-203); there is no requirement that the overall module root is wrapped. The tzrec BaseModel/DMP root stays untouched and only model.llm is sharded. Corollary: never run a wrapped sub-block's forward standalone before the first full forward through the topmost wrapped module, or lazy init raises (_fsdp_state.py:184-189).
  • The composability filter does NOT know TorchRec. _get_managed_modules only refuses to recurse into modules registered with the replicate contract (_fsdp_common.py:101-106); a DMP-sharded table under the wrapped module would be claimed and mis-sharded. Rule: never call fully_shard on an ancestor of TorchRec-sharded modules, or pass their params via ignored_params (covered by test_fully_shard_ignore_params.py).
  • MixedPrecisionPolicy(param_dtype=torch.bfloat16, reduce_dtype=torch.float32) gives the standard bf16-compute/fp32-reduce recipe while sharded params and optimizer state stay in the original dtype (_fsdp_api.py:14-53).
  • set_requires_gradient_sync(False) is the gradient-accumulation lever (FSDP1 no_sync equivalent; controls reduce-scatter and all-reduce), needed for RL mini-epoch accumulation; set_reshard_after_forward(False) flips to a rollout/eval-friendly mode at runtime.
  • register_fsdp_forward_method(llm, "generate") (_fully_shard.py:662): .generate() bypasses module-call hooks, so eval-time generation under sharding requires registering it as an FSDP forward method (or manual unshard()/reshard()). transformers' own Trainer does exactly this after FSDP2 prep.
  • No deepcopy. FSDPModule explicitly forbids deepcopy (_fully_shard.py:234) — RL reference models must be built as a second instance plus state-dict load, never copy.deepcopy(actor).

DeviceMesh from the existing WORLD

tzrec already owns dist.init_process_group (tzrec/utils/dist_util.py:57-75). init_device_mesh adopts that PG rather than fighting it: it only calls init_process_group() if none exists (device_mesh.py:427), and for a mesh dim spanning the entire world the dim group is the default PG — the zero-extra-PG fast path at device_mesh.py:511-529. A pure-FSDP run therefore creates no new NCCL communicator: the same WORLD PG that DMP uses backs the FSDP all-gathers, with correctness guaranteed by deterministic program-order issue (same argument as today's DMP+DDP). DeviceMesh.from_group (line 1027) exists for adopting externally created PGs if ever needed. The one structural rule for FSDP+TP: mesh["dp"] and mesh["tp"] must be slices of one root mesh, because FSDP's 2D placement path calls DeviceMesh._concatenate which requires an identical _flatten_rank_map (_fsdp_param.py:299, device_mesh.py:1389) — never build separate dp and tp meshes.

Tensor parallelism and loss_parallel

DTensor TP is applied before sharding: parallelize_module(llm, mesh["tp"], plan) with ColwiseParallel/RowwiseParallel/SequenceParallel styles (tensor/parallel/style.py); Qwen3 even ships a ready base_model_tp_plan in its config. The composition order verified in test_2d_composability.py:179-256 is: one global 2D mesh, then TP, then bottom-up fully_shard on mesh["dp"]. loss_parallel() (tensor/parallel/loss.py:31) computes cross-entropy directly on vocab-sharded logits (Colwise lm_head with use_local_output=False) — the memory lever for the 176,384-row head (§7), avoiding an all-gather of [tokens, 176K] logits; note label smoothing is unsupported inside the context, and backward must run inside it too.

DCP: one checkpoint for both worlds

DCP's planner treats plain Tensor, DTensor, and ShardedTensor uniformly (planner_helpers.py:380-432), so a single state dict mixing TorchRec ShardedTensor tables and FSDP2 DTensor params saves and loads correctly — this is verified planner behavior, not an assumption, and tzrec's checkpoint_util.py:27-35 already builds on these APIs. The dense optimizer joins via get_optimizer_state_dict(llm, dense_optim), which converts param ids to canonical FQNs and yields reshardable DTensor state. Two HF-format bridges matter operationally: HuggingFaceStorageReader (hf_storage.py:202) loads a sharded-safetensors Qwen3 checkpoint straight into sharded DTensor params, each rank reading only its slice ranges; HuggingFaceStorageWriter(save_distributed=True, enable_consolidation=True) exports a tzrec-trained dense tree back to consolidated HF safetensors plus index — the artifact vLLM and verl consume. async_save exists (stage-to-CPU then background write) and matches tzrec's async-I/O preference, with the usual caveats: all ranks must pass identical keys on load (mismatch hangs), and StateDictOptions(broadcast_from_rank0=True) is DTensor-only — usable for the dense tree, not for TorchRec tables.

DDP/DMP composability — resolving R-DMP

The exact hybrid pattern tzrec needs is an in-tree test: test_fully_shard_ignore_params.py::test_ddp_A_fsdp_B_ddp_C (line 232) builds one model on one default PG, applies fully_shard to an inner subtree, then registers the FSDP param FQNs via DDP._set_params_and_buffers_to_ignore_for_model before DDP-wrapping the rest, and steps optimizers under implicit_replication() (which lets ops mix DTensor and plain tensors). TorchRec's DMP dense wrapping uses the same _ddp_params_and_buffers_to_ignore mechanism inside its DefaultDataParallelWrapper, and DMP's data_parallel_wrapper is pluggable with init_data_parallel=False available for ordering control — tzrec's own DMP wrapper already forwards it (dist_util.py:172). This closes R-DMP: the coexistence is not a bespoke hack but the composition the upstream test suite pins.

Pipelining and torch.compile status

torch.distributed.pipelining ships full schedules but its docs state "currently in alpha state and under development"; with ≤8B dense models and ZeRO-style sharding available, PP is out of scope for v1 (see also the tied-embedding complications it creates on the Megatron side, §11.3). For compile: torch._dynamo.config.skip_fsdp_hooks = True is the default, so compiling a wrapped module graph-breaks at FSDP boundaries; the supported pattern is regional compile — torch.compile each transformer block, comms stay eager (the torchtitan recipe). Helpfully, skip_torchrec = True means dynamo will not trace TorchRec internals, keeping compiled dense regions clean. DeviceMesh slicing under compile is unverified upstream — slice meshes at setup time, never inside compiled code.

11.3 Megatron-core 0.18 as an embeddable library

megatron-core is usable as a pure library: no megatron.training, no argparse globals, no process ownership. initialize_model_parallel(...) asserts torch.distributed.is_initialized() (parallel_state.py:728) and never calls init_process_group; it only carves TP/PP/CP/EP/DP subgroups out of the existing WORLD via new_group. This is exactly how verl drives it inside Ray workers (verl/workers/engine/megatron/transformer_impl.py:148-159), so the embedding pattern is production-proven. New in 0.18, rank_offset/local_world_size can build the Megatron layout over a subset of the world, leaving other ranks for sparse-only or rollout roles. An mpu-free path exists — ProcessGroupCollection is accepted by GPTModel, DDP, the optimizer, schedules, and finalize_model_grads — but it is not airtight: the unfused vocab_parallel_cross_entropy still reads the global TP group (cross_entropy.py:130-158), and helper fallbacks substitute mpu globals once initialized. The pragmatic call: initialize mpu globally on the existing WORLD, start at tp=pp=1 (all subgroups trivially sized, so the "mcore island" is cheap), and treat pg_collection as an optional explicitness layer rather than the mechanism.

GPTModel and the Qwen3 mapping

GPTModel(config, transformer_layer_spec, vocab_size, max_sequence_length, position_embedding_type='rope', rotary_base=1e6, share_embeddings_and_output_weights=..., ...) (gpt_model.py:86-109) covers Qwen3 dense with stock fields — no custom modules: num_query_groups = HF num_key_value_heads, kv_channels = HF head_dim (128, decoupled from hidden/heads), qk_layernorm=True (the per-head q/k RMSNorm that is Qwen3's signature), gated_linear_unit=True with SiLU, RMSNorm, no biases. This mapping is cross-checked against verl's converter (config_converter.py:167-184). The padded model vocab 176,384 = 2⁸×689 (§7) divides evenly for TP ∈ {1,2,4,8,…}, and both mbridge and Megatron-Bridge build GPTModel(vocab_size=176384) with no extra padding, so Megatron-side and HF-side shapes match exactly — the ceil(v/(128·tp)) formula is a megatron.training script convention tzrec never enters [mbridge][megatron-bridge]. T5Model exists with the same library shape for the OneRec V1 encoder–decoder topology [onerec-v1]; at PP=1 its legacy encoder/decoder placement scheme is moot.

The minimal external loop and its mandatory wiring

The canonical embedded loop is the QuickStart skeleton plus the DDP/optimizer wiring that megatron/training normally does (training.py:2879-2897):

# build (once): mpu init on existing world; model_parallel_cuda_manual_seed(seed)
model = DDP(config, ddp_config, GPTModel(...))           # mcore DDP: flat grad buffers, param.main_grad
optimizer = get_megatron_optimizer(opt_cfg, [model])     # DistributedOptimizer = ZeRO-1

# per step:
model.zero_grad_buffer(); optimizer.zero_grad()
losses = forward_backward_func(forward_step_func=fwd_step, data_iterator=it, model=model,
                               num_microbatches=k, seq_length=S, micro_batch_size=b)
ok, grad_norm, _ = optimizer.step()

The forward_step_func contract (schedules.py:63-94) — return (output_tensor, partial(loss_func, loss_mask)), where loss_func can return (loss_sum, num_tokens, report) for per-token normalization — consumes any iterator, so tzrec's dataset pipeline plugs in without Megatron dataset code. DistributedOptimizer is ZeRO-1: optimizer state and fp32 master params are sharded across DP over flat grad-buffer ranges, grads reduce-scatter, params all-gather after step. finalize_model_grads (finalize_model_grads.py:438-543) is the per-step synchronization point whose ordering matters for D7: DP reduce-scatter, layernorm/non-TP allreduce over TP, tied word-embedding grad allreduce over the embd group, then per-token scaling. Stage-1 itemic-row freezing must zero param.main_grad rows after this function and before optimizer.step(), with a wd_mult=0.0 param-group override — the OpenOneRec restore-after-step trick is broken under DistributedOptimizer because the fp32 master shard drifts and overwrites restored rows every step; full mechanics and recipe in §8 (D7).

dist_checkpointing, packed sequences, MoE, and the gotchas

dist_checkpointing.save/load is TP/PP/DP-reshardable (save at tp=4, load at tp=8) over a DCP-based torch_dist backend, with allow_shape_mismatch=True on embedding weights tolerating vocab-padding differences — the Megatron-mode dense checkpoint lives as a sibling directory in the tzrec checkpoint layout (§11.7). Packed THD sequences (PackedSeqParams with cu_seqlens) flow into TE's varlen attention, but the local (non-TE) spec hard-asserts packed_seq_params is None (dot_product_attention.py:153) — 32K sample packing (D6) on the Megatron backend requires the TE layer spec, mirroring the FA-only packing rule on the HF side (§8). The MoE surface is config-only and covers the OneRec-style recipe: num_moe_experts, moe_router_topk, moe_grouped_gemm, moe_token_dispatcher_type='alltoall', DeepSeek-V3 aux-loss-free expert-bias balancing [dsv3]; EP>1 with TP>1 requires sequence_parallel=True. Three operational gotchas: (1) megatron-core HEAD requires Python ≥3.12 — tzrec pins a released megatron-core wheel rather than tracking HEAD; (2) model_parallel_cuda_manual_seed must run before model build for TP-correct init; (3) the inference engines have no beam search — the REST server rejects it outright (text_generation_server.py:46-47) — so SID beam decode never runs inside mcore; it stays in the host-side beam driver over an exported single-step scorer (D13, §10) or in vLLM via verl (D11, §9).

11.4 Transformers: model classes yes, Trainer no

tzrec depends on transformers for exactly four things: (1) the Qwen3 model classes — GQA with per-head-dim q/k RMSNorm, the strict input_ids XOR inputs_embeds forward (the context-embedding injection surface of §7), and the FlashAttentionKwargs conduit that carries packed-sequence cu_seq_lens untouched into the kernel; (2) the tokenizer plus resize_token_embeddings(mean_resizing=True) for itemic vocab expansion — with the 267-stale-row re-init correction (R-VOCAB-INIT, §7); (3) from_pretrained sharded-safetensors loading with meta-device init; (4) generate() for offline eval — the v5-refactored beam search is fully vectorized with static shapes, applies logits processors to log-softmax scores (so a trie's -inf mask composes with beam scores), and auto-enables synced_gpus under FSDP so sharded eval generation does not deadlock.

tzrec deliberately does not adopt Trainer. Trainer always constructs its own Accelerator internally (create_accelerator_and_postprocess, trainer.py:752-861), bringing the exact singleton conflicts of §11.5, plus its own dataloader and checkpoint formats. One Trainer behavior must be replicated in the tzrec loop, though: gradient-accumulation loss normalization. Trainer prefetches all accumulation micro-batches and counts labels.ne(-100).sum() across them (num_items_in_batch, trainer.py:2095-2140), normalizing the summed CE by total non-masked tokens — the "GA loss bug" fix. With response-masked CE and packing (D6), naive per-micro-batch mean loss biases gradients toward short-response micro-batches; tzrec's SFT loop adopts the same total-token normalization (and on the Megatron side, calculate_per_token_loss with the 3-tuple loss_func return achieves the equivalent).

Version pin. tzrec pins the v4.53 line now (the OpenOneRec-verified line, and the line the export experiment of §10 was proven on) and records the v5 migration deltas as a checklist rather than chasing main: torch_dtype → dtype with the default flipping from fp32 to checkpoint-sniffed "auto"; rope_theta/rope_scaling → unified rope_parameters; _tied_weights_keys list → dict; meta-init always on (low_cpu_mem_usage dropped); and group-beam/constrained-beam generation modes moved out to Hub custom_generate repos requiring trust_remote_code — another reason the production constrained decode path does not depend on HF generation modes (D12).

11.5 Accelerate: port the recipe, never call prepare

The verdict is categorical: tzrec never constructs an Accelerator or calls Accelerator.prepare inside its training path. The reasons are specific and verified, not stylistic:

  • PG lifecycle ownership. PartialState does adopt an already-initialized PG (a correction to the prior spec's wording), but the conflict is lifecycle and policy: Accelerator.end_training() destroys the default PG (accelerator.py:3402, state.py:848-856) that DMP still uses; PG kwargs such as tzrec's PROCESS_GROUP_TIMEOUT_SECONDS apply only when accelerate creates the PG; and all three state objects are process-global singletons.
  • Optimizer dummy-swap machinery cannot round-trip TorchRec. _prepare_fsdp2 requires model+optimizer in one call, swaps every param_group entry to torch.empty(1) dummies and re-points them at sharded DTensors by data_ptr mapping, raising KeyError on any miss (accelerator.py:1714-1726, fsdp_utils.py:563-591). TorchRec fused embedding params live in DMP-sharded modules, are excluded from the dense optimizer by in_backward_optimizer_filter, and are invisible to accelerate's parallel-model extraction — the hybrid model cannot pass through. AcceleratedOptimizer additionally gates step() on accelerate-owned GradientState that tzrec's TrainPipelineSparseDist does not drive.
  • Forward monkeypatching breaks tzrec machinery. Under native AMP, prepare_model rebinds model.forward via MethodType (accelerator.py:1818-1829) — incompatible with TorchRec train-pipeline forward rewriting and the torch.export paths tzrec's export relies on.
  • Dataloader rewrap is wrong for tzrec datasets. Prepared IterableDatasets become DataLoaderDispatcher (rank-0 iterates and broadcasts — catastrophic for throughput) or IterableDatasetShard (re-shards round-robin on top of tzrec's already rank-sharded ODPS/Parquet readers, duplicating or dropping data).
  • ParallelismConfig cannot express the hybrid mesh. Validation enforces total_size == num_processes (parallelism_config.py:372-377); there is no vocabulary for "dense FSDP mesh plus TorchRec model-parallel embedding world on the same ranks".

What accelerate does contribute is the best-curated FSDP2 layering recipe in the ecosystem, and it is nearly free of the singletons: tzrec ports fsdp2_prepare_model / _prepare_fsdp2 (~150 lines, fsdp_utils.py:645-807, with an in-code citation per tzrec convention) into the Fsdp2Parallelizer. The load-bearing details of the port: apply activation checkpointing (checkpoint_wrapper, preserve_rng_state=False) at the same boundary as the wrap policy before sharding; regional compile before sharding; fully_shard children bottom-up; give an untied input embedding its own unit; shard the tail group [final_norm, lm_head] together with reshard_after_forward=False (its end-of-forward re-gather cannot be overlapped); root fully_shard last with the default reshard heuristic; fp32 master-weight upcast under mixed precision; meta-device materialization with rank-0 full-state-dict broadcast load (fsdp2_load_full_state_dict), re-registering non-persistent buffers and re-tying weights after load. One simplification falls out for free: because tzrec controls ordering, it creates the dense optimizer after sharding, sidestepping accelerate's entire dummy-param swap dance. Direct accelerate use remains acceptable in exactly one place: a TorchRec-free pure-LLM entrypoint (e.g. a standalone text-only SFT utility), provided it never calls end_training() while tzrec components share the process — and even there the ported recipe makes it largely redundant.

11.6 verl: the RL harness, used as-is

verl 0.8 is adopted wholesale as the RL stage's orchestrator and is detailed in §9 (D9–D11); here only the framework-leverage summary. All four tzrec requirements — custom rec dataset, trie-constrained rollout, composable rewards, Megatron-backed actors — land through documented extension surfaces with zero verl source patches: data.custom_cls dataset plug-in, a custom AgentLoop registered via rollout.agent.agent_loop_config_path YAML, the engine-level vLLM v1 logits-processor injection via engine_kwargs.vllm.logits_processors, reward.custom_reward_function, and the policy-loss/advantage registries for ECPO/GBPO plugins (D10). verl runs as a separate process tree with a parquet/HF-checkpoint handoff, in its own conda env — its ray/vllm dependency set conflicts with tzrec's grpcio-tools<1.63/protobuf-4.x pin, which is R-ENV, mitigated by env isolation rather than pin negotiation. The Megatron engine mandates mbridge (assert engine_config.use_mbridge) and consumes the SID-expanded HF checkpoint directly, including the 176,384-row embedding (§11.3). Candidate upstream contributions from this work: the CSR-trie constrained-beam logits processor and server glue (the gap verl itself has — constrained beam rollout — is the one feature absent in-tree), and the SID agent loop as a reference recipe.

Decision D8 — dense parallelism: Fsdp2Parallelizer default, MegatronParallelizer for scale-out

Initialization order

The order is rigid and extends tzrec's existing sequence (train_and_evaluate, tzrec/main.py:543-788): (1) dist.init_process_group + torch.cuda.set_device (unchanged, dist_util.py:57-75); (2) DenseParallelizer pre-init — DeviceMesh creation (FSDP2 mode) or initialize_model_parallel + model_parallel_cuda_manual_seed (Megatron mode); (3) model build, with the dense backbone on meta device; (4) dense parallelization + HF weight load (DCP HuggingFaceStorageReader or mbridge); (5) apply_optimizer_in_backward on sparse params (must precede planning, unchanged); (6) planner + DMP with the LLM params excluded from dense DDP wrapping; (7) optimizer tree assembly; (8) checkpoint restore. Keeping tzrec's set_device first matters: init_device_mesh would otherwise set the device itself from LOCAL_RANK.

FSDP2-native pattern (default)

In FSDP2 mode the train step shape is unchanged from today's tzrec: one backward populates both worlds' gradients; the sparse fused optimizer has already applied its updates in-backward by the time TZRecOptimizer.step() runs the dense AdamW over DTensor params. The dense optimizer joins tzrec's existing tree — CombinedOptimizer([fused_sparse, dense, *part]) (main.py:693-755) — through the in_backward_optimizer_filter split plus the part_optimizers regex seam (optimizer_builder.py:219-237), which is precisely the "LLM params get AdamW + cosine, rec params keep Adagrad" mechanism; grad-norm math over mixed DTensor/plain params runs under implicit_replication(). Sequencing risk R-GRAD is mostly about accumulation asymmetry: in-backward sparse updates fire on every backward while set_requires_gradient_sync(False) defers only dense reduction — accumulated-step recipes must either scale sparse LR accordingly or keep sparse tables out of accumulated phases; the RFC default is accumulation=1 for joint phases.

Megatron pattern (recsys-examples-proven)

The Megatron backend follows NVIDIA recsys-examples' integration, which has trained HSTU and SID-GR models on exactly this stack [repo-recsysexamples]. Wrapper order is DistributedModelParallel( McoreDDP( Float16Module( model ) ) ) — DMP outermost, mcore DDP excluding TorchRec table shards from its DP broadcast (sharding.py:62-146). The DMP↔TP impedance mismatch (DMP treats every rank, including TP peers, as an embedding shard; TP needs all TP ranks on the same batch) is resolved at the activation level: each TP rank looks up its own micro-batch, then jagged-allgathers embeddings over the TP group with a custom GradScale autograd op multiplying gradients by tp_size — undoing TorchRec's division by world = tp×dp when only the dp division is wanted — followed by dmp_batch_to_tp allgathering labels (dmp_to_tp.py:15-68, ranking_gr.py:118-120). DP-sharded tables instead get a TP-group allreduce in a custom finalize_model_grads pre-hook before the stock mcore finalize. The step sequence — zero_grad_buffer(); optimizer.zero_grad(); forward; (loss × dp_size / global_tokens).backward(); finalize_model_grads; dense optimizer.step() — cannot be expressed inside TrainPipelineSparseDist.progress, so the Megatron backend ships its own pipeline class returned from tzrec's single pipeline-selection seam, create_train_pipeline (dist_util.py:335), mirroring recsys-examples' JaggedMegatronTrainPipelineSparseDist. Known limitation inherited from this stack: no unified gradient clipping across worlds (the sparse update happens inside backward; dynamicemb tracks "decouple backward from optimizer update" as future work).

Planner accounting and the unified checkpoint

R-PLANNER: TorchRec's planner budgets HBM for embedding shards only — HeuristicalStorageReservation(percentage=0.15) is the sole headroom for the LLM's parameters, gradients, optimizer state, and activations, and the precedent is real (a large pooled table planned table-wise onto rank 0 OOM'd a V100 because the planner ignored dense activations; fixed by a global row_wise constraint). Mitigation is twofold: the DenseParallelizer computes its per-rank dense budget (sharded params + grads + optimizer shards + an activation estimate from micro-batch × seq len × hidden) and raises the reservation via STORAGE_RESERVE_PERCENT — surfaced as the planner env knob TZREC_DP_DENSE_RESERVE_PERCENT per tzrec's env-naming convention, read at the call site; and generative configs default to a global row_wise sharding constraint so no single rank concentrates table storage. Checkpointing stays a single tzrec checkpoint: the FSDP2 mode extends the existing DCP dict — {"model": ..., "sparse_optim": ..., "dense_optim": get_optimizer_state_dict(llm, dense_optim)} — relying on DCP's verified mixed ShardedTensor+DTensor handling (§11.2); the Megatron mode attaches a dist_checkpointing subdirectory beside model/ and optimizer/ at the single write site save_model (checkpoint_util.py:873), the same side-by-side pattern dynamicemb already uses. HF-format export for rollout/serving handoff goes through HuggingFaceStorageWriter (FSDP2) or the mbridge exporter (Megatron).

11.8 Parallelism selection guide

Efficiency anchors for capacity planning, from the published systems rather than projections: OneRec V1 reports 23.7% training MFU on a 720-GPU DP+ZeRO-1 deployment and 28.8% inference MFU on its L20/TensorRT serving stack — versus 4.6%/11.2% for the legacy cascade ranking stack — establishing what a well-engineered generative recommender achieves without model parallelism [onerec-v1]. OneRec V2's lazy decoder then cut total training computation ~94% and real training resources ~90% at matched quality, and serves at 62% inference MFU (1B model, beam 512, 36 ms on L20 — a lazy-decoder-specific figure, context KV shared across beams by construction, not a transferable target for Qwen3-class backbones; §10.4) — evidence that architecture-level compute allocation dominates parallelism-level tuning at this scale [onerec-v2]. recsys-examples' HSTU benchmark gives the realistic optimization trajectory for the two-world stack itself: 6.38% baseline MFU → 9.73% with Karmarkar–Karp jagged-batch balancing (1.53×) → 22.96% with CUTLASS attention; and the TP=2 regression above [repo-recsysexamples]. PLUM contributes the throughput heuristics tzrec adopts as training-loop defaults for the dense world: maximize batch size into HBM (halving batch required more than 2× the steps for equal quality), constant LR scaled inversely with model size for never-ending production training, and iso-FLOPS curves showing the eval-loss frontier shifts to larger models sooner than train loss — favoring larger dense models for next-day generalization at a fixed budget [plum].

12.1 Design rules and registry coupling

Three mechanical couplings constrain every message below. First, proto message name == Python class name: config_util.which_msg(model_config, "model") returns the message class name of the set oneof arm and BaseModel.create_class(name) resolves it against the registry metaclass (config_util.py:73-75, main.py:127-159) — adding a model is one message, one tag, one identically named class, and zero orchestration edits. One registration caveat: auto_import scans tzrec/models/*.py non-recursively (load_class.py:53-99), so the classes under tzrec/models/generative/ register through a one-line re-export shim placed directly under tzrec/models/ — auto_import itself is not modified (§4, §7). Second, proto field names == module keyword names: config_to_kwargs is MessageToDict(..., including_default_value_fields=True, preserving_proto_field_name=True) (config_util.py:66-70), so field naming is API design, not cosmetics. including_default_value_fields is the protobuf-4.x spelling, which is why the codegen toolchain stays pinned at grpcio-tools<1.63.0; after any .proto edit, regenerate with bash scripts/gen_proto.sh (the script globs new files automatically). Third, the SID tokenizer messages must carry exactly the field names the already-merged BaseSidModel reads off its model config — embedding_feature_name, input_dim, normalize_residuals, codebook (tzrec/models/sid_model.py:54-74).

The minimal-proto rules applied throughout: alternatives are a named oneof (the arm name routes behavior); booleans and scalars are flat fields on the owning message rather than nested option messages; knobs that tzrec does not itself interpret are not modeled — for RL they pass through a verl_overrides string map of dotted Hydra paths, and for anything else they do not exist until a consumer does (fail loud beats silent defaults).

13. Evaluation

Evaluation of the generative stack spans four altitudes: training-free tokenizer gates (owned by §5), offline model metrics computed over constrained beam decode, general-capability retention probes around every training stage, and parity against external benchmarks and reference repositories. Three principles unify the design: every retrieval metric is computed over the many-to-many SID↔item catalog (D4), never over raw SID strings alone; generation-based metrics are expensive and therefore run on a cadence decoupled from checkpointing; and evaluation must consume the same versioned decode artifacts (trie, catalog index) that serving consumes, or its numbers do not predict production behavior.

13.1 Tokenizer-level gates

The tokenizer quality battery is training-free (§5.4, decision D3) and gates the pipeline before any LLM training tokens are spent: Gini coefficient over SID assignment counts (used-SIDs convention, histogram closed form), per-level and full-SID codebook utilization, unique-SID ratio (the batch-granular tzrec/metrics/unique_ratio.py metric plus a global offline variant), Embedding HitRate over the pre-quantization input embeddings, and a collision histogram reporting σ-capacity violations. FORGE validates that Gini, utilization, and Embedding HitRate correlate with downstream generative-retrieval HitRate across its variant matrix, while reconstruction-fidelity and consistency metrics do not [forge].

Never model-select a tokenizer by reconstruction loss

FORGE's anti-metric finding is explicit: Multiple-VQ wins two fidelity metrics yet is the worst retriever; RQ-Kmeans beats the RQ-VAE base on fidelity but loses overall [forge]. PLUM adds the complementary caution that uniqueness alone is also insufficient — its multi-embedding ablation raised SID uniqueness (96.9%) while lowering downstream Recall@10 (12.8 vs 14.4) [plum]. When comparing tokenizer families (not just hyperparameters within one family), pair the gate battery with PLUM's cheap fixed-recipe probe: a small fixed model, no co-pretraining, minimal prompt, downstream Recall@10.

13.2 Offline model evaluation

Canonical constrained-beam protocol

Protocol pin — the canonical offline retrieval metric is constrained-beam, multi-item-expanded, per-sample macro Recall/HR/NDCG@K

tzrec gr eval computes retrieval metrics in six steps: (1) decode K_beam SID sequences per prompt using the same SidConstrainedLogitsProcessor and CSR trie artifact used at serving (D12); (2) expand each SID into its item list via SidCatalogIndex in beam order — σ-capacity collision buckets (D2) make this expansion one-to-many, following the calc_hr pattern in FORGE's released code [repo-alsid]; (3) deduplicate item ids, keeping the best-ranked occurrence; (4) truncate to K items; (5) compute per-sample recall |top-K ∩ GT| / |GT|, a hit indicator, and NDCG from the rank of the first hit; (6) macro-average over samples. Item-id match is the primary semantics; exact SID-tuple match is reported as a secondary column.

prompts --> constrained beam (K_beam, serving trie) --> ranked SIDs
                                                   | (beam order)
                                                   v
                SidCatalogIndex expansion: SID -> [item ids]  (sigma buckets)
                                                   |
                       dedup by item id (keep best rank), truncate to K
                                                   |
                                                   v
            per-sample Recall / HR / NDCG @ K  -->  macro-average over samples

Three protocol pins deserve justification. Per-sample macro averaging: FORGE notes that its per-sample-averaged recall differs from corpus-pooled HR [forge]; OpenOneRec's harness is also per-sample [repo-openonerec]. We pin per-sample macro and label every reported number with it. Item-id vs exact-tuple match: with a RandomCollisionResolver (D2) the last SID level is non-semantic and buckets hold up to σ items, so an exact-tuple hit criterion is ill-defined for retrieval quality; OpenOneRec's benchmark computes both SID-string and pid-deduplicated variants (utils_by_pid.py) and we keep the tuple variant only as a tokenizer-drift diagnostic [repo-openonerec]. Jagged ground truth: targets are sets, not single items — OneReason's R3 tasks average 13.92 target items per user in the Video domain [onereason] — so the metric consumes jagged GT id lists (Batch.jagged_labels), and Recall@K uses the true set size as denominator. When Latte latent tokens are enabled (D16), beams reaching the same item through different latent paths are aggregated (sum/max) inside SidCatalogIndex scoring before ranking, leaving the metric layer unchanged [latte]. A fourth pin applies to instruction-tuned arms: they are evaluated per template paraphrase, reporting mean/min/max across paraphrases — LC-Rec's released harness runs all 17 seqrec phrasings and reports exactly this spread [repo-lcrec] — so template overfitting surfaces as a wide min/max gap instead of hiding inside a single-phrasing number (this is the protocol the one-sampled-template-per-datum rule of §6 exists to support).

Metric menu

The hallucination-rate row doubles as a decode-correctness assertion: under constrained decoding it must be exactly zero, which is MiniOneRec's CC counter — their released harness treats any nonzero value as a broken trie or a prompt/template mismatch [repo-minionerec]. In unconstrained mode (a supported configuration per D12), GRID and PLUM both report that small hallucination rates are tolerable and accuracy is near-identical to constrained decoding [grid] [plum], so the rate is reported, trended, and alarmed above 5%.

Reasoning-mode diagnostics

Every reasoning-capable checkpoint is evaluated in both thinking and non-thinking modes on the same prompts. This is non-negotiable: OneReason shows SFT-stage thinking loses to non-thinking on all four of its domains, and only RL plus rejection-sampling fine-tuning flips the sign [onereason]; OpenOneRec's SFT-stage "non-thinking" scores are inflated by instruction drift (the model thinks despite /no_think) [openonerec]. A single-mode eval would have hidden both failure modes. We adopt OneReason's diagnostics as cheap standard probes: ΔLL = log p(GT | x, CoT) − log p(GT | x), two teacher-forced log-prob passes that quantify whether the generated trace helps or actively distracts (negative for SFT-only models, −2.7 to −5.2 in OneReason; positive after RFT); γ_legal, the fraction of SIDs referenced inside the trace that exist in the catalog; and γ_hist|legal, the fraction of legal references that appear in the user history — both pure string/set operations over parsed traces [onereason]. These attach to the RL health dashboard in §9 but are equally useful as offline SFT-quality gates.

Expressiveness probe

As a cheap structural audit, we port Latte's probability-correlation probe: sample item pairs at varying SID tree distance, score them across a user population, and measure the correlation of scores (TIGER-style models show Pearson ρ ≈ 1.0 for sibling leaves at tree distance 2, still above 0.8 at distance 4), plus the Kendall correlation between tree distance and model item-similarity [latte]. High coupling means the decode tree, not user preference, is dictating relative item scores — the direct symptom that motivates latent tokens (D16) and multi-SID indexing (D4). The probe needs only an item-scoring pass over sampled pairs, no generation.

Temporal protocol and loss/metric decoupling

For production-shaped corpora the split is strictly temporal: train on days 1..N, evaluate on day N+1, as in PLUM's scaling study (train 7 consecutive days, evaluate day 8) [plum]. Leave-last-out splits remain only for academic-benchmark parity (§13.4). PLUM also supplies the coverage metric for comparing against an embedding-retrieval incumbent — effective vocabulary size, where PLUM scored 2.6×/13.2× its production baseline — which we adopt for pre-launch A/B framing [plum].

Do not early-stop generative recommenders on eval loss

PLUM observes that eval Recall@10 keeps improving after eval loss saturates, and that the eval-loss frontier shifts toward larger models sooner than the train-loss frontier [plum]. Checkpoint selection and stage-exit criteria must therefore key on the generation metrics of the menu above, never on loss alone. Loss curves remain useful for divergence detection, not for model selection — the same shape of mistake as selecting tokenizers by reconstruction loss (§13.1).

13.3 General-capability retention

Every stage boundary in the SFT/RL ladder (§8, §9) runs a fixed general-capability probe suite — OpenOneRec's sanity set is the template: MMLU-Pro, IFEval, GSM8K/MATH-500, GPQA-Diamond, AIME, LiveCodeBench [openonerec]. The evidence that this must be instrumented per stage, not once at the end: OpenOneRec's 8B model drops MMLU-Pro from 0.72 to 0.53 and IFEval from 0.86 to 0.62 after multi-task SFT; on-policy distillation (D10) recovers most of the reasoning and instruction-following loss (AIME 0.51 → 0.71, IFEval → 0.77), though MMLU-Pro world knowledge remains the largest residual gap (0.52 final vs the 0.724 base) [openonerec]. The downside scenario is LC-Rec: its LaTeX source contains a cut table showing MMLU falling 35.2 → 25.8 after recommendation fine-tuning [lcrec], and OneReason measures an LC-Rec-style 8B SFT-only model at 9.73 MMLU-Pro versus its own RFT model at 72.08 ≈ base Qwen3-8B (72.35) [onereason] — retention is decided by the recipe (general-data mix in CO_PRETRAIN, distillation repair), not by the backbone. The probes run on the external LLM-eval harness against exported HF checkpoints, with per-stage regression budgets set in config (§12) and the OpenOneRec trajectory as the reference envelope. This instrumentation is the early-warning system for R-FORGET; the repair tool is D10's on-policy distillation (§9). Where the product surface is retrieval-only, the suite degrades to monitoring-only, which is PLUM's documented posture [plum].

13.4 Benchmarks and parity targets

Amazon protocol. The canonical academic configuration is Amazon Reviews with 5-core user/item filtering, per-user chronological ordering, leave-last-out splitting (test = last item, validation = second-to-last), a history cap of 20–50 items per preset, and sliding-window augmentation in training — GRID measures the sliding window as the single largest training-recipe effect (Beauty R@10 0.0597 with vs 0.0447 without) [grid]. Reference repos diverge from this in load-bearing ways (MiniOneRec splits 8:1:1 over the interaction list rather than leave-one-out [repo-minionerec]), so parity runs replicate the reference repo's exact protocol, not the tzrec canonical one, and published numbers are never compared across protocols.

AL-GR adapter. A dataset adapter under tzrec/datasets/gr/ reads FORGE's public AL-GR artifacts from HuggingFace (final/base SIDs, Item-SID, 512-d Item-EMB), with AL-GR-Tiny sized for CI; the eval writer can emit calc_hr-compatible JSONL so tzrec runs compare directly against published AL-GR numbers (base S3 HR@20 4.33% → Final 5.44%) [forge] [repo-alsid].

RecIF-Bench and successors. RecIF-Bench's 8-task/4-layer taxonomy (alignment, fundamental rec, instruction following, reasoning; Pass@1/Pass@32/Recall@32, AUC-from-logprobs for label prediction, LLM-judge double-weighted F1 for captioning/explanation) is the broadest public capability benchmark for this stack [openonerec]. Its harness stays an external tool, not in-tree: it pins Ray + vLLM 0.7.3 + a Gemini judge client [repo-openonerec] — a dependency set incompatible with tzrec's pins (the same isolation argument as R-ENV). tzrec's responsibility ends at exporting HF-format checkpoints whose prompt/SID conventions the harness accepts. OneReason-Bench (R0–R3) is the more diagnostic successor; an adapter is deferred until its artifacts ship [onereason].

Parity oracles. Each generative model class ships with a parity target against its reference implementation, in two modes: inference parity (load the reference's released checkpoint through tzrec's model code and match its eval outputs) and training parity (retrain with the tzrec pipeline and land within ±2% of published numbers). Inference parity isolates model-code bugs from data-pipeline bugs and should be established first wherever public checkpoints exist.

13.5 In-tzrec wiring

Metric modules. Teacher-forced metrics ride the standard init_metric/update_metric contract of tzrec/models/model.py: token-level loss, next-token accuracy on itemic positions, and the itemic-vs-text loss split (the per-token-class monitoring OpenOneRec implements in its trainer [repo-openonerec]) register as ordinary train/eval metrics. The generation-based recall family needs a new metric class: the existing RecallAtK consumes dense [B, N] score matrices (tzrec/metrics/recall_at_k.py), whereas constrained-beam eval produces jagged ranked item-id lists per prompt. A SidRecallAtK-style metric therefore takes (jagged predicted id lists, jagged GT id sets from Batch.jagged_labels) and implements ID-tuple hit semantics directly; per-beam rows written through the predict writer follow the established TARGET_REPEAT_INTERLEAVE_KEY convention so per-prompt columns expand correctly (§3).

Cadence. Generation eval is decoupled from checkpoint cadence. Teacher-forced metrics (and ΔLL, which costs two extra forward passes) run at every evaluation round; full constrained-beam HR@K runs on a sampled prompt subset at checkpoint multiples and on the full set only at stage exits, invoked explicitly via tzrec gr eval (§12). The cost asymmetry is large — RecIF's short-video task alone is 38,781 prompts at beam 32 [repo-openonerec] — and coupling it to every checkpoint would either starve training or pressure teams into skipping generation eval entirely.

Beam-width gap. Offline eval defaults to beam 32 (the OpenOneRec convention) while production serves far wider beams — OneRec-V1/V2 serve beam 512 [onerec-v2] and FORGE's deployment returns 1200 candidates [forge] — so small-beam recall systematically understates serving-time coverage. Beam width is part of every reported metric's label, and each launch is preceded by a serving-width parity run executed through the serving decode path itself — the exported AOTI single-step scorer and host-side beam driver (§10, D13), or the tzrec-gr-serving runtime for widths beyond the host loop's KV budget (§10.6) — which simultaneously catches export drift between the training graph and the serving graph.

Artifact identity. The eval job must load the same versioned (catalog snapshot, CSR trie, item↔SID index) triple that export produced — never rebuild its own — and the eval harness can drive the GR runtime's SGLang-compatible /generate endpoint directly, keeping eval and serving on the same decode path (§10.6). A skew between the tokenizer snapshot that generated the training data and the catalog used at eval silently deflates recall with no error raised; the eval entry point asserts snapshot-hash equality between the trie artifact and the catalog index, and re-tokenization cadence for streaming corpora is tracked as R-RETOK in §15.

14. Roadmap, testing, and dependencies

This section turns the design into an execution plan: six phases P0–P5, each a vertical slice that ships a runnable CLI verb (§12) and closes against a measurable exit gate, followed by the testing strategy (mapped onto tzrec's existing unittest conventions) and the dependency matrix. It updates the prior internal spec's roadmap (09_roadmap_testing_risks.md) with the evidence gathered for this RFC: the single-step export path is empirically de-risked, the tensorized trie is resolved by an existing open-source design, and the FSDP2-default decision (D8, §11) reorders which backend each phase exercises first. Reference repositories remain validation oracles, never dependencies.

14.1 Phases P0–P5

Revision vs prior spec 09 — three plan changes

(a) P1 now targets Qwen3 on FSDP2, not Megatron: the OpenOneRec-proven recipe [openonerec] lands first and the Megatron backend moves wholesale to P4, shrinking the riskiest early integration surface. (b) P2 starts from a proven baseline instead of an open risk: the AOTI single-step scorer was smoke-tested end to end on this machine (R-EXPORT de-risked) and the constrained-decode data structure is adopted from STATIC's published CSR design (R-TRIE resolved) [static][static-repo], so P2 is integration work, not research. (c) CO_PRETRAIN is budgeted as a first-class stage inside P1 rather than an optional mix knob, following PLUM's 2×2 ablation showing continued pre-training matters more than base-LLM choice (Recall@10 0.27 random-init+CPT vs 0.23 Gemini-init without CPT) [plum].

P0 is CPU/single-GPU work measured in minutes per experiment, which is exactly why D1 makes RQ-KMeans the default and D3 makes the quality gates training-free: the P0 exit is a metric sweep, not a GPU-week. The asset contract (the item_id→SID parquet/ODPS table plus inverse index, §5) is the only seam later phases consume, so P0 can be re-run (new embeddings, new geometry) without invalidating downstream code.

P1 carries the two-world integration risks that are pattern-proven but not yet integrated in tzrec: R-DMP (shard only the LLM subtree; the upstream ddp_A_fsdp_B test plus the recsys-examples wrapper-order precedent [repo-recsysexamples] define the pattern) and R-GRAD (sparse fused in-backward step, then dense grad finalize, then dense optimizer step — the exact sequencing recsys-examples runs in production code). Both get dedicated 2-GPU integration tests before any model-quality work is trusted. P1 deliberately uses token-only prompts (SIDs as added tokenizer tokens), deferring context-embedding injection to P4 so that parity against LC-Rec/MiniOneRec is apples-to-apples.

P2 is the phase whose risk profile changed most since the prior spec. The decode/export design (D13) was validated empirically on this machine: a Qwen3 single decode step with StaticCache exports via torch.export and compiles via AOTInductor on torch 2.11.0+cu129 / transformers 4.53.3 (tiny Qwen3, bf16, sdpa, batch 8), with host-side index_select KV-cache beam reorder bit-exact between exported-step calls, and a dynamic Dim("batch") explicit-cache-I/O variant serving B∈{4, 8, 24} from one artifact. The remaining P2 work is therefore productization: the real-scale artifact (vocab 176,384), the prefill-graph export (the smoke ran decode steps only), the trie build job, and the two known serving gotchas (mandatory .contiguous() input guard; GLIBCXX_3.4.30 on the dlopen path) wired into the export runner and PAI image checks.

The serving-implementation analysis (2026-06-10) adds seven obligations to the export/serving ledger (§10). (1) The L20 bake-off — the sid-gr-inference-seeded runtime vs RTP-LLM, both behind SGLang-compatible APIs under the nearline envelope — runs before the token-only production default is declared. (2) The tensorized CSR trie gather (D12) replaces sid-gr-inference's per-beam python trie walk, which asserts batch_size=1, before any batch > 1 or W=512 use. (3) Context-length bucketing lands in the adopted runtime's scheduler, which currently groups decode batches by exact context length. (4) The batched-beam decode module gets a Qwen3-scale smoke (real head counts, 3K context, W=512) for both implementations — gr_decode_atten (c1) and the two-phase flex_attention + LSE-combine route (c2), whose export+AOTI smoke passes at toy scale only; the module itself stays a demand-gated contingency (§10.7), so this smoke is a readiness gate, not a committed production milestone. (5) The python EAS custom container (M-a) gets a QPS benchmark before it is trusted beyond eval/shadow/nearline. (6) 8B synchronous serving is prefill-bound (~0.4–0.8 s at 3K contexts on one L20) and requires TP on every path — an engine flag for RTP-LLM/vLLM, a new sharded-serving subsystem for the native runtime — so the 8B sync arm stays engine-first until that subsystem exists. (7) Nearline capacity economics — tokens/user/day × DAU × refresh rate, with active-user tiering and trigger policy as the sizing levers — are tabulated before the first production exposure (order-of-magnitude: 100M refreshes/day at ~0.2 s GPU each ≈ 230 sustained L20s). The serving-runtime extension plan (re-verified at recsys-examples 5dc46a217d, 2026-06-30; zero drift in either serving tree since 2550a9c4) adds five more: (8) the FORGE-dec qwen2_5 adapter lands at P2 as the BackboneAdapter contract proof (~0.5–1 wk); (9) the decode kernel is validated to W=1024, so FORGE's W=1200 gets its own validation run (seam S5); (10) the embedding-ingress extension (seam S1: request-side fp16 context tensors first, then session ContextKV reuse keyed user_id+feature-version) passes design review before P4; (11) /generate per-token logprobs move to post-mask emission so gen_prob_logged receives beam-faithful scores (today they are pre-mask); (12) the per-family checkpoint-manifest export tool (tzrec DCP→safetensors: config + weights + tokenizer/SID manifest) defines the train-to-serve contract for every gr_models family (§10.6).

P3 lands on the verl 0.8 line with the zero-patch contract of D9; the GRPO defaults are adopted from OpenOneRec and the exit is deliberately a behavior gate (reward-up, legality, bounded KL) rather than a benchmark number, because RL gains are dataset-dependent. ECPO and GBPO follow as registered estimator/policy-loss plugins once the GRPO loop is green; GBPO additionally exercises the no-rollout data path (logged feedback with gen_prob_logged, §6). P4 then scales out: the Megatron backend (including the backend-specific stage-1 freezing of D7, whose grad-zeroing recipe is already worked out against megatron-core 0.18 source), MoE/EP arms, the 250M-item dynamicemb path, and the one remaining open design task R-INJECT. P5 is parity and frontier work; it intentionally contains everything with research risk (reasoning RL, PIT/Latte, MTP) so that P0–P4 form a shippable production core without it.

14.2 Testing strategy

All new code follows tzrec's existing conventions: colocated *_test.py files under unittest, run via tzrec/tests/run.py, formatted and linted by the standard pre-commit stack. The GR-specific additions are organized around five rules, all of which already have precedents in the tree.

  • Case tables via @parameterized.expand. Tokenizer geometries (num_levels × codebook_size), collision resolvers, vocab-extension sizes, reward shapes, and trie traversals are classic case-table tests: one body, many input rows — never N near-identical methods or subTest loops.
  • Capability skips at the right granularity. GPU-dependent tests use the existing @unittest.skipIf(gpu_unavailable) probes from tzrec/utils/test_util.py:38. CI scope markers (@mark_ci_scope("h20"), test_util.py:151) go on classes or methods only — never module-level scope lists — so a single file can mix CPU-lane trie math with H20-lane Megatron tests. CI lanes reuse scripts/ci/ci_test.sh with --scope forwarding; no sibling CI scripts.
  • Optional-wheel probes via importlib.util.find_spec. The pattern is already in test_util.py:43-51 (the hstu and torch_fx_tool probes, with the explicit comment that importing test_util must not drag heavyweight stacks into every test process). New probes: faiss_unavailable, transformers_unavailable, megatron_unavailable, vllm_unavailable, verl_unavailable. A bare pip install tzrec without any GR extra must still pass the full default test suite.
  • Subprocess isolation for global state. The SUBPROC_TEST_PATTERN list (tzrec/tests/run.py:23) gains the GR tests that mutate process globals: faiss/CUDA initialization in tokenizer training, set_static_max_seq_lens (process-global HSTU sizing), Megatron parallel_state (one parallel layout per process; must be destroyed between configurations), and HF tokenizer caches.
  • Numerical parity oracles as nightly H20-scope tests. The highest-value tests assert agreement with reference implementations rather than self-consistency: tokenizer code assignments vs OpenOneRec ResKmeans and AL-GR; LC-Rec eval parity (±2% relative) on a fixed Amazon split; trie legality sets vs MiniOneRec's hash-dict trie (illegal count exactly 0, per the no-defensive-tolerance rule); GRPO advantages vs verl's compute_grpo_outcome_advantage at the pinned commit. These run nightly on the H20 lane, not per-PR.

The P2 export tests have a concrete template: the smoke matrix from the de-risking study (export → multi-step eager parity → beam-reorder parity → AOTI compile → AOTI multi-step parity → dynamic-batch artifact) becomes a parameterized test class over {torch.export module, AOTI fp32, AOTI bf16} × {buffers cache, explicit-I/O cache}, with the tolerance discipline this codebase already uses for TRT: record pre-activation logit-level max_abs_diff, gate bf16 at the measured fusion-noise band, and require fp32 exactness rather than loosening a single global tolerance. A stride/contiguity assertion test reproduces the non-contiguous-input corruption gotcha so a regression in the .contiguous() guard fails loudly. End-to-end smoke configs live under tzrec/tests/configs/gr/, one per phase (tokenizer, SFT, export/serve, RL), mirroring the existing integration-test layout; the RL smoke runs the verl process tree from its own env and asserts only on artifacts (parquet handoff in, checkpoint + metrics out), keeping the env boundary (R-ENV) inside the test design.

14.3 Dependency matrix

Core pins are unchanged from master (requirements/runtime.txt): existing tzrec users see no new mandatory dependency. Everything GR-specific is extras-gated (pip install tzrec[gr], tzrec[gr-megatron], …) and probed with find_spec at import/test time; the RL stack is not an extra at all but a separate environment (R-ENV), because ray/vllm dependency trees conflict with tzrec's grpcio-tools<1.63.0 / protobuf 4.x pin and verl always runs as its own process tree with a parquet/checkpoint handoff.

Two cross-cutting rules complete the matrix. First, every new wheel is first installed into a clone of the baseline conda env, never upgraded in place, so a bad build cannot poison the working environment. Second, any dependency whose API surface this RFC quotes (verl, vllm, transformers, megatron-core) is referenced with an explicit version tag in code comments and docs; the pins above are the only versions for which the claims in §9–§11 were verified.

15.2 Alternatives considered (and why rejected)

Each alternative below was seriously considered — most are the stack some reference implementation actually uses. The recurring rejection reason is ownership: tzrec already owns the process group, the train loop, the optimizer tree, and a proto-based config system, and any framework that insists on owning one of those becomes a second source of truth.

  • DeepSpeed (LC-Rec's training path). LC-Rec fine-tunes its LLM under DeepSpeed ZeRO-3 [repo-lcrec], so it is a proven recipe for exactly our P1 workload. But ZeRO-3 is functionally redundant with in-tree FSDP2 (D8) while adding a second engine that wants to own initialization, optimizer construction, and checkpoint format — all places TorchRec DMP and tzrec's CombinedOptimizer tree already sit. One sharding runtime for the dense world is enough; the prior spec's non-goal stands.
  • TRL / Unsloth (MiniOneRec and semantic-ids-llm precedent). MiniOneRec runs its whole GRPO loop as a forked TRL GRPOTrainer under accelerate, and semantic-ids-llm uses the same family of single-node tooling [repo-minionerec][repo-semanticids] — excellent existence proofs (we adopt their trie and reward mechanics) but a single-node, ≤8-GPU ceiling with HF generate rollouts, no server-mode inference engine, no Megatron path, and no story for bridging TorchRec sparse tables. verl provides the same algorithms with a scale path and zero-patch extension points (D9).
  • PyTorch Lightning + Hydra (GRID's stack). GRID organizes tokenizer and retrieval training under Lightning modules with Hydra config composition [repo-grid]. Both layers are orchestration owners: Lightning wants the loop (and its own checkpoint/callback model), Hydra wants config authority. tzrec's loop is tzrec/main.py + protobuf configs, and the GR pipeline must stay a peer of every other tzrec model family — adding a second orchestration layer for one model family would fork operational knowledge. GRID remains a P0 validation oracle.
  • Building on NVIDIA recsys-examples directly. It is the closest existing two-world codebase, and this RFC mirrors several of its patterns outright (wrapper order, grad sequencing, beam KV-cache design) [repo-recsysexamples]. As a base, though, it is Megatron-pinned (parallel_state as the single source of parallel truth, gin-config, no FSDP2 path), has no proto config surface, no ODPS/Kafka/PAI integration, and its checkpoint format is world-size-locked for the dense part. We import the patterns with citations, not the framework. That rejection is scoped to the training base and stands. The serving verdict changed on re-study at 2550a9c4 (v26.05): PR #414's examples/sid-gr-inference — a standalone, pip-installable, pure-python SID-GR serving runtime with none of the Megatron coupling above — is now adopted as the seed of the native GR serving path — the tzrec-gr-serving fork, since scoped to all softmax-attention backbones behind the BackboneAdapter contract (§10.5, §10.6) — fork-owned with its alpha maturity tracked as R-SIDGR above. The re-study also recorded a cautionary tale: upstream's own examples/sid_gr trainer and examples/sid-gr-inference runtime are disjoint stacks with no checkpoint bridge (TorchRec-KJT, effectively-NoPE training vs a hardcoded HF-Qwen3 loader) — exactly the train-to-serve rift tzrec's per-family checkpoint-manifest contract (WS4) is designed to prevent; examples/sid_gr remains a kernel-and-concept donor, never a checkpoint source.
  • Pure-Megatron training loop (megatron.training.pretrain). The harness assumes it owns the process: argparse-driven global args, its own tokenizer/vocab-padding flow (which would silently disagree with the HF-side 176,384 vocab), its own dataloaders and checkpoint cadence. megatron-core as a library (models, parallel groups, DDP, DistributedOptimizer, dist-checkpointing) is exactly what D8's MegatronParallelizer embeds; megatron.training is not (§11) [megatron].
  • accelerate as a runtime dependency. Rejected for PG/optimizer ownership reasons; the useful part — the ~150-line FSDP2 layering recipe — is ported with citation instead. Full analysis in §11.5 [accelerate].
  • HF Trainer for SFT. Brings its own Accelerator, dataloader sharding, and checkpoint layout; incompatible with the two-world step sequencing and the unified DCP checkpoint. Covered in §11.4 [transformers].

A final non-alternative worth recording: not building the generative path inside tzrec at all, i.e., running OpenOneRec or a fork as a sidecar system. This fails the requirement that motivated the RFC — feature pipelines, samplers, 250M-item embedding infrastructure, checkpointing, and PAI deployment already live in tzrec, and every reference stack would need all of them rebuilt or bridged. The two-world design exists precisely so the LLM joins tzrec's world rather than the reverse (§4).
