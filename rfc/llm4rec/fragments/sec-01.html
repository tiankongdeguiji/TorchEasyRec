1. Executive summary

This RFC proposes extending TorchEasyRec (tzrec) into a production framework for LLM- and SID-based generative recommendation: predicting the next item directly, as a short sequence of Semantic-ID (SID) tokens emitted autoregressively by a transformer language model, instead of scoring candidates through a retrieval/ranking cascade. The design covers the full pipeline — SID tokenization, supervised fine-tuning (SFT) and continued pre-training, reinforcement learning (RL), and constrained-decoding inference/serving — and specifies precisely how tzrec leverages PyTorch 2.11, Megatron-core 0.18, HuggingFace Transformers, Accelerate, and verl for efficient multi-GPU and distributed training.

The proposal is grounded in a systematic study of the OneRec family (V1, V2, OneRec-Think, OneReason [onerec-v1][onerec-v2][onerec-think][onereason]), the open reproductions OpenOneRec and MiniOneRec [openonerec][minionerec], industrial SID-construction work (ALGR/FORGE at Alibaba, PLUM at YouTube [forge][plum]), the academic line (LC-Rec, GRID, Latte, PIT, RPG [lcrec][grid][latte][pit][rpg]), and source-level reviews of the five infrastructure stacks plus NVIDIA's recsys-examples two-world integration [repo-recsysexamples]. It supersedes the internal v1 spec under experiments/llm4rec/: the architecture and canonical names are retained, while a small number of decisions are revised with new evidence (each marked Revision in place; see §4).

1.1 The shape of the design

Two-world training. Recommendation models have two kinds of parameters with incompatible scaling behavior. Sparse embedding tables (item IDs, user features, SID codebooks — up to 250M rows) stay in the TorchRec world tzrec already owns: model-parallel sharding via DistributedModelParallel, fused in-backward optimizers, dynamicemb hash tables. The dense LLM trunk (0.1B–8B+ parameters) joins a new dense world: FSDP2 fully_shard by default, Megatron-core tensor/expert parallelism at scale. Both worlds share one torch.distributed WORLD, one DeviceMesh, two optimizer trees, and one unified DCP checkpoint. The bridge between them is activation-level: TorchRec produces context embeddings that enter the LLM as inputs_embeds. This is the same integration shape NVIDIA's recsys-examples proves out for HSTU at scale, adapted to tzrec's existing seams (§11).

Four subsystems, all config-driven. (1) A SID tokenizer subsystem (tzrec/sid/) builds on the ResidualQuantizer/BaseSidModel base classes already on master: RQ-KMeans and RQ-VAE quantizers, pluggable collision resolution, training-free quality metrics, and a many-to-many item↔SID catalog (§5). (2) An SFT/pretraining pipeline runs the staged curriculum every successful system converges on — itemic-text alignment → co-pretraining → multi-task SFT → rejection-sampling fine-tuning — inside tzrec's existing train loop via a new pipeline class (§8). (3) An RL subsystem targets verl 0.8 with zero framework patches for Qwen3-class backbones, plus a tzrec-native rollout loop for architectures vLLM cannot serve, and GBPO-style learning from logged real feedback with no rollouts at all (§9). (4) A decoding/serving stack pairs the tzrec GR serving runtime (a BackboneAdapter fork of recsys-examples sid-gr-inference) with the AOTI scorer path — an AOTInductor-exported single-step scorer under a host-side beam driver — both decoding through a STATIC-style CSR tensorized trie (§10).

1.2 What is new since the v1 spec

  • FSDP2 becomes the default dense backend (D8); Megatron-core is the scale-out path for MoE/EP and ~8B+. OpenOneRec trains all stages on raw fully_shard; PyTorch 2.11's DCP, DeviceMesh, and DDP-composability APIs make the TorchRec bridge native (§11).
  • Constrained decoding is resolved, not open: YouTube's STATIC flattens the SID trie into CSR tensors — one batched gather per decode step, ~1.2–3.1 GB at 3×8192 over 250M items [static] (D12). Unconstrained beam with post-grounding is a supported config, with GRID/PLUM evidence that it is often sufficient.
  • Single-step scorer export is de-risked empirically: a Qwen3 decode step with a static KV cache was validated on this machine (torch 2.11, transformers 4.53) — the torch.export module matches eager bit-exactly, host-side beam reordering included; the AOTInductor compile is fp32-exact, with bf16 logits within ≈5e-3–8e-3 fusion-reorder noise — with two sharp edges (input contiguity, GLIBCXX) documented in §10 (D13).
  • The vocabulary-extension math is corrected (total-vocab padding to 176,384, not block padding), and a latent OpenOneRec bug is fixed by contract: HF mean_resizing leaves the first 267 itemic rows uninitialized (§7).
  • Collision resolution defaults to RandomCollisionResolver: FORGE's Random-5 strictly dominates every KNN setting on both AL-GR and Amazon [forge], so the default flips from KNN-σ; KNN-σ remains an option where last-level semantics matter (§5).
  • Stage-1 embedding freezing is backend-specific (D7): the DTensor restore-after-step trick is proven on FSDP2 but broken under Megatron's DistributedOptimizer fp32 masters; the correct Megatron mechanism is gradient zeroing after finalize_model_grads plus a weight-decay override (§8).
  • Item↔SID is many-to-many from day one (D4): FORGE collision buckets, PIT's dynamic top-B index, and Latte's latent trees all break one-SID-per-item; the catalog, decoder, and metrics are specified accordingly (§5, §10).
  • New algorithm coverage: OneRec-Think/OneReason reasoning recipes (two-stage rollouts, stage-wise clipping, RFT, on-policy distillation), PLUM's co-pretraining and evaluation lessons, PIT/Latte/RPG extensions — folded into D10, D15, D16.
  • Serving implementation is analyzed to a decision (2026-06-10): a GR service decomposes into six layers with an engine-invariant core that stays tzrec code under every topology; wide beams turn on sharing KV bandwidth, not storage — beams-in-query measured 135× over per-sequence attention in a kernel-level A10 microbenchmark; recsys-examples' sid-gr-inference (PR #414) is adopted as a fork behind the BackboneAdapter contract — the tzrec GR serving runtime, serving all softmax-attention GR backbones (Qwen3/Qwen2.5 token-only now; OneRec-V1/V2 via planned enc_dec/lazy_decoder adapters), with HSTU the sole exception on tzrec's own stack [repo-recsysexamples]; and W=512-synchronous serving on Qwen3-class backbones is demoted to a demand-gated contingency (§10).

1.4 Non-goals (v1)

  • No raw-pixel/multimodal encoders inside tzrec: the multimodal item embedding is an upstream artifact consumed as a parquet/ODPS column (§5.1).
  • No general-purpose LLM pretraining from scratch; every recipe starts from an open checkpoint (Qwen3 family).
  • No replacement of TorchRec for sparse sharding; no DeepSpeed; no pipeline parallelism in v1 (PP=1).
  • Reward models (P-Score multi-tower) are trained with tzrec's existing multi-task stack and consumed by RL as a scoring service; their training is not redesigned here.
  • No synchronous LLM reasoning on the request path: CoT models serve via nearline/Think-Ahead caching patterns (§10.7).
