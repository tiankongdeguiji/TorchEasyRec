10. Decoding, inference, and serving

This section specifies how a trained generative recommender becomes a serving artifact: the constrained-decoding layer that keeps beams inside the catalog (D12), the export contract for a single-step decode scorer driven by a host-side beam loop (D13), the KV-cache strategy ladder, the serving-engine landscape, and the serving topologies that wrap them. The structural rule from §4 holds throughout: the autoregressive decode loop is never traced into an exported graph — it lives in host code invoking an export-clean scorer, exactly as TDM retrieval already does (tzrec/tools/tdm/retrieval.py:121-414). Both decisions owned here changed status during research: the single-step export path was empirically proven on this machine (R-EXPORT de-risked), and the tensorized-trie design from STATIC [static] resolves the constrained-decoding cost concern (R-TRIE resolved). A third update is the engine landscape: all three OSS serving engines were studied at source level (§10.5), with two headline moves — RTP-LLM's OSS build turns out to ship the AL-GR/FORGE constrained-decoding production path, and SGLang enters as arguably the best-equipped RL sampling-rollout engine. Engine-native constraint paths are additional consumers of the same SidTrie artifact, not replacements for the D12/D13 defaults.

10.1 Requirements

The decode subsystem must cover a wide operating range: published systems disagree by an order of magnitude on beam width, and SID geometry is a config, not a constant (D5).

Latency anchors. OneRec-V2 serves a 1B-parameter lazy decoder at context length 3000 with beam 512 in 36 ms on L20 GPUs at 62% inference MFU [onerec-v2] — the bar for a purpose-built architecture with trivial (3-token) self-attention. For decoder-only transformer backbones, recsys-examples measures its KV-cached generate_beam_decode at 2.1× / 4.7× / 11–12× over cacheless full re-encoding at history lengths 256/512/1024, with the decode loop itself ~15 ms and dominated by small-batch dense projections rather than attention [repo-recsysexamples]. The lessons: KV caching across SID levels is mandatory for transformer backbones, and below width ~512 the GEMMs, not cache management, set the floor. FORGE adds a third anchor of a different kind: its nearline path writes 1200 candidates to a cache table within 200 ms — generative retrieval need not sit on the synchronous request path at all (§10.6) [forge]. Engine cost models at beam 512 sit an order of magnitude higher at 3000-token contexts (§10.5): context length, not beam width, dominates.

Grounding through the many-to-many catalog (D4). Decoded SIDs are not items. Three mechanisms adopted in §5 make the SID↔item map many-to-many, and the decode layer must honor all of them: (a) collision buckets — up to σ items share one SID under FORGE-style post-processing, so evaluation and serving must expand each generated SID into its item list in beam order before truncating to top-K (FORGE's calc_hr.py protocol: 100 beams × σ=5 yields up to 500 candidates [repo-alsid]); (b) Latte latent tokens — beams reaching the same item through different latent roots have their scores aggregated (agg_sum or agg_max; empirically equivalent [latte]); (c) PIT-style dynamic indices keep top-B SIDs per item [pit]. SidCatalogIndex (§5) owns this: the decode driver hands it (sid_tuple, score) pairs in beam order and receives a deduplicated, score-aggregated item ranking — dedup is item-level, not SID-level.

Level-agnostic driver. Per D5, the host driver delegates the number of scorer invocations to BaseSidDecoder (3 AR steps default, 1 for RPG-style parallel decode, T for diffusion — §7); nothing in this section may hard-code three levels or AR-only assumptions. The default 3×8192 geometry (§5) coexists with LC-Rec's 4×256 and semantic-ids-llm's 3+1 dedup-digit scheme [repo-semanticids]; trie, head, and catalog index all take the level structure from the tokenizer artifact.

10.2 Constrained decoding (D12)

Decision D12 — per-prefix legality via a STATIC-style CSR tensorized trie; unconstrained beam is a supported config

tzrec implements catalog-grounded constrained decoding as a packed-CSR tensorized trie following STATIC [static][static-repo]: the catalog's SID set is flattened offline into start_mask + dense head tables + a packed_csr [token, next_state] array + indptr row pointers + per-depth layer_max_branches, so one constrained decode step costs a single batched gather. The trie is the canonical tzrec artifact; the masking logic ships as SidConstrainedLogitsProcessor with one consumer surface per execution environment — an HF LogitsProcessorList drop-in, a vLLM v1 engine-level processor, SGLang custom-LP and grammar-backend bindings, an RTP-LLM prefix_dict rendering, and a torch-native path inside the host beam driver — so engine-native constraint paths consume the same artifact rather than replacing it. Unconstrained beam search with post-hoc catalog grounding is a supported configuration, not a degraded mode: GRID found free-form decoding matches or beats constrained [grid]; PLUM launched at YouTube unconstrained with <5% hallucination [plum]. This resolves R-TRIE.

Why flat per-depth masks are insufficient

The cheap constraint — masking step-l logits to the level-l itemic token range [151669 + l·8192, 151669 + (l+1)·8192) — guarantees only well-formed SIDs, not that the SID denotes an item: at 3×8192 the leaf space holds 8192³ ≈ 5.5×10¹¹ combinations against ~2.5×10⁸ cataloged items, so 99.95% of well-formed leaves are vacant and beam mass parked on them is wasted width. Catalog-grounded generation needs per-prefix legality: given the tokens emitted so far, which children lead to at least one real item. The naive implementations both fail at scale: HF's prefix_allowed_tokens_fn is a per-row Python callback (a CPU stall per beam per step — the original R-TRIE concern), and GRID's vectorized alternative brute-force compares each candidate prefix against the full (N, H) codebook tensor in 100k-row chunks, O(C·b·H) per step, with an in-code TODO conceding a trie is needed (tiger_generation_model.py:202-251 [repo-grid]).

The STATIC CSR design

STATIC ("Vectorizing the Trie", YouTube/Google) is deployed at 100M–1B-item scale; its open-source repo ships JAX and PyTorch reference implementations [static-repo]. build_static_index(sorted_sids (N,L), vocab_size, dense_lookup_layers=d) (static_decoding/csr_utils.py:21-203) emits five read-only tensors: a start_mask (V,) of valid level-0 tokens; dense (V,)×d mask/state tables for the first d "hot head" layers where nearly every prefix exists; a row-major packed_csr (E+V, 2) int32 of [token, next_state] pairs in which each state's children are contiguous and token-sorted (V padding rows for clamp-safe out-of-bounds reads); indptr (S+2,) int32 row pointers; and layer_max_branches, the per-depth maximum fan-out K fixing the static gather width. The constrained step (decoding_pt.py:45-116) is one batched "burst read":

starts = indptr[states]; lens = indptr[states + 1] - starts        # (B*W,)
pairs  = packed_csr[(starts[:, None] + arange(K)).clamp(max=last)] # (B*W, K, 2) one gather
cand_tok, cand_next = pairs[..., 0], pairs[..., 1]
cand_lp = logprobs.gather(1, cand_tok.clamp(max=V-1).long())
cand_lp = where(arange(K) < lens[:, None], cand_lp, -inf)          # invalid slots masked

Crucially this operates on candidate log-probs and never materializes a (B·W, V) mask: beam-score fusion (scores = cum[:, :, None] + cand_lp.view(B, W, K) then flat topk(W)) is strictly cheaper than mask-then-topk whenever K « V. The repo's sparse_transition_torch (decoding_pt.py:146-304) is a complete torch beam harness over this kernel — step 0 via start_mask, dense tables for depths < d, CSR elsewhere — and is the direct template for tzrec's host driver (§10.3). STATIC's deployed numbers (+0.033 ms/step, 948× over a CPU trie) are TPU v6e measurements at V=2048/L=8 — not quotable as GPU expectations, but our V=8192/L=3 burst reads are smaller per step.

One adaptation is required: STATIC assumes trie tokens are the model vocabulary, while tzrec's itemic tokens live at offsets inside a ~176K LLM vocab. The trie therefore stores per-level local ids 0..8191 (token id minus the level base derived from itemic_id_range, §7), and consumers add the offset back when masking LLM logits — dense tables at V=176K would be absurd; at V=8192 they are 64 MB.

NaN guard. A row masked to all -inf makes the subsequent log_softmax emit NaN, silently corrupting beam scores. By construction every reachable trie state has ≥1 child, so the guard is an invariant assertion, not a fallback: arm only on states reached through the trie itself, and decode fixed-length with ignore_eos so EOS candidates are never siphoned into vLLM's completed-set rescoring.

SidConstrainedLogitsProcessor consumer surfaces

The same artifact backs five consumers. (i) The HF variant is a LogitsProcessorList-compatible class for generate()-based eval — implemented as the vectorized CSR gather above, explicitly not as a prefix_allowed_tokens_fn closure, which HF executes as a Python loop over rows. (ii) The vLLM v1 variant implements the engine-level interface (vllm/v1/sample/logits_processor/interface.py: apply, update_state, is_argmax_invariant() = False), registered via --logits-processors and armed per request either through SamplingParams.extra_args (RL rollouts, §9) or by detecting an <|sid_begin|> tail in prompt_tok_ids and walking the trie over the partial-SID suffix. Prompt-tail arming is what makes one processor cover vLLM's emulated beam search too, since each beam step is a fresh request whose prompt carries the full prefix (§10.5). Two engine flags are load-bearing — logprobs_mode=processed_logprobs (the default raw_logprobs snapshots logprobs before sampler-stage processors run, making the mask invisible to beam ranking) and max_logprobs ≥ 2×beam_width — and two composability limits (rejected under spec decode; forces the V1-runner fallback) are detailed in §10.5. (iii) The SGLang bindings come in two grades: a per-request CustomLogitProcessor (dill-serialized class behind --enable-custom-logit-processor; it sees each request's live state per step and lazy-loads the trie from a custom_params path) works today with zero server code, while the production-grade route registers a sidtrie grammar backend via the public register_grammar_backend hook — one file implementing fill_vocab_mask/accept_token over the same CSR, inheriting scheduler integration, spec-decode rollback, and reasoning-gated activation (§10.5) [sglang]. (iv) RTP-LLM consumes the artifact as configuration: the build job renders the trie into the engine's tree_decode_config prefix_dict JSON for its in-engine TreeLogitsProcessor — viable for small and medium catalogs; 250M items needs the loader patch named in §10.5 [rtp-llm]. (v) The host-driver variant (§10.3) calls the gather directly on the scorer's logits, sliced to the level-local range before log_softmax.

Unconstrained mode and post-grounding

The trie is a config knob, not an architectural dependency. GRID's ablation found constrained and free-form beam search nearly identical, with free-form sometimes better (Beauty R@10 0.0609 vs 0.0597) and cheaper — the model learns valid SID structure from SFT alone [grid]. PLUM shipped YouTube production retrieval unconstrained, tolerating <5% invalid SIDs [plum], and OneRec likewise serves without a trie [onerec-v1]. In unconstrained mode, generated SIDs are grounded post-hoc by SidCatalogIndex with an exact-then-prefix fallback: try the full tuple, then back off to 2-token and 1-token prefix buckets — the graceful-degradation trick from semantic-ids-llm, exploiting the hierarchy so even a hallucinated leaf resolves to semantically nearby items [repo-semanticids]. Recommended defaults: constrained for RL rollouts (legality directly shapes reward, §9) and for serving at 10⁸-item scale; unconstrained as the cheap baseline for eval ablations and small catalogs.

Trie artifact build job

The trie is built offline from the same item_id → SID assignment table that feeds SidCatalogIndex (§5): sort the deduplicated (N, L) code array, run build_static_index (vectorized NumPy — minutes at 250M rows), and write an .npz containing the five tensors plus per-level bases and the tokenizer fingerprint. On request the builder also emits the RTP-LLM prefix_dict JSON rendering; because that format enumerates per-prefix candidate lists explicitly, it is bounded to small/medium catalogs (§10.5). The artifact is versioned with, and invalidated by, the SID assignment snapshot — under streaming corpora the re-tokenization cadence (R-RETOK, §15) governs trie refreshes, and serving must atomically swap trie, catalog index, and any cached prefix sets as one unit. With Latte latent tokens enabled, the builder prepends the latent level as depth 0 (M′ extra root branches); nothing else changes (D16).

10.3 Single-step scorer export (D13)

Decision D13 — host-side beam driver over an AOTI-exported single-step scorer with explicit KV-cache I/O

The serving artifact for the default Qwen3 backbone is a torch.export + AOTInductor package exposing one decode step with the KV cache as explicit inputs, plus a prefill graph; the beam loop (level masking, trie gather, top-k, cache reorder) runs host-side in the serving processor. This is empirically proven on this machine (torch 2.11.0+cu129, transformers 4.53.3, tiny Qwen3, bf16, sdpa): the step exports strictly and matches eager StaticCache decoding bit-exactly over 5 sequential steps; compiles and runs under AOTI (fp32-exact; bf16 within ≈5e-3–8e-3 fusion noise); survives host-side index_select beam reorder; and the explicit-I/O variant compiles with Dim("batch") so one artifact serves any users×beams product. R-EXPORT is thereby de-risked from design assumption to validated recipe with enumerated residuals. The legacy FX-trace + TorchScript path is a NO-GO for HF models.

The proven recipe

The export rides transformers' own supported path. On the 4.53.x pin, TorchExportableModuleForDecoderOnlyLM (integrations/executorch.py, v4.53.3 L30-72) wraps the model with a StaticCache whose per-layer [batch, num_kv_heads, max_cache_len, head_dim] tensors are registered as non-persistent buffers, and swaps attention to sdpa_without_vmap — 4.53's default mask construction uses torch.vmap, which does not trace under torch.export (on the v5 line mask creation is export-safe by default and the shim disappears). forward(input_ids, cache_position) derives positions, RoPE, and the causal mask from cache_position inside the graph; the cache's in-place index_copy_ mutation is preserved through export and AOTI — PyTorch 2.11's AOTI suite includes a literal KV-cache buffer-mutation test (test/inductor/test_aot_inductor.py:2835) plus bf16 SDPA coverage [pytorch][transformers]. Vocab extension to 176,384 rows (§7) is invisible to the recipe — it only resizes embed_tokens and the untied lm_head.

The HF recipe fixes batch×beam at export time (cache buffers are module state). The preferred tzrec variant therefore moves the cache to explicit I/O, which was also smoke-tested end to end including dynamic batch:

def forward_step(
    input_ids:      torch.LongTensor,   # [N, 1]   N = B_users x W_beams  (Dim("batch"))
    cache_position: torch.LongTensor,   # [1]      absolute position = prefix_len + step
    *kv_caches:     torch.Tensor,       # 2L x [N, H_kv, S_max, D] bf16, mutated in place
) -> torch.Tensor:                      # logits [N, 1, vocab ~ 176384]

Explicit I/O keeps all state host-owned (no hidden tensors in the artifact), matches tzrec's "dense module + explicit inputs" export style, and enables one artifact across beam widths. The buffers variant remains a proven fallback: AOTICompiledModel.load_constants({"key_cache_i": ..., ...}, user_managed=True) adopts host tensors by storage pointer, so the host can still reorder cache rows between calls. Prefill needs no separate cache input — a prefill graph (sequence-length Dim is the documented dynamic-shape example) or host-side eager prefill writes positions [0, prefix_len) of the same tensors, once per user followed by repeat_interleave(W, dim=0) or directly at N with repeated prompts.

Host beam loop and measured parity

Per SID step the host masks logits to the legal level range (plus the trie gather of §10.2), takes log_softmax, accumulates beam scores, runs a flat topk(W) over [B, W·K] (parent = idx // K, token = idx % K), and reorders every cache tensor with kv.copy_(kv.index_select(0, beam_idx)). For 3-token SIDs only steps 2 and 3 reorder — step 1 is pure expansion — so total reorder traffic is two copies of [B·W, H_kv, S_max, D] per layer per request: microscopic at W ≤ 32, affordable at 512. The smoke matrix (tzrec120 env, A10):

Gotcha 1 — AOTI silently corrupts on non-contiguous inputs

Feeding the AOTI artifact a strided slice such as generated[:, -1:] (shape (B,1), stride (S,1)) produced garbage logits with no error — on CPU and CUDA, fp32 and bf16, even with zero transformer layers. AOTI reads the raw data pointer assuming example-input strides; ep.module() handles the same input fine, which makes the bug easy to miss in eager-parity testing. Host beam loops naturally produce exactly such slices. The tzrec step-runner must .contiguous() every input and assert strides before each call. (Bisected empirically; reproducible.)

Gotcha 2 — the AOTI .so requires GLIBCXX_3.4.30

dlopen of the compiled package failed on a system libstdc++ lacking GLIBCXX_3.4.30; fixed via LD_LIBRARY_PATH=$CONDA_PREFIX/lib. PAI serving images must carry a matching toolchain runtime on the dlopen path — this is a deployment-image checklist item, not a code change.

Residual proof obligations (tracked in §14): (1) a real-scale artifact — Qwen3-0.6B/1.7B with the 176,384-row vocab — measuring compile time, .pt2 size, load time, and per-step latency on PAI GPUs (the smoke test used a toy config); (2) per-row cache_position [N] for mixed prefix lengths in one batch (only the scalar-position variant was tested; left-padding within a user group sidesteps it); (3) prefill-graph export with a dynamic sequence dim (documented HF recipe, not exercised here); (4) re-running the matrix if the transformers pin moves to v5, where the drift is real: key_cache lists become cache.layers[i].keys, a new cumulative_length_{i} buffer must be included in constant handling and reset per generation, and v5's Cache.reorder_cache rebinds attributes via index_select — which breaks exported-buffer aliasing, so reorder must stay copy_-based. Cache-internal access goes through a thin version shim from day one.

Fit with tzrec's existing export machinery

The AOTI step package is a sibling of what tzrec already produces: tzrec/acc/aot_utils.py:export_model_aot performs torch.export.export with Dim("batch", min=1, max=499999999) (L177) and aoti_compile_and_package into aoti/aoti_model.pt2 (L226-235), with load_model_aot on the predict side — the scorer reuses the same compile config and loading path. The default FX route is not viable here: export_model_normal (tzrec/utils/export_util.py:148) is symbolic_trace + torch.jit.script (L288, L292), neither of which survives HF's config-dependent control flow and cache objects. The RTP export path is likewise out of scope for the LLM trunk — embedding-table-oriented, hard-rejecting DTensor parameters (export_util.py:484) — though its batch_size==1 convention ("when batch_size=1, we assume gr model", export_util.py:1010-1011) already anticipates GR-shaped serving. The host-driver pattern itself is established tzrec practice: TDM retrieval runs a Python beam loop over a scripted scorer; the SID driver swaps GraphLearn child expansion for trie children and stateless re-encode for explicit KV-cache I/O.

10.4 KV-cache strategies

Four cache designs form a ladder from "ship now" to "optimize later"; the host-driver contract (scorer in, logits out, cache reorderable by row) is identical across them, so they swap without touching decode logic.

(a) HF StaticCache with host index_select reorder — the default. Per-layer [N, H_kv, S_max, D] tensors, reordered by index_select at beam pruning. Proven bit-exact above; trivial host logic; two reorder copies total for 3-token SIDs. At W ≤ 512 with GQA models this traffic sits far below the dense-projection floor recsys-examples measured — no performance argument against it in v1.

(b) recsys-examples append-only beam KV + ancestry indices — the later optimization. The SID-GR example never reorders: context KV (prefill over history+BOS) is a shared slab, per-step beam KV is appended step-major, and ancestry is resolved at attention time by BeamSearch.build_beam_topk_indices, which traces parent_indices backwards into a gather tensor consumed by the beam_decode_attn kernel (corelib/gr_decode_atten) [repo-recsysexamples]. This avoids all reorder copies and avoids replicating history W times — but it requires a nonstandard ancestor-gather attention signature that HF Qwen3 layers do not compute (i.e. exporting a custom-attention model, forfeiting the HF-recipe guarantees), and the kernel requires uniform widths with beam_width ≤ min(codebook_sizes). recsys-examples' own profiling shows the kernel saved ~0.15 ms of a 6.7 ms forward — dense projections dominate. Adopt only if decode latency at high B·W becomes a measured bottleneck.

(c) delta_hstu_mha jagged cache — the in-house backbone's native path. For InhouseHstuGRBackbone (D5), tzrec already ships the prefill/decode cache machinery: STULayer holds per-layer k_cache/v_cache with jagged offsets, prefill populates them via kv_caching_lengths, and cached_forward runs delta_hstu_mha over cache+delta (tzrec/ops/hstu_attention.py:199-267, tzrec/modules/gr/stu.py:564-641). Constraints to respect: the delta size must be uniform per row (beam steps append exactly one token per row, which fits), CUTLASS falls back to Triton on the delta path, and the cache lives as Python module attributes — fine precisely because the loop is host-side; beam forks reorder by re-indexing k_cache/v_cache and offsets, mirroring (a).

(d) recsys_kvcache_manager for cross-request session reuse. For high-QPS deployments where a user's history prefix recurs across requests, NVIDIA's standalone paged GPU KV cache with host/FlexKV tiered offload keys cache identity by user_id — recommendation histories never share prefixes across users, so engine-style token-prefix matching (vLLM's block hash, SGLang's radix tree) buys nothing while user-keyed reuse skips re-prefilling thousands of history tokens. Measured on the HSTU inference example: 1.3–2.6× end-to-end with CUDA graphs [repo-recsysexamples]. This slots under topology (a) of §10.6 as a phase-2 optimization for the HSTU and lazy-decoder backbones.

10.5 Serving engines: vLLM, SGLang, RTP-LLM

All three OSS engines were studied at source level on 2026-06-10: vLLM at HEAD fe1d923af (v0.22.1+ dev, ten minor releases past the earlier v0.12 snapshot), SGLang at HEAD b40f365732 (v0.5.12.post1), and RTP-LLM at OSS v0.2.0 b0c09a344 [vllm][sglang][rtp-llm]. None replaces the D13 host driver — topology (a) of §10.6 remains the default and the only path for non-HF backbones — but each earns a distinct role, and all three consume the D12 SidTrie artifact through their own constraint surfaces.

vLLM — eval harness, emulated beams, and a rescoring channel

The seven claims established at v0.12 all hold at HEAD [vllm], with refinements. BeamSearchParams still carries no constraint fields and no vllm#21546-class fix ever landed (the beam code moved to vllm/entrypoints/generate/beam_search/); logprobs_mode landed in v0.10.0 (PR #21398), settling the version question; and grammar bitmasks are applied in the model runner before the sampler, so structured-output masks are visible even in default raw_logprobs — only sampler-stage processors (custom LPs, allowed_token_ids, logit_bias) need processed_logprobs. Two new hard conflicts: custom logits processors are rejected when speculative decoding is enabled (structured outputs do compose, making a grammar-borne constraint the only single-engine EAGLE-CoT pairing — PoC, not assumable), and the Model Runner V2 now default for Qwen3 dense supports neither custom LPs nor prompt embeds, silently falling back to the V1 runner. tzrec uses vLLM for offline eval at widths 10–50, OpenOneRec-parity checks, and SID-shortlist rescoring: the new generative scoring API plus SamplingParams.logprob_token_ids (≤128 ids) and flat_logprobs score pre-retrieved candidates teacher-forced — one prefill per user, no decode loop. Constrained beams always go through tzrec's custom client loop (~200 lines modeled on the online beam mixin) carrying per-step allowed_token_ids or extra_args, which also unlocks FORGE-style variable widths; prompt embeds now hash into prefix-cache keys (an embeds-input custom loop keeps reuse) but remain blocked in the beam API. Operational keys: --max-logprobs ≥ 2W (1024 at width 512), --max-num-seqs ≥ 512 (default 128; CUDA-graph capture extends to min(2×max_num_seqs, 512), keeping beam-512 batches on captured graphs), --logprobs-mode processed_logprobs, --logits-processors; async scheduling is default-on. R-VLLM-PIN is re-anchored at v0.22.1+ (§15).

SGLang — the RL-rollout engine, not a beam engine

SGLang still has no beam search of any kind — a repo-wide grep at HEAD returns zero hits, not even a vestigial field [sglang] — so it stays off the beam-faithful eval and production paths. For GRPO sampling rollouts (§9), however, it is arguably the best-equipped engine. A per-request dill-serialized custom logit processor (behind --enable-custom-logit-processor) sees the live request state each step and can walk the full 250M-item trie lazy-loaded from a custom_params path — trie-constrained sampling with zero server code; the cleaner server-side route is the registered sidtrie grammar backend (§10.2); per-level itemic masks alternatively fit in a ~0.4 MB EBNF compiled once and cached — provided the itemic tokens are registered as non-special added tokens, or xgrammar cannot emit them. The ReasonerGrammarBackend keeps any grammar dormant inside <think>…</think> and arms it after, with a per-request thinking_budget — CoT-then-constrained-SID in one request, the exact two-stage rollout shape. The radix cache at page_size=1 lets N CoT samples pay the ~3K-token prefill once, with K constrained continuations radix-hitting prompt+CoT; returned logprobs are post-constraint and renormalized (the semantics GBPO logging wants); and the verl weight-update surface (update_weights_from_*, memory release/resume, --rl-on-policy-target fsdp bit-exact inference) is the richest of the three. Caveats: input_embeds forces --disable-radix-cache server-wide, so embedding-input rollouts lose all prefix reuse; the enc-dec whitelist is three audio/vision models; emulated beam-512 (~100–300 ms idle vs ~30–50 ms for a native beam) monopolizes the scheduler — a diagnostic, not a topology.

RTP-LLM — engine-native beams plus an OSS generative-rec constraint stack

The deep study refutes this RFC's earlier claim that no logits-processor or constrained-decoding hook exists in the OSS build [rtp-llm]. OSS RTP-LLM ships a C++ logits-processor framework running inside Sampler::forward before beam top-k: TreeLogitsProcessor implements catalog-trie constrained decoding configured purely by --tree_decode_config (a JSON prefix_dict of per-prefix candidate lists), with per-beam DFA state forked and reordered on every beam transition; RecommendationLogitsProcessor (landed 2026-04-29) is purpose-built for SID generative rec — generate_config.combo_token_size treats every k consecutive tokens as one item, banned_combo_token_ids/auto_parse_banned_combo filter already-exposed items parsed from the prompt, generated items are deduplicated across beam steps, and a Qwen3 think-prelude is skipped via DFA. This is the AL-GR/FORGE production path, in OSS [forge]. The native beam path is stronger than assumed: variable_num_beams gives per-step widths (step 1 always width 1; the TRT-derived kernels extended to padded width 4096); the KV cache forks block-level copy-on-write — full blocks refcount-shared, only the last partial block copied per fork, so a 3000-token prompt is held once and 512 beams add about one block each; and masks apply before log_softmax, so cum_log_probs renormalize over the allowed set. ThinkModeLogitsProcessor bounds CoT deterministically; the 64-token-block prefix cache (REUSE_CACHE=1, 3-tier) serves two-stage CoT-then-beam request pairs; gRPC UpdateWeights with verl name maps plus SetPause/SetRestart covers RL. Limits, stated plainly: speculative decoding (including Qwen3-MoE EAGLE3/MTP) does not compose with beam requests; the beam path is deterministic — per-beam temperature is ignored, so RL rollout diversity must come from the CoT sampling stage; and three patch surfaces gate large-scale adoption — (1) BaseLogitsProcessor::generateVocabMask builds a dense CPU [B, V] mask and H2D-copies it every step (~90 MB/step at beam 512; ~100–200 LoC fix following the Recommendation processor's sparse-write pattern), (2) PrefixToCandidateTokens::loadPrefixDict loads JSON plus a string-keyed hash map and does not scale to 250M items (compact binary format plus a reload RPC for catalog refresh), (3) single-request CoT-then-beam is inexpressible because the width schedule indexes absolute output position — patch numBeams to index from the think-end DFA position, or run two requests over the prefix cache. Cost model: ~0.8–1.2 s/request at beam 512 over 3000-token contexts, bf16 on L20 — attention-read bound, since shared-prefix KV is re-read per beam; fp8 KV halves it, and FORGE's 200 ms envelope corresponds to ~500-token contexts. The trie build job (§10.2) renders the SidTrie additionally as prefix_dict JSON for small/medium catalogs.

10.6 Serving topologies

(a) PAI-EAS + AOTI scorer + host driver — the zero-dependency default. The processor loads the AOTI step package (and prefill graph), the trie .npz, and the SidCatalogIndex snapshot; the beam loop, trie gather, and item grounding run in the processor process. This is the only topology requiring no engine patches and no new serving dependency, the only one serving all backbones including encoder-decoder and HSTU (which no OSS LLM engine speaks), and it inherits tzrec's predict-side conventions — per-beam outputs ride the TARGET_REPEAT_INTERLEAVE_KEY flatten/regroup path into ListArray columns. Its ceiling is host-loop Python overhead at very large B·W, exactly where (b) takes over.

(b) RTP-LLM — engine-native beams with an OSS constraint stack: the FORGE production shape. Per §10.5, everything FORGE's serving contract needs is config-only in the OSS engine: variable_num_beams=[300,600,1200] reproduces the dynamic schedule, tree_decode_config grounds beams in the catalog trie, and RecommendationLogitsProcessor supplies the prompt-parsed exposure filter and cross-step item dedup [rtp-llm][forge][repo-alsid]. At small/medium catalog scale this is the lowest-code path to production beam-512: tzrec exports the SidTrie as prefix_dict JSON, the engine returns per-beam cum_log_probs renormalized over the allowed set, and SidCatalogIndex grounds the result. At 250M items the two named patches (sparse mask writes; compact trie loader + reload RPC) gate adoption, and 3000-token contexts cost ~0.8–1.2 s/request bf16 on L20 — fp8 KV or shorter serialized contexts buy back the 200 ms envelope. Recommended where Taobao-side infra alignment matters or production beam widths rule out emulation; the engine is PAI-ecosystem native (iGraph, vipserver, kmonitor in-tree).

(c) vLLM serving — eval harness and emulated beams for token-only Qwen3 models. vLLM's beam search remains a pure-Python emulation at the API layer: one 1-token request per live beam per step — 2W+1 requests per 3-token decode — with prefix caching bounding per-beam recompute to a ≤16-token tail; block-aligning the prompt so SID tokens start a fresh block cuts step cost ~4× [vllm]. BeamSearchParams still propagates no constraints, so constrained beams run through tzrec's custom loop plus the engine-level SidConstrainedLogitsProcessor with prompt-tail arming and processed_logprobs (§10.2). At width 512 on L20: ~0.2–1.4 s/step for Qwen3-8B (~1.5–3.5 s/request) — fine for offline eval and OpenOneRec-parity at W ≤ 50, marginal for synchronous production at 8B; 0.6–1.7B models drop to ~30–100 ms/step and are production-plausible. The scoring API offers a decode-free alternative: rescore a pre-retrieved SID shortlist teacher-forced (§10.5). Cautions from §10.5 apply (V2-runner fallback; the spec-decode/custom-LP conflict). API drift is R-VLLM-PIN (§15), re-anchored at v0.22.1+. The OpenAI-endpoint + custom-LP + processed_logprobs combination is source-verified but not yet GPU-smoke-tested — it gets a validation run before this topology is declared supported (§14).

SGLang's role — RL rollouts and offline CoT, never beams. SGLang is deliberately absent from the beam topologies: with no beam path it cannot serve beam-faithful retrieval, and emulation monopolizes its scheduler (§10.5). Its place is the training side: the recommended GRPO rollout engine (§9) — trie-constrained sampling via custom LP or the sidtrie grammar backend, single-request CoT-then-constrained-SID via ReasonerGrammar, deterministic on-policy logprobs, the most complete verl weight-update surface — and the natural harness for offline CoT-generation jobs feeding distillation and nearline reasoning corpora.

(d) Nearline patterns — how reasoning-class models actually ship. No published system serves LLM reasoning synchronously; three deployed patterns cover the spectrum. FORGE async-trigger cache: generation fires on a user event (item-detail click), the RTP-LLM-served path writes 1200 candidates to a cache table within 200 ms (the cache write is application-side, not in-engine), and the existing ranking stage consumes the cache — generative retrieval as a feeder, off the latency path [forge]; the engine-side half is now visible in OSS — RecommendationLogitsProcessor bans already-exposed items at generation time, exactly the contract a refreshed candidate cache needs [rtp-llm]. OneRec-Think Think-Ahead: an offline reasoning job beam-searches only the first 2 SID levels of T reasoning paths per user into a per-user prefix cache; online, a small realtime OneRec decodes the final level constrained to the cached prefixes [onerec-think] — requiring the constrained-decode utility to accept "constrain to cached prefix set" as an alternative trie root, a small generalization of the CSR design. OneReason fast–slow thinking: nearline batch inference with hierarchical decoding (beam at the top level, greedy below) feeds a Redis candidate pool with automatic fallback to the realtime model, while representation distillation ("Thinking Token" supervising the realtime decoder's BOS hidden state) carries reasoning value into the synchronous path [onereason]. tzrec v1 ships (a) plus the FORGE-style cache-table pattern as a documented deployment recipe; Think-Ahead prefix caches reuse the trie machinery; the distillation bridge is a §14 roadmap item.

Exposure-time generation-probability logging. Every topology above also owns the producing side of a training contract: GBPO (§9) populates pi_old from the family-4 column gen_prob_logged : ARRAY<DOUBLE> (§6) — the per-token generation probabilities of the exposed item's SID tokens, logged at exposure time. The values are already computed in every topology and need only be carried through: the host driver (a) takes per-token log_softmax scores during beam accumulation, so it attaches them to each (sid_tuple, score) pair handed to SidCatalogIndex and emits them per grounded candidate in the response (collision-bucket items expanded from one SID share that SID's token probabilities, D4); engine topologies (b)/(c) return per-token logprobs natively — in RTP-LLM and SGLang these are post-mask by construction (masking precedes the softmax; neither engine exposes a pre-constraint channel), while vLLM needs processed_logprobs for sampler-stage masks — so in all cases the logged value reflects the masked distribution actually decoded from (§10.5); the nearline patterns (d) persist them alongside candidates in the cache table so the downstream exposure×feedback join can recover them. The fallback is part of the contract: items exposed through non-generative traffic carry no generation probability, the column is absent, and the trainer falls back to ratio = 1 — the regime GBPO is designed to keep stable — so logging is mandatory only on generative impressions, never a backfill obligation on the full exposure log.
