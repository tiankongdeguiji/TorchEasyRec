10. Decoding, inference, and serving

This section specifies how a trained generative recommender becomes a serving artifact: constrained decoding (D12), the single-step scorer export driven by a host-side beam loop (D13), the KV-cache ladder, the engine landscape, and the serving topologies. The structural rule from §4 holds throughout: the autoregressive decode loop is never traced into an exported graph — it lives in host code invoking an export-clean scorer, exactly as TDM retrieval already does (tzrec/tools/tdm/retrieval.py:121-414). Research moved both owned decisions: the single-step export path is empirically proven on this machine (R-EXPORT de-risked), and STATIC's tensorized trie [static] resolves the constrained-decoding cost concern (R-TRIE resolved). The engine study (§10.5) found RTP-LLM's OSS build ships the AL-GR/FORGE constrained-decoding production path and SGLang the best-equipped RL sampling-rollout surface — engine-native constraint paths consume the same SidTrie artifact, they do not replace the D12/D13 defaults. The serving-implementation analysis (2026-06-10) adds the six-layer decomposition with its engine-invariant core (§10.1) and adopts recsys-examples' sid-gr-inference as the seed of tzrec's own runtime; the extension plan (2026-07-03) specifies that runtime as the tzrec-gr-serving fork behind the BackboneAdapter contract, serving all softmax-attention GR backbones (§10.6).

10.1 Requirements

The decode subsystem must cover a wide operating range: published systems disagree by an order of magnitude on beam width, and SID geometry is a config (D5).

The six-layer decomposition and the engine-invariant core

A GR inference service is six layers. L1 prompt frontend — user features → prompt and/or context embeddings, shared code with the training-side task builder (train/serve skew is the classic rec killer); serving assembles token ids directly: static template segments pre-tokenized offline, itemic ids computed arithmetically, never string-tokenized. L2 tokenizer — hot-path only if L1 assembles strings. L3 model execution — prefill, decode, batching, KV cache, quantization, TP: the only layer where the engine question lives. L4 decode control — beam widths, constraint masking, logprobs, dedup/exposure filters. L5 grounding — SidCatalogIndex, response format, gen_prob_logged. L6 ops — EAS deployment, versioned artifact swap, A/B, monitoring, fallback. The engine-invariant core (L1, the L4 artifacts, L5, L6) is tzrec code under every topology in §10.7: the shared prompt frontend and token-id assembly; the trie builder and CSR runtime (~500 LOC owned from STATIC's 836-LOC, zero-server reference [static-repo]); SidCatalogIndex grounding and gen_prob_logged emission; versioned swap of (tokenizer, SID assignment, trie, catalog) as one unit; an eval harness on the same L1/L4/L5 code as serving. RTP-LLM's in-tree recommendation_parser.py proves even engine deployments need this python layer [rtp-llm].

The GR workload also has structure general engines cannot exploit: decode length is fixed and tiny (3–4 SID tokens; no EOS uncertainty, no streaming); requests are homogeneous, so gang/bucketed batching replaces continuous batching, preemption, and paged eviction; cross-user token-prefix reuse is ~0 — the reuse that matters is beams sharing one user's context within a request (§10.4) and user-keyed session reuse; and beam search is the primary decode mode (RTP-LLM's native rec-purpose beam path is the one engine exception). A general engine imports machinery this workload does not need while missing what it does: L1, L4 fidelity, L5, batched-beam attention.

Latency anchors. OneRec-V2 serves a 1B lazy decoder at context 3000, beam 512, in 36 ms on L20 at 62% inference MFU [onerec-v2] — read carefully: the lazy decoder shares context KV across beams by construction and cuts ~94% of decode compute, so 36 ms is not a target a Qwen3-class backbone can reach (§10.4). For decoder-only transformer backbones, recsys-examples measures KV-cached generate_beam_decode at 2.1× / 4.7× / 11–12× over cacheless re-encoding at history 256/512/1024, the decode loop itself ~15 ms and dominated by small-batch dense projections [repo-recsysexamples]: KV caching across SID levels is mandatory; below width ~512 the GEMMs set the floor. FORGE's nearline path writes 1200 candidates to a cache table within 200 ms — generative retrieval need not sit on the synchronous request path at all (§10.7) [forge]. Engine cost models at beam 512 over 3000-token contexts sit at ~0.8–1.2 s/request — 3–30× off a purpose-built path depending on L2 residency (§10.4): context length, not beam width, dominates.

Grounding through the many-to-many catalog (D4). Decoded SIDs are not items. Three mechanisms from §5 make the SID↔item map many-to-many, and the decode layer honors all of them: (a) collision buckets — up to σ items share one SID under FORGE-style post-processing, so each generated SID expands into its item list in beam order before truncating to top-K (FORGE's calc_hr.py protocol: 100 beams × σ=5 yields up to 500 candidates [repo-alsid]); (b) Latte latent tokens — beams reaching the same item through different latent roots aggregate scores (agg_sum/agg_max; empirically equivalent [latte]); (c) PIT-style dynamic indices keep top-B SIDs per item [pit]. SidCatalogIndex (§5) owns this: the decode driver hands it (sid_tuple, score) pairs in beam order and receives a deduplicated, score-aggregated item ranking — dedup is item-level, not SID-level.

Level-agnostic driver. Per D5, the host driver delegates the number of scorer invocations to BaseSidDecoder (3 AR steps default, 1 for RPG-style parallel decode, T for diffusion — §7); nothing here may hard-code three levels or AR-only assumptions. The default 3×8192 geometry (§5) coexists with LC-Rec's 4×256 and semantic-ids-llm's 3+1 dedup-digit scheme [repo-semanticids]; trie, head, and catalog index take the level structure from the tokenizer artifact.

10.2 Constrained decoding (D12)

Decision D12 — per-prefix legality via a STATIC-style CSR tensorized trie; unconstrained beam is a supported config

tzrec implements catalog-grounded constrained decoding as a packed-CSR tensorized trie following STATIC [static][static-repo]: the catalog's SID set is flattened offline into CSR tensors (start_mask, dense head tables, packed_csr, indptr, layer_max_branches), so one constrained decode step costs a single batched gather. The trie is the canonical tzrec artifact; the masking logic ships as SidConstrainedLogitsProcessor with one consumer surface per execution environment — an HF LogitsProcessorList drop-in, a vLLM v1 engine-level processor, SGLang custom-LP and grammar-backend bindings, an RTP-LLM prefix_dict rendering, and a torch-native path inside the host beam driver — so engine-native constraint paths consume the same artifact rather than replacing it. Unconstrained beam search with post-hoc catalog grounding is a supported configuration, not a degraded mode: GRID found free-form decoding matches or beats constrained [grid]; PLUM launched at YouTube unconstrained with <5% hallucination [plum]. This resolves R-TRIE.

Why flat per-depth masks are insufficient

The cheap constraint — masking step-l logits to the level-l itemic range [151669 + l·8192, 151669 + (l+1)·8192) — guarantees only well-formed SIDs: at 3×8192 the leaf space holds 8192³ ≈ 5.5×10¹¹ combinations against ~2.5×10⁸ cataloged items, so 99.95% of well-formed leaves are vacant and beam mass parked on them is wasted width. Catalog-grounded generation needs per-prefix legality: which children of the emitted prefix lead to at least one real item. The naive implementations fail at scale: HF's prefix_allowed_tokens_fn is a per-row Python callback (a CPU stall per beam per step — the original R-TRIE concern), and GRID's vectorized alternative brute-force compares each candidate prefix against the full (N, H) codebook tensor, O(C·b·H) per step, with an in-code TODO conceding a trie is needed [repo-grid].

The STATIC CSR design

STATIC ("Vectorizing the Trie", YouTube/Google) is deployed at 100M–1B-item scale; its repo ships JAX and PyTorch references [static-repo]. build_static_index(sorted_sids (N,L), vocab_size, dense_lookup_layers=d) (static_decoding/csr_utils.py:21-203) emits five read-only tensors: start_mask (V,) of valid level-0 tokens; dense (V,)×d mask/state tables for the first d "hot head" layers where nearly every prefix exists; a row-major packed_csr (E+V, 2) int32 of [token, next_state] pairs, each state's children contiguous and token-sorted (V padding rows for clamp-safe reads); indptr (S+2,) int32 row pointers; and layer_max_branches, the per-depth maximum fan-out K fixing the static gather width. The constrained step (decoding_pt.py:45-116) is one batched "burst read":

starts = indptr[states]; lens = indptr[states + 1] - starts        # (B*W,)
pairs  = packed_csr[(starts[:, None] + arange(K)).clamp(max=last)] # (B*W, K, 2) one gather
cand_tok, cand_next = pairs[..., 0], pairs[..., 1]
cand_lp = logprobs.gather(1, cand_tok.clamp(max=V-1).long())
cand_lp = where(arange(K) < lens[:, None], cand_lp, -inf)          # invalid slots masked

Crucially this operates on candidate log-probs and never materializes a (B·W, V) mask: beam-score fusion (cum[:, :, None] + cand_lp.view(B, W, K), flat topk(W)) is strictly cheaper than mask-then-topk whenever K « V. The repo's sparse_transition_torch (decoding_pt.py:146-304) is a complete torch beam harness over this kernel — step 0 via start_mask, dense tables for depths < d, CSR elsewhere — the direct template for tzrec's host driver (§10.3). STATIC's deployed numbers (+0.033 ms/step, 948× over a CPU trie) are TPU v6e measurements at V=2048/L=8 — not quotable as GPU expectations, though our V=8192/L=3 burst reads are smaller per step.

One adaptation: STATIC assumes trie tokens are the model vocabulary; tzrec's itemic tokens live at offsets inside a ~176K LLM vocab. The trie stores per-level local ids 0..8191 (token id minus the level base, itemic_id_range, §7); consumers add the offset back when masking LLM logits — dense tables at V=176K would be absurd, at V=8192 they are 64 MB.

NaN guard. A row masked to all -inf makes log_softmax emit NaN, silently corrupting beam scores. Every reachable trie state has ≥1 child by construction, so the guard is an invariant assertion, not a fallback: arm only on states reached through the trie, and decode fixed-length with ignore_eos so EOS candidates are never siphoned into vLLM's completed-set rescoring.

SidConstrainedLogitsProcessor consumer surfaces

The same artifact backs five consumers. (i) HF: a LogitsProcessorList-compatible class for generate()-based eval — the vectorized CSR gather, explicitly not a prefix_allowed_tokens_fn closure (a Python loop over rows). (ii) vLLM v1: the engine-level interface (apply, update_state, is_argmax_invariant() = False), registered via --logits-processors, armed per request through SamplingParams.extra_args (RL rollouts, §9) or by detecting an <|sid_begin|> tail in prompt_tok_ids — prompt-tail arming also covers vLLM's emulated beam search, where each beam step is a fresh request carrying the full prefix. Two flags are load-bearing — logprobs_mode=processed_logprobs (default raw_logprobs snapshots logprobs before sampler-stage processors, hiding the mask from beam ranking) and max_logprobs ≥ 2×beam_width; composability limits in §10.5. (iii) SGLang, two grades: a per-request dill-serialized CustomLogitProcessor (zero server code; lazy-loads the trie from a custom_params path), or the production route — a sidtrie grammar backend registered via the public register_grammar_backend hook, one file implementing fill_vocab_mask/accept_token over the same CSR, inheriting scheduler integration, spec-decode rollback, and reasoning-gated activation [sglang]. (iv) RTP-LLM consumes the artifact as configuration: the build job renders the trie to tree_decode_config prefix_dict JSON for the in-engine TreeLogitsProcessor — small/medium catalogs; 250M items needs the loader patch of §10.5 [rtp-llm]. (v) The host driver (§10.3) calls the gather directly on the scorer's logits, sliced to the level-local range before log_softmax.

Unconstrained mode and post-grounding

The trie is a config knob, not an architectural dependency. GRID's ablation found free-form beam search matches or beats constrained (Beauty R@10 0.0609 vs 0.0597) and is cheaper — the model learns valid SID structure from SFT alone [grid]; PLUM shipped YouTube production retrieval unconstrained with <5% invalid SIDs [plum]; OneRec likewise serves without a trie [onerec-v1]. Unconstrained output is grounded post-hoc by SidCatalogIndex with an exact-then-prefix fallback (full tuple, then 2-token and 1-token prefix buckets) — semantic-ids-llm's graceful degradation, exploiting the hierarchy so even a hallucinated leaf resolves to semantically nearby items [repo-semanticids]. Recommended defaults: constrained for RL rollouts (legality shapes reward, §9) and serving at 10⁸-item scale; unconstrained as the cheap baseline for eval ablations and small catalogs.

Trie artifact build job

The trie is built offline from the same item_id → SID assignment table that feeds SidCatalogIndex (§5): sort the deduplicated (N, L) code array, run build_static_index (vectorized NumPy — minutes at 250M rows), and write an .npz with the five tensors, per-level bases, and the tokenizer fingerprint. On request the builder also emits the RTP-LLM prefix_dict JSON rendering — a format that enumerates per-prefix candidate lists explicitly, so bounded to small/medium catalogs (§10.5). The artifact is versioned with, and invalidated by, the SID assignment snapshot — the re-tokenization cadence (R-RETOK, §15) governs trie refreshes, and serving swaps trie, catalog index, and cached prefix sets atomically as one unit. With Latte latent tokens, the builder prepends the latent level as depth 0 (M′ extra root branches); nothing else changes (D16).

10.3 Single-step scorer export (D13)

Decision D13 — host-side beam driver over an AOTI-exported single-step scorer with explicit KV-cache I/O

The serving artifact for the default Qwen3 backbone is a torch.export + AOTInductor package exposing one decode step with the KV cache as explicit inputs, plus a prefill graph; the beam loop (level masking, trie gather, top-k, cache reorder) runs host-side in the serving processor. This is empirically proven on this machine (torch 2.11.0+cu129, transformers 4.53.3, tiny Qwen3, bf16, sdpa): the step exports strictly and matches eager StaticCache decoding bit-exactly over 5 sequential steps; compiles and runs under AOTI (fp32-exact; bf16 within ≈5e-3–8e-3 fusion noise); survives host-side index_select beam reorder; and the explicit-I/O variant compiles with Dim("batch") so one artifact serves any users×beams product. R-EXPORT is thereby de-risked from design assumption to validated recipe with enumerated residuals. The legacy FX-trace + TorchScript path is a NO-GO for HF models.

Revision — serving-implementation analysis (2026-06-10)

The host-loop recipe is hereby scoped by its KV memory. Qwen3 GQA KV (8 kv-heads × head_dim 128, bf16) costs 4 KB per token per layer — 112 KB/token at 28 layers (0.6B/1.7B), 144 KB at 36 (4B/8B) — so an S=3000 context holds ~344 MB (1.7B) / ~442 MB (8B) of KV per sequence. The D13 design duplicates that per beam (repeat_interleave prefill, one StaticCache row per beam): W=32 costs ~11 GB — eval-friendly on an L20-48G — but W=512 costs ~176 GB: impossible on any single GPU tzrec targets. The naive host loop is therefore an eval/small-W recipe (W ≤ ~32–64 at 3K contexts), not a beam-512 production path; wide synchronous beams need the batched-beam decode module of §10.4 or the tzrec-gr-serving runtime (§10.6), whose context KV is shared across beams by construction. Two clarifications: the smoke ran no prefill (prefill-graph export remains residual obligation 3), and per-beam duplication is the D13 design, not something the smoke proved. None of this displaces the decision: the AOTI single-step scorer plus host beam driver remains the export-clean default, the only path for HSTU, and the path for encoder-decoder and lazy-decoder backbones until the corresponding gr_models families land (§10.6, P4/P5).

The proven recipe

The export rides transformers' own supported path. On the 4.53.x pin, TorchExportableModuleForDecoderOnlyLM (integrations/executorch.py, v4.53.3 L30-72) wraps the model with a StaticCache whose per-layer [batch, num_kv_heads, max_cache_len, head_dim] tensors are registered as non-persistent buffers, and swaps attention to sdpa_without_vmap (4.53's default mask construction uses torch.vmap, which does not trace under torch.export). forward(input_ids, cache_position) derives positions, RoPE, and the causal mask from cache_position inside the graph; the cache's in-place index_copy_ mutation is preserved through export and AOTI — PyTorch 2.11's AOTI suite includes a literal KV-cache buffer-mutation test (test/inductor/test_aot_inductor.py:2835) plus bf16 SDPA coverage [pytorch][transformers]. Vocab extension to 176,384 rows (§7) is invisible to the recipe — it only resizes embed_tokens and the untied lm_head.

The HF recipe fixes batch×beam at export time (cache buffers are module state). The preferred tzrec variant therefore moves the cache to explicit I/O, which was also smoke-tested end to end including dynamic batch:

def forward_step(
    input_ids:      torch.LongTensor,   # [N, 1]   N = B_users x W_beams  (Dim("batch"))
    cache_position: torch.LongTensor,   # [1]      absolute position = prefix_len + step
    *kv_caches:     torch.Tensor,       # 2L x [N, H_kv, S_max, D] bf16, mutated in place
) -> torch.Tensor:                      # logits [N, 1, vocab ~ 176384]

Explicit I/O keeps all state host-owned (no hidden tensors in the artifact), matches tzrec's "dense module + explicit inputs" export style, and enables one artifact across beam widths. The buffers variant remains a proven fallback: AOTICompiledModel.load_constants(..., user_managed=True) adopts host tensors by storage pointer, so the host can still reorder cache rows between calls. Prefill needs no separate cache input — a prefill graph (dynamic sequence Dim) or host-side eager prefill writes positions [0, prefix_len) of the same tensors, once per user then repeat_interleave(W, dim=0), or directly at N with repeated prompts.

Host beam loop and measured parity

Per SID step the host masks logits to the legal level range (plus the trie gather of §10.2), takes log_softmax, accumulates beam scores, runs a flat topk(W) over [B, W·K], and reorders every cache tensor with kv.copy_(kv.index_select(0, beam_idx)). For 3-token SIDs only steps 2 and 3 reorder — step 1 is pure expansion — so total reorder traffic is two copies of [B·W, H_kv, S_max, D] per layer per request: microscopic at W ≤ 32 — and at 512 the copies would be affordable, but the duplicated cache itself is not (revision callout above). The smoke matrix (tzrec120 env, A10; R-EXPORT follow-up study, 2026-06-10) adds to the callout's results: strict export at batch 8; max_abs_diff = 0 over 5 steps; bit-exact next-step parity after host reorder; AOTI with user_managed host-owned KV; explicit-I/O in-place kv mutation correct under both ep.module() and AOTI; one Dim("batch", 1, 512) artifact serving B ∈ {4, 8, 24}.

Gotcha 1 — AOTI silently corrupts on non-contiguous inputs

Feeding the AOTI artifact a strided slice such as generated[:, -1:] (shape (B,1), stride (S,1)) produced garbage logits with no error — on CPU and CUDA, even with zero transformer layers. AOTI reads the raw data pointer assuming example-input strides; ep.module() handles the same input fine, so the bug hides from eager-parity testing — and host beam loops naturally produce exactly such slices. The tzrec step-runner must .contiguous() every input and assert strides before each call.

Gotcha 2 — the AOTI .so requires GLIBCXX_3.4.30

dlopen of the compiled package failed on a system libstdc++ lacking GLIBCXX_3.4.30; fixed via LD_LIBRARY_PATH=$CONDA_PREFIX/lib. PAI serving images must carry a matching toolchain runtime on the dlopen path — this is a deployment-image checklist item, not a code change.

Residual proof obligations (tracked in §14): (1) a real-scale artifact — Qwen3-0.6B/1.7B with the 176,384-row vocab — measuring compile time, .pt2 size, load time, and per-step latency on PAI GPUs; (2) per-row cache_position [N] for mixed prefix lengths in one batch (only scalar positions were tested; left-padding within a user group sidesteps it); (3) prefill-graph export with a dynamic sequence dim (documented HF recipe, not exercised here); (4) re-running the matrix before any transformers-v5 pin move — real drift: key_cache lists become cache.layers[i].keys, a new cumulative_length_{i} buffer needs constant handling and per-generation reset, and v5's Cache.reorder_cache rebinds attributes via index_select, breaking exported-buffer aliasing (reorder stays copy_-based). Cache-internal access goes through a thin version shim from day one.

Fit with tzrec's existing export machinery

The AOTI step package is a sibling of what tzrec already produces: tzrec/acc/aot_utils.py:export_model_aot runs torch.export.export with a dynamic Dim("batch") (L177) and aoti_compile_and_package into aoti/aoti_model.pt2 (L226-235), with load_model_aot on the predict side — the scorer reuses the same compile config and loading path. The default FX route (export_model_normal, export_util.py:148: symbolic_trace + torch.jit.script) does not survive HF's config-dependent control flow and cache objects. The RTP export path is likewise out of scope for the LLM trunk — embedding-table-oriented, hard-rejecting DTensor parameters (export_util.py:484) — though its batch_size==1 convention ("when batch_size=1, we assume gr model", export_util.py:1010-1011) already anticipates GR-shaped serving.

10.4 KV-cache strategies

Four cache designs form a ladder from "ship now" to "optimize later"; the host-driver contract (scorer in, logits out, cache reorderable by row) is identical across them, so they swap without touching decode logic. The organizing question for wide beams: does a design share KV storage or KV bandwidth across one user's W beams? The geometry (revision, §10.3): 4 KB/token/layer bf16, so an S=3000 context is ~344 MB (1.7B) / ~442 MB (8B) per sequence.

Sharing storage vs sharing bandwidth

Storage sharing is what engines do, and it does not touch the decode wall. RTP-LLM forks beam KV copy-on-write and vLLM's prefix cache dedups blocks — memory is shared well; attention still runs per sequence, every beam reading the full context KV each step. At S=3000, W=512 that is 176 GB (28 layers) / 227 GB (36 layers) of KV reads per user per step — 204–262 ms/step at L20's 864 GB/s DRAM as the upper bound. L20's 96 MB L2 can soften this (one user's ~12 MB/layer working set fits when few users are in flight), so a per-sequence engine beam sits 3–30× off a purpose-built path depending on L2 residency — the DRAM-bound branch matching RTP-LLM's own ~0.8–1.2 s/request cost model (§10.5). vLLM's emulated beam pays a further, different tax: per-step request re-issue and tail prefill put it at roughly 10–50× a native fork-KV beam step [vllm].

Bandwidth sharing is the mechanism that removes the wall: beams in the query dimension. Batching the W beams as query rows against the user's shared context KV — q [B, H, W, D] over kv [B, H, S, D] — reads context KV ~once per user per layer per step (~0.34 GB/user/step at 28 layers: sub-ms). This is measured, not modeled: on A10 at S=3000, W=512, one layer, beams-in-query ran 0.105 ms vs 14.2 ms for per-sequence attention over duplicated KV — 135× at the kernel level. It is also why OneRec-V2 serves beam 512 in 36 ms [onerec-v2] — with the mandatory caveat that V2 is a 1B lazy decoder: context KV shared across beams by construction, ~94% of compute cut. A Qwen3-class backbone with batched-beam attention removes the KV wall but does not approach 36 ms — the GEMM/weight-read floor plus full prefill remains.

(a) HF StaticCache with host index_select reorder — the default. Per-layer [N, H_kv, S_max, D] tensors, reordered by index_select at beam pruning. Proven bit-exact above; trivial host logic; two reorder copies total for 3-token SIDs, far below the dense-projection floor recsys-examples measured. The binding constraint is capacity, not traffic: per-beam duplication caps it at W ≤ ~32–64 at 3K contexts on an L20-48G (revision, §10.3) — the eval and narrow-beam default, not a wide-beam path.

(b) The batched-beam decode module — bandwidth sharing on a Qwen3-class backbone (demand-gated). Decode steps only; prefill and weights untouched; parity-testable against the naive path at small W. W=512-synchronous on a Qwen3-class backbone has no production precedent — OneRec-V1 served at W=128 [onerec-v1], V2's 512 rides the lazy decoder, OpenOneRec uses 32 [openonerec], FORGE's 1200 is nearline [forge] — so this module is a demand-gated contingency, never a committed milestone (§10.7). Three implementations of the mechanism:

  • (c1) recsys-examples gr_decode_atten — the kernel-proven implementation. Exactly this split: K1 runs context attention with beams as the query tile (plus split-KV) over the shared prefill slab, K2 gathers per-beam step-major generated KV by ancestry indices without reordering, K3 merges via LSE combine [repo-recsysexamples]. GQA-generic (head_q % head_kv == 0; Qwen3's 16/8 in-envelope), bf16/fp16, W tested to 1024, L20 (SM89) dispatches the SM80 kernel class; ragged contexts need the 3-kernel backend. Sober scoping: it is not a wheel (CuTe-DSL JIT, cutlass-dsl 4.4/4.5 pins); it is softmax-only (the LSE combine), hence inapplicable to HSTU's silu attention — recsys-examples' own HSTU stack never uses it; the flat beam-KV layout needs uniform per-step widths; its nonstandard attention signature forfeits the plain-HF export recipe; and their own Nsight profile shows the entire decode-attention bucket at 1.593 ms of a 23.3 ms decode stage — non-attention kernels dominate. The kernel is one implementation of the mechanism, not the magic: K1 is plain flash attention at Sq=W (what the two-phase flex path reproduces), and K2's ancestry-gather avoids a reorder worth ≤4 MB/layer at 3-token SIDs — microseconds via torch.gather.
  • (c2) Two-phase flex_attention + exact LSE combine — the torch-native default. Phase 1: flex_attention of the W beam queries over the shared context KV, requesting the log-sum-exp via return_aux (return_lse is deprecated); phase 2: per-beam attention over the ≤3 generated tokens (tiny); merge by the standard flash-decoding LSE combine, which is exact (verified to 6e-16 in fp64), not approximate. An export + AOTI smoke of flex_attention-with-LSE passes at toy scale on torch 2.11 [pytorch]; the Qwen3-scale artifact (real head counts, 3K context, W=512) remains a D13-style proof obligation (§14).
  • (c3) Stride-0 expanded-KV SDPA. Expanding context K/V across beams as a zero-stride view works on all SDPA backends and stays a view under export+AOTI (no materialization) — but it shares storage, not bandwidth: beams-in-batch still reads KV per batch row and measured ~100× slower than beams-in-query. A memory-saving fallback only.

(c) delta_hstu_mha jagged cache — the in-house backbone's native path. For InhouseHstuGRBackbone (D5), tzrec already ships the prefill/decode cache machinery: STULayer holds per-layer k_cache/v_cache with jagged offsets, prefill populates them via kv_caching_lengths, and cached_forward runs delta_hstu_mha over cache+delta (tzrec/ops/hstu_attention.py:199-267, tzrec/modules/gr/stu.py:564-641). Constraints: delta size uniform per row (beam steps append one token per row — fits), CUTLASS falls back to Triton on the delta path, and the cache lives as Python module attributes — fine because the loop is host-side; beam forks reorder by re-indexing k_cache/v_cache and offsets, mirroring (a).

(d) recsys_kvcache_manager for cross-request session reuse. For high-QPS deployments where a user's history prefix recurs across requests, NVIDIA's standalone paged GPU KV cache with host/FlexKV tiered offload keys cache identity by user_id — histories never share prefixes across users, so engine-style token-prefix matching buys nothing while user-keyed reuse skips re-prefilling thousands of history tokens. Measured on the HSTU inference example: 1.3–2.6× end-to-end with CUDA graphs [repo-recsysexamples]; the paged attention entry it feeds belongs to the same fbgemm_gpu_hstu wheel lineage tzrec already ships. This slots under topology (a) of §10.7 as a phase-2 optimization for the HSTU and lazy-decoder backbones.

10.5 Serving engines and runtimes

Three OSS engines and one purpose-built runtime were studied at source level: vLLM at HEAD fe1d923af (v0.22.1+ dev), SGLang at b40f365732 (v0.5.12.post1), RTP-LLM at OSS v0.2.0 b0c09a344, and recsys-examples at 2550a9c4 (v26.05+; re-verified unchanged at 5dc46a217d, 2026-06-30 — zero drift in either serving tree) [vllm][sglang][rtp-llm][repo-recsysexamples]. The last is a re-study with a methodological lesson: the earlier snapshot a4d3b01 (v26.04) predated PR #414 (2026-06-02), which added examples/sid-gr-inference and changes the verdict on the repo. None displaces the D13 scorer — topology (a) of §10.7 remains the export-clean default, and non-engine backbones route through the tzrec-gr-serving runtime of §10.6 (HSTU through tzrec's own stack) — but each earns a distinct role, and all four consume the D12 SidTrie artifact.

vLLM — eval harness, emulated beams, and a rescoring channel

The seven claims established at v0.12 hold at HEAD [vllm], with refinements. BeamSearchParams still carries no constraint fields and no vllm#21546-class fix ever landed; logprobs_mode landed in v0.10.0; grammar bitmasks apply before the sampler, so structured-output masks are visible even in default raw_logprobs — only sampler-stage processors need processed_logprobs. Two hard conflicts: custom logits processors are rejected under speculative decoding (structured outputs do compose), and Model Runner V2, now default for Qwen3 dense, silently falls back to the V1 runner under custom LPs or prompt embeds. tzrec uses vLLM for offline eval at widths 10–50, OpenOneRec-parity checks, and SID-shortlist rescoring: the generative scoring API with SamplingParams.logprob_token_ids (≤128 ids) scores pre-retrieved candidates teacher-forced — one prefill per user, no decode loop. Constrained beams always run through tzrec's custom client loop (~200 lines) carrying per-step allowed_token_ids or extra_args, which also unlocks FORGE-style variable widths; prompt embeds now hash into prefix-cache keys but remain blocked in the beam API. Operational keys: --max-logprobs ≥ 2W, --max-num-seqs ≥ 512, --logprobs-mode processed_logprobs, --logits-processors. R-VLLM-PIN is re-anchored at v0.22.1+ (§15).

SGLang — the RL-rollout engine, not a beam engine

SGLang still has no beam search of any kind — a repo-wide grep at HEAD returns zero hits, not even a vestigial field [sglang] — so it stays off the beam-faithful eval and production paths. For GRPO sampling rollouts (§9), however, it is arguably the best-equipped engine. The per-request custom logit processor walks the full 250M-item trie with zero server code; the registered sidtrie grammar backend is the cleaner server-side route (both in §10.2); per-level itemic masks alternatively fit a ~0.4 MB EBNF — provided the itemic tokens are registered as non-special added tokens, or xgrammar cannot emit them. ReasonerGrammarBackend keeps any grammar dormant inside <think>…</think> and arms it after, with a per-request thinking_budget — CoT-then-constrained-SID in one request, the exact two-stage rollout shape. The radix cache at page_size=1 lets N CoT samples pay the ~3K-token prefill once; returned logprobs are post-constraint and renormalized (the semantics GBPO logging wants); the verl weight-update surface (update_weights_from_*, memory release/resume, --rl-on-policy-target fsdp bit-exact inference) is the richest of the three. Caveats: input_embeds forces --disable-radix-cache server-wide; the enc-dec whitelist is three audio/vision models; emulated beam-512 monopolizes the scheduler — a diagnostic, not a topology.

RTP-LLM — engine-native beams plus an OSS generative-rec constraint stack

The deep study refutes this RFC's earlier claim that the OSS build has no logits-processor or constrained-decoding hook [rtp-llm]. OSS RTP-LLM ships a C++ processor framework inside Sampler::forward, before beam top-k: TreeLogitsProcessor does catalog-trie constrained decoding configured purely by --tree_decode_config (a JSON prefix_dict of per-prefix candidate lists), per-beam DFA state forked and reordered on every beam transition; RecommendationLogitsProcessor (landed 2026-04-29) is purpose-built for SID rec — combo_token_size treats every k consecutive tokens as one item, banned_combo_token_ids/auto_parse_banned_combo filter already-exposed items parsed from the prompt, items dedup across beam steps, a Qwen3 think-prelude is skipped via DFA. This is the AL-GR/FORGE production path, in OSS [forge]. The native beam path is stronger than assumed: variable_num_beams gives per-step widths (step 1 always width 1; kernels padded to width 4096); beam KV forks block-level copy-on-write, so a 3000-token prompt is held once; masks apply before log_softmax, so cum_log_probs renormalize over the allowed set. Limits, stated plainly: speculative decoding does not compose with beam requests; the beam path is deterministic (per-beam temperature ignored — RL rollout diversity must come from the CoT sampling stage); and three patch surfaces gate large-scale adoption — (1) generateVocabMask builds a dense CPU [B, V] mask and H2D-copies it every step (~90 MB/step at beam 512; ~100–200 LoC sparse-write fix), (2) loadPrefixDict (JSON plus a string-keyed hash map) does not scale to 250M items — compact binary format plus a reload RPC, (3) single-request CoT-then-beam is inexpressible because the width schedule indexes absolute output position — patch numBeams to index from the think-end DFA position, or run two requests over the 64-token-block prefix cache (REUSE_CACHE=1, 3-tier). Cost model: ~0.8–1.2 s/request at beam 512 over 3000-token contexts, bf16 on L20 — the per-sequence KV wall of §10.4; fp8 KV halves it; FORGE's 200 ms envelope corresponds to ~500-token contexts. The trie build job (§10.2) also renders the SidTrie as prefix_dict JSON for small/medium catalogs.

recsys-examples sid-gr-inference — the purpose-built SID-GR runtime, adopted as seed

PR #414 added examples/sid-gr-inference: a standalone, pip-installable, pure-python SID-GR serving framework whose sole dependency is huggingface-hub [repo-recsysexamples]. Anatomy in one line: a minimal Qwen3 nn.Module over stock HF safetensors (tzrec's 176,384-row vocab flows through as config), dense ContextKV/BeamKV lease pools — exactly the beams-share-context split the bandwidth analysis of §10.4 demands — a continuous-batching scheduler grouping decode ticks by (step, W, next_W, ctx_len), decode/prefill CUDA graphs, an SGLang-compatible /generate server, TokenTrie + JSONL catalog with /catalog hot reload/rollback, and dynamic beam-width policies (fixed / scheduled / score-margin — the FORGE shape). Benchmarked against SGLang's beam-search PR branch (Qwen3-1.7B, ctx 1000/5000, W=256, 3 tokens): 1.90–2.27× offline, 1.85× online, top1 exact match 1.00. Caveats: a self-declared single-node, single-GPU alpha, validated on H100/bf16 only; the trie step-mask is a batch-1 per-beam python walk; the 176K-vocab topk is unoptimized; no CI; no torch.export/AOTI story — the maturity gap is tracked as R-SIDGR (registered in §15). Adoption is no longer a posture but a plan: §10.6 specifies the tzrec-gr-serving fork, its verified extension seams, and the per-backbone workstreams.

10.6 The tzrec GR serving runtime (BackboneAdapter over sid-gr-inference)

Scope and packaging — tzrec-gr-serving, a fork of sid-gr-inference behind the BackboneAdapter contract

The tzrec GR serving runtime is a fork of sid-gr-inference, shipped as a separate deliverable — tzrec-gr-serving, consuming tzrec export artifacts — and serving all softmax-attention GR backbones: Qwen3/Qwen2.5 token-only, the lazy decoder, and encoder-decoder. This retires the earlier "HF-Qwen3 family" scoping: the seam audit below shows the scheduler, KV pools, CUDA-graph runners, and server are model-agnostic, so the fork keeps ~13K lines of runtime while the engine-invariant core of §10.1 stays tzrec code in front of and behind it. HSTU is the sole exception — silu attention falls outside seam S3 — and stays on tzrec's own stack (topology (a), §10.7; delta_hstu_mha cache, §10.4). tzrec/acc/gr/ keeps the library pieces both paths consume: the AOTI scorer, the trie builder, SidCatalogIndex, and artifact versioning. examples/sid_gr is a kernel+concept donor only, never a checkpoint source (below).

Why the base holds: the verified extension seams

The fork bet rests on source-verified seams, not optimism. The model contract is duck-typed — the engine holds model: Any, and exactly one Qwen3 import exists outside gr_models (an optional JIT beam-KV write kernel): a new backbone is a new gr_models family plus a bring-up script, zero engine edits. "Prefill" is a model method, not a scheduler concept — the scheduler flips a stage flag and the executor calls the model's prefill — so a model that fills ContextKV by projection, with no causal prefill at all, is legal. allocate_context_kv is model-owned, making per-family pool schemas a model-side declaration. step is passed into the decode step, so position-specific heads are legal model-side. Beam widening is mechanically supported (next_W ≤ W·V; the scheduler already groups by (step, W, next_W, ctx_len); config defaults cap width 128 / 3 steps; the kernel is tested to W=1024). CUDA-graph capture wraps the whole decode step model-agnostically. RoPE is model-side and KV is stored post-rope, so non-RoPE architectures simply skip it. The CuTe decode kernel is optional per layer — SDPA fallback costs ≤1.6 ms per their own Nsight profile — so the kernel bounds performance, never feasibility. And decode control is beam-faithful and extensible: trie masks renormalize beam scores post-mask, and /generate ignores unknown JSON fields.

Five seams are genuinely fixed and bound the extension work:

  • S1 — token-ids-only ingress. /generate accepts input_ids only, and ctx_len := input_ids.shape[-1] drives pool leases and batch grouping. The one runtime extension both OneRecs force.
  • S2 — causal-prefill assumption in the stock prefill backends and prefill CUDA graph; bypassable model-side, since the model owns its prefill method.
  • S3 — softmax-only single-merged-softmax decode kernel, no attention-bias argument: cannot express enc-dec two-softmax sublayers, T5 relative bias, or HSTU silu. Kept optional; SDPA/flex fallback (§10.4 c2) for the enc-dec families.
  • S4 — batch-1 CPU python trie — replaced by tzrec's tensorized CSR gather (§10.2) regardless (R-SIDGR).
  • S5 — kernel W validation ceiling 1024: FORGE's 1200 needs a validation run (or a torch-topk path).

Per-target suitability

Two rows deserve emphasis. OneRec-V2 is the best architectural fit: its ContextProcessor emits per-layer K/V directly — it is the ContextKV abstraction minus the prefill — and with L_kv=1, K=V, and MQA the entire per-request context state is one [B, 3000, G_kv, d_head] slab, ~0.77 MB/request at S=3000 [onerec-v2]. It goes first in the extension sequence: the best fit and the production W=512 architecture. And the sid_gr row is a cautionary tale, not a work item: examples/sid_gr and examples/sid-gr-inference are two disjoint stacks in one repo — the table row's KJT/NoPE/per-rank training side against an HF-dirs-only loader with hardcoded Qwen3 anatomy (mandatory qk-norm, RoPE) and input_ids-only /generate — with no converter anywhere in the repo. The train-to-serve checkpoint contract must be designed, not assumed; WS4's export tool is the structural answer.

Workstreams WS0–WS6 and sequencing

  • WS0 — fork governance. Fork under tzrec test conventions; pin the upstream commit; send generic fixes upstream.
  • WS1 — the existing R-SIDGR items. CSR trie gather (S4), context-length bucketing, L20 validation, EAS packaging, the prompt frontend (token-id assembly shared with training).
  • WS2 — formalize BackboneAdapter, an ABC over the duck protocol, kept upstream-compatible: encode_context (causal prefill | encoder + per-decoder-layer cross-KV precompute | ContextProcessor projection), step-aware decode_step, allocate_context_kv, and head/vocab-layout metadata — flat 176,384 vs level-local 8192, the local layout matching the CSR trie's per-level local ids (§10.2).
  • WS3 — S1 embedding ingress. First: request-side fp16 context tensors (~0.8–1 MB/request), with admission/memory/grouping keyed on embedding count and session ContextKV reuse keyed user_id+feature-version instead of token hash. Phase 2: in-process tzrec EmbeddingGroup/fg for co-located tables.
  • WS4 — model zoo + checkpoint contract. gr_models families: qwen2_5 (QKV bias=True, no qk-norm; ~150–300 LOC), lazy_decoder (slab ContextKV, 3 position-specific heads; ~600–1000 LOC), enc_dec (bespoke SDPA attention, two softmaxes per layer, optional T5 rel-bias, MoE FFN; ~800–1200 LOC). Per family, a tzrec DCP→safetensors manifest export tool emits config.json + weights + tokenizer/SID manifest — the train-to-serve contract that prevents a sid_gr-style rift by construction.
  • WS5 — decode control. Level-offset mask processor (~50 LOC — a static range mask, not the trie); tensorized σ-bucket SID→item expansion (~100 LOC) in SidCatalogIndex grounding; beam caps raised plus the S5 W=1200 validation; post-mask per-token gen_prob emission (below).
  • WS6 — perf/ops. Per-family CUDA-graph validation (including SDPA-fallback graphs), the L20 matrix per family, session ContextKV reuse.

Sequencing. P2 delivers WS0–WS2 plus the FORGE-dec qwen2_5 adapter as the cheap contract proof (~0.5–1 wk for the adapter itself): a second family is what proves BackboneAdapter is an abstraction, not a Qwen3 rename. P4/P5 deliver WS3 plus lazy_decoder first (OneRec-V2: best fit and the production W=512 architecture; ~2–3 wk including ingress), then enc_dec (OneRec-V1; ~3–5 wk); T5 parity is optional.

gen_prob logging. The runtime's response per-token logprobs are currently pre-mask (true unconstrained probabilities), while beam scores renormalize post-mask. gen_prob_logged (§6) must reflect the masked distribution actually decoded from (the logging contract of §10.7; GBPO, §9), so WS5 adds a post-mask per-token variant — a small change, since the renormalized scores already exist at selection time.

10.7 Serving topologies

(a) The tzrec GR serving runtime — the engine-free default, now a three-rung ladder. The foundation is unchanged: load the AOTI step package (and prefill graph), the trie .npz, and the SidCatalogIndex snapshot; run the beam loop, trie gather, and grounding in the serving process; per-beam outputs ride the TARGET_REPEAT_INTERLEAVE_KEY flatten/regroup path into ListArray columns. It remains the only topology requiring no engine patches and the only one serving all backbones, including encoder-decoder and HSTU (which no OSS LLM engine speaks). M-a: a python EAS custom container running tzrec-gr-serving (§10.6), covering every softmax-attention backbone as its gr_models families land — gang/bucketed batching over the fixed 3–4-token decode, with honest ops caveats: the hot path holds python between kernels (one straggler stall inflates a gang batch's p99); per-replica assets are multi-GB (~1–1.5 GB CSR trie plus ~2–4 GB grounding map at 250M items, in tension with fractional-GPU pooling); EAS python containers lack the warmup/rolling-update/health conventions of the C++ processors. M-b: a C++ TorchRecProcessor extension — the AOTI runner plus a ~200-line libtorch beam loop — demand-gated on measured M-a QPS shortfall. M-c: the batched-beam decode module (§10.4) — demand-gated on a demonstrated W=512-synchronous product need, which no Qwen3-class production system has yet exhibited.

(b) RTP-LLM — engine-native beams with an OSS constraint stack: the FORGE production shape. Per §10.5, everything FORGE's serving contract needs is config-only: variable_num_beams=[300,600,1200] reproduces the dynamic schedule, tree_decode_config grounds beams in the catalog trie, and RecommendationLogitsProcessor supplies the exposure filter and cross-step item dedup [rtp-llm][forge][repo-alsid]. At small/medium catalog scale this is the lowest-code path to production-width beams in the nearline/short-context envelope — FORGE's actual deployment shape: tzrec exports the SidTrie as prefix_dict JSON, the engine returns per-beam cum_log_probs renormalized over the allowed set, and SidCatalogIndex grounds the result. At 250M items the two named patches gate adoption, and 3000-token synchronous contexts pay the per-sequence KV-read wall (§10.4; fp8 KV or shorter serialized contexts buy back the 200 ms envelope). Recommended where Taobao-side infra alignment matters or production beam widths rule out emulation; the engine is PAI-ecosystem native (iGraph, vipserver, kmonitor in-tree).

(c) vLLM serving — eval harness and emulated beams for token-only Qwen3 models. vLLM's beam search remains a pure-Python emulation at the API layer: one 1-token request per live beam per step — 2W+1 requests per 3-token decode — with prefix caching bounding per-beam recompute to a ≤16-token tail; block-aligning the prompt so SID tokens start a fresh block cuts step cost ~4× [vllm]. Constrained beams run through tzrec's custom loop plus the engine-level SidConstrainedLogitsProcessor with prompt-tail arming and processed_logprobs (§10.2). At width 512 on L20: ~0.2–1.4 s/step for Qwen3-8B (~1.5–3.5 s/request) — fine for offline eval and OpenOneRec-parity at W ≤ 50, marginal for synchronous production at 8B; 0.6–1.7B models drop to ~30–100 ms/step and are production-plausible. The cautions of §10.5 apply (V2-runner fallback; the spec-decode/custom-LP conflict; R-VLLM-PIN API drift, §15). The OpenAI-endpoint + custom-LP + processed_logprobs combination is source-verified but not yet GPU-smoke-tested — it gets a validation run before this topology is declared supported (§14).

SGLang's role — RL rollouts and offline CoT, never beams. With no beam path SGLang cannot serve beam-faithful retrieval, and emulation monopolizes its scheduler (§10.5). Its place is the training side — the recommended GRPO rollout engine (§9) — and the natural harness for offline CoT-generation jobs feeding distillation and nearline reasoning corpora.

(d) Nearline patterns — how reasoning-class models actually ship. No published system serves LLM reasoning synchronously; three deployed patterns cover the spectrum. FORGE async-trigger cache: generation fires on a user event (item-detail click), the RTP-LLM-served path writes 1200 candidates to a cache table within 200 ms (the cache write is application-side), and the existing ranking stage consumes the cache — generative retrieval as a feeder, off the latency path [forge]; the engine-side half is visible in OSS: RecommendationLogitsProcessor bans already-exposed items at generation time [rtp-llm]. OneRec-Think Think-Ahead: an offline reasoning job beam-searches the first 2 SID levels of T reasoning paths per user into a per-user prefix cache; online, a small realtime OneRec decodes the final level constrained to the cached prefixes [onerec-think] — "constrain to cached prefix set" becomes an alternative trie root, a small CSR generalization. OneReason fast–slow thinking: nearline hierarchical decoding (beam at the top level, greedy below) feeds a Redis candidate pool with realtime-model fallback, while "Thinking Token" representation distillation (supervising the realtime decoder's BOS hidden state) carries reasoning value into the synchronous path [onereason]. Think-Ahead prefix caches reuse the trie machinery; the distillation bridge is a §14 roadmap item.

First production for token-only Qwen3: the nearline envelope first; the substrate settled by a bake-off. The first production exposure goes behind the FORGE-style async cache table regardless of substrate — it removes the latency-wall problem class and is production-proven at Taobao scale [forge]. Underneath, the substrate is a genuine tie: (b) RTP-LLM — ops-hardened, PAI-native, per-sequence beams, two named patches at 250M catalogs — versus (a)/M-a running tzrec-gr-serving (§10.6) — beam/trie fidelity, dynamic widths, torch-native ownership, template parity by construction, upstream alpha maturity (R-SIDGR). An L20 bake-off settles it cheaply; both expose SGLang-compatible APIs. Capacity, not latency, is the nearline sizing question: 100M candidate refreshes/day at ~0.2 s GPU each ≈ 230 sustained L20s — active-user tiering and trigger policy, not engine choice, are the sizing levers. One bound holds on every path: 8B synchronous serving is prefill-bound (~0.4–0.8 s at 3K contexts on one L20) and needs TP — a flag for the engines, a new sharded-serving subsystem for the native runtime. In-house backbones are unaffected: they stay tzrec-native on (a), borrowing at most recsys_kvcache_manager and the paged fbgemm_gpu_hstu attention arguments — the same wheel lineage tzrec already ships (§10.4).

Exposure-time generation-probability logging. Every topology also owns the producing side of a training contract: GBPO (§9) populates pi_old from the family-4 column gen_prob_logged : ARRAY<DOUBLE> (§6) — the per-token generation probabilities of the exposed item's SID tokens, logged at exposure time. The values are computed in every topology and need only be carried through: the host driver (a) attaches per-token log_softmax scores from beam accumulation to each (sid_tuple, score) pair handed to SidCatalogIndex and emits them per grounded candidate (collision-bucket items expanded from one SID share that SID's token probabilities, D4) — under M-a via the post-mask emission of WS5 (§10.6); engines (b)/(c) return per-token logprobs natively — post-mask by construction in RTP-LLM and SGLang (masking precedes the softmax; no pre-constraint channel), via processed_logprobs in vLLM — so the logged value always reflects the masked distribution actually decoded from (§10.5); nearline patterns (d) persist them alongside candidates in the cache table for the downstream exposure×feedback join. The fallback is contractual: non-generative impressions carry no column and the trainer falls back to ratio = 1 — the regime GBPO is designed to keep stable — so logging is mandatory only on generative impressions, never a backfill of the full exposure log.
