4. Target architecture overview

This section fixes the end-to-end shape of the system: the canonical pipeline from offline SID tokenization through two-world training, RL, and constrained-decode serving; the rationale for keeping sparse parameters in TorchRec while moving the dense LLM to FSDP2/Megatron-core; the package layout and the model-class taxonomy; and the artifact contracts that connect the lifecycle stages. It closes with an explicit accounting of what is retained from the prior internal spec (experiments/llm4rec/00_master_spec.md v1.0) and what this RFC revises. Detailed designs live in the owning sections; everything here is the binding skeleton they hang on.

4.1 Canonical system diagram

The architecture mirrors the OpenOneRec blueprint end to end — build SIDs, extend the LLM vocabulary with itemic tokens, co-pretrain/SFT, RL, constrained-decode serving [openonerec] — realized on tzrec's existing orchestration seams (§3). 4.2 Two-world training: why the split is shaped this way

Sparse stays TorchRec. The sparse world's defining workloads have no dense-world equivalent: a 250M-row item table only fits via dynamicemb HKV hash tables with ROW_WISE sharding (dynamicemb_util.py:216), per-token SID and id lookups need EmbeddingCollection semantics with all-to-all exchange, bounded-HBM exact-id mapping needs ZCH, and the fused in-backward optimizer is what makes embedding training throughput-viable. tzrec already owns all of this (§3) [torchrec]. A strategic simplification keeps the burden tractable: after SID assignment, day-to-day LLM training is dominated by dense compute — the giant item tables are stressed mainly offline during tokenization and in HSTU/encoder-style context paths, while the itemic vocabulary itself is a small 24,576-row block living inside the LLM's embedding matrix.

Dense leaves the DMP/DDP world. Today DMP wraps the entire TrainWrapper and DDP-replicates every non-sharded parameter (dist_util.py:164-195). For a 1.7B–8B trunk that means a full parameter copy, full gradients, and full AdamW state per rank — DDP shards nothing. FSDP2's fully_shard shards parameters, gradients, and optimizer state as DTensors over a DeviceMesh derived from the same WORLD process group, composes with TorchRec's DTensor-aware DCP checkpointing, and is exactly how OpenOneRec trains every stage of its 1.7B/8B models [openonerec][pytorch]. Megatron-core 0.18 remains the scale-out backend where FSDP2 runs out of road: expert parallelism for the OneRec MoE arms, tensor-parallel vocab embedding and vocab_parallel_cross_entropy for the 176K-row vocabulary at 8B+, and alignment with verl's megatron engine [megatron][verl].

Revision pointer (D8). This inverts the prior spec's decision 3(f) (Megatron-core default, FSDP2 fallback); the canonical rationale lives in the revision callout in §11.1, and §11 owns the full decision.

The bridge and its risks. The two worlds meet at exactly one activation boundary: TorchRec produces all-to-all'd context embeddings (user towers, HSTU-encoded histories, SID-history embeddings), and the dense backbone consumes them as inputs_embeds (HF path) or decoder_input (mcore path), overwriting designated feature slots without disturbing rope position ids — R-INJECT. Optimizer trees stay disjoint, partitioned by in_backward_optimizer_filter, but step inside one CombinedOptimizer; the ordering of Megatron's finalize_model_grads versus the fused in-backward sparse step is R-GRAD. DMP must shard the sparse subtree without DDP-wrapping the FSDP/TP-managed dense parameters — R-DMP — and the TorchRec planner must be told about dense memory it cannot see — R-PLANNER. All four are tracked with proof obligations in §15; the concrete mechanisms live in §11.

4.3 Package layout

The package contract is retained verbatim from the prior spec (§4.1 there), with the SID tokenizer reshaped around the BaseSidModel/ResidualQuantizer base classes that landed on master in PR #538:

  • tzrec/sid/ — quantizer backends (k-means and VQ subclasses of ResidualQuantizer), pluggable CollisionResolvers, training-free quality metrics (Gini, utilization, Embedding HitRate), SidCatalogIndex and the trie data structure (§5).
  • tzrec/models/generative/ — BaseGenerativeModel plus the five model arms (§7); concrete SID models SidRqkmeans/SidRqvae register alongside. Because auto_import scans tzrec/models/ non-recursively (load_class.py:53-99), the subpackage is re-exported through a one-line shim so registry side effects fire.
  • tzrec/modules/llm/ — backbone wrappers (Qwen3GRBackbone, LazyDecoderGRBackbone, EncoderDecoderGRBackbone, InhouseHstuGRBackbone), SidPredictionHead, vocab-extension tool, HF↔mcore weight bridge (§7).
  • tzrec/rl/ — verl-pluggable policy-loss/advantage estimators (ECPO, GBPO, on-policy distillation), composable rewards, the SID-constrained agent loop (§9).
  • tzrec/distributed/ — DenseParallelizer with Fsdp2Parallelizer (default) and MegatronParallelizer backends, DeviceMesh construction from the existing WORLD (§11).
  • tzrec/acc/gr/ — SidConstrainedLogitsProcessor (HF and vLLM v1 variants), CSR trie build/load, beam decoders, the host-side decode driver, AOTI export helpers (§10).
  • tzrec/datasets/gr/ — instruction/prompt builders and the 32K SequencePacker (§6).
  • tzrec/features/semantic_id_feature.py — SemanticIdFeature/SequenceSemanticIdFeature owning their _fg_json (§6).
  • tzrec/gr/ — seven CLI verbs (train_tokenizer | assign_sid | sft | rl | eval | export | serve), thin argparse wrappers (§12).
  • tzrec/main_gr.py — orchestration sibling of main.py: same config/feature/dataloader/checkpoint plumbing, plus the dense-parallelizer hook, the GR train pipeline, and the decode driver that main.py has no seam for.
  • tzrec/protos/ — four new proto files; §12 is the source of truth.

4.4 Model classes and how the literature maps onto them

Five generative model classes and two SID tokenizer-training classes enter the oneof model registry; the proto message name equals the Python class name, so registration needs no main.py edits (§3). MiniOneRec is deliberately not a class: it is a config preset over LCRec plus GRPO [minionerec].

The newly surveyed literature maps onto this taxonomy without new classes. TIGER and GRID configure the encoder-decoder shape at small scale [tiger][grid]; PLUM's lessons (co-pretraining dominates base-model choice, reward-weighted SFT sampling) land as stage and sampling knobs in D6/D14 [plum]; OneRec-Think and OneReason contribute RL recipes (two-stage rollouts, RFT, MOPD, stage-wise clipping) and the Think-Ahead serving pattern to D10/D13 [onerec-think][onereason]; Latte's latent tokens and PIT's tokenizer–recommender co-evolution are scoped extensions under D16 [latte][pit]; RPG-style parallel decode and LLaDA-Rec diffusion are future work that D5's level-agnostic interfaces must not preclude (D15) [rpg][llada-rec].

4.5 Pipeline lifecycle and artifact hand-offs

Every stage is a separate launch with a file-level artifact contract; no stage holds another's in-process state. Checkpoints chain through the existing --fine_tune_checkpoint mechanism, and decode-time assets (catalog, trie, extended tokenizer) ship through --additional_export_config into model_acc.json alongside the exported model, following the TDM precedent of separately serialized artifacts (§3).

Two properties of this chain are deliberate. First, the SID asset table is the single integration seam between the tokenizer subsystem and everything downstream — training joins, vocabulary id arithmetic, trie construction, and HR@K expansion all read the same table, so a re-tokenization (cadence risk R-RETOK) is a table swap plus dependent-artifact rebuild, not a code change. Second, every numbered stage is restartable from its inputs: the pipeline has no hidden coupling that would force a full rerun when, say, only the RL stage changes.
