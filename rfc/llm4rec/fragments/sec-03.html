3. TorchEasyRec today: baseline and gap analysis

This section inventories the tzrec baseline at master commit 7886e4c from the perspective of the generative-recommendation requirements distilled in §2: what the orchestration, data, and model layers already provide, which seams the design in §4 deliberately exploits, and which baseline assumptions break and must be engineered around. Every claim below is grounded in a source location at this commit. The headline finding is that tzrec already ships most of the sparse-world machinery (DMP sharding, dynamicemb, jagged data plumbing, a host-side beam-search precedent, and the PR #538 SID base classes), while the dense LLM world — HF/Megatron backbones, FSDP2, packed-sequence training, RL, and autoregressive serving — is entirely absent and is exactly what this RFC adds.

3.1 Orchestration: one loop, one process group, one DMP wrap

All CLIs (tzrec/train_eval.py, eval.py, export.py, predict.py) are thin argparse wrappers over tzrec/main.py (1488 lines), which owns the entire lifecycle. The training entry train_and_evaluate (main.py:543-788) executes a fixed order of operations:

load EasyRecConfig (text proto)            main.py:568-580
  -> init_process_group: ONE global WORLD PG, device=cuda:{LOCAL_RANK}   dist_util.py:57-75
  -> create_features (BaseFeature registry)                              main.py:94-112
  -> create_dataloader: IterableDataset yields full Batch objects        dataset.py:751-840
  -> _create_model: oneof-name -> BaseModel.create_class -> TrainWrapper main.py:127-159
  -> apply_optimizer_in_backward over model.sparse_parameters()          main.py:661-668
  -> planner.collective_plan(model, sharders, dist.GroupMember.WORLD)    main.py:682
  -> DistributedModelParallel(module=TrainWrapper, ...)  [whole model]   main.py:686-691
  -> TZRecOptimizer( CombinedOptimizer([fused_sparse, dense, *parts]) )  main.py:693-755
  -> per epoch: pipeline = create_train_pipeline(model, optimizer)       main.py:430-434
  -> step: losses, predictions, batch = pipeline.progress(iterator)
  -> ckpt_manager.maybe_save(...) ; on save -> run_eval(step, epoch)     main.py:490-497

Three structural facts here shape the whole design. First, the process-group topology is flat: init_process_group creates only the global WORLD with backend nccl and never builds TP/PP/EP sub-groups; the planner receives dist.GroupMember.WORLD directly (main.py:682). Any dense-parallel backend must layer its meshes on top of this WORLD after init, which is precisely the contract D8 (§11) adopts. Second, sparse optimization is fused into backward: BaseModel.sparse_parameters() (models/model.py:153-192) BFS-collects every EmbeddingBagCollectionInterface/EmbeddingCollectionInterface parameter and apply_optimizer_in_backward attaches the fused optimizer before planning; per-table trainable=false features are frozen by attaching SGD with lr=0.0 while keeping the fused kernel (main.py:666-668) — an existing precedent for parameter-group-level freezing relevant to D7. Third, one DistributedModelParallel wraps the entire TrainWrapper, and with init_data_parallel=True every non-sharded parameter is DDP-replicated by DMP's default DataParallelWrapper (dist_util.py:164-195) [torchrec]. tzrec's DMP wrapper does forward a custom data_parallel_wrapper argument (dist_util.py:172), which is the hook the FSDP2 integration uses (§11).

The optimizer tree is already a three-way composite, which is the single most important reuse point for a second (dense LLM) optimizer world:

TZRecOptimizer (grad_scaler + gradient_accumulation)        optim/optimizer.py:26-68
 +- [GradientClippingOptimizer]                             main.py:723-741
     +- CombinedOptimizer([
          model.fused_optimizer,   # sparse, in-backward; .step() is a no-op
          dense_optimizer,         # KeyedOptimizerWrapper(remaining params)
          *part_optimizers,        # one per regex_pattern over DMP FQNs
        ])                                                  main.py:693-755

Parameters are split by in_backward_optimizer_filter(model.named_parameters()) and then by re.fullmatch per dense_optimizer.part_optimizers[i].regex_pattern (optimizer_builder.py:219-237); each node gets its own BaseLR schedule, and base_lr==0 freezes a group outright (optim/lr_scheduler.py:34-41). Anything that speaks the KeyedOptimizer protocol can be appended to the CombinedOptimizer list. Two caveats matter for the LLM loop: gradient accumulation only gates the dense side (in-backward sparse optimizers step on every backward, tzrec-core notes §4), and the entire fwd+bwd+step sequence is owned by pipeline.progress with no hook between backward and step — the origin of R-GRAD (§15).

Checkpointing

Checkpoints are torch DCP directories model.ckpt-{step}/{model, optimizer, plan, dataloader_state.json, dynamicemb?} written by the free functions save_model (checkpoint_util.py:873-913) and save_dataloader_state (checkpoint_util.py:922); saves are synchronous, with only pruning offloaded to a daemon thread that keeps the newest keep_checkpoint_max plus the best checkpoint by eval metric (checkpoint_util.py:267-588). Save triggers include step/epoch intervals and Kafka event-time quorum reconciled across ranks over a side gloo group (checkpoint_util.py:392-456). Restore goes through PartialLoadPlanner (checkpoint_util.py:50-148), which (a) skips keys missing from checkpoint metadata with a warning — partial restore for fine-tuning works today, (b) remaps names via the fine_tune_ckpt_param_map file ({new}\t{old} lines), and (c) is DTensor-aware because DCP load itself is. dynamicemb already demonstrates the “foreign parameter world” pattern: a side-by-side custom dump (DynamicEmbDump) keyed by module discovery, living next to model/ and optimizer/. This is exactly how the dense LLM checkpoint root attaches in §11.

Export and predict paths

Export (main.py:891-1007, utils/export_util.py) runs on rank 0 only and forces WORLD_SIZE=1 (export_util.py:159-168), restoring DCP weights into an unsharded module. The default artifact is FX symbolic_trace followed by torch.jit.script; under ENABLE_TRT/ENABLE_AOT, split_model (export_util.py:1068-1206) FX-splits the graph at marker boundaries into a sparse GM and a dense GM, compiling the dense part with torch_tensorrt or torch.export + aoti_compile_and_package with a dynamic torch.export.Dim("batch"). The RTP export path hard-rejects DTensor state (raise ValueError("DTensors are not considered yet."), export_util.py:484) — relevant because FSDP2 produces DTensor parameters; the AOTI path has no such rejection and is the route D13 builds on (§10). Notably, the predict driver already supports variable-cardinality “GR-style” outputs: when TARGET_REPEAT_INTERLEAVE_KEY (constant.py:40) appears in predictions, flattened per-candidate outputs are regrouped into pa.ListArray rows via fbgemm.asynchronous_complete_cumsum offsets (main.py:1017-1037) — beam outputs of K SIDs per user ride this convention unchanged.

3.2 Data and feature layer

The data layer is an Arrow-centric streaming pipeline: readers yield Dict<str, pa.Array> micro-batches, the dataset (an IterableDataset) applies sampling/masking and feature generation, and DataParser.to_batch assembles a fully formed Batch per iteration — the DataLoader runs with batch_size=None, collate_fn=identity (dataset.py:827-833). The Batch dataclass (datasets/utils.py:298-512) implements TorchRec's Pipelineable protocol and carries dense_features (KeyedTensor), sparse_features (KeyedJaggedTensor), sequence_dense_features, labels, jagged_labels: Dict<str, JaggedTensor>, sample_weights, additional_infos: Dict<str, Tensor> (a free-form tensor side-channel), plus serving/streaming metadata (tile_size, dummy, checkpoint_info, data_timestamp). One discipline applies to any extension: a new payload must be threaded through to(device), pin_memory(), record_stream(), and to_dict() or it silently drops on the H2D copy (utils.py:344-512).

The jagged_labels slot is the established precedent for variable-length, per-token supervision: DataParser.parse routes list<int>/list<float> label columns into values+lengths pairs (data_parser.py:231-250, 458-467), and both HSTUMatch._get_label (models/hstu.py:392-394) and DlrmHSTU._get_label (models/dlrm_hstu.py:246-258) consume them. This is the carrier for batch.jagged_labels["response_sid_ids"] and the separate batch.jagged_labels["loss_mask"] defined in §6; no Batch schema change is needed. What does not exist is any per-token mask semantics — masks today are sample-level feature dropout or implicit in jagged lengths (tzrec-data notes §8) — hence the loss-mask label convention rather than an overload of .weights().

Feature generation has four modes (data.proto:47-60): FG_NONE (inputs already encoded), FG_NORMAL/FG_DAG (pyfg C++ feature generation), and FG_BUCKETIZE. Two properties matter for the LLM route. First, FG_NONE accepts native Arrow list columns: _parse_fg_encoded_sequence_sparse_feature_impl takes list<int> directly (feature.py:262-272), ODPS ARRAY<BIGINT> maps onto it (odps_dataset.py:65), and jagged label columns likewise — so pre-tokenized prompt ids and precomputed SID columns flow end-to-end with zero new transport code. Second, TokenizeFeature (tzrec/features/tokenize_feature.py) already runs BPE/sentencepiece tokenization inside pyfg against a HF tokenizer.json or spiece.model vocab, and with tokens_as_sequence=true (feature.proto:757) emits the token stream as a jagged sequence feature (tokenize_feature.py:185-217). However, it offers no chat template, no BOS/EOS/special-token control, and no verified byte-level parity with the HF Qwen3 tokenizer — which is why §6 standardizes on offline pre-tokenized ARRAY<BIGINT> columns for v1 and treats any fg-side SID join as future work (R-FGJOIN).

The reader fleet is production-hardened and reused wholesale: OdpsReader (MaxCompute Storage-API Arrow sessions created on rank 0 and broadcast, LZ4 transport, row-offset checkpoint resume, odps_dataset.py:364-642), ParquetReader with global row-range rebalancing, CsvReader, and KafkaReader (each message a serialized Arrow IPC RecordBatch; per-row offsets and event timestamps injected and popped into Batch.checkpoint_info/data_timestamp, driving the event-time checkpoint quorum, kafka_dataset.py:128-485). Two adjacent primitives seed generative needs: cost-based batching (sample_cost_field + batch_cost_size, dataset.py:567-585) is a greedy token-budget batcher when cost = token count, and the GraphLearn sampler cluster (sampler.py) provides distributed “fetch attributes for sampled ids and append rows” — reusable for SID-level contrastive negatives and eval candidate expansion, though string-attr-centric and CPU-side. Known limitations the design must respect: one reader schema per dataloader (no ratio-based mixing of heterogeneous corpora, odps_dataset.py:424-438), batch-granular rather than row-granular shuffling, and the sequence-feature default value "0" colliding with SID code 0 unless the vocab layout reserves it (feature.py:549-561; the flat itemic-offset encoding in §6 addresses this).

3.3 Model and module layer

Models auto-register by class name via the metaclass in load_class.py:117-145; _create_model resolves config_util.which_msg(model_config, "model") — the protobuf message class name of the set oneof model arm — to BaseModel.create_class(name) (main.py:127-159, config_util.py:73-75). Adding a model is therefore one proto message plus one identically named class in tzrec/models/*.py (the import is non-recursive, load_class.py:53-99), with config_to_kwargs bridging proto fields to module kwargs. The ModelConfig oneof currently occupies tag blocks 100 (ranking), 200 (multitask, including DlrmHSTU=205), 300 (matching, including HSTUMatch=304), 400 (TDM), and 500 (RocketLaunching); the 600 block is free and is where the five generative arms and the 610 SID-tokenizer arms land (§12). The BaseModel contract is narrow — predict(batch), init_loss()/loss(), init_metric()/update_metric()/compute_metric() — and the eval loop only ever calls predict once per batch and feeds (losses, predictions, batch) to metrics (main.py:162-226).

EmbeddingGroup (modules/embedding.py:139-519) routes feature groups to pooled EmbeddingBagCollections or sequence EmbeddingCollections and emits a uniform Dict<str, Tensor>: DEEP/WIDE groups as dense [B, sum_dims], SEQUENCE/JAGGED_SEQUENCE groups as "g.query"/"g.sequence"/"g.sequence_length", with JAGGED_SEQUENCE keeping jagged values [sum(len_i), D] (embedding.py:1370-1375). A SID-token history is representable today as a JAGGED_SEQUENCE group over an EC table — bringing TorchRec sharding, ZCH, and dynamicemb along for free — and these grouped outputs are the payload that context-embedding injection (D8's bridge, R-INJECT, §7) feeds into the LLM.

The HSTU stack is the in-house generative backbone precedent. Its jagged convention — (values [total, D], seq_offsets [B+1], max_seq_len, num_targets [B]) — pervades tzrec/ops; hstu_mha (ops/hstu_attention.py:43-196) is the full causal kernel and delta_hstu_mha (:199-267) the incremental/cached variant. STULayer already implements a two-phase KV cache: prefill splits just-computed k/v at kv_caching_lengths into per-layer caches (modules/gr/stu.py:128-175), and cached_forward concatenates cache plus delta and runs delta_hstu_mha (stu.py:564-641). Constraints to design around: the delta size must be uniform per row (L % B == 0, hstu_attention.py:221) — compatible with stepwise SID decode where the host replicates beam rows — the cache is plain Python instance state (not traceable, reinforcing the host-side-loop rule), and CUTLASS falls back to Triton on the delta path. DlrmHSTU contributes two conventions reused verbatim: predictions[TARGET_REPEAT_INTERLEAVE_KEY] = num_targets for jagged per-candidate outputs (dlrm_hstu.py:192-244, key at constant.py:40), and enable_global_average_loss, which all-reduces the average batch size to weight variable per-rank token counts (dlrm_hstu.py:64-69, 280-284) — the same normalization a globally averaged token CE needs in §8.

TDM retrieval (tzrec/tools/tdm/retrieval.py:121-414) is the codebase's proof that autoregressive decode belongs on the host: a scripted, stateless scorer is driven by a Python loop that expands candidates per tree level, re-materializes features in a worker process, runs torch.topk beam pruning at width 2*recall_num, and keeps all beam state in pyarrow on the host. A SID beam decoder is structurally identical — children become the trie-constrained next codes instead of GraphLearn tree children, and the exported scorer gains explicit KV-cache I/O instead of full re-encoding (§10, D13). The TDM export also set the two-artifact precedent (embedding/ + model/, main.py:981-998) for splitting a prefix encoder from a per-step scorer.

SID baseline from PR #538 — and what is deliberately not there yet

Commit 7dc1c18 landed the tokenizer-training substrate this RFC builds on. ResidualQuantizer (modules/sid/residual_quantizer.py) implements the residual chain with precise gradient semantics — quantized vectors are subtracted detached so the residual chain carries no gradient, while the aggregate sum keeps codebook gradients flowing (_residual_pass, rq.py:119-153) — plus get_codes/decode_codes as the offline SID-assignment API and abstract _quantize_layer/_lookup_code primitives for backends. BaseSidModel (models/sid_model.py:54-74) consumes a precomputed item-embedding dense feature (embedding_feature_name/input_dim/normalize_residuals/codebook config fields), declares the internal-loss convention (losses computed in predict and passed through predictions; init_loss() is a no-op), and registers mse plus unique_sid_ratio (metrics/unique_ratio.py — explicitly a batch-level diversity proxy, not global codebook coverage). What is not in the repo: no concrete quantizer backend (the docstring names the planned k-means and VQ backends; neither exists), no SID proto messages in the oneof model (the 600/610 blocks are empty), no collision resolution, no item↔SID catalog or trie, no quality gates beyond UniqueRatio, and no decoder or generative model of any kind. §5 fills the tokenizer half (D1–D4); §7 the model half (D5).

3.4 Gap analysis

Table 3.1 maps the requirement clusters of §2 onto existing assets, the concrete gap, and the owning section of this RFC. The pattern is consistent: transport, sharding, checkpointing, and the host-side decode precedent exist; everything that touches a HF transformer, a packed 32K sequence, an RL rollout engine, or an autoregressive serving loop is new.

The four integration seams the design exploits

Four seams, zero forks

The architecture in §4 deliberately routes every two-world integration through four existing seams rather than forking the loop:

  • Pipeline factory. create_train_pipeline (dist_util.py:335-376) is the single pipeline-selection point, called at exactly three sites (main.py:176, 430; predict uses PredictPipelineSparseDist directly). The loop relies only on pipeline.progress(iterator) -> (losses, predictions, batch) plus StopIteration semantics, and a no-sparse fallback to TrainPipelineBase already exists — a custom GR pipeline that orders sparse dist, dense backward, finalize_model_grads, and the two optimizer steps slots in here (§11).
  • Optimizer tree. part_optimizers regex groups (optimizer_builder.py:219-237) and the open-ended CombinedOptimizer list (main.py:693-755) give the LLM its own AdamW + schedule without touching sparse Adagrad; per-node BaseLR registration covers warmup-cosine (§8, §12).
  • Checkpoint roots. save_model (checkpoint_util.py:873) is the single write site; the dynamicemb side-dump and PartialLoadPlanner's skip-missing/remap behaviors make an additional dense-world subtree (and warm-start from converted HF weights via fine_tune_ckpt_param_map) additive rather than invasive (§11).
  • Batch side-channels. jagged_labels carries response_sid_ids and loss_mask; additional_infos carries packing metadata; TARGET_REPEAT_INTERLEAVE_KEY carries variable-K outputs through loss, metrics, and the predict writer — all without a Batch schema change (§6).

Assumptions that break

Baseline assumptions the design must engineer around

  • Single DMP wrap vs. an FSDP2 subtree. DMP wraps the whole TrainWrapper and DDP-replicates every non-sharded parameter; naively placing a fully_sharded LLM inside would have DMP's DataParallelWrapper attempt to DDP-wrap DTensor modules. The custom data_parallel_wrapper hook (dist_util.py:172) and DDP-ignoring the FSDP FQNs are the mitigation (D8, R-DMP; §11). Side effects extend to tooling: _log_train special-cases ShardedTensor but not DTensor histograms (main.py:229-314), and RTP export rejects DTensor outright (export_util.py:484).
  • The planner ignores dense memory. The TorchRec planner budgets HBM from device total_memory minus HeuristicalStorageReservation(0.15); a multi-billion-parameter dense model plus activations and optimizer state is invisible to it, so plans that fit today OOM under the LLM (R-PLANNER). Mitigations — raising STORAGE_RESERVE_PERCENT, forcing row-wise constraints — are specified in §11.
  • pipeline.progress owns fwd+bwd+step; accumulation is asymmetric. There is no hook between backward and step, and in-backward sparse optimizers step every backward while TZRecOptimizer accumulates the dense side — the two-world step-sequencing problem (R-GRAD) is resolved by the GR pipeline in §11.
  • FX/JIT cannot trace a HF transformer. The default export (symbolic_trace + jit.script) will not swallow dynamic control flow and cache objects; the AOTI single-step-scorer route (D13) with a host-side decode loop is mandatory, never tracing the autoregressive loop (§10; R-EXPORT, de-risked).
  • Eval-after-save is too expensive for generation. The loop runs a full eval after every checkpoint save (main.py:490-497), and the eval contract assumes one cheap predict per batch. Constrained beam decode over the catalog per save is unaffordable; generative eval cadence and decode-driver wiring are decoupled in §13.
  • One flat process group. init_process_group creates only WORLD and the planner assumes it; Megatron's initialize_model_parallel and FSDP2 meshes must layer on top after init without re-creating groups (D8; §11).

In sum, the baseline supplies a stable chassis — registry-driven models, Arrow-jagged transport, fused sparse optimization, DCP checkpointing, and a proven host-side beam pattern — and the gaps concentrate precisely where the two-world design of §4 introduces its new packages. The remainder of this RFC fills those gaps without modifying the contracts cataloged above.
