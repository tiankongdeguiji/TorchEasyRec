12. Configuration and API design

This section is the source of truth for the new protobuf surface and CLI. Four proto files are added — tzrec/protos/models/generative_model.proto, tzrec/protos/models/semantic_id.proto, tzrec/protos/sft.proto, tzrec/protos/rl.proto — plus seven new oneof model arms in model.proto. The design follows tzrec's established minimal-proto idiom: no wrapper messages, no enums for single-mode operations, and name-based routing wherever a choice among alternatives exists. Two complete text-proto examples close the section.

12.1 Design rules and registry coupling

Three mechanical couplings constrain every message below. First, proto message name == Python class name: config_util.which_msg(model_config, "model") returns the message class name of the set oneof arm and BaseModel.create_class(name) resolves it against the registry metaclass (config_util.py:73-75, main.py:127-159) — adding a model is one message, one tag, one identically named class, and zero orchestration edits. One registration caveat: auto_import scans tzrec/models/*.py non-recursively (load_class.py:53-99), so the classes under tzrec/models/generative/ register through a one-line re-export shim placed directly under tzrec/models/ — auto_import itself is not modified (§4, §7). Second, proto field names == module keyword names: config_to_kwargs is MessageToDict(..., including_default_value_fields=True, preserving_proto_field_name=True) (config_util.py:66-70), so field naming is API design, not cosmetics. including_default_value_fields is the protobuf-4.x spelling, which is why the codegen toolchain stays pinned at grpcio-tools<1.63.0; after any .proto edit, regenerate with bash scripts/gen_proto.sh (the script globs new files automatically). Third, the SID tokenizer messages must carry exactly the field names the already-merged BaseSidModel reads off its model config — embedding_feature_name, input_dim, normalize_residuals, codebook (tzrec/models/sid_model.py:54-74).

The minimal-proto rules applied throughout: alternatives are a named oneof (the arm name routes behavior); booleans and scalars are flat fields on the owning message rather than nested option messages; knobs that tzrec does not itself interpret are not modeled — for RL they pass through a verl_overrides string map of dotted Hydra paths, and for anything else they do not exist until a consumer does (fail loud beats silent defaults).

Revision — sid_vocab_size is removed; vocab quantities are derived

The prior spec carried a sid_vocab_size field defined as the padded itemic block length (24,832), with total vocab = itemic_id_range_begin + block — arithmetic that was internally inconsistent (§4.6). Under the corrected math everything derives from three fields: K = num_sid_levels × codebook_size itemic tokens occupy [itemic_id_range_begin, begin+K) = [151669, 176245); the two specials <|sid_begin|>/<|sid_end|> follow at 176245/176246; and the model vocab pads the total: align(176247, 256) = 176384, which divides evenly for TP ∈ {1,2,4,8,…}. A redundant stored size would only invite drift; instead main_gr recomputes the quantities and fails loud if they disagree with the checkpoint's embedding-row count and tokenizer length. The stale-row initialization gotcha for rows [151669, 151936) is owned by §7 (R-VOCAB-INIT).

The five model messages are thin: a backbone plus optional stage configs, and model-specific knobs only where a paper defines one.

12.4 semantic_id.proto

The two tokenizer models share the BaseSidModel field quartet verbatim and inline their quantizer-specific knobs (a dedicated message per model already exists; wrapping its own options again would be a wrapper for its own sake). Metrics need no new fields: BaseSidModel.init_metric already registers mse and unique_sid_ratio unconditionally, and the training-free quality gates (Gini, per-level utilization, Embedding HitRate) are computed by assign_sid tooling over the finished asset table (D3, §5, §13) — they are properties of an artifact, not of a training step, so they live behind CLI flags (--output_table_path, --quality_report_path) rather than the model proto.

12.5 sft.proto

// tzrec/protos/sft.proto
enum SftStage {
    ITEMIC_TEXT_ALIGN = 0;   // stage 1: train itemic rows only (D7)
    CO_PRETRAIN = 1;         // stage 2: full-param, rec + general-text mix (D6)
    MULTI_TASK_SFT = 2;      // stage 3: instruction tuning, response-masked CE
    RSFT = 3;                // optional: top-fraction sessions, annealed LR
}

message MixtureSource {      // one weighted MixtureReader source (Sec.6.4)
    required string input_path = 1;
    optional float weight = 2 [default = 1.0];
}

message SftConfig {
    optional SftStage stage = 1 [default = MULTI_TASK_SFT];
    // freezing: first trainable embedding row; rows below stay frozen in stage 1
    optional uint32 start_optimize_embedding_index = 2;
    // CO_PRETRAIN mix: per-source interleave weights for the MixtureReader (Sec.6.4)
    repeated MixtureSource mixture_sources = 3;
    // packing (D6): requires a varlen flash-attention backend (validated, see below)
    optional uint32 pack_max_length = 4 [default = 32768];
    // loss; no masking knob here: jagged_labels["loss_mask"] is fixed at build time (Sec.6/8)
    optional bool use_chunked_loss = 5 [default = true];   // 176K-vocab lm_head CE
    optional uint32 loss_chunk_size = 6 [default = 16384];
    // RSFT session filter: keep top fraction by the named reward column
    optional float rsft_keep_fraction = 7 [default = 0.5];
    optional string rsft_score_column = 8;
    // PLUM-style reward-weighted sample selection (D14); 0 = uniform
    optional float reward_weighted_sampling_alpha = 9 [default = 0.0];
}

Validation (D6, fail loud). When packing is enabled (pack_max_length > 0, the default), main_gr.sft requires the backbone's resolved attention implementation to be a varlen flash-attention backend — flash_attention_2/3 on the HF route, the TransformerEngine layer spec on the Megatron route — and aborts at startup otherwise: SDPA and eager attention ignore cu_seqlens and silently cross-attend across packed documents (§6.5, §8). The packing×backend pair is checked in config validation, never discovered mid-train.

Deliberate omissions: optimizer, LR schedule, and mixed-precision settings are not duplicated here — they remain in TrainConfig, with the LLM's AdamW expressed through the existing adamw_optimizer/cosine_annealing_learning_rate arms; instruction-task suite construction and chat templating are data-build-time concerns owned by §6's builders, not training-time proto (only the corpus-interleave weights above are train-time, because the MixtureReader enforces them at the sampler). A stage flip between runs is a one-field --edit_config_json override, and stage checkpoints chain via --fine_tune_checkpoint (§8).

12.6 rl.proto

// tzrec/protos/rl.proto
enum RlAlgorithm {
    GRPO = 0;   // default; stock verl estimator (D10)
    ECPO = 1;   // OneRecV1 early-clip; registered policy-loss plugin
    GBPO = 2;   // OneRecV2 gradient-bounded; no rollouts, logged feedback
    PPO = 3;    // stock verl, critic enabled
}

message RewardModelReward {     // external P-Score multi-tower RM service (D14)
    required string score_service = 1;
}
message DurationAwareReward {
    optional float log_bucket_base = 1 [default = 2.0];
    optional float percentile_threshold = 2 [default = 0.25];
}
message SidLegalityReward {}    // OneRecV1 random-K legality credit; diagnostic under
                                // constrained decode (never penalize-and-train: discard)
message FormatReward {}
message DiversityReward {}      // group-level first-digit diversity factor

message RewardConfig {
    oneof reward {
        RewardModelReward reward_model = 1;
        DurationAwareReward duration_aware = 2;
        SidLegalityReward sid_legality = 3;
        FormatReward format = 4;
        DiversityReward diversity = 5;
    }
    optional float weight = 10 [default = 1.0];
}

message RlConfig {
    optional RlAlgorithm algorithm = 1 [default = GRPO];
    // rollout (vLLM is the pinned engine, D9/D11 — no engine enum)
    optional uint32 group_size = 2 [default = 8];
    optional uint32 beam_width = 3 [default = 32];     // SID-stage beam (two-stage rollout)
    optional bool constrained_decoding = 4 [default = true];
    // objective
    optional float kl_loss_coef = 5 [default = 0.001];
    optional float clip_ratio_low = 6 [default = 0.2];
    optional float clip_ratio_high = 7 [default = 0.28];
    // on-policy distillation (selected via verl_overrides adv estimator; D10)
    optional string distill_teacher_path = 8;
    optional float distill_adv_min = 9 [default = -30.0];
    optional float distill_adv_max = 10 [default = 5.0];
    repeated RewardConfig rewards = 11;
    // pass-through dotted Hydra overrides; tzrec does not interpret these
    map<string, string> verl_overrides = 20;
}

main_gr.rl renders this message into a verl Hydra config: algorithm selects the registered estimator, constrained_decoding: true injects the CSR-trie logits processor through engine_kwargs.vllm.logits_processors and registers the SID agent loop via rollout.agent.agent_loop_config_path, rewards become a custom_reward_function, and verl_overrides wins last-write for anything else (e.g. actor_rollout_ref.rollout.gpu_memory_utilization) [verl]. Two prior-spec arms are dropped: the rollout-engine enum (vLLM is pinned — SGLang has no beam search, D11 [sglang]) and the DPO arm (no v1 implementation; an enum value without a consumer violates fail-loud). Rendering is preceded by validation: RlConfig is cross-checked against the D11 backbone×rollout matrix (§9.4) — GBPO is the only legal algorithm for the lazy_decoder backbone (no rollout engine), rollout-based algorithms require a token-LLM backbone, and encoder_decoder/inhouse_hstu take the tzrec-native in-process loop — so an unsupported (backbone, algorithm) pair is a startup config error, never a silent degradation.

12.7 CLI entry points

All verbs are thin argparse wrappers over tzrec/main_gr.py and reuse the standard flags --pipeline_config_path, --model_dir, --edit_config_json, --fine_tune_checkpoint.

12.8 Example (a): RQ-KMeans tokenizer training

The tokenizer is an ordinary tzrec model run: the item-embedding table is the training input, the precomputed embedding arrives as a dense raw_feature with value_dim = input_dim (consumed directly by BaseSidModel._extract_feature, bypassing EmbeddingGroup), and no labels are needed. 12.9 Example (b): OpenOneRec-style SFT (stage 3, FSDP2)

Training samples are pre-tokenized offline with the extended Qwen3 tokenizer (§6): the prompt column is an ARRAY<BIGINT> of token ids whose history SIDs are already itemic tokens, and the two label columns carry the target SID ids and the per-token supervision mask. With fg_mode: FG_NONE these flow through existing parsing paths into sparse_features and jagged_labels with no new transport. The prompt feature is transport-only — the Qwen3 backbone consumes raw jagged token ids and owns the token embedding itself (§7); embedding_dim and expression are set only because the IdFeature schema requires them (both unused in FG_NONE transport-only mode), and §6 defines how transport-only features skip table creation.

            T_max: 5000
            warmup_size: 500

    num_steps: 5000

    save_checkpoints_steps: 500
}
eval_config {
    num_steps: 50
}
