7. Model layer: backbones, vocabulary, and context injection

This section specifies the dense half of the two-world architecture: the BaseGenerativeModel contract that plugs generative recommenders into tzrec's existing registry/loss/metric machinery, the four backbone modules behind decision D5, the corrected itemic-vocabulary contract that every other section consumes, the prediction-head and decode-strategy abstractions that keep the design level-count- and decode-strategy-agnostic, and the context-injection bridge from TorchRec sparse embeddings into the LLM (R-INJECT). Everything here is a module-layer specification; training-loop mechanics live in §8 and §11, and decode/serving in §10.

7.1 BaseGenerativeModel(BaseModel) contract

Generative models are ordinary tzrec models. A new abstract base, BaseGenerativeModel(BaseModel) in tzrec/models/generative/, follows the same registry pattern as BaseSidModel (tzrec/models/sid_model.py, PR #538): the metaclass from tzrec/utils/load_class.py:117-145 auto-registers every subclass under its class name, and tzrec/main.py:_create_model resolves the class from the protobuf oneof model arm via config_util.which_msg (config_util.py:73-75). The concrete classes and their proto tags are fixed by this RFC: OpenOneRec=600, OneRecV1=601, OneRecV2=602, ALGR=603, LCRec=604 (message name == class name; see §12 for the proto surface). MiniOneRec is a config preset over LCRec + GRPO, never a class. One registration caveat from the baseline study: auto_import scans tzrec/models/*.py non-recursively (load_class.py:53-99), so classes under tzrec/models/generative/ must be re-exported from a module directly under tzrec/models/ (or auto_import gains the subpackage — a one-line change owned by §12).

The base contract mirrors BaseModel's three surfaces with generative-specific semantics:

  • predict(batch) -> Dict[str, Tensor]. In training/eval mode this is the teacher-forced scoring pass: assemble inputs (token ids and/or injected context embeddings, §7.5), run the backbone, emit per-level logits through SidPredictionHead (§7.4), and compute the token-level cross-entropy inside predict, passing loss tensors through the predictions dict. This adopts the BaseSidModel convention ("SID models compute their losses internally and pass them through predictions", sid_model.py:91-97 init_loss docstring) rather than the RankModel scaffolding, which assumes per-row scalar labels and cannot express jagged per-token CE.
  • loss(predictions, batch) selects the loss tensors from predict's output and applies global token normalization. Per-rank token counts vary under packing; the established pattern is DlrmHSTU.enable_global_average_loss (dlrm_hstu.py:64-69, 280-284): all-reduce the global valid-token count and weight the local sum accordingly, which is also exactly the HF Trainer num_items_in_batch fix replicated in §8.
  • Labels. Target SID token ids arrive as batch.jagged_labels["response_sid_ids"] and the response mask as a separate batch.jagged_labels["loss_mask"]. The mask is never smuggled through JaggedTensor.weights() — weights remain reserved for sample weighting. Both are jagged over response tokens; the model never calls to_padded_dense on the packed path.
  • Generated outputs. In offline predict/eval mode, generation runs through the host-side BaseSidDecoder driver in eager mode (the decode loop is never traced into a graph; §10). The model emits one prediction row per generated candidate — jagged over the top-K beam results — and sets predictions[TARGET_REPEAT_INTERLEAVE_KEY] (tzrec/constant.py:40) to the per-user candidate count, exactly the DlrmHSTU variable-output convention. PredictWrapper passes the key through unconditionally (model.py:332-335) and the predict writer already expands per-user reserved columns by it, so beam outputs need zero writer changes.
  • init_metric registers generative-retrieval metrics (HitRate@K / Recall@K over decoded item ids after SID→item expansion; §13 owns definitions). Because metrics consume item ids, swapping decoders (§7.4) is metric-transparent.

7.2 Backbones (D5)

Decision D5 — four backbones behind one interface; decode strategy owned by the decoder, not the model

v1 ships four backbone modules under tzrec/modules/llm/, all implementing a common GRBackbone forward contract (inputs: token ids and/or context embeddings plus packing metadata; output: final hidden states): Qwen3GRBackbone (HF weights, FSDP2 default or Megatron-core scale-out), LazyDecoderGRBackbone (OneRec-V2), EncoderDecoderGRBackbone (OneRec-V1), and InhouseHstuGRBackbone (export-clean, reuses STUStack). Two abstractions keep the model layer decode-agnostic: SidPredictionHead emits per-level logits (never hard-coding 3 levels), and BaseSidDecoder owns the number of scorer invocations (3 AR steps, 1 RPG-style parallel pass, or T diffusion steps). No interface in this section may assume AR-only decoding or num_decode_steps == num_sid_levels (see D15/D16 in §7.4).

Qwen3GRBackbone

The Qwen3 architecture is GQA with a signature twist: per-head-dim RMSNorm on q and k (q_norm = Qwen3RMSNorm(head_dim), applied over the last axis before RoPE; modeling_qwen3.py:222-291 [transformers]). head_dim is an explicit config field (128 for every released size), not derived from hidden_size/num_heads. SwiGLU MLPs, RMSNorm eps 1e-6, rope theta 1e6, full attention on all layers in released checkpoints. Embedding tying is size-dependent and load-bearing for §7.3: Qwen3-0.6B/1.7B/4B ship tie_word_embeddings: true; 8B/14B/32B ship false. The HF vocab is 151,936 rows of which only 151,669 are used tokens. One memory lever worth wiring from day one: Qwen3ForCausalLM.forward(logits_to_keep=...) slices hidden states before the lm_head (modeling_qwen3.py:467), which matters greatly at a 176K-row vocabulary; the training-side equivalent is the chunked-lm_head CE owned by §8.

Two construction routes, selected by the parallelism config (§11):

  • FSDP2 route (default). from_pretrained loads the vocab-expanded checkpoint directly (meta-init + sharded safetensors), then fully_shard is applied per Qwen3DecoderLayer and root — the layering OpenOneRec uses in production (pretrain/onerec_llm/training/distributed.py [repo-openonerec]). tzrec pins the transformers 4.53 line (OpenOneRec's pin, and the version the AOTI export proof in §10 was validated on); the v5 line renames torch_dtype→dtype and nests rope_theta under rope_parameters, so version assumptions must not leak across the pin.
  • mcore route. Qwen3-dense maps onto a stock TransformerConfig with no custom modules — verified field-by-field in the research notes: num_query_groups = num_key_value_heads, kv_channels = head_dim (decoupled from hidden size, exactly Qwen3's convention), qk_layernorm=True (the layer spec's q_layernorm/k_layernorm = TENorm(kv_channels) reproduces per-head q/k RMSNorm), gated_linear_unit=True + SiLU, normalization="RMSNorm", add_bias_linear=False; GPTModel(position_embedding_type="rope", rotary_base=1e6, share_embeddings_and_output_weights=tie_word_embeddings) [megatron]. The same mapping is what verl's converter applies (qk_layernorm=True if "Qwen3" in architectures, verl/models/mcore/config_converter.py:167-184 [verl]). Weight conversion goes through mbridge's AutoBridge.from_config(hf_config, dtype) [mbridge], which handles the q/k/v→fused linear_qkv interleave and gate+up→linear_fc1 concat, and — critically for §7.3 — builds the model at exactly the HF vocab size with no extra padding.

LazyDecoderGRBackbone (OneRec-V2)

OneRec-V2's lazy decoder removes the encoder entirely: in V1, context encoding consumed 97.66% of FLOPs at context length 512 while the loss-bearing target decode was 2.34% [onerec-v2]. The lazy design embeds the heterogeneous context (user profile + short/long behavior pathways) once in a Context Processor, projects every context position to a single dimension d_context = S_kv · L_kv · G_kv · d_head, partitions the feature dimension into S_kv·L_kv chunks of G_kv·d_head each, and uses RMSNorm'ed chunks directly as cross-attention K/V: k_l = RMSNorm(C_{l·S_kv}), with v_l = k_l when S_kv=1. There are no learned W_k/W_v matrices over the context anywhere in the decoder — the only per-layer learned pieces on the context path are RMSNorm gains. Layer l selects its KV slice via l_kv = floor(l · L_kv / N_layer); since the paper's ablations show the aggressive settings are loss-neutral (loss flat at 3.27 across (L_kv, S_kv); MQA G_kv=1 shrinks context KV 13×), tzrec defaults to L_kv=1, S_kv=1, G_kv=1. All transformer compute (pre-norm blocks ordered cross-attn → causal self-attn → FFN/MoE) runs only on the 3-token target [BOS, s1, s2]; output is 3 position-specific RMSNorm+Linear heads over 8192-way softmaxes with loss = mean of the 3 CEs (V1 used the sum — loss values are not comparable across versions). The result: 18.89 GFLOPs/sample at 1B, independent of context length, versus 634.8 for a naive decoder-only at near-identical loss (the naive baseline was trained only at 0.1B, where the comparison is 3.54 vs 3.57). Scaling anchors for config validation: 1B = d_model 1792, 18 layers, 14 heads, SID embedding dim 90. The Context Processor maps onto tzrec feature groups via per-pathway projections; data-side assembly, including request-grouped "common context compression", is owned by §6.

EncoderDecoderGRBackbone (OneRec-V1)

The V1 architecture is a pre-norm encoder over a four-pathway context — user static (1 token), short-term (20 interactions), positive-feedback (256), and lifelong (up to 100K items compressed TWIN-V2-style, then QFormer'ed to 128 query tokens) — concatenated to a 405-token encoder input with learned positions, each pathway ending in a Dense(LeakyReLU(Dense)) projection to d_model [onerec-v1]. The decoder interleaves causal self-attention, cross-attention to the encoder output, and an MoE FFN (SwiGLU experts, top-k routing, DeepSeek-V3 aux-loss-free balancing — worth −0.2 loss in their ablation); NTP loss is the sum of the 3 token CEs. Reference family points: 0.935B = 8 layers, d_model 1024, 24 experts top-2 in the decoder only; 2.633B = 24 layers, 24/4 experts in both encoder and decoder. Construction routes: mcore T5Model (which takes separate encoder/decoder configs and layer specs, t5_model.py:136-159 [megatron]; PP=1 keeps the legacy encoder/decoder pipeline-placement scheme out of play) or an in-house pre-norm module set when mcore is not in the build. The pathway feature engineering is §6 territory; rollouts for this backbone use the tzrec-native in-process beam loop since no serving engine speaks this architecture (D11, §9).

InhouseHstuGRBackbone

The fourth backbone needs no new dependencies: STUStack (tzrec/modules/gr/stu.py) is already a causal, target-aware, jagged-native transformer with per-layer KV caching and an incremental-decode path (cached_forward over delta_hstu_mha, hstu_attention.py:199-267) whose uniform per-row delta-size constraint matches stepwise SID decode exactly (each beam row appends one token per step). SID history enters as a JAGGED_SEQUENCE feature group; per-level codebooks are fused into one embedding table with cumulative per-level id offsets, the layout proven in NVIDIA's SID-GR example (sid_sequence_dataset.py:95-141 [repo-recsysexamples]), with a learnable BOS embedding parameter. With a fused vocab of 24,576 the head is small enough that no vocab parallelism is ever needed. This backbone is the export-clean lane: DlrmHSTU already exports through tzrec's AOTI path, and the same machinery carries the single-step decode scorer of §10 (D13). Its limits are inherited and documented: the delta path falls back from CUTLASS to Triton, and beam forks require host-side re-indexing of the per-layer k_cache/v_cache plus offsets (a small helper this RFC adds; §10).

7.3 Itemic vocabulary: the corrected contract

Revision — vocab padding applies to the total, not the itemic block (supersedes prior spec §4.7.4)

The prior internal spec padded the itemic block to a TP-divisible size (24,578 → 24,832). That formula is wrong and is retired: padding must apply to the total vocabulary, matching OpenOneRec's tool (expand_qwen3_vocab.py:169: align(151669 + 24578, 256) = 176384 [repo-openonerec]) and required for exact HF↔mcore shape parity (below). Padding the block instead of the total would shift every special-token id, break the id = 151669 + level·8192 + code arithmetic, and desynchronize tzrec checkpoints from the OpenOneRec-compatible layout. All sections of this RFC use the numbers in the table below.

Token ordering is load-bearing. Itemic tokens are added in strict layer-major order — all <s_a_0>..<s_a_8191>, then <s_b_*>, then <s_c_*>, then the two specials last — so that id = 151669 + level·8192 + code holds; OpenOneRec hard-codes this arithmetic in at least four places across training, RL metrics, and distillation configs [repo-openonerec], and tzrec's RL legality and loss-split code relies on it too. A rendered SID is the string <|sid_begin|><s_a_{c0}><s_b_{c1}><s_c_{c2}><|sid_end|>. The tokens are added as ordinary added tokens, not special tokens, so they survive skip_special_tokens=True decoding. The vocab-expansion tool — python -m tzrec.gr.extend_vocab, the stage-3 lifecycle entry point of §4 (offline single-process, deliberately not one of the §12 CLI verbs) — asserts padded_vocab % 256 == 0; the training driver asserts vocab % tp == 0 at startup — fail loud, no silent re-padding.

New-row initialization and the 267-row gotcha

Initialization uses HF resize_token_embeddings(new_num_tokens=176384, mean_resizing=True). The exact mechanics (modeling_utils.py:3071-3094 [transformers]): cast the old embedding to fp32, compute the row mean μ and covariance Σ = (W−μ)ᵀ(W−μ)/N, then sample new rows from MultivariateNormal(μ, 1e-9·Σ) — i.e. the mean plus tiny correlated noise (the Hewitt vocab-expansion trick, which keeps pre/post-resize next-token KL small) — falling back to plain mean-fill if Σ is not positive-definite. The lm_head receives the same treatment, and tie_weights() re-ties afterwards so for tied checkpoints the input-embedding init is what survives. Because the algorithm needs the full unsharded [V, H] matrix in fp32, the resize runs offline, in the HF representation, before any bridge conversion or sharding — OpenOneRec loads the model with torch_dtype=torch.float32 explicitly for this reason, and both mbridge and Megatron-Bridge then copy row values verbatim into the TP-sharded model.

Gotcha (R-VOCAB-INIT) — 267 itemic tokens land on stale dummy rows

resize_token_embeddings(mean_resizing=True) initializes only rows ≥ old_embedding_rows = 151936. The first 267 itemic tokens occupy ids [151669, 151936) — Qwen's shipped dummy rows — and therefore keep Qwen's untrained values after the resize. OpenOneRec shipped this bug and silently trained through it [repo-openonerec]. tzrec's expansion tool must explicitly re-initialize rows [len(tokenizer_before_add), old_embedding_rows) from the same N(μ, 1e-9·Σ) distribution, in both the input embedding and (for untied models) the lm_head, and the tool's self-check must verify that no row in the itemic range is bit-identical to the pre-resize dummy rows.

Two initialization options layer on top. First, codebook-vector init: after the mean-resize, itemic rows may be overwritten with embeddings derived from the RQ codebook centroids (a plain in-place write of get_input_embeddings().weight.data[151669:176245]). Since centroids live in the item-embedding space (e.g. 4096-d Qwen3-Embedding-8B vectors for the OpenOneRec-compatible tokenizer) and generally do not match the LLM hidden size, this requires a projection or truncation step; it ships as an off-by-default option — the OpenOneRec evidence is that plain mean init trains fine through stage 1. Second, per-domain begin tokens: reasoning/multi-domain recipes use domain-specific begin markers (e.g. separate begin tokens per vertical) in the OneReason style [onereason]; tzrec supports an optional list of extra special tokens appended after <|sid_end|>. The 137 alignment-dummy rows absorb small additions without changing the padded size; if the total ever exceeds 176,384, the tool re-aligns to the next multiple of 256 and the new padded size propagates everywhere (it is computed, never hard-coded).

Tied-weight handling per model size. For tied checkpoints (0.6B–4B) there is one logical [V, H] matrix: the resize path updates the input embedding and re-ties; under mcore with PP=1 the output layer is built with skip_weight_param_allocation and shares the embedding Parameter, so row-level operations (init, freezing) apply exactly once. For untied checkpoints (8B+), embed_tokens and lm_head are independent [V, H] matrices with identical row geometry: both receive the 24,448 resize-initialized rows covering the 24,578 added tokens, both need the 267-row re-init for the tokens landing on pre-existing dummy rows, and both must be covered by the stage-1 freezing recipe (OpenOneRec's masker matches both by name).

TP sharding compatibility

Megatron's VocabParallelEmbedding splits rows via VocabUtility.vocab_range_from_global_vocab_size, whose divide() hard-asserts padded_vocab % tp == 0 (tensor_parallel/utils.py:113-121 [megatron]). 176,384 = 2^8 · 689 satisfies this for every realistic TP degree, and — verified across both bridges — mbridge and Megatron-Bridge build GPTModel(vocab_size=176384) with no additional padding, so Megatron-side and HF-side shapes match exactly [mbridge] [megatron-bridge]. Do not route any tzrec flow through megatron.training's tokenizer-padding convention (ceil(v/(128·tp))·128·tp), which would yield 176,640 at tp=4 and 177,152 at tp=8 and desynchronize checkpoints. For stage-1 row freezing (D7, mechanics in §8), the per-rank frozen-row count follows from the module attributes vocab_start_index / num_embeddings_per_partition: n_frozen_local = clamp(151669 − vocab_start_index, 0, rows_per_rank).

7.4 Prediction heads and decode-strategy abstractions

SidPredictionHead is the single module boundary between backbone hidden states and per-level token distributions: it takes hidden states plus the level count m and emits per-level logits — conceptually [B, m, vocab_per_level], concretely a list of per-level logit tensors so unequal per-level codebook sizes stay legal. Loss code consumes the list, never the literal 3. Three implementations ship in v1:

  • Shared-lm_head slice (Qwen3 path): the head is the stock 176K-row lm_head; "per-level logits" are views of the itemic block [151669 + l·8192, 151669 + (l+1)·8192). Training CE for this path runs over the full vocabulary (the response tokens are itemic, but the model must keep competing them against text tokens), so this implementation is a thin indexing adapter used by decoding and metrics rather than by the SFT loss.
  • Position-specific heads (OneRec-V2): m independent RMSNorm+Linear heads over per-level codebooks, teacher-forced at the m target positions [onerec-v2].
  • Per-hierarchy heads over a fused vocab (in-house path): either one shared linear over ΣC_l with per-level slicing or a ModuleList of per-level linears — both layouts proven in the GRID-tokenized SID-GR example [grid] [repo-recsysexamples].

BaseSidDecoder owns generation: generate(scorer, n_return) -> item ids, where the decoder — not the model — decides how many scorer invocations occur. ConstrainedBeamDecoder (3 AR steps over the CSR trie, D12) is the v1 default; an unconstrained variant with post-grounding is a supported config (§10). The abstraction exists because the literature has diverged from one-token-per-step AR: RPG predicts all m unordered PQ digits from one forward and walks a precomputed item-similarity graph (+12.6% NDCG@10, ≈15× decode speedup, constant in catalog size [rpg]); LLaDA-Rec denoises SIDs in T ≤ m diffusion steps [llada-rec]. A future GraphPropagationDecoder (1 invocation) and a diffusion decoder (T invocations) are peers behind the same interface; because metrics consume item ids (§7.1), decoder swaps are metric-transparent. Decode-strategy-specific artifacts (trie, similarity graph) are optional members of the export bundle (§10).

Decision D15 — MTP auxiliary loss as a v1.5 flag; parallel/speculative/diffusion decode as future work the interfaces must not preclude

DeepSeek-V3-style MTP — D sequential modules, each a shared embedding/output-head plus one transformer block and a d×2d projection, with CE weighted λ/D (λ = 0.3 early, 0.1 late) — is a pure training regularizer that is discarded at inference and showed consistent benchmark gains at identical inference cost in DeepSeek's ablations [dsv3]. On the mcore route this is config-only (TransformerConfig.mtp_num_layers / mtp_loss_scaling_factor + get_gpt_mtp_block_spec [megatron]), so tzrec exposes it as an optional v1.5 flag. MTP-based speculative beam decoding is explicitly out of v1 scope: with only 3 SID levels the ceiling is ~1.5× (3 sequential calls → 2) and AtSpeed's own N-to-K acceptance degrades past K=20 beams [atspeed] — at L=3 the structurally better answer to decode cost is RPG-style parallel decode, which D5's interfaces already accommodate (§14 for roadmap placement).

Latte latent tokens (D16, model-layer slice). Latte breaks the AR decoding tree's probability coupling by prepending one of M′ latent tokens (uniformly sampled per example) to the target SID only — never to the history — and aggregating item scores over latent paths at inference (sum or max; near-identical) [latte]. Model-layer cost is M′ extra vocabulary rows (M′ ∈ {4, 8}; fits the spare-row headroom on the Qwen3 path, or M′ extra fused-vocab rows in-house), beam step-0 restricted to the latent vocabulary, and item-level score aggregation across beams — the same multi-path item scoring that SidCatalogIndex already provides for collision buckets (D4, §5). The data-side hook (sampled virtual token column) is one line in §6; the +3.45% avg NDCG@10 evidence makes this a cheap optional extension rather than a default.

7.5 Context injection (R-INJECT)

The two-world bridge moves TorchRec-computed context embeddings (user profile, behavior-sequence side features) into the dense LLM as inputs. The injection surface differs per backbone, which is why R-INJECT concentrates on the Qwen3 path: LazyDecoderGRBackbone and EncoderDecoderGRBackbone consume context embeddings natively (the context K/V and encoder pathways are the architecture), and InhouseHstuGRBackbone reads jagged grouped features through the existing EmbeddingGroup contract. For Qwen3 the mechanisms are:

  • HF route — inputs_embeds. Qwen3Model.forward enforces a strict XOR between input_ids and inputs_embeds (modeling_qwen3.py:388-389 [transformers]), and — verified in the notes — RoPE consumes position_ids only, never token ids, so an embeds-only forward is fully supported. Two generation caveats are inherited: generate() uses inputs_embeds only on the first iteration before switching to generated ids, and the attention mask cannot be inferred from embeds, so it is always passed explicitly.
  • mcore route — decoder_input. GPTModel.forward(decoder_input=...) bypasses the embedding lookup and accepts pre-computed hidden inputs in mcore's internal [s, b, h] layout (gpt_model.py:479-507 [megatron]); partial injection computes the token embeddings first, scatters context rows, then passes the result.

Two placement strategies are supported, selected per feature group in config (§12):

  • Itemic-slot overwrite (default for text+SID SFT). The prompt template reserves placeholder token positions (e.g. a context slot adjacent to each history SID, or fixed user-profile slots); the model computes embed_tokens(input_ids) and index_copy_s projected context vectors into those rows. Token count is unchanged, so packing math, cu_seqlens, loss masks, and the chat template all survive untouched — the property that makes this composable with the 32K packed path of §8.
  • Prefix prepend (soft prompt). N_ctx projected vectors are prepended to the sequence; positions shift by N_ctx and the injected positions are excluded from loss via the mask. This is the natural form for prompt-light rec-only models and is the degenerate case of V1/V2's "context first, target last" layout.

Bridge conversions are mechanical but specified once: sparse lookups emit fp32 (tzrec's embedding output dtype; recsys-examples likewise pins FP32 qcomms), cast to bf16 at the bridge; jagged [sum_len, D] values are gathered directly into flat packed positions — never via to_padded_dense on the packed path; and the projection MLP adapting embedding dims to the LLM hidden size belongs to the dense model, so its parameters live in the dense optimizer tree (§11). When TP is enabled (mcore lane), the DMP↔TP impedance mismatch is solved at the activation level, exactly as recsys-examples does: each TP rank performs the sparse lookup on its own micro-batch, then applies GradScale × tp (compensating TorchRec's division of model-parallel embedding grads by tp×dp when only dp is wanted) followed by an autograd-aware jagged allgather over the TP group, with labels allgathered via dmp_batch_to_tp (commons/distributed/dmp_to_tp.py:15-68 [repo-recsysexamples]).

R-INJECT is an explicit proof-obligation task, not an assumed-working hook

The first milestone of the model-layer work item is a test battery, run before any full training: (1) gradient reach — a probe embedding table injected via inputs_embeds must show its fused in-backward optimizer firing (parameter delta after one step), proving autograd crosses the bridge; (2) packing equivalence — injected forward over a packed batch is bit-comparable to the same samples unpacked; (3) generate parity — first-step inputs_embeds + subsequent token-id steps reproduce teacher-forced logits on the same prefix; (4) TP equivalence (mcore lane only) — GradScale + allgather matches the tp=1 reference. Sequencing of the two optimizer trees around this bridge is R-GRAD/R-DMP territory, owned by §11; the consolidated register is in §15.

7.6 MoE

MoE matters here for two reasons: OneRec-V1/V2 both report MoE as the parameter-scaling lever (V1: 24-expert top-2 decoder MoE at 0.935B; V2: 53 routed + 1 shared experts, top-3, replacing dense FFNs after the first 2 blocks — its 4B-total/0.5B-active MoE beats a 2B dense model at 0.5B-dense compute [onerec-v1] [onerec-v2]), and both adopt DeepSeek-V3's aux-loss-free load balancing [dsv3]. On the mcore route this entire design space is configuration, not code: TransformerConfig exposes num_moe_experts, moe_router_topk, moe_ffn_hidden_size, moe_shared_expert_intermediate_size, moe_grouped_gemm, moe_token_dispatcher_type (allgather/alltoall/flex), moe_router_load_balancing_type ∈ {aux_loss, seq_aux_loss, global_aux_loss, sinkhorn, none}, and the aux-loss-free pair moe_router_enable_expert_bias + moe_router_bias_update_rate=1e-3 (transformer_config.py:604-822 [megatron]). tzrec's MoE presets default to the OneRec recipe (aux-loss-free balancing, alltoall dispatcher, grouped GEMM); one verl-sourced nuance is recorded for the RL stage: verl sets moe_router_load_balancing_type="none" during RL because the aux loss hurts policy optimization [verl]. Hard constraints to validate at config time: EP>1 requires num_moe_experts set, and TP>1 combined with EP>1 requires sequence_parallel=True.

On the HF side, Qwen3-MoE exists and is complete (Qwen3MoeForCausalLM: identical Qwen3 attention, top-k softmax router with optional renorm, 3D expert parameter layout, Switch-style aux loss via output_router_logits [transformers]), so the FSDP2 lane can fine-tune MoE checkpoints — but without expert parallelism or grouped-GEMM dispatch its efficiency ceiling is low. The decision boundary is therefore the one already drawn in D8: MoE and EP are the Megatron lane's reason to exist; expert-parallel sizing, placement, and its interaction with the TorchRec planner are owned by §11.
