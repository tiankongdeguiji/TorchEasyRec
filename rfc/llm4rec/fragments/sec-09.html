9. Reinforcement learning pipeline

RL is the stage where the generative recommender stops imitating the logged exposure distribution and starts optimizing a reward: next-token prediction can only fit what the legacy cascade chose to expose, so it inherits that cascade's ceiling [onerec-v1]. This section owns four decisions: the algorithm family (D10, §9.1), the verl integration contract (D9, §9.2), the composable reward library (D14, §9.3), and the per-backbone rollout decision matrix (D11, §9.4); it closes with on-policy distillation and reasoning RL (§9.5) and operational guidance (§9.6). The constrained-decoding artifacts the rollout depends on (CSR trie, SidConstrainedLogitsProcessor) are specified in §10 (D12); the dense training engines verl reuses are covered in §11 (D8); the RlConfig proto surface is in §12.

9.1 Why RL, and the algorithm family (D10)

The empirical case for RL in this domain is consistent across the OneRec line. OneRec-V1 measured that RL concentrates quality at the top of the generated list — the gain over no-RL is largest at small serving K (Pass@32 App Stay Time +0.49% vs +0.09% at Pass@512) — and that beam-search rollouts beat top-k/top-p sampling for SID generation, with group size G ≈ 4× the serving K as the saturation rule of thumb [onerec-v1]. OpenOneRec showed the staging effect: on-policy distillation dips recommendation metrics while restoring general capability, and the Rec-RL GRPO stage then recovers and surpasses the SFT checkpoint (video-rec Recall 0.0324 → 0.0304 → 0.0370) [openonerec]. At the same time, PLUM is the production proof that RL is an additive phase, not a launch prerequisite: SID + continued pretraining + reward-weighted SFT alone beat a tuned embedding-table retrieval stack [plum]. tzrec therefore treats RL as phase P3 of the roadmap (§14) and designs it so the SFT pipeline of §8 is independently shippable.

Decision D10 — GRPO is the default; ECPO and GBPO are registered plugins, never forks

The default algorithm is GRPO with the OpenOneRec-proven configuration (group-relative outcome advantage, KL-in-loss low_var_kl with coefficient 0.001, asymmetric clipping with clip_ratio_high=0.28) — this is stock verl, zero tzrec code [repo-openonerec][verl]. ECPO (OneRec-V1) and GBPO (OneRec-V2) ship in tzrec/rl/ as user-side registrations against verl's @register_adv_est / @register_policy_loss registries, which verl explicitly supports for out-of-tree estimators [verl] (verl/trainer/ppo/core_algos.py: ADV_ESTIMATOR_REGISTRY, POLICY_LOSS_REGISTRY). GBPO additionally has a verl-free implementation in the tzrec-native loop (§9.4) because it needs no rollout engine at all. DAPO-style asymmetric clipping and GSPO sequence-level ratios are configuration toggles on the GRPO path (clip_ratio_low/high, policy_loss.loss_mode=gspo), mirroring MiniOneRec's ablation surface [minionerec].

The three core objectives, in the notation of the source papers (sg = stop-gradient, G = group size, A_i = advantage of rollout i):

# GRPO (verl compute_grpo_outcome_advantage; norm_adv_by_std_in_grpo=True)
A_i = (r_i - mean(r_1..G)) / (std(r_1..G) + eps)
L   = PPO_clip(ratio, A_i; clip_low=0.2, clip_high=0.28) + kl_loss_coef * KL_low_var(pi_theta || pi_ref)

# ECPO — Early-Clipped GRPO (OneRec-V1 Eq. 32-34), delta = 0.1, no KL term
J_ECPO = E[ (1/G) * sum_i min(rho_i * A_i, clip(rho_i, 1-eps, 1+eps) * A_i) ]
rho_i  = pi_theta(o_i|u) / pi'_old(o_i|u)
pi'_old = max( sg(pi_theta(o_i|u)) / (1 + eps + delta),  pi_old(o_i|u) )

# GBPO — Gradient-Bounded Policy Optimization (OneRec-V2 Eq. 11-12), no clipping
J_GBPO = -E[ (1/G) * sum_i (pi_theta(o_i|u) / pi'_old(o_i|u)) * A_i ],   A_i in {+1, -1}
pi'_old = max(pi_old, sg(pi_theta))        if A_i >= 0
        = max(pi_old, 1 - sg(pi_theta))    if A_i <  0

The mechanics matter for stability on 3-token SID outputs. ECPO's early clip bounds the denominator rather than the ratio: for negative-advantage rollouts where pi_theta >> pi_old, plain GRPO lets the effective ratio explode; substituting pi'_old = max(sg(pi_theta)/(1+eps+delta), pi_old) caps it near 1+eps+delta, which OneRec-V1 identifies as the fix for the gradient explosion that otherwise collapses legality (§9.3) [onerec-v1]. GBPO goes further: no clipping at all, full sample utilization, and a sign-dependent denominator that reproduces BCE gradient magnitudes — positives bounded like 1/pi, negatives like 1/(1-pi), so negative gradients shrink exactly where the policy already assigns low probability. Its operational profile is unique: it trains on logged real feedback (top-quartile duration-shaped positives, dislikes as negatives, advantage in {+1, −1}, deliberately not normalized), needs no rollout engine, no critic, and no reward model — but it requires that serving logs the per-token generation probability of every exposed item; impressions from non-generative traffic fall back to pi_old := sg(pi_theta) (ratio ≡ 1), the regime GBPO is specifically designed to keep stable [onerec-v2]. This logging requirement is a serving contract tracked in §10.

MiniOneRec contributes the group-equals-beams construction: one constrained beam-search call with num_beams = num_return_sequences = G and do_sample=True produces the whole GRPO group, guaranteeing valid and mutually distinct items per group, and the beam rank doubles as a ranking signal for an NDCG-shaped reward (high-ranked wrong items penalized more, and only in groups where the target was findable) [repo-minionerec]. OneReason contributes the reasoning-era refinements that tzrec adopts as configuration rather than new algorithms: (i) two-stage rollout — sample N reasoning traces, then K parallel SID completions per trace, paying for N long generations while getting N×K group members (Kuaishou reports per-step time dropping from ~350 s to 150–200 s with quality gains); (ii) stage-wise clipping — a looser clip range on CoT tokens than on SID tokens, because a shared clip lets the 3-token SID distribution move too aggressively and triggers entropy collapse (R-ENTROPY); (iii) negative-rollout down-weighting — a response-level loss weight β<1 on non-hit rollouts (advantages untouched) so abundant zero-reward rollouts do not dominate gradients; and (iv) the specialize-then-unify schedule — per-domain RL teachers consolidated by rejection-sampling fine-tuning (RFT), the only method in their study that guarantees thinking mode beats non-thinking across all domains and K [onereason]. Items (i)–(iii) require token-span and group-structure metadata in the loss; verl's driver-assignable loss callable (set_loss_fn) is the sanctioned injection point [verl].

9.2 verl integration: the zero-patch plan (D9)

verl 0.8 (studied at commit 8232babc, version 0.8.0.dev) is a Ray single-controller harness: a TaskRunner builds worker groups and drives RayPPOTrainer.fit, while the legacy per-backend workers have been replaced by one unified ActorRolloutRefWorker/TrainingWorker pair parameterized by a model-engine registry (fsdp/fsdp2/megatron/torchtitan/veomni) [verl]. Rollout is exclusively async server mode: each replica boots a full vllm serve (or SGLang) on the trainer GPUs, and CPU-side agent loops drive token-in/token-out generation per sample through a load-balanced server client. Rewards run in a parallel reward-loop worker pool that can score streamingly as rollouts finish; reference and teacher models are fused into the same worker or split into dedicated resource pools. Weight resync after each actor update goes through a pluggable CheckpointEngine (colocated all-gather by default; NCCL/NIXL/Mooncake for disaggregated layouts; a Qwen3-30B-A3B full resync benchmarks at ~7 s on 4×8 H100). The Megatron engine path requires mbridge (assert engine_config.use_mbridge) and consumes HF checkpoints directly — including our vocab-resized Qwen3, since embedding sizes come from the HF config [mbridge]. A TransferQueue-based successor trainer (main_ppo_sync.py) coexists with the classic path and shares the same extension surfaces, including first-class multi-trajectory agent-loop outputs — relevant for group-equals-beams; tzrec builds against the classic path and keeps its custom pieces portable to both.

Decision D9 — RL runs on the verl 0.8 line with zero framework patches

Every tzrec RL requirement lands through documented verl extension surfaces; we fork nothing and patch nothing. The user-side inventory, all living under tzrec/rl/ and a config bundle:

  • Dataset: an offline tzrec job exports RL parquet in verl's schema (prompt chat messages with SID history rendered as itemic tokens, data_source, reward_model.ground_truth, extra_info); where on-the-fly construction is needed, one TzrecRLDataset class registered via data.custom_cls.{path,name}.
  • Rollout: a sid_agent AgentLoop registered via rollout.agent.agent_loop_config_path YAML ({name, _target_} entries; no fork) and selected by rollout.agent.default_agent_loop or a per-row agent_name column.
  • Constraints: the vLLM-v1 variant of SidConstrainedLogitsProcessor (§10, D12) injected engine-wide via ++actor_rollout_ref.rollout.engine_kwargs.vllm.logits_processors=[...] — verl assembles vLLM CLI args from this passthrough — with per-request arming via SamplingParams.extra_args, which verl forwards verbatim (vllm_async_server.py:502).
  • Rewards: reward.custom_reward_function.{path,name} plus a tzrec RewardManager loaded via reward.reward_manager.source=importlib for raw-token-id scoring (§9.3).
  • Engines: FSDP2 (model_engine=dp, strategy fsdp2) for ≤8B dense, Megatron (model_engine=megatron, mbridge mandatory) for MoE/EP — config-only, aligned with D8 (§11). Checkpoint-engine weight resync is used as-is.

Preconditions: vLLM is the pinned rollout backend for beam-faithful OneRec-parity rollouts (SGLang still has no beam search, though it is a viable alternative for sampling-only GRPO rollouts — §9.6, §10.5); the engine must run with logprobs_mode="processed_logprobs" (verl 0.8's default, verl/workers/config/rollout.py:203); and engine_kwargs.vllm.max_logprobs must be raised to ~2× the beam width. verl's OOV guard masks logits at len(tokenizer), so the padded dummy rows above 176247 (§7) can never be sampled — itemic tokens must be genuine tokenizer-added tokens.

Constrained beam search: the one true gap

verl 0.8 has no beam search anywhere in its rollout path, and vLLM's beam search is not an engine feature: in both the sync LLM.beam_search entrypoint and the OpenAI-server path (use_beam_search: true on completions/chat), it is a client-side emulation that, per token step, submits one fresh 1-token request per live beam with SamplingParams(logprobs=2*beam_width, max_tokens=1, temperature=0) and re-ranks candidates host-side [vllm]. Two consequences follow. First, the emulation never copies constraint fields into its per-step SamplingParams — so per-request constraints do not reach beam steps. Second, and more subtly, the logprobs the ranking consumes are by default computed from raw logits, before any logits processor runs:

logprobs_mode is the linchpin of constrained beam ranking

With vLLM's default logprobs_mode="raw_logprobs", an engine-level logits processor masks what gets sampled but is invisible to the logprobs that emulated beam search ranks by — the beam happily ranks illegal continuations. With logprobs_mode="processed_logprobs", beam steps are all-greedy, so the returned logprobs are the log-softmax over masked logits — exactly the renormalized-over-legal-set distribution constrained beam search needs. verl 0.8 defaults to processed_logprobs (rollout.py:203); plain vllm serve and offline LLM() do not, and must set it explicitly (§10, §13). The option landed in vLLM v0.10.0 (PR #21398) [vllm] and does not exist on the 0.8.5 line OpenOneRec pinned — their beam ranking was unconstrained by construction [repo-openonerec]. Companion knob: engine max_logprobs defaults to 20, failing validation for beam width >10; OpenOneRec sets 320 (width ≤160), and width 512 needs ≥1024.

Because each emulated beam step is a fresh request whose prompt ends with the partial SID, an engine-level trie processor can derive its state from the prompt tail (walk back to <|sid_begin|>) instead of requiring extra_args — one processor then constrains verl rollouts, offline LLM.beam_search, and OpenAI-server beam requests without forking any beam loop. With that in place, three designs were evaluated for verl-side generation (full analysis in the research notes; ranked):

The canonical recipe: OpenOneRec two-stage rollout

The reference rollout recipe for reasoning-capable Qwen3 backbones is OpenOneRec's two-stage generation, reimplemented as a sid_agent AgentLoop (their OneRecvLLMRollout subclassed the retired SPMD rollout and cannot be ported directly) [repo-openonerec]. Stage 1 samples one CoT with stop=["</think>"] (max 1024 tokens), filters any OOV ids ≥ tokenizer length, and splices the literal \n<|sid_begin|>; stage 2 runs beam width 32 for exactly 3 itemic tokens. All 32 beams are returned as separate trajectories that share one GRPO uid — the trainer assigns uids before beam expansion, so the group is the beam set (rollout.n=1). Run parameters worth copying verbatim: actor LR 2e-6, kl_loss_coef=0.001 (low_var_kl), clip_ratio_high=0.28, temperature 1.0, max_logprobs=320, prompt length 10240 / response length 2048. One structural caveat: group members share a single sampled CoT, so the advantage differentiates items, not reasoning — CoT diversity comes only across prompts and epochs. OneReason's N CoT × K SID two-stage rollout is the generalization that restores reasoning-level credit assignment and is the configuration tzrec exposes for reasoning RL (§9.5) [onereason].

9.3 Composable rewards (D14)

Decision D14 — rewards are composable tzrec-side functions, scored on raw token ids

All rewards live in tzrec/rl/reward.py as plain callables composed by weighted sum, surfaced to verl through reward.custom_reward_function and — for anything that needs exact SID matching or group structure — a tzrec RewardManager registered via reward.reward_manager.source=importlib whose run_single reads the raw responses token ids. The same callables are reused by the tzrec-native rollout loop (§9.4) and the eval harness (§13), so reward semantics never fork between training and evaluation.

verl's custom-function contract (as of 0.8.0.dev): compute_score(data_source, solution_str, ground_truth, extra_info, **kw) -> float | {"score": float, ...}, where dict extras land in reward_extra_info for logging and for GDPO's per-component keys; sync and async functions are both supported [verl]. The gotcha that forces the RewardManager path: solution_str is decoded with skip_special_tokens=True. Our itemic tokens are deliberately ordinary added tokens (§7) and survive that decode — OpenOneRec relies on exactly this and regex-parses <s_a_(\d+)><s_b_(\d+)><s_c_(\d+)> — but string-level matching is fragile against template drift and cannot see token positions; id-level matching against itemic_id_range = [151669, 176245) is exact and is the tzrec default. Token-level shaped rewards (e.g., per-level credit at specific positions) also require the manager, since the naive path writes one scalar at the last valid token.

The reward inventory, each item grounded in its source system:

  • RewardModelReward (P-Score). OneRec-V1's P-Score reward model is, structurally, a stock tzrec multi-task ranking model: SIM-style sequence backbone, one BCE tower per objective (ctr/lvtr/ltr/vtr/…), and a fused P-Score tower trained with weighted BCE against all labels [onerec-v1]. tzrec trains it with the existing multi-task stack (§3) and serves it externally; the reward callable decodes generated SIDs to item ids via SidCatalogIndex (§5, D4) and batch-scores over RPC. In verl this maps to either the RM resource pool (reward.reward_model.enable_resource_pool, RM served on rollout infrastructure) or an external endpoint called from the custom function, which receives reward_router_address for exactly this purpose. The investment is justified by V1's online result: pure generative was only at parity with the cascade; the reward model contributed most of the win (+0.54%/+1.24% App Stay Time with RM selection).
  • DurationAwareReward (OneRec-V2). Log-bucket the video duration (F(d) = floor(log_beta(d + eps))), rank the target's playing time as a percentile within the user's own history for that bucket, then assign +1 to batch top-quartile non-disliked samples, −1 to explicit dislikes, and filter the rest — deliberately without advantage normalization, which V2 found degrading [onerec-v2]. This is a data/label-pipeline job (per-user duration/playtime history, dislike labels) feeding an advantage column; it pairs with GBPO and with the hybrid arm (independent streams: rollouts → RM reward, logged exposures → feedback reward) that V2 found best-balanced.
  • SidLegalityReward / FormatReward. OneRec-V1's format reward is the canonical treatment of legality: under ECPO the legality ratio collapsed below 50% via the squeezing effect, and the fix is to take a random K of the G rollouts, assign A=+1 to legal ones, and discard illegal ones entirely — never penalize them, since negative updates squeeze probability mass further onto the argmax. Random-K selection climbs to ~95% legality and stays; top-K-by-probability rises then falls [onerec-v1]. Under constrained decoding (D12) the legality reward remains enabled as a diagnostic: MiniOneRec's CC counter precedent shows a non-zero illegal count means the constraint machinery silently regressed [repo-minionerec]. Legality-rate monitoring is mandatory in every RL run — R-LEGALITY. FormatReward additionally checks the surface form (a well-formed <|sid_begin|>…<|sid_end|> block of exactly num_levels itemic tokens; non-trivial <think>…</think> when reasoning is required).
  • DiversityReward (OneReason). A trace-level multiplicative factor R_div = max(0, m1 - 1)/(K - 1), where m1 is the number of distinct first-level SID digits among the K completions of one CoT — the first digit approximates the coarse category, so this rewards category-diverse generation and helps most at large K [onereason]. It needs the rollout grouping structure, not just (prompt, response) — RewardManager territory.
  • Hierarchical partial credit. OpenOneRec's partial_hit_reward grants 100 for a full triple match, 10 for the first two levels, 1 for the first level, averaged over predicted SIDs; in their released config it is a logged extra while the trained score is first-SID Pass@1 [repo-openonerec]. tzrec exposes it as an optional trained arm (MiniOneRec's HEPO 0.2/0.5/1.0 is the same idea) — useful against reward sparsity on large catalogs, with the caveat that prefix credit is only meaningful when the tokenizer hierarchy is semantically loaded (§5).
  • Multi-component normalization (GDPO). When mixing accuracy, diversity, and business components, verl's in-tree GDPO estimator (algorithm.adv_estimator=gdpo + algorithm.gdpo_reward_keys) normalizes each component's advantage separately — preferable to hand-tuning weights on a summed scalar [verl].
  • Reward-weighted SFT sampling (PLUM). Not an RL reward but part of the same design space: sample SFT examples proportionally to a handcrafted engagement reward, then weight them equally. Production-proven at YouTube at near-zero infrastructure cost [plum]; exposed as a sampling knob on the SFT data pipeline (§8) so teams get reward alignment before any RL is stood up.

9.4 Rollout decision matrix (D11)

Decision D11 — the rollout path is chosen per backbone, not globally

There is no single rollout engine that serves all five backbones of D5. tzrec routes each generative model class to the cheapest rollout path that is actually expressible: verl + vLLM for token-LLM backbones, an out-of-tree vLLM wrapper model only when per-request dense context embeddings must reach the engine, GBPO (no engine) for the lazy decoder, and a tzrec-native in-process loop for encoder-decoder and HSTU backbones. The matrix below is binding; RlConfig validation (§12) rejects unsupported combinations rather than degrading silently.

The tzrec-native in-process rollout loop

For backbones that cannot ride vLLM, the rollout runs inside the training process on the already-sharded model — a pattern with two existence proofs: recsys-examples' generate_beam_decode executes beam search on a DMP-sharded Megatron-style model with TorchRec sparse lookups in the loop (split context-KV / beam-KV caches, beam-ancestry index propagation) [repo-recsysexamples], and MiniOneRec runs a complete colocated GRPO loop (forked TRL trainer, constrained HF generate) on ≤7B models across 4–8×80 GB GPUs [minionerec]. The tzrec design, per training step modulo a rollout interval: (1) switch to eval mode and run generate_beam(batch, G) under no_grad on a user subsample (OneRec-V1 routed 1% of users; G ≈ 4× serving K); (2) take pi_old directly from the beam's per-token logprobs — the beam already computes them, no extra forward; (3) score rewards in-process (P-Score as a second eval-only DMP model on the same ranks, or RPC) plus the format/legality channel; (4) add the ECPO or GBPO loss to the streaming RSFT loss in the same backward — the fused sparse optimizers and the dense path are untouched, it is one extra loss term, exactly OneRec-V1's joint RSFT+RL recipe [onerec-v1]. A teacher-forced recompute of old logprobs is needed only when taking multiple gradient steps per rollout batch (mini-epochs); ECPO needs no reference model at all, and if a KL variant is wanted, only the dense parameters can be cheaply frozen-copied — duplicating the sparse world is not viable, one more reason to follow the no-KL design. At 3–4 decode steps even the naive re-forward generation is acceptable (context prefill dominates); the envelope is roughly ≤3B dense with short decodes, beyond which paged KV and continuous batching are missed badly enough to mandate the verl path. Longer term, this loop is expected to converge onto the tzrec GR serving runtime once the lazy_decoder and enc_dec adapter families land (§10.6), so rollout-in-validation and serving share one decode stack.

Sparse tables cannot ride weight resync

verl's checkpoint engines stream (name, full_tensor) pairs — name-agnostic transport, name semantics at the endpoints. A tzrec dense backbone could resync to a tzrec rollout server over this channel, but the sparse world cannot: a per-tensor stream has no representation for dynamicemb/TorchRec sharded hash tables with billions of rows. The production-proven answer is decoupling: push dense parameters to the rollout/inference side and serve embeddings from a parameter server or incremental dump — OneRec-V1 pushed dense params over a message queue every 1000 steps to its external beam-search service [onerec-v1], which is structurally verl's STANDALONE replica mode plus an NCCL/NIXL checkpoint engine. The SID-as-input representation (history rendered as itemic tokens rather than per-item sparse embeddings, OneRec-V1 §4.2.2) shrinks the "sparse" surface to an ordinary dense embedding matrix and makes the model fully resync-friendly — an additional argument, beyond the modeling ones in §6, for SID-everything inputs in RL-bound configurations.

9.5 On-policy distillation and reasoning RL

On-policy distillation is the capability-retention stage (R-FORGET): OpenOneRec's SFT checkpoint loses general capability hard (AIME'24 0.79 → 0.51, IFEval 0.86 → 0.62), distillation against the original Qwen3 recovers most of it (AIME 0.71, IFEval 0.77), and the subsequent Rec-RL roughly preserves it [openonerec]. The mechanism is reverse-KL-as-advantage run through the standard policy-gradient machinery: the student samples on-policy from 200K general-domain prompts, and each token's advantage is A_t = -(logp_student - logp_teacher), clamped to [-30, +5]; the teacher is the unmodified base Qwen3 of the same scale, loaded as the reference worker with an overridden model path. Sampling is deliberately hot (temperature 1.1, top_p 0.95, top_k 200) so that itemic-token leaks onto general prompts actually get sampled — and punished.

The vocab mismatch is the crux: the teacher has never seen ids ≥ 151669. The OpenOneRec recipe, adopted with one definitional change: (1) build a mask over response tokens whose id falls in itemic_id_range = [151669, 176245) — the same single range used by loss splitting and RL legality (§7) — plus, stated explicitly, the two SID specials <|sid_begin|>/<|sid_end|> (176245/176246), which the teacher equally does not know and which must therefore be covered by the teacher-side replacement and override of steps (3)–(4). One-time divergence note: OpenOneRec's released mask is the open-ended id ≥ 151669, which sweeps in its specials implicitly; tzrec deliberately reuses the shared itemic_id_range definition rather than introducing a second ad-hoc threshold; (2) truncate the trajectory after the first itemic token (response_mask zeroed beyond it; the itemic token itself stays in-loss so it absorbs the penalty); (3) before the teacher forward, replace masked input ids with <|im_end|> (id 151645) to avoid out-of-bounds embedding lookups; (4) override the teacher's logprob at masked positions with a strongly negative constant (−100 in the released config — workable only because of the −30 advantage clamp). Net effect: emitting an itemic token on a general-domain prompt earns the maximum negative advantage, pulling general behavior back to the teacher without unlearning the SID head [repo-openonerec]. verl 0.8 has on-policy distillation in-tree — a distillation config block with teacher resource pools, per-sample teacher routing, and a family of distillation losses (k3, reverse/forward KL, top-k variants) [verl] — so OpenOneRec's fork-era recipe maps onto configuration plus tzrec-side hooks: the trajectory truncation and teacher-query id swap land in the sid_agent loop (which owns response_mask and the teacher-logprob fetch), and the loss-side clamping lands in the driver-assigned loss callable. No verl patches, consistent with D9.

For reasoning models, tzrec adopts OneReason's pipeline shape rather than inventing one [onereason]. Single-domain GRPO teachers are trained with the two-stage N×K rollout, stage-wise clipping, and the diversity-factored reward (§9.1, §9.3); unification is then RFT — batch-sample trajectories from each domain teacher, keep only verified hits passing trace-quality filters, and consolidate by plain SFT. RFT is an offline pipeline (the §13 eval harness generates, the §8 SFT loop consumes); it requires no new RL machinery, and it is the only method in OneReason's study with a thinking-beats-non-thinking guarantee across domains and K. OneRec-Think's rollout-beam reward (beam search width 32 inside the reward to grant per-digit credit if any beam hits) is supported by the same server-side beam mechanism as design (b) of §9.2, with the reward worker calling the rollout replica's HTTP endpoint [onerec-think]. MOPD — multi-teacher on-policy distillation with single-point reverse-KL advantages, truncated importance weights, and information-gain trajectory filtering — is explicitly future work: verl's multi-teacher routing makes it reachable, but RFT covers the v1 need at far lower complexity.

9.6 Operational concerns

Environment isolation (R-ENV). The RL stage runs in its own conda environment or container image, as a separate process tree from tzrec training. This is not optional: tzrec pins grpcio-tools<1.63.0 and the protobuf 4.x API line, while ray/vllm/verl pull newer protobuf and grpcio. The handoff surfaces are files, not imports — RL datasets as parquet, policies as HF checkpoints, the trie as a versioned .npz artifact (§10) — so the two environments never share a Python process. The tzrec.gr.rl CLI is a thin launcher that materializes the verl Hydra config and submits the Ray job into that environment (§12).

Version pins (R-VLLM-PIN). verl 0.8.0.dev's declared pin vllm>=0.8.5,<=0.12.0 is stale — the code carries live branches for vLLM ≥0.13 and ≥0.17 — and the constrained-decoding parameter surface drifted across that span (guided_decoding → structured_outputs; logprobs_mode landed only in v0.10.0; per-request logits processors rejected by V1). tzrec pins an exact (verl commit, vLLM release) pair per image, records it in the §14 dependency table, and tags every API-level claim with its verl version, as the prior spec already mandated. vLLM remains the pinned backend for beam-faithful OneRec-parity rollouts: SGLang still ships no beam search of any kind (a repo-wide grep at the HEAD studied returns zero hits) [sglang]. For sampling-only GRPO rollouts, however, SGLang is a viable and often superior alternative — its per-request custom logit processor sees the live request each step and can walk the SID trie, ReasonerGrammarBackend natively keeps constraints dormant until </think>, and its verl weight-update surface is the richest of the three engines (§10.5).

Entropy collapse (R-ENTROPY). The 3-token SID segment is the collapse-prone span. Standing dashboards: per-token entropy split into itemic vs language tokens at the 151669 boundary (OpenOneRec added exactly this metric to its trainer [repo-openonerec]), Validity@K, per-level token entropy and codebook utilization (shared with §5 tokenizer health), legality rate (R-LEGALITY), and the degenerate-group ratio (fraction of GRPO groups whose rewards are all zero — OpenOneRec's grpo_all_zero_hit_reward_group_ratio). Mitigations in order of preference: stage-wise clipping (tighter on SID tokens), ECPO/GBPO denominator bounds, and reward densification (partial credit, N×K rollouts) before any KL-coefficient surgery.

Sequence-length sizing. rollout.prompt_length and rollout.response_length drive padded tensor shapes through the whole verl pipeline. Pure SID responses are 3 itemic tokens plus markers — size response length in the tens, not the LLM-default thousands, or pay the memory for nothing; generation dominates RL step time, so this is the cheapest large win. Two-stage CoT configurations need response length sized for the CoT (OpenOneRec: 1024-token stage-1 cap inside a 2048 response budget, 10240 prompt). max_num_batched_tokens and max_model_len follow the same logic.

Reward hacking. OneRec-V1 found on-policy rollouts beat a frozen behavior model (vtr +5.61%) but attributed the residual gap to reward hacking against the imperfect P-Score model, flagging it as open [onerec-v1]. tzrec's mitigations: hybrid reward streams (logged real feedback alongside the RM, per OneRec-V2's best-balanced arm [onerec-v2]), the legality/diversity monitors above, and periodic held-out evaluation against non-RM metrics (§13) so an RM-gamed policy is caught by metrics it cannot influence.

GRPO uid discipline. Group identity is the per-prompt uid assigned in the fit loop before any expansion: beams of one prompt must share it (group = beams), repeated prompts must not. Custom samplers or datasets that duplicate rows must preserve uid semantics, and the seqlen-balancing pass reorders rows freely — safe only because advantage computation is uid-keyed, a property any custom estimator registered under D10 must also maintain.
