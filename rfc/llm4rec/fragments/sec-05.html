5. SID tokenizer subsystem (tzrec/sid)

This section specifies the subsystem that turns a catalog of item embeddings into the Semantic-ID space everything downstream consumes: quantizers (D1), collision resolvers (D2), training-free quality gates (D3), and the assignment/asset/catalog layer publishing the item↔SID mapping as a many-to-many index (D4). The package is tzrec/sid/ plus two tokenizer-training model classes (SidRqkmeans=610, SidRqvae=611) extending the BaseSidModel/ResidualQuantizer substrate already on master from PR #538 (§3). One empirical theme governs effort allocation throughout: the quantizer algorithm is a second-order choice, while collision post-processing, the embedding's collaborative signal, and the evaluation protocol are first-order [forge] [grid].

5.1 Item embedding sources

tzrec does not run image or video encoders. The tokenizer's input is an upstream artifact: a parquet/ODPS table with an item-id column and a fixed-width fp32 vector column (parquet list<float32>, ODPS ARRAY<FLOAT> or a base64-packed string per the AL-GR release format [repo-alsid]). This is exactly the surface BaseSidModel already consumes — a RawFeature with value_dim=input_dim read via _extract_feature (tzrec/models/sid_model.py:76-89) — so the embedding contract requires no new data-layer code. The asset must carry an embedding_version tag, because every downstream artifact (codebooks, SID asset table, trie, vocabulary alignment) is keyed to it (§5.5).

Two evidence points steer how teams should produce this artifact. First, collaborative signal dominates content: FORGE's modality ablation (Table 6) shows an i2i-co-occurrence-only embedding reaching HR@20 4.63% versus 3.01% text-only and 3.06% image-only, with the full fusion at 4.89% — content embeddings mainly add tail coverage at large K (HR@1000 29.01% vs 23.15% for i2i-only) [forge]. Second, not all alignment positives help: aligning the encoder with query–item pairs or same-category positives hurt retrieval by 4.6–15% relative HR, while i2i co-occurrence positives and side-info text concatenation helped (FORGE Table 5). PLUM independently found the co-occurrence InfoNCE term to be the single largest tokenizer-side win — removing it dropped SID uniqueness from 96.7% to 91.8% and downstream Recall@10 from 14.4% to 12.6% (PLUM Table 4) [plum]. PLUM also makes the operational case for injecting behavior as a training objective on content-only inputs rather than fusing CF embeddings into the quantizer input (the LETTER-style alternative [letter]): CF embeddings drift with popularity and would couple the re-tokenization cadence to CF retraining (§5.5, R-RETOK). The RQ-VAE trainer therefore exposes an optional i2i co-occurrence InfoNCE auxiliary loss (§5.2), and the recommended upstream recipe is an i2i-aligned multimodal encoder — with the caveat that a plain i2i-only embedding is already a strong, cheap baseline.

Encoder size, by contrast, is not worth GPU budget: GRID's sweep from Flan-T5-Large to XXL (14× parameters) moved Amazon-Beauty Recall@10 only from 0.0639 to 0.0646 [grid]. Marginal effort belongs in co-occurrence pair mining (the same pairs double as Embedding-HitRate ground truth in §5.4), not in a larger encoder.

5.2 Quantizers

Decision D1 — RQ-KMeans default, RQ-VAE option; both packaged as tzrec models

The default tokenizer is ResidualKMeansQuantizer (3×8192, faiss residual k-means over a frozen embedding, training-free), matching the OneRec lineage and the GRID benchmark verdict that residual k-means meets or beats RQ-VAE at a fraction of the complexity [onerec-v1] [openonerec] [grid]. ResidualVectorQuantizer (gradient-trained RQ-VAE) is a supported option with the knobs the strongest implementations converge on: k-means codebook init, commitment loss, dead-code reset, optional rotation-trick estimator, no EMA (every strong implementation rejected it), optional per-level Sinkhorn (last level in practice), and an optional co-occurrence InfoNCE auxiliary loss; progressive prefix masking is specified but deliberately cut from the v1 proto surface (Table 5.2). Both are exposed as tzrec models — SidRqkmeans (proto tag 610) and SidRqvae (611) over BaseSidModel — so tokenizer training reuses the stock dataloader, distributed, metric, and checkpoint machinery instead of growing a parallel trainer.

The class plan extends the abstract ResidualQuantizer from PR #538 (tzrec/modules/sid/residual_quantizer.py), whose contract is already final: subclasses populate one layer per level and implement _quantize_layer/_lookup_code; the shared _residual_pass (rq.py:119-153) fixes the gradient semantics (residual chain detached, aggregate sum carries codebook gradients) and get_codes/decode_codes (rq.py:155-168, 197-213) are the offline assignment/reconstruction API. Per-level codebook sizes are already first-class (normalize_n_embed, rq.py:21-37), which is deliberate: PLUM's multi-resolution geometry (halving codebook size per level) improved both uniqueness and recall [plum], so the config must not hard-code "L levels × fixed K" even though the shipped default is uniform 3×8192. The 3×8192 default itself is FORGE's empirically supported sweet spot for 1e8-scale corpora: under-capacity geometries are catastrophic (3×512 cost 19–21% HR@20 on a 251M-item corpus), 2-level variants trade HR@20 for HR@1000, and 4×4096 is accuracy-neutral at strictly higher decode cost (FORGE Tables 8/10) [forge]. For small academic corpora the GRID result (3×256 optimal at TIGER scale [grid]) is reachable through the same config fields.

The RQ-VAE knob set is the union of what the strongest open implementations found necessary to avoid codebook collapse, each having rediscovered a subset independently: al_sid ships gradient codebooks with dead-code restart and EMA disabled (configs/rqvae_i2v.yml) [repo-alsid]; semantic-ids-llm implemented EMA, rejected it, and relies on k-means init + rotation trick + frequent dead-code resets [repo-semanticids]; GRID needed embedding whitening (RQ-VAE) and residual normalization (RK-Means/R-VQ) [grid]. LC-Rec contributes the per-level Sinkhorn epsilon list — balanced optimal-transport assignment on the last level only (ε=0.003, 50 iterations), a training regularizer while inference stays greedy argmin (index/models/vq.py, get_indices(use_sk=False)) [repo-lcrec]. The co-occurrence InfoNCE head follows the al_sid construction (three symmetric terms with learnable temperatures, positives from an i2i pair table, embeddings all-gathered with gradient across ranks); pairs arrive as a second input table on the same item-id space. Progressive prefix masking (train against a randomly truncated prefix of levels so every SID prefix is a usable coarse representation) is cheap and directly benefits prefix-decodable beam search [plum]; it is deliberately left out of the minimal v1 proto, and adding progressive_mask_prob later is a backward-compatible field extension.

Tokenizer-as-model: SidRqkmeans / SidRqvae

BaseSidModel already defines the packaging: the proto message carries embedding_feature_name, input_dim, normalize_residuals, and repeated codebook; init_loss() is a no-op because SID models compute losses internally and return them through predictions (sid_model.py:54-74); init_metric() registers mse and unique_sid_ratio (tzrec/metrics/unique_ratio.py). SidRqvae.predict runs the encoder + _residual_pass + decoder and emits {"reconstruction": ..., "codes": ..., "loss_recon": ..., "loss_commit": ..., "loss_infonce": ...}; loss() pulls the loss tensors out of predictions, so the stock TrainWrapper sums them unchanged. Both classes add the new gate metrics of §5.4 (per-level utilization and entropy as torchmetrics; Gini and Embedding HitRate run offline over the assignment table, not in-loop). Following LC-Rec, the eval-time checkpoint selector for SidRqvae is collision rate / unique-SID ratio, never reconstruction loss (§5.4, D3) [repo-lcrec].

Distributed training for large corpora

The tokenizer model is tiny — three 8192×64 codebooks plus a small MLP is a few million parameters — so the distributed problem is data volume, not model size. SidRqvae trains as ordinary data-parallel: no sharded embedding module exists, so tzrec's pipeline factory selects the plain TrainPipelineBase path and DDP-replicates the model; the only cross-rank coupling is the InfoNCE all-gather, which al_sid already demonstrates at 2-node scale [repo-alsid]. Reference batch/optimizer settings come straight from the published configs (al_sid: batch 2048, AdamW, cosine; LC-Rec: batch 1024, AdamW 1e-3, k-means init [repo-lcrec]).

SidRqkmeans is gradient-free, so "training" through the stock loop means reusing its plumbing, not its optimizer: the tzrec.gr.train_tokenizer entry (a thin wrapper over tzrec/main_gr.py, §12) iterates the standard dataloader to fill a per-rank reservoir of embedding rows up to a fixed sample budget (faiss's default per-centroid training-point cap; deliberately not a v1 proto field), gathers the reservoir to rank 0, runs faiss residual k-means level by level (the OpenOneRec train_kmeans loop: fit on residuals, assign, subtract, recurse [repo-openonerec]), and broadcasts the centroids into the quantizer's non-trainable ParameterList; a standard eval pass then computes the in-loop metrics, and the checkpoint saves through the normal DCP path. A subsample is sufficient and intended — k-means quality saturates far below full-corpus scale (faiss itself caps training points per centroid by default), and OpenOneRec fits its production tokenizer in one in-memory CPU pass — whereas the full-corpus work is the encode in assign_sid, chunked and embarrassingly parallel (§5.5). For a streaming/trainer-integrated k-means (not needed in v1), GRID's trick of expressing mini-batch k-means as a weighted-MSE loss under SGD — letting DDP gradient averaging perform the cross-GPU reduction — is the documented fallback [repo-grid].

5.3 Collision resolution

Decision D2 — collision resolution is pluggable post-processing, decoupled from the quantizer

Collision handling lives in tzrec/sid/collision.py as a separate pipeline stage operating on the raw assignment table, never inside the quantizer. Resolvers: RandomCollisionResolver (round-robin last-level reassignment, default for GR targets), KnnCollisionResolver (capacity-σ walk over ranked last-level candidates), UniquenessDigitResolver (TIGER-style appended digit), and SinkhornGroupResolver (LC-Rec collision-group re-encoding). For 1e8-scale catalogs the resolver compiles to FORGE's iterative ODPS SQL pattern rather than running in-process.

FORGE's central finding is that this stage, not the quantizer, is where retrieval quality is won: with everything else fixed, no collision handling costs 18% relative HR@20 versus a σ=25 baseline, tightening to σ=5 gains 15.8%, and the non-semantic Random-5 policy gains 29.4% HR@20 / 4.8% HR@1000 — a strictly monotone ordering from "no handling" through KNN to Random (FORGE Table 7; on sparse Amazon data the Random-5 gain exceeds 100%) [forge]. The reading: leaf fairness beats leaf semantics — the third residual level carries little information anyway (LC-Rec's level-wise case study agrees [lcrec]), so equalizing the SID-occupancy distribution matters more than assigning the semantically nearest leaf. This is also why the FORGE "RQ-VAE beats RQ-Kmeans" headline does not move D1: the comparison is confounded (only RQ-VAE received the Final post-processing), and at equal post-processing the encoders are within noise of each other (FORGE Table 9).

Revision vs prior spec — default resolver flips from KNN-σ to Random

The prior spec (3a) made KnnCollisionResolver (σ=5) the default. Given the FORGE Table 7 evidence above — Random-5 strictly dominates every KNN setting on both AL-GR and Amazon — v1 defaults to RandomCollisionResolver with the σ=5 capacity convention. KNN-σ remains the recommended choice when last-level code embeddings are reused as semantic features elsewhere (Random deliberately destroys level-3 semantics), and is required anyway as the fallback for quantizers that emit ranked candidates.

The four resolvers, concretely:

  • RandomCollisionResolver (FORGE Algorithm 2): keep the first L−1 semantic levels; assign the last level by a round-robin counter keyed on the (c_1, …, c_{L-1}) prefix — IndexMap[key] = (IndexMap[key]+1) mod n_L — dispersing items maximally. With n_L = 8192 and the σ=5 convention, a bucket only overflows past σ once a prefix holds >40,960 items. The counter is insertion-order dependent, so determinism requires (a) a pinned snapshot ordering — the resolver sorts by item_id before assignment — and (b) persisted IndexMap state for incremental arrivals: the counter table is part of the published asset so that tomorrow's new items continue the rotation instead of restarting it (§5.5).
  • KnnCollisionResolver (FORGE Algorithm 1): requires the quantizer to emit a ranked last-level candidate list (al_sid's get_sorted_index — the argsort of all last-level distances; production exports the top 200 [repo-alsid]); walk candidates nearest-first and take the first bucket with fewer than σ items. ResidualKMeansQuantizer.get_codes therefore gains an optional return_topk_last_level argument; this ranked emission is the one quantizer-side hook the resolver layer needs.
  • UniquenessDigitResolver (TIGER / GRID / semantic-ids-llm): append one extra digit numbering items within each collision group, guaranteeing strict uniqueness [tiger] [repo-semanticids]. Useful when downstream requires a bijection, but it lengthens decode by one step, the digit is purely positional (observed group sizes up to 50 against a reserved range of 256 [repo-semanticids]), and GRID found it near-indistinguishable from simply tolerating collisions (Recall@10 0.0597 vs 0.0591) [grid]. tzrec validates group sizes against the digit range, fails loud on overflow, and uses 0-based positions uniformly (GRID's released tags are inconsistently 1-based).
  • SinkhornGroupResolver (LC-Rec): re-encode only the colliding groups as single batches with last-level Sinkhorn assignment enabled, repeating up to 20 rounds; transport balancing splits the ties while keeping the last level semantic [repo-lcrec]. Only applicable to SidRqvae (needs the trained codebook distances); residual collisions among exact-duplicate items are tolerated, as in the original.

Scaling to 250M items: the iterative ODPS SQL pattern

An in-memory resolver is fine to ~1e7 items; beyond that, tzrec ships the resolver logic as a templated MaxCompute SQL job, directly modeled on the AL-GR production pipeline that processed 251M items (SID_generation/ID_collision.sql [repo-alsid]): (1) a first pass keeps up to σ random items per SID via ROW_NUMBER() OVER (PARTITION BY sid ORDER BY rand()); (2) overflow items join against their exploded top-200 ranked candidate codes (TRANS_ARRAY, priority 1..200); (3) an iterated LEFT ANTI JOIN loop removes already-assigned items and full buckets, assigns by priority with capacity truncation, and merges, until empty. The Random variant replaces the ranked candidate generator with a random/rotating code. The output schema — (item_id, origin_codes, final_codes, bucket_pos) — is the asset-table schema of §5.5 verbatim, so the SQL job and the in-process resolver are interchangeable producers.

5.4 Training-free quality gates

Decision D3 — tokenizers are selected by training-free gates, never by reconstruction loss

A sid_eval step (part of assign_sid, §5.5) computes four gates from the assignment table and the embedding table alone: Gini coefficient over SID occupancy, codebook utilization per level and for full SIDs, global unique-SID ratio, and Embedding HitRate against held-out co-occurrence pairs. These correlate with downstream GR HitRate; reconstruction fidelity and consistency metrics do not, and must never drive model selection [forge].

The motivation is cost: validating a tokenizer by training the GR model is an 80-hour, 32-GPU experiment per configuration at FORGE's scale [forge], while every gate below is a SQL aggregation plus a small ANN job. FORGE validated the correlation empirically (its Figure 3: lower Gini and higher Embedding HitRate track higher downstream HR across the whole noco→Random-5 family), and PLUM's harness independently arrived at the same two-axis protocol — uniqueness plus a cheap downstream recall probe [plum].

Gini coefficient. With SID occupancy counts a_(1) ≤ … ≤ a_(N) and Lorenz curve L(i) = C(i)/C(N), G = (2/N) · Σ_i (i/N − L(i)); lower is fairer. One convention gotcha must be pinned: the FORGE paper defines N over all possible SID slots (8192^3, including zero-count slots) while the released script uses used SIDs only — the two produce very different absolute values [forge] [repo-alsid]. tzrec adopts the cleaner decomposition: Gini over used SIDs (fairness among occupied buckets) plus the separate utilization metric below (how much of the space is occupied at all). The implementation is the al_sid histogram closed form — one ODPS GROUP BY builds the count→frequency histogram; arithmetic-series rank sums then compute G in O(#distinct counts) memory — so it runs on a 250M-item table without moving the data.

Codebook utilization — the fraction of codes used per level and the fraction of realized full SIDs — plus per-level coverage and entropy diagnostics logged during training, following GRID's _compute_output_stats (distinct-ids/n_clusters per level, categorical entropy of the per-level code distribution, batch unique-SID fraction, residual-norm ratios) [repo-grid]. These are the early-warning signals for dead codes and level collapse, registered as torchmetrics on both SidRqkmeans and SidRqvae.

Unique-SID ratio. The in-batch UniqueRatio metric already exists (tzrec/metrics/unique_ratio.py) and is documented as a batch-level diversity proxy; sid_eval adds the global ratio (distinct full SIDs / corpus size before resolution; exactly 1.0 after a uniqueness-guaranteeing resolver) computed over the full assignment table. PLUM supplies the essential caveat: uniqueness alone is an insufficient selection metric — its multi-embedding ablation produced higher uniqueness (96.9% vs 96.7%) with lower downstream recall (12.8% vs 14.4%) [plum]. Gates are evaluated jointly, never as a single scalar.

Embedding HitRate. FORGE shows the metric predicts downstream HR but never publishes a precise protocol [forge], so tzrec defines one: build a faiss inner-product index over the item vectors, query each item in a held-out set, retrieve top-K neighbors (excluding self), and score hits against held-out i2i co-occurrence pairs (the same pair table used for the InfoNCE loss, split before training); report HR@K for K ∈ {20, 100, 500} with per-query macro averaging. Two variants run under one flag: embedding-gate mode indexes the raw input embeddings (validating the upstream artifact before any quantization, FORGE's published usage), and quantized mode indexes decode_codes reconstructions (validating that quantization preserved neighborhood structure, the reading in the al_sid release [repo-alsid]). The notes for the two FORGE sources disagree on which variant the paper measured, which is precisely why the tzrec protocol names its variant in the metric key (emb_hitrate@K/raw vs /quantized).

Anti-metric: reconstruction loss does not predict retrieval quality

FORGE tested feature fidelity, style consistency, and homogeneous consistency against downstream HR and found no correlation: Multiple-VQ wins two fidelity metrics yet is the worst retriever in the comparison (FORGE §4.3.5, Figure 4) [forge]. Selecting RQ-VAE checkpoints by validation reconstruction loss — the common practice the default HF/Lightning training loop encourages — silently optimizes the wrong objective. tzrec's SidRqvae therefore keys best-checkpoint tracking to collision rate / unique-SID ratio (LC-Rec's best_collision_model precedent [repo-lcrec]) and surfaces the gate metrics, with reconstruction MSE logged for debugging only.

5.5 SID assignment, asset table, and catalog index

Decision D4 — the item↔SID mapping is many-to-many from day one

The catalog layer (SidCatalogIndex) treats both directions as one-to-many: one SID resolves to up to σ items (collision buckets), and one item may carry multiple SID paths (PIT's top-B dynamic beam index [pit], Latte's latent-token trees [latte]). All consumers — HR@K evaluation, trie construction, generation grounding, RL legality — go through this index and its multi-path scoring/dedup API rather than assuming a bijection.

assign_sid: chunked batch inference

tzrec.gr.assign_sid loads a tokenizer checkpoint and runs get_codes over the full embedding table in chunks — the al_sid full-inventory pattern [repo-alsid], but through tzrec's standard predict pipeline: the ODPS/parquet reader streams the embedding column, the model emits codes [B, L] (plus ranked last-level candidates when a KNN resolver is configured), and the predict writer lands the raw assignment table. The collision resolver then runs as a second pass (in-process or the ODPS SQL job of §5.3), followed by the sid_eval gates. Encoding is embarrassingly parallel across workers, so 1e8-item catalogs are a throughput problem, not a design problem.

Asset table schema

Training data consumes this asset by an offline join: a batch job maps each interaction history's item ids through the asset table and materializes SID sequences as plain ARRAY<BIGINT> columns, which flow through the existing FG_NONE path as fg-encoded integer sequence features with zero new transport code (§6). Performing the join inside the feature-generation pipeline instead (fg-side SID lookup at training/serving time, which would let one asset update propagate without re-materializing datasets) is explicitly future work tracked as R-FGJOIN (§15).

SidCatalogIndex

SidCatalogIndex (tzrec/sid/catalog.py) is the single in-memory view of the asset table, loaded once per job. Its API: items_for_sid(codes) → List[item_id] (bucket members in bucket_pos order), sids_for_item(item_id) → List[path], expand_beams(beam_paths_in_order, k) → top-k item ids (walk completed beams in score order, expand each SID into its bucket, de-duplicate, truncate — the AL-GR calc_hr protocol, where 100 beams × σ=5 yields up to 500 candidates [repo-alsid]), and aggregate_item_scores(scored_paths, agg=sum|max) → Dict[item_id, score] for the multi-path case where several beams — latent-token trees, PIT index entries, or plain collision siblings — reach the same item. Consumers: HR@K evaluation (§13) must expand SID→items in beam order rather than assume one item per SID; the CSR trie of §9/§10 (D12) is built from the set of valid code paths; unconstrained-decode grounding maps generated SIDs back to catalog items, with the prefix-fallback trick (exact path, else longest valid prefix 3→2→1) as a graceful-degradation option [repo-semanticids]; and RL legality rewards query the same index (§9, D14). Multi-path scoring is forced anyway: collision buckets alone (σ>1) make the mapping non-bijective, and PIT's ablation shows its multi-view index is the single largest contributor (removing it: −14.9% Recall@10) [pit].

Re-tokenization cadence for streaming corpora

New items arrive continuously in production; the SID space must follow without invalidating a trained LLM. The v1 policy is frozen-codebook incremental assignment: between major versions, codebooks are pinned; new items get codes via the frozen get_codes plus the collision resolver running with persisted state (the Random resolver's IndexMap counters and bucket occupancies are part of the asset, so incremental assignment is deterministic and capacity-correct). Full re-tokenization — retraining codebooks and regenerating every SID — redefines token semantics and forces an LLM re-alignment pass (at minimum the CO_PRETRAIN stage of §8); it is scheduled with major model refreshes, not on a timer. The cadence question is tracked as R-RETOK (§15), bounded by three evidence points. First, under the Random resolver the leaf codes of late-arriving items are semantically meaningless by construction — acceptable for GR targets, but leaf-code semantics then degrade monotonically between re-tokenizations, so no consumer may reuse level-3 code embeddings as features under this policy [forge]. Second, PLUM's two choices — behavioral signal as an objective over static content-only inputs, and progressive masking so every prefix stays a usable coarse code — are exactly what decouples tokenizer freshness from CF drift; tzrec inherits the co-occurrence objective as a v1 RQ-VAE knob and earmarks progressive masking as its follow-up field (§5.2) [plum]. Third, genuinely streaming tokenizer maintenance (incremental codebook merges without a global re-fit) exists in the literature but is unproven in any system we studied; noted as future work (§14) [awesome-list].

Decision D16 — Latte latent tokens as a cheap option; PIT tokenizer co-evolution as a phase-2 streaming extension

Two published extensions of the SID layer are adopted at different speeds. Latte latent tokens ship in v1 as a cheap option, off by default (n_latent_tokens=0): the asset and catalog layers need nothing new, and the model-layer slice (extra vocabulary rows, restricted beam step 0, multi-path score aggregation) is specified in §7.4. PIT tokenizer–recommender co-evolution is deferred to phase 2 streaming training — it requires train-time generation and a recommender-as-critic path that only the RL machinery of §9 provides — while v1 commits to the seams it needs: weighted multi-SID sid_paths (D4), a versioned asset table, and a reusable SID-emission head (D5). The two hooks below elaborate.

Hook: PIT tokenizer–recommender co-evolution (D16, phase 2)

PIT replaces the frozen tokenizer with a small autoregressive item-to-token model that co-evolves with the recommender in streaming training: the tokenizer proposes B candidate SIDs per item by beam search, the recommender-as-critic picks the most predictable one as a pseudo-label, both update jointly, and a momentum-weighted top-B beam index replaces the static asset table — deployed online against the OneRec-V2 baseline with positive A/B results [pit]. tzrec deliberately scopes this to phase 2, streaming-only: it requires train-time generation inside the training loop (eager mode, incompatible with graph capture) and a critic-scoring path that only the RL machinery of §9 provides, and its offline form degenerates anyway (PIT freezes the tokenizer on academic datasets). What v1 does commit to are the seams PIT needs later: SidCatalogIndex already supports weighted multi-SID entries per item (Table 5.3's sid_paths), the asset table is versioned for streaming sync, and the decode-strategy abstraction of §7 (D5) keeps the SID-emission head reusable as a standalone item-to-token model.

Hook: Latte latent tokens (D16, cheap option)

Latte addresses a structural expressiveness limit of single-tree SID decoding — items sharing a prefix have strongly coupled probabilities across users, provably suppressing user-specific rank reversals between tree-close items — by prepending one of M′ latent tokens (M′ ∈ {4,8}) to each training target and aggregating item scores over latent paths at inference, lifting NDCG@10 by ~3.5% on average with identical SIDs [latte]. The integration cost in tzrec is deliberately near-zero: training needs only a uniformly sampled latent-token id prepended to the target SID (one extra CE position, a dataset-builder flag in §6); the vocabulary needs M′ extra rows, which fit in the existing alignment headroom after the SID specials without changing the padded vocab total (§7); decoding restricts beam step 0 to the latent vocabulary and then proceeds normally (trie root fan-out in §10); and score aggregation across latent paths reuses SidCatalogIndex.aggregate_item_scores — the same component collision buckets and PIT already require (sum vs max aggregation is empirically a wash [latte]). The one real cost is beam budget: with M′ trees the effective per-tree beam shrinks, so eval keeps ≥50 beams and final retrieval ~500, per the reference configuration [repo-latte]. n_latent_tokens=0 (the default) disables the feature entirely.
